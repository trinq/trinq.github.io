Lesson 0004
Outer Loop

Reading time: ~20 minutes. No coding required.

In the previous lesson, you learned how a single agent completes a single task — the inner loop. Now we zoom out. The outer loop is the layer that decides which task to work on next, when to run agents, and how to coordinate across sessions. This is where Loop Engineers do their most important work.

The Outer Loop: The Orchestrator

While the inner loop optimizes how an agent does a task, the outer loop optimizes what task gets done next. Think of it as the manager that:

  • Scans the environment for work
  • Prioritizes tasks based on signals and dependencies
  • Assigns work to the right agent
  • Monitors results and updates state

Without an outer loop, agents are reactive — they wait for a human to tell them what to do. With an outer loop, the system is proactive — it continuously scans, decides, and acts.

The Outer Loop Cycle

Scan → Prioritize → Assign → Monitor → Scan
  1. Scan — Read signals, check backlog, look for new triggers
  2. Prioritize — Rank tasks by urgency, impact, and dependencies
  3. Assign — Give the top-priority task to the right agent
  4. Monitor — Watch for completion, errors, and new signals

State Tracking: The Backbone of the Outer Loop

The outer loop needs to know the current state of everything. This is called state tracking. The system must always know:

  • Which tasks are pending (waiting to be done)
  • Which tasks are in progress (currently being worked on)
  • Which tasks are completed (done and verified)
  • Which tasks are blocked (waiting on a dependency)

The shared file system stores this state. A simple state file looks like this:

# signals/backlog.md

## Status Legend
- [ ] Pending
- [>] In Progress
- [x] Done
- [!] Blocked

## Backlog
- [ ] Research CVE-2024-XXXX (priority: high)
- [x] Write Q3 security report (priority: medium)
- [>] Review vendor proposals (priority: high)
- [!] Update threat model (priority: medium) — blocked on vendor response

Trigger Types in the Outer Loop

The outer loop decides when to act based on triggers. Four main types:

Time-Based

Scheduled triggers run at fixed intervals. Example: every morning at 7am, scan for new CVEs. Useful for recurring, predictable work. Be careful: scheduled runs can pile up if tasks take longer than expected.

Event-Based

Reactive triggers fire when a specific file or signal appears. Example: when signals/urgent.md is created, immediately triage. This is the most responsive trigger type.

Dependency-Based

Chained triggers fire when a previous task completes. Example: after the research agent finishes, the summary agent starts. This is how you create pipelines of agents.

Request-Based

Human-initiated triggers start when someone asks. Example: a customer submits a ticket. This keeps humans in control of high-priority work.

Prioritization: How to Decide What Comes First

Not all tasks are equal. The outer loop needs a way to rank them. A simple prioritization framework:

Priority Signal Type Example Action
P0 — Critical Incident, breach alert New zero-day CVE detected Interrupt current work, handle immediately
P1 — High Deadline approaching Proposal due in 24h Complete before scheduled work
P2 — Medium Standard backlog item Weekly security digest Process in normal order
P3 — Low Nice-to-have Blog post draft Do when capacity available

Signals in the shared file system carry their own priority. The outer loop reads priority and acts accordingly.

Cross-Session Coordination

The outer loop must handle work that spans multiple sessions. A task might take days to complete across many agent runs. The key to cross-session coordination:

  • Checkpoint files — At the end of each session, the agent writes what it did and what needs to happen next. The next session reads this before starting.
  • Artifact versioning — Each work product gets a version number. This makes it clear which is the latest and what changed.
  • Dependency locks — A task that depends on another task gets locked until the dependency is complete. The outer loop reads these locks to decide scheduling.

The Outer Loop is Not Magic

A common misconception: the outer loop is an AI that magically decides everything. In reality, the outer loop is deterministic logic built on top of the shared file system. It is not a model — it is a set of rules:

IF signals/urgent.md exists
  THEN run triage agent immediately
ELIF tasks due in 24h exist
  THEN prioritize those tasks
ELIF time is 7am
  THEN run morning research agent
ELSE
  WAIT

The sophistication comes from well-designed signals, clear contracts, and good state management — not from a smarter AI.

Key Takeaways

  • The outer loop optimizes what task gets done next — it is the orchestrator of the whole system.
  • Outer loop cycle: Scan → Prioritize → Assign → Monitor.
  • State tracking is the backbone — the system must always know which tasks are pending, in progress, done, or blocked.
  • Four trigger types: time-based, event-based, dependency-based, and request-based.
  • The outer loop is deterministic logic, not an AI. It reads the file system and follows rules.
  • Cross-session coordination requires checkpoints, artifact versioning, and dependency locks.

Questions to Think About

  1. What are the top 3 tasks you do repeatedly at work? How would an outer loop prioritize them?
  2. What signals in your workflow would trigger an outer loop to take action? (A new email? A calendar event? A CVE alert?)
  3. How would you encode your current workflow as a set of IF-THEN rules in the shared file system?
If anything is unclear, ask. The outer loop is the hardest concept in Loop Engineering. Make sure this clicks before moving to compound loops.
