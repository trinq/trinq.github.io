Lesson 0007
Tool Use & MCP

Reading time: ~20 minutes. No coding required.

In the previous lessons, you learned how agents work (inner loop), how systems of agents work (outer loop and compound loops), and how to make your codebase agent-friendly. Now let's cover the final piece: how agents interact with the outside world through tools and the Model Context Protocol (MCP).

Why Tools Matter

Without tools, an agent is just a text generator. It can reason, plan, and write — but it cannot take action in the real world. Tools give agents the ability to:

  • Read and write files
  • Run terminal commands
  • Search the web
  • Call APIs
  • Send emails
  • Control browsers

Every tool a harness provides expands what the agent can do. The art of Loop Engineering is choosing the right tools, describing them clearly, and orchestrating them effectively.

What Makes a Good Tool?

Not all tools are created equal. A well-designed tool has 5 properties:

1. Clear Name

The tool name describes exactly what it does. "read_file" not "tool_1." "search_cve" not "search." Ambiguous names cause tool misuse — the agent picks the wrong tool.

2. Explicit Description

A description that tells the agent when to use this tool and when not to. Include edge cases: "Use this when X, but not when Y."

3. Typed Inputs

Each parameter has a type (string, number, boolean) and a description. The agent must know what to pass in. Avoid ambiguous parameters like "filter" without explaining the format.

4. Predictable Output

The agent knows what to expect back. If a tool sometimes returns text, sometimes JSON, and sometimes an error, the agent cannot plan effectively.

5. Single Responsibility

Each tool does one thing well. Avoid "do everything" tools. A tool that reads one file is better than a tool that "manages files."

The Model Context Protocol (MCP)

MCP is an open protocol that standardizes how agents connect to external tools and data sources. Think of it as USB for AI agents — a universal connector that lets any MCP-compatible agent talk to any MCP-compatible tool.

Before MCP, each agent framework had its own tool format. A tool built for LangChain did not work with CrewAI. MCP solves this by defining a common language:

  • Tools — Functions the agent can call
  • Resources — Data the agent can read (files, databases, configs)
  • Prompts — Reusable prompt templates

MCP servers expose tools and resources. MCP clients (agents) connect to servers and use them. One agent can connect to multiple servers simultaneously.

+---------------------------------------------+
|  Agent (MCP Client)                         |
|                                             |
|  +----------+  +----------+  +----------+  |
|  | File     |  | Web      |  | Database |  |
|  | Server   |  | Server   |  | Server   |  |
|  +----------+  +----------+  +----------+  |
+---------------------------------------------+

Common MCP Tool Categories

MCP has a growing ecosystem of pre-built tools. Here are the most useful categories:

Category Tools Use Case
File System read_file, write_file, list_directory, glob Reading configs, writing outputs, navigating project structure
Web fetch_url, search_web, scrape_html Researching CVEs, competitor analysis, market data
Developer run_command, git operations, docker Running tests, deployments, CI/CD pipelines
Communication send_email, post_slack, create_ticket Customer notifications, team alerts, issue creation
Data query_database, read_api, webhook CRM data, metrics, third-party integrations

Tool Orchestration: How to Chain Tools Effectively

Individual tools are powerful. Chained tools are transformative. Tool orchestration is the skill of combining tools into sequences that accomplish complex tasks.

A simple chain for CVE research:

1. search_cve(keyword="zero trust")     --> finds relevant CVEs
2. fetch_cve_detail(id=CVE-XXXX)       --> gets full CVE data
3. search_web(query=affected_products)   --> finds affected product versions
4. write_file(path="signals/cve-xxxx.md", content=summary)  --> saves signal

The key to good orchestration is explicit tool descriptions. If the agent knows what each tool outputs, it can chain them automatically. If descriptions are vague, the agent will misuse or ignore tools.

Security Considerations for Tool Use

Tools give agents power — and power requires responsibility. When designing a harness, consider:

  • Least privilege — Give each agent only the tools it needs. A research agent does not need write access to production systems.
  • Read-only by default — Most agents should operate in read-only mode. Write access is a higher privilege that requires justification.
  • Audit logging — Log every tool call: who called it, when, with what parameters, and what it returned. You need this for debugging and security review.
  • Rate limiting — Prevent runaway loops that call the same tool thousands of times. Set max iterations and timeouts.
  • Confirmation gates — For destructive actions (delete, overwrite, send), require a human confirmation before executing.
Security consultant perspective. As someone who works in security, you already understand least privilege and audit logging. Apply the same principles to your agent tools. An agent with too many tools is like a user with too many permissions — a breach waiting to happen.

MCP in Practice: A Real Security Loop Example

Here is how MCP tools could power a real compound loop system for a security consulting team:

+---------------------------------------------------+
|  Security Loop System                               |
|                                                   |
|  +---------------+  MCP Server 1: NVD / CISA  |
|  | CVE Research  |-> Tools: fetch_cve, search_nvd |
|  | Loop          |                                 |
|  +-------+-------+                                 |
|          | signal: new_cve                        |
|          +->                                     |
|  +-------+-------+  MCP Server 2: Internal CRM  |
|  | Impact        |-> Tools: query_customers,        |
|  | Assessment    |           check_exposure         |
|  +-------+-------+                                 |
|          | signal: high_impact_cve               |
|          +->                                     |
|  +-------+-------+  MCP Server 3: Email / Docs |
|  | Proposal      |-> Tools: generate_pdf,           |
|  | Loop          |           send_email,          |
|  +---------------+                                 |
+---------------------------------------------------+

Each loop connects to its own MCP server with the specific tools it needs. The loops communicate through the shared file system, not through the MCP connections.

Key Takeaways

  • Tools give agents the ability to act in the real world — without them, agents are just text generators.
  • A good tool has: clear name, explicit description, typed inputs, predictable output, and single responsibility.
  • MCP is the standard protocol that lets any MCP-compatible agent connect to any MCP-compatible tool — the "USB for AI agents."
  • Tool orchestration chains tools into sequences. Explicit descriptions enable the agent to chain tools automatically.
  • Apply security principles: least privilege, read-only by default, audit logging, rate limiting, and confirmation gates.
