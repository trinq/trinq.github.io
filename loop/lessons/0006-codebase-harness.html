Lesson 0006
Codebase Harness

Reading time: ~25 minutes. Light coding — optional exercises.

So far, you have learned about harness design, inner loops, outer loops, and compound loops. Now let's bring it all together with a practical, hands-on topic: making your codebase work with agents, not against them. This is the Codebase Harness — the set of practices that make a codebase agent-friendly.

The Problem: Codebases Are Built for Humans

Most codebases are designed for human collaborators who have context built up over months of working in the codebase. They know:

  • Where to find the authentication logic
  • Why a certain pattern exists
  • Which file to modify for a specific feature
  • How to run tests and what they mean

Agents do not have this context. A fresh agent dropped into a codebase sees a wall of files. Without guidance, it will:

  • Read irrelevant files, wasting context budget
  • Modify the wrong files
  • Reinvent existing patterns instead of following them
  • Run commands incorrectly or miss critical setup steps

The solution: build a codebase harness that provides the context humans take for granted.

The Three Qualities of an Agent-Friendly Codebase

Legible

The agent can easily understand the structure, find the right files, and know where to make changes. This is achieved through AGENTS.md, clear directory organization, and self-documenting code.

Executable

The agent can spin up the dev environment, run tests, and verify work with minimal friction. This requires one-command setup, clear startup scripts, and documented dependencies.

Verifiable

The agent has tools to check its own work before presenting it to humans. This means tests, linters, and explicit acceptance criteria so the agent knows when it is actually done.

Making It Legible: AGENTS.md

AGENTS.md is a file at the root of your project that tells agents everything they need to know. Think of it as a README written for an AI colleague.

A good AGENTS.md includes:

  • Project overview — What does this project do? Who are the stakeholders?
  • Directory structure — A map of the codebase with explanations of what each folder contains
  • Key patterns — The coding conventions, naming rules, and architectural patterns used
  • Where to find things — Common tasks and which files to look at
  • What to avoid — Known anti-patterns, deprecated files, sensitive areas

Example structure:

# AGENTS.md

## Project Overview
This is the internal security proposal automation system.
It generates customized security proposals for enterprise clients.

## Directory Structure
- /src/agents/        # Agent prompt definitions and loop configurations
- /src/services/     # Business logic (proposal generation, PDF export)
- /src/data/          # Client data, templates, signals
- /tests/             # Playwright e2e tests, unit tests

## Key Patterns
- All agent prompts are in /src/agents/prompts/
- Signal files follow schema in /src/data/signals/README.md
- Proposals use the template system in /src/data/templates/

## Where to Find Things
- To add a new proposal type: /src/services/proposals/
- To modify signal behavior: /src/agents/signals/
- To change the output format: /src/data/templates/

Making It Executable: Dev Scripts

Agents need to run the project without manual setup. Every project should have a dev.sh (or Makefile) at the root that handles:

  • Setup — Install dependencies, set environment variables
  • Start — Spin up the dev server
  • Test — Run the test suite
  • Verify — Run linting and type checking

Example dev.sh:

#!/bin/bash
# dev.sh — Agent-facing project commands

case "$1" in
  setup)
    npm install
    cp .env.example .env
    ;;
  start)
    npm run dev
    ;;
  test)
    npm test
    npx playwright test
    ;;
  verify)
    npm run lint
    npm run typecheck
    ;;
  *)
    echo "Usage: ./dev.sh {setup|start|test|verify}"
    ;;
esac

Making It Verifiable: Self-Check Systems

Agents need to know when their work is actually correct. There are 3 verification layers:

Automated Tests

Unit tests and integration tests that run automatically. The agent should be able to run the full test suite with one command. If tests fail, the agent knows the work is not done.

Linters and Type Checkers

Static analysis tools (ESLint, Prettier, TypeScript, Ruff) that catch style and type errors. These run fast and give the agent immediate feedback.

Acceptance Criteria

Explicit checklists in the task prompt: "This task is complete when: [1] tests pass, [2] no lint errors, [3] the proposal renders correctly in the browser."

Critical rule. Do not let an agent verify its own work without a separate verification step. The source video says: "it just generally doesn't work that well." Always spawn a verifier agent — a separate, read-only agent that checks the output and reports back.

Verification Agent Pattern

Instead of letting the agent self-verify, spawn two agents:

  1. Worker agent — Does the actual work (write code, create documents)
  2. Verifier agent — Read-only. Checks the output against acceptance criteria, runs tests, and reports findings
Task prompt
    |
    +-- Worker Agent --> Changes to codebase
    |                         |
    |                         +-- Verifier Agent (read-only)
    |                                    |
    |                                    +-- Report: pass / fail / issues
    |
    +-- Human reviews the report

The verifier agent should be given only the acceptance criteria and the ability to read files and run tests — not the ability to modify files.

Directory Structure for a Loop-Aware Codebase

Here is a recommended structure for a codebase designed for agents:

project/
  AGENTS.md              # Agent-facing documentation
  dev.sh                 # One-command dev workflow
  .cursor/
    rules/               # Cursor-specific agent rules
  src/
    agents/              # Agent configurations
      prompts/           # System prompts for each loop
      contracts/         # Loop contracts
    loops/               # Loop implementations
    services/            # Business logic
  tests/
    e2e/                # Playwright tests (verifiable by agent)
    unit/                # Unit tests
  signals/               # Shared file system signals
  artifacts/             # Generated outputs
  worklog.md             # Global activity log

Key Takeaways

  • A codebase harness makes your codebase Legible (agents understand structure), Executable (agents can run it), and Verifiable (agents can check their work).
  • AGENTS.md is the agent's map of the codebase — include directory structure, patterns, and where to find things.
  • dev.sh provides one-command workflows: setup, start, test, verify.
  • Use a verifier agent — a read-only agent that checks work — not self-verification.
  • Verification has 3 layers: automated tests, linters/type checkers, and explicit acceptance criteria.

Questions to Think About

  1. Look at a codebase you work with. Is it legible to a fresh agent? What would an agent miss on day one?
  2. Create a draft of what your AGENTS.md would look like for one of your projects.
  3. What verification tools do you already have in place? What is missing?
If anything is unclear, ask. This lesson bridges theory and practice. Before moving to the final lesson, make sure you can explain what makes a codebase legible, executable, and verifiable.
