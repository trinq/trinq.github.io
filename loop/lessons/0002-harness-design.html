Lesson 0002
Harness Design

Reading time: ~20 minutes. No coding required.

In the previous lesson, you learned that a Loop Engineer works at the outer loop — the layer surrounding agents. That layer is called the Harness. This lesson breaks down what the harness actually is and how to design one.

What is the Harness, exactly?

The harness is everything that is not the model. It is the scaffolding that makes agents useful, reliable, and autonomous. Without a harness, an agent is just a chatbot that answers one question and forgets everything.

The harness has 5 core components:

1. Prompt Design

How you write instructions that tell the agent what to do. This includes system prompts, user templates, and output formats. A well-written prompt is precise; a bad one is vague and produces unpredictable results.

2. Context Management

How you control what information the agent sees at any given moment. This includes context injection, truncation strategy, and how you prioritize relevant information over noise. Context is finite — you must manage it actively.

3. Orchestration Logic

The workflow that determines when the agent runs, what triggers it, and what happens after. This is the "brain" of the harness — the part that decides the agent's next move based on current state.

4. Hooks

Injection points in the workflow where you can intercept, log, or modify behavior. Hooks let you add observability, enforce policies, or pause/stop the agent mid-loop without rewriting its core logic.

5. Shared File System

The persistent storage layer that lets multiple agent sessions share state. This is the most overlooked component and the most critical one. Without it, each session starts from scratch every time.

The Shared File System: The Foundation

Let's focus on component 5, because it underpins everything else.

When many agent sessions work on the same project, each session needs answers to two questions:

  • "Where are we?" — What is the current state of the project? What has been done?
  • "What just happened?" — What did the last session accomplish? What should I know before starting?

The shared file system provides these answers. It stores:

  • Signals — Triggers and priorities generated by other loops
  • Artifacts — Outputs from agent work (docs, code, reports)
  • Contracts — Defined goals and workflows for each loop
  • Logs — A history of what was done, readable by both humans and agents
Analogy. Think of the shared file system as a shared whiteboard in an office. Instead of each person memorizing everything, everyone writes on and reads from the same board. As long as the board is accurate, any person (or agent) can pick up where the last one left off.

A Minimal Harness Architecture

You do not need a complex system to start. A minimal harness has this structure:

project/
  signals/           # Triggers and priorities
    README.md        # Signal schema definition
  artifacts/         # Agent outputs
    [artifact-001]/
      meta.json
      content.md
  contracts/         # Loop definitions
    loop-name/
      README.md
      backlog.md
  worklog.md         # Global history log
  AGENTS.md          # Agent-facing documentation
  dev.sh             # One-command setup and verify

Each folder has a README that defines the schema and expected format. This is how agents understand the structure without you explaining it every time.

Prompt Design: The First Line of Defense

A good harness prompt does 3 things:

  1. Tells the agent who it is — Role, responsibilities, boundaries
  2. Tells the agent where it is — File system structure, current project state
  3. Tells the agent what to do when done — Output format, where to save, what to log

Bad harness prompt: "You are an agent. Fix bugs in the codebase."

Good harness prompt: "You are the Support Loop agent for the security proposal team. Your job is to research new CVEs every morning and create a summary in signals/cve-daily.md using the schema defined in signals/README.md. After completing, update worklog.md with what you did."

Context Management: Feeding the Agent Wisely

Context has a hard ceiling. Even with 1M token context windows, effective useful context is ~128K–200K tokens. Beyond that, quality degrades. The harness must manage this actively.

Three context strategies:

Top-up

Feed the agent only the most relevant context for the current task. Do not dump everything. Use file summaries and excerpts rather than full files.

Summarization

When history grows too long, summarize it. Compress older conversation turns into brief notes and keep only recent, critical context.

Delegation

If a task needs deep context, spawn a focused sub-agent with exactly the files and instructions it needs. Do not try to feed everything to one agent.

Orchestration: Trigger Types

How does the harness decide to wake up an agent? There are 4 common trigger types:

Trigger When it fires Example
Time-based At a scheduled interval Every morning at 8am, run the CVE research agent
Event-based When a specific file or signal appears When signals/high-priority.md is created, run the triage agent
Request-based When a human asks User submits a ticket, agent picks it up from the queue
Dependency-based When another loop finishes After the SEO loop creates content, the Ads loop prioritizes that keyword

A mature loop system uses all 4 types in combination.

Key Takeaways

  • The harness is everything that is not the model: prompt design, context management, orchestration logic, hooks, and shared file system.
  • The shared file system is the foundation — it lets multiple sessions share state and continue work without starting from scratch.
  • A minimal harness needs: signals/, artifacts/, contracts/, worklog.md, and agent-facing docs.
  • Good harness prompts tell the agent: who it is, where it is, and what to do when done.
  • Context is finite. Use top-up, summarization, and delegation to manage it.

Questions to Think About

  1. In your current work as a Pre Sales Security Consultant, what information do you wish was always "on the whiteboard" so you did not have to re-explain it every time?
  2. What triggers could wake up an agent in your workflow? (Time? A new CVE? A customer request?)
  3. How much context do you need to give an agent to do a specific task well? Is it more or less than you think?
If anything is unclear, ask. Harness design is the foundation for everything else. Make sure this clicks before moving to Lesson 0003.
