Lesson 0005
Compound Loops

Reading time: ~20 minutes. No coding required.

In the previous lessons, you learned about the inner loop (one agent, one task) and the outer loop (orchestration of tasks). Now we reach the real power of Loop Engineering: compound loops — when multiple loops run simultaneously, share signals, and amplify each other's work.

The Difference: Simple Loop vs Compound Loop

A simple loop is what most people imagine when they hear "loop": one agent running forever with the same prompt. It is reactive, self-contained, and limited.

A compound loop is different: many loops running at the same time, all reading from and writing to the same shared file system, creating a network effect where each loop's output becomes another loop's input.

Support Loop → Signals → SEO Loop → Product Loop → Ads Loop

The key insight: each loop does not need to know about the others. They only need to follow the signal contract. The shared file system does the coordination implicitly.

How Signals Work in a Compound System

Signals are the "language" between loops. A signal is a structured message that:

  • Describes something that happened or needs attention
  • Has a clear schema so any loop can read it
  • Has a priority level that drives the outer loop
  • Optionally carries metadata (source loop, timestamp, urgency)

Example signal file:

# signals/high-ctr-keyword.md
---
type: keyword-opportunity
priority: P1
source: seo-loop
created: 2024-03-15T08:00:00Z
---

keyword: "zero trust architecture"
ctr: 4.2%
current_rank: 8
organic_content_exists: false
---

This keyword has high commercial intent but no organic content.
Recommend: create in-depth guide.

Now, the SEO loop reads this signal and creates content. The Ads loop reads the same signal and starts a campaign. The Product loop reads it and prioritizes documentation. All three react to the same signal — without knowing about each other.

Loop Types in a Compound System

Different loops serve different purposes. In a mature system, you typically see these types:

Research Loop

Scans external data sources for new information. Examples: CVE research, market analysis, competitor monitoring. Produces signals about findings.

Production Loop

Takes signals and produces artifacts. Examples: content creation, code generation, proposal writing. Consumes signals, produces content.

Distribution Loop

Takes artifacts and publishes or promotes them. Examples: SEO loop (publishes content), Ads loop (runs campaigns), Email loop (sends newsletters).

Support Loop

Handles incoming requests and issues. Examples: customer support triage, bug triage, ticket management. Produces signals that trigger other loops.

The Compounding Effect

Compound loops create value through compounding — the output of one loop becomes the input of another, and the effect grows over time:

  1. The Research loop finds a high-ctr keyword with no content → creates a signal
  2. The Production loop reads the signal → creates a 5,000-word guide
  3. The SEO loop reads the new content → publishes and optimizes it
  4. The Distribution loop sees the published content → promotes it via social and ads
  5. The Research loop sees traffic growth on that keyword → creates a signal for related keywords
  6. Back to step 2 — the cycle continues, each iteration building on the last

No single agent or loop could do all of this. But together, they create a self-reinforcing system that grows organically.

Designing Signal Contracts

The power of compound loops depends entirely on well-designed signal contracts. A contract is a README in the signals folder that defines:

  • Schema — What fields does a signal of this type have?
  • Priority levels — What do P0, P1, P2, P3 mean for this signal type?
  • Expected actions — What should a loop do when it reads this signal?
  • Lifetime — How long is this signal valid? When should it be archived?
Rule of thumb. If a loop needs to understand another loop's code to read its signals, the signal contract is broken. Any loop should be able to read any signal by following the schema in signals/README.md.

Real Example: A Security Loop System

Here is how compound loops could work for a security consulting team:

Loops:
  - CVE Research Loop: monitors NVD, CISA, security feeds daily
  - Impact Assessment Loop: reads new CVE signals, assesses customer exposure
  - Proposal Loop: reads impact signals, drafts customer-specific proposals
  - Customer Comms Loop: reads proposal completions, formats and sends emails
  - Metrics Loop: reads all loop outputs, generates weekly summary report

Each loop is simple on its own. Together, they automate the entire workflow from CVE discovery to customer proposal — without anyone manually chasing each step.

Managing Loop Dependencies

With many loops running, you need to manage dependencies carefully:

No Direct Coupling

Loops should never call each other directly. All communication goes through the shared file system. Direct coupling creates spaghetti and makes the system fragile.

Contract Versioning

When you change a signal schema, increment the version. Old loops reading new signals (or vice versa) should gracefully handle mismatches.

Loop Isolation

Each loop should be independently deployable. One loop failing should not crash the others. Use timeouts, error boundaries, and circuit breakers.

Key Takeaways

  • Compound loops are many loops running simultaneously, all reading and writing to the shared file system.
  • Signals are the language between loops — structured messages with a clear schema.
  • Four common loop types: Research, Production, Distribution, and Support.
  • The compounding effect happens when one loop's output becomes another loop's input — creating a self-reinforcing cycle.
  • Loops should never couple directly — all communication goes through the shared file system.
  • A good signal contract lets any loop read any signal without understanding the producing loop's code.
