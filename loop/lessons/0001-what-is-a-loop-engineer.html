Lesson 0001
What is a Loop Engineer?

Reading time: ~25 minutes. No coding required.

One day, you are out but a pile of PRs is still rolling into the repo. A whole bunch of agents are running in the background, automatically finding issues and doing the work. One SEO agent has generated 20–40 high-quality content pages over 2 days — without you lifting a finger. This is the product of Loop Engineering.

This lesson introduces the concept, distinguishes it from Prompt Engineering and Agent Engineering, and gives you the full picture of the loop system.

Three Layers of AI Engineering

The source video argues there are 3 layers of AI Engineering, each at a different layer:

Prompt Engineer

Optimize commands for a model. Give one input, get a better output. Focus on context injection techniques to steer model behavior — like "always return text in CAPS" or "write in a journalistic style."

Agent Engineer

Design an agent to complete a task well. Focus on tool use, memory, and context management of a single agent. "How do we make this agent do task X well?"

Loop Engineer

Set up the environment so many agents run automatically, share state, and self-improve. "How do we get agents woken up at the right time, doing the right work, and logging results?" This is the outermost layer — the layer surrounding agent runtime.

In short: Prompt Engineer works with 1 prompt. Agent Engineer works with 1 agent. Loop Engineer works with a system of many agents running together.

The Three Core Components of an Agent

Any agent — ChatGPT, Claude, Cursor, Copilot — has 3 components:

Agent

The language model (LLM) receives input, thinks, decides on the next action. It is the "brain" of the system.

Loop

The thinking loop — call tool → receive result → think again → call tool or answer. The loop continues until the agent decides the task is done.

Memory Layer

The memory storing conversation history, files, and state. The memory layer is what distinguishes an agent from a plain chatbot — the agent remembers "what it did before."

There is a fourth component: Tool Access — the ability to call APIs, read files, send requests. But the 3 above are what everyone has in common.

The Inflection Point: From Task Completion to Cross-Session Work

Mid-2024, context windows grew (4K → 128K → 1M tokens), models got smarter, and tool use became common (MCP). This changed how we use LLMs:

2023 2024–2025 2025–2026
Task 1 turn, small 30 minutes – 2 hours Many sessions, many agents
Context ~4K tokens 128K–1M tokens Effective: 128K–200K
Work pattern Prompt → Output Agent + Tools Loop + State tracking

The larger context window does not mean "use 1M tokens." The effective context is only about 128K–200K tokens. This creates a problem: how can many different agent sessions work on the same large task without losing state between calls?

Answer: Harness.

What is the Harness?

The harness is everything that is not the model in an agent system. It includes:

  • How you write prompts
  • How you manage context
  • Orchestration logic
  • Hooks
  • Shared file system — the most important thing most people overlook

Why is the shared file system so important? Because when many agent sessions run on the same task, each session needs to know "where things are" and "what was done." Without a shared file system, each session is like a goldfish that has to remember everything in context — but the context window is finite.

Inner Loop vs Outer Loop

Inner Loop

Optimize how one agent completes a task. Focus on: prompt optimization, tool selection, context management. Question: "How do we make the agent do this task well?"

Outer Loop

Optimize how the system decides which task to work on next. Focus on: trigger setup, state tracking, cross-session coordination. Question: "Which task should we do next?"

Loop Engineers work mainly at the outer loop — setting up the environment so agents get woken up at the right time, do the right work, and log results — without anyone prompting them by hand.

The Picture: This Is Not Just One Loop

When people hear "loop," they think of a simple while loop: "run the agent forever with the same prompt." But the source video talks about something different: compound loops — many different loops all reading and writing to a shared file system, creating a compounding effect.

Support Loop → Signals → Product Loop → SEO Loop → Ads Loop

Example: SEO loop discovers a keyword with high CTR but no organic content → creates a signal. Ads loop reads this signal → prioritizes creating ads for that keyword. Product loop reads the signal → prioritizes building content. They all share one "brain" — the signals folder.

Codebase Harness: Making Agents Work Autonomously

A practical part the source video emphasizes: the codebase must be agent-friendly. Three qualities:

  • Legible — The agent can easily understand the structure, know where to make changes, know where to find information. This can be through AGENTS.md, documentation, or custom linting rules.
  • Executable — The agent can spin up the dev server, run tests, verify work with just one command. Include dev.sh, startup scripts.
  • Verifiable — The agent has tools to check results. Playwright for browser automation and video recording. End-to-end tests for critical flows. PR checklist to ensure quality before human review.
Key point. Do not let the agent self-verify. The source video says: "it just generally doesn't work that well." Instead, spawn a read-only verifier agent to check results and report.

Three File Types in a Loop System

Artifacts

The output of each agent's work. Can be docs, signals, tasks, tickets, campaigns. Each artifact has its own folder, a README defining the schema and process, and a metadata + timeline file.

Contracts

The README in each loop folder, defining: the loop's goal, workflow, backlog, timeline. Each time the loop runs, it reads the contract first to understand "where things are."

Logs

Global worklog.md. The agent logs completed work, readable by both humans and agents. Contains information that does not belong in any artifact but needs to be kept.

Key Takeaways

After this lesson, you know:

  • Loop Engineer is someone who sets up a system of many agents running automatically, sharing state, and self-improving. This is the outermost layer (outer loop), different from Prompt Engineer (1 prompt) and Agent Engineer (1 agent).
  • Harness is everything that is not the model — especially the shared file system so many agent sessions can all understand "where things are."
  • Compound loops are when many different domain loops (support, SEO, product) all read/write signals, creating a compounding effect.
  • Codebase harness needs 3 qualities: legible, executable, verifiable. Do not let the agent self-verify.
