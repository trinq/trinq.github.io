The 0-to-90-to-100 Accuracy Trajectory — How Real AI Systems Actually Improve

"The last mile is always the hardest mile." — Zheng Xue, JP Morgan Chase Private Bank

Prerequisites: All four prior lessons. You should be able to name the six Ask David components, explain the supervisor's four jobs, design a sub-agent evaluation plan, and specify HITL triggers before starting. This lesson synthesizes everything.

What this lesson teaches. By the end of this lesson you will understand the empirical accuracy trajectory of production AI systems — from raw model output through three distinct improvement phases, each requiring different techniques and yielding diminishing returns. You will also know the three meta-learnings that govern the entire course.

1. The Empirical Reality — 0 to 50% is the Starting Point

Ask David opens with a number that surprises most engineers: when a general-purpose model is applied to a specific financial domain, accuracy typically starts below 50%. This is not a failure of the model — it is the nature of the problem. A model trained on the entire internet does not know the difference between a fund termination for underperformance versus regulatory action versus voluntary wind-down. It has never seen your specific database schema, your document format, or your domain's conventions for answering "Why was this fund terminated?"

The trajectory that follows is not linear. Each phase of improvement requires different techniques, and the effort to move from 90% to 95% is categorically different from the effort to move from 50% to 80%. Treating them the same way wastes enormous effort.

2. The Four-Phase Trajectory

Phase 1
0–50%
Base model on domain task
Phase 2
50–80%
Engineering + chunking + search
Phase 3
80–90%
Workflow subgraphs + fine-tuning
Phase 4
90–100%
Last-mile: targeted fixes

Phase 1: 0 to 50% — The Baseline (No Special Techniques)

The model is doing its best with general knowledge. The gap between its training data and your domain is large. Examples of failure modes at this stage:

  • Confusing fund termination reasons (performance vs. regulatory vs. voluntary)
  • Answering based on publicly available information rather than internal records
  • Using wrong field names when attempting structured queries
  • Hallucinating compliance details that don't exist in your documents

This phase is expected. Don't try to engineer your way out of it — that's Phase 2 work.

Phase 2: 50 to 80% — Quick Engineering Wins

This is where most teams spend too little time. The gains in this phase come from engineering, not model changes. Ask David identifies two primary levers:

Chunking strategy
How you split documents into retrievable chunks is the single highest-leverage change for RAG agents. Fixed-size chunking (e.g., 512 tokens) misses semantic boundaries — a sentence that completes a thought at token 510 gets cut off. Semantic chunking (split on paragraphs or sections) preserves meaning. For a financial corpus: annual reports chunked by section, emails chunked by thread, compliance documents chunked by regulatory reference. Each document type has an optimal chunking strategy. This is pure engineering — no model change required.
Search algorithm improvements
Moving from pure vector similarity to hybrid search (dense + sparse), adding re-ranking, or tuning the similarity threshold. For structured data: better schema prompts, few-shot examples in the SQL agent's context. These are all engineering improvements. The model stays the same — you're just giving it better inputs and better retrieval.

The Phase 2 mindset shift: Stop asking "which model should I use?" and start asking "how is my retrieval pipeline structured?" In Phase 2, the bottleneck is almost never the model — it's the engineering between the user and the model.

Phase 3: 80 to 90% — Subgraph Isolation

Once engineering is exhausted, the remaining errors are structural. Ask David's insight: create separate workflow subgraphs for different query types, so that improvements to one subgraph don't accidentally degrade another.

In Ask David: general QA queries go through a different subgraph than specific fund queries. Why? Because optimizing the general QA flow to improve "What are the top-performing funds this quarter?" should not risk degrading the specific fund flow for "Why was fund XYZ terminated?" Each subgraph can be independently tuned, evaluated, and deployed.

Subgraph isolation
Each subgraph is a self-contained LangGraph (or equivalent) with its own sub-agents, prompts, and evaluation dataset. When you improve the RAG chunking for fund documents, you run the fund subgraph eval suite. The general QA subgraph isn't affected. This prevents the most expensive type of regression: fixing one thing and breaking another.
Fine-tuning for specific question types
Once subgraphs are isolated, you can fine-tune a smaller model for a specific subgraph's query distribution. Ask David fine-tunes models for the analytics agent (ReAct tool calling) separately from the RAG agent (document synthesis). Fine-tuning at the subgraph level is cheaper and more targeted than fine-tuning a single model for everything.

Phase 4: 90 to 100% — The Last Mile

Ask David is explicit: this is where the work becomes categorically different. The last 10% requires targeting specific error categories one at a time. There is no general technique — only relentless per-case analysis.

Error category analysis
Run your evaluation results through a classification: which errors are retrieval failures (wrong chunks retrieved), which are synthesis failures (right chunks, wrong answer), which are routing failures (sent to wrong subgraph), which are personalization failures (right answer, wrong format for role)? Each category has a different fix. Retrieval failures → improve chunking or embedding. Synthesis failures → improve synthesis prompt or add few-shot examples. Routing failures → improve supervisor's intent classification. Personalization failures → expand role-specific examples in the personalization node.
Ground truth expansion
At 90%+ accuracy, the remaining errors are rare events — edge cases, unusual fund structures, novel regulatory situations. The evaluation dataset must expand to cover these. This is expensive: you need domain experts to label each new edge case. But it is the only path to the last mile. Every new labeled example in the evaluation dataset is a potential fix for one more error category.

The temptation to skip phases: Teams that skip Phase 2 (engineering) and jump straight to fine-tuning (Phase 3) waste significant resources. A model fine-tuned on poorly chunked data learns the chunking artifacts, not the domain. Fix the engineering foundations first. The phase order is not a suggestion — it reflects what actually moves the needle at each accuracy level.

3. The Three Meta-Learnings — What Ask David Teaches About AI Development

Zheng Xue closes with three principles that apply far beyond the Ask David system. These are the meta-learnings of the entire course:

The three lesson learned

1. Start simple and refactor often.

The biggest wasted effort in AI system development is premature complexity. Building a sophisticated multi-agent architecture before you understand which queries fail and why is backwards. Start with one agent that handles the happy path. Evaluate. Find the failure modes. Then split — and split only where evaluation shows the split is justified.

Ask David's architecture is not what it started as. The planning node, the separate subgraphs, the reflection node — these are all refactors that emerged from evaluation results, not initial designs. The initial design was much simpler.

2. Evaluation-driven development.

Every technique in every phase is justified by measurement. Chunking strategy changes are justified by Precision@K improvements. Subgraph isolation is justified by routing accuracy improvements. Last-mile fixes are justified by per-error-category analysis. Without measurement, you are guessing — and in probabilistic systems, guessing is expensive.

The evaluation flywheel from lesson 0003 is what makes this sustainable: evaluation generates ground truth, ground truth makes evaluation better, better evaluation makes the system better.

3. Human in the loop.

Some decisions permanently justify human involvement regardless of AI accuracy. The 90–100% journey doesn't eliminate the need for HITL — it refines the triggers. As the AI improves, fewer queries need human review. But the architecture for human involvement never goes away. The goal is not to remove humans; it is to make human time increasingly precious — reserved for cases that genuinely need it.

"Number one: start simple and refactor often. Number two: evaluation-driven development. Number three: human in the loop."
— Zheng Xue, JP Morgan Chase Private Bank
