The Supervisor + Sub-Agent Pattern — How to Split Work Across Agents

Lesson 0001 showed you the architecture. This lesson shows you how to design it — where to draw agent boundaries, what each sub-agent actually does, and how the supervisor delegates without becoming a bottleneck.

Prerequisites: Lesson 0001 — you should be able to name the six components of Ask David before starting.

1. The Core Problem — One Agent Can't Do Everything

A single agent answering investment research questions would fail for a simple reason: "Why was fund XYZ terminated?" requires three different capabilities simultaneously — querying a structured database of fund records, retrieving documents about past performance, and running an analytics model — and a single agent that tries to do all three ends up doing all three poorly.

The supervisor + sub-agent pattern solves this by splitting the problem along capability boundaries:

  • Each sub-agent is an expert in one thing — one agent talks to SQL, another talks to a vector database, another talks to an API.
  • The supervisor's job is not to do the work, but to route the work to the right expert, aggregate the results, and decide when to escalate.
  • Each sub-agent can be built, tested, and improved independently.

The key insight from Ask David: the supervisor is not a "super agent" that knows everything. It is a routing layer with memory. It delegates; it doesn't execute.

2. The Supervisor — Routing, Not Doing

The supervisor in Ask David has four jobs:

  1. Understand intent. Classify the user's question: general QA or specific fund question?
  2. Route to sub-agents. Send the task to one or more specialized agents. Optionally wait for responses before routing to the next.
  3. Manage memory. Short-term (session context) and long-term (user preferences, past queries).
  4. Decide escalation. Know when to invoke human-in-the-loop.

In LangGraph terms, the supervisor is a graph node, not an agent in the traditional sense. It decides which edges to traverse next. Here's the pseudo-logic:

Supervisor Decision Logic (Pseudo-code)

state = { question, intent, memory, answer }

# Step 1: Classify intent
intent = classify_intent(state.question)
# → "general_qa" or "specific_fund"

# Step 2: Route to correct sub-graph
if intent == "specific_fund":
    # Enter the specific fund sub-graph
    fund_data = structured_data_agent(query=state.question, memory=state.memory)
    doc_data = doc_agent(query=state.question, fund_context=fund_data)
    analytics_data = analytics_agent(query=state.question, fund_id=extract_fund_id(fund_data))
    raw_answer = synthesize(
        structured=fund_data,
        docs=doc_data,
        analytics=analytics_data
    )
else:
    # Enter the general QA sub-graph
    raw_answer = general_qa_agent(query=state.question, memory=state.memory)

# Step 3: Quality gate
if not reflection_judge(raw_answer):
    # Retry — send back to sub-agents
    raw_answer = retry(...)

# Step 4: Personalize and return
final_answer = personalize(raw_answer, role=state.memory.user_role)
return { answer: final_answer, memory: update_memory(state.memory, session) }

Common mistake: Building a supervisor that tries to answer questions itself, only calling sub-agents as "tools." This creates a bottleneck — the supervisor becomes the limiting factor. The supervisor should route, not answer.

3. Sub-Agent Types — What Goes Inside Each Box

3a. Structured Data Agent

Handles: relational databases, APIs with structured responses, anything with a schema.

How it works:

  1. Receives a natural language question.
  2. Translates it into a SQL query or API call using an LLM with a schema prompt.
  3. Executes the query against the database or API.
  4. Uses an LLM to summarize the structured result into natural language.

In Ask David: "Why was fund XYZ terminated?" → SELECT status, termination_date, reason FROM funds WHERE id = 'XYZ' → summarized as "The fund was terminated due to performance issues on 2024-03-15."

Key pattern: Schema-supplied prompting
The LLM that generates SQL needs to know the database schema. In practice, this means injecting the schema into the system prompt — table names, column names, types, relationships. This is a fixed cost: you build it once, the agent uses it forever.

3b. Unstructured Doc Agent (RAG)

Handles: PDFs, emails, meeting notes, video/audio transcripts, any text that isn't in a database.

How it works:

  1. Pre-processing (upfront): Documents are ingested, chunked, embedded, and stored in a vector database.
  2. Query time: User question → embedded → similarity search against vector DB → top-k chunks retrieved → LLM synthesizes a response from the chunks.

Critical: Pre-processing is the hidden work. Ask David mentions this explicitly — the RAG agent only works because someone already ingested, chunked, and vectorized years of documents. This is a data engineering problem, not an AI problem. Don't underestimate the upfront cost.

Key decisions for a RAG agent:

  • Chunking strategy. How big are the chunks? Overlapping or not? Fixed size or semantic (paragraph, section)? The answer depends on the query type — broad questions need larger chunks; specific questions need smaller, targeted chunks.
  • Embedding model. Which model creates the vector representation? Different models encode different semantics.
  • Retrieval strategy. Pure similarity search? Hybrid (keyword + vector)? Re-ranking? Parent-document retrieval?
  • Synthesis prompt. How does the LLM use the retrieved chunks to answer? This is separate from the retrieval step.

3c. Analytics Agent

Handles: proprietary models, internal APIs, complex computations that don't fit SQL or RAG.

Ask David uses two approaches:

  • ReAct agent for simple queries: the agent reasons about which API to call, calls it, observes the result, and continues until it has an answer. Think tool-use in a loop.
  • Text-to-code generation for complex queries: the agent writes Python or R code to run the analytics, but a human reviews before execution. This is where human-in-the-loop is explicit and appropriate.
Pattern: Two-tier analytics
Simple queries use fully autonomous ReAct agents. Complex queries use human-supervised code generation. The threshold between "simple" and "complex" is a configuration decision — you can tune it based on how much risk each query type carries.

4. Designing Agent Boundaries — Where to Split

The hardest design decision: where do you split the work? Get this wrong and you end up with either:

  • Too few agents: One agent that tries to do everything, leading to mediocre performance across the board.
  • Too many agents: An explosion of agents that are hard to coordinate, hard to evaluate, and introduce latency at every hop.

Use these three heuristics from Ask David:

Heuristic 1: Split by data type
If two sub-tasks require fundamentally different data sources (structured DB vs. document corpus vs. API), they belong to different agents. Each agent gets exactly one data source as its primary input.
Heuristic 2: Split by output type
If two sub-tasks produce outputs that can't be meaningfully merged by a simple concatenation, they belong to different agents. A SQL result and a document excerpt need a synthesis step — put that in the supervisor or a dedicated synthesis node.
Heuristic 3: Split by evaluation metric
If you can't define a separate evaluation metric for a sub-task, it's not a separate agent. "The SQL agent is good if its queries are correct." "The RAG agent is good if it retrieves relevant documents." If you can't name the metric, the split isn't justified yet.

5. Memory — The Supervisor's Second Job

Ask David uses two types of memory in the supervisor:

Type Scope Content Example
Short-term Session Current conversation, recent queries, pending context "In this conversation, the user asked about fund XYZ, then asked a follow-up about its performance history."
Long-term Persistent User preferences, role, past interactions "This user is a financial advisor. They prefer concise summaries. They've asked about terminated funds twice this month."

The practical implementation: short-term memory is the conversation history in the LLM context window. Long-term memory is a separate retrieval step — before answering, the supervisor retrieves relevant facts from persistent storage. This is often called a RAG-for-memory pattern: embed the user's known facts, retrieve relevant ones at query time.
