Ask David: The Architecture of an Enterprise Multi-Agent System

"D.A.V.I.D. — Data Analytics Visualization Insights and Decision Making Assistant." When a financial advisor asks "Why was this fund terminated?", a multi-agent system answers — with human oversight, personalized for the audience, and evaluated at every step.

Source: David Odomirok & Zheng Xue (JP Morgan Chase Private Bank) — "How JP Morgan Built An AI Agent for Investment Research with LangGraph", LangChain Interrupt, YouTube, 14:11, 2025-05-29. Watch it before or after this lesson.

What this lesson teaches. By the end of this lesson you will be able to draw the Ask David architecture from memory, explain why each component exists, and name the three lesson learned. You will also produce a one-paragraph description of a multi-agent system you could build in your own domain.

1. The Problem — Thousands of Products, One Research Team

JP Morgan Chase Private Bank's investment research team manages lists of thousands of financial products — each backed by decades of data. When an advisor has a question ("Why was this fund terminated?"), the process is manual: reach out to a human researcher, dig through databases and documents, piece together an answer, and build a presentation — all while the client is waiting.

This doesn't scale. A team of researchers can't answer thousands of questions per day. The business case for automation is obvious: not to replace the researchers, but to give every advisor the research capability of a specialist — in real time, in a client meeting.

The solution is not a single chatbot. It's a multi-agent system — a team of specialized AI agents, orchestrated by a supervisor, with access to different data types, memory, reflection, and human oversight. This is Ask David.

2. The Architecture — A System, Not a Script

Most AI demos show a single agent that takes a question and returns an answer. Ask David has six distinct components that must work together. Here is the full picture:

Why each component exists

Planning node. Every user question is first classified: is this a general question ("how do I invest in gold?") or a specific fund question ("why was fund XYZ terminated?"). These route to different sub-graphs with different agents. Without routing, a general model answers fund-specific questions badly.

Supervisor agent. The single entry point that understands user intent, delegates to sub-agents, and manages both short-term memory (session context) and long-term memory (persistent preferences). It also knows when to invoke human-in-the-loop — this is not a fully autonomous system.

Structured Data Agent. Translates natural language into SQL queries or API calls against structured databases. The LLM summarizes the tabular result. This handles the backbone of production financial data.

Unstructured Doc Agent (RAG). Handles the vast library of documents — PDFs, emails, meeting notes, video/audio transcripts. Documents are pre-processed, vectorized, and stored. The RAG agent retrieves and synthesizes. Without pre-processing, unstructured data is inaccessible at query time.

Analytics Agent. Runs proprietary models and analytics APIs. For simple queries, uses a ReAct agent with API tools. For complex queries, uses text-to-code generation with human supervision before execution. This is where the "white glove" expert analysis gets democratized.

Personalization node. The same underlying data gets presented differently depending on who's asking. A due diligence specialist needs full detail. A financial advisor needs a concise summary. The personalization layer tailors depth and framing without changing the underlying answer.

Reflection node (LLM-as-judge). Before the answer is returned, an LLM evaluates whether the answer makes sense. If it fails the check, the system retries. This is automated quality control — catching errors before the user sees them.

Summarization node. Wraps up: summarizes the conversation, updates memory, returns the final answer.

3. Three Lesson Learned — Iterate, Evaluate, Keep Humans In

Lesson 1: Start Simple, Refactor Often

The Ask David team did not build the full architecture on day one. They started with a vanilla ReAct agent, validated it, added a RAG agent, then integrated both under a supervisor, then added sub-graphs for specific intents. Each iteration built confidence before adding complexity. The lesson: resist the temptation to architect the complete system upfront. Ship a working agent, get it into production, refactor.

Lesson 2: Evaluate Early, Evaluate Continuously

Enterprise AI projects have short development phases but long evaluation phases. The Ask David team started evaluation before they had ground truth — using LLM-as-judge and trajectory evaluation from day one. The key insight: "once you start doing evaluation, you will have review; once you start doing review, you will accumulate ground truth examples." The evaluation drives the ground truth, not the other way around. Each sub-agent is also evaluated independently — the evaluation data for the structured data agent is different from the doc agent's evaluation data.

Lesson 3: Human-in-the-Loop is a Design Choice, Not a Failure Mode

The financial stakes are real: billions of dollars of assets. The team acknowledges that 90–100% accuracy is "the last mile, and the last mile is always the hardest mile." For high-stakes domains, human-in-the-loop is not a workaround for unreliable AI — it's a deliberate architectural decision. Ask David still consults with real David whenever needed. The system knows when to escalate.

4. The Accuracy Trajectory — 0 to 100

The speakers described the journey from a general-purpose model to a production-grade system as a predictable trajectory:

<50%
Base model
General LLM, no domain tuning
~80%
Quick wins
Chunking, search, engineering
~90%
Workflow chains
Sub-graphs, intent routing
~100%
Last mile
Human-in-the-loop

The 0–50% step is using a general model off the shelf. The 50–80% step is proper engineering: chunking strategies, retrieval algorithms, prompt engineering. The 80–90% step is architectural: splitting into sub-graphs so you can tune specific question types without breaking others. The 90–100% step is human judgment — and you never fully close that gap in high-stakes domains.

Do not skip the 0–80% engineering. The temptation is to jump straight to multi-agent orchestration before getting the fundamentals right. A supervisor agent over three bad sub-agents is still a bad system. Get each sub-agent to 80%+ before wiring them together.
