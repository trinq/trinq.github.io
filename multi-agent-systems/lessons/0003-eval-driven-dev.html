Evaluation-Driven Development — Measure Before You Build, Improve What You Measure

"Once you start doing evaluation, you will have review. Once you start doing review, you will accumulate more of the ground truth examples." — Zheng Xue, JP Morgan

Prerequisites: Lesson 0001 (Ask David architecture) and Lesson 0002 (supervisor + sub-agent patterns). You should understand the three sub-agent types and the evaluation data from Ask David before starting.

What this lesson teaches. By the end of this lesson you will be able to design an evaluation plan for any sub-agent — choosing the right metric, the right ground truth source, and the right evaluation method. You will also understand why evaluation should start on day one, even before ground truth exists.

1. The Inversion — Evaluation Doesn't Follow Development

In traditional software development, the instinct is: build first, measure later. Write the code, ship it, see if it works. This is fine for deterministic systems — a function either returns the right output or it doesn't. But multi-agent systems are probabilistic. A supervisor that routes 80% of questions correctly today might route 60% correctly next week after a model update or a data drift event. You need to know before it matters, not after.

The Ask David team's key insight is an inversion of the conventional sequence:

Evaluate early → Find weak links → Improve iteratively → Accumulate ground truth → Better evaluation

This is a flywheel — evaluation drives ground truth, which drives better evaluation, which drives better systems.

The critical reversal: you do not wait for ground truth to start evaluation. Ground truth accumulates from evaluation, not the other way around. You start with imperfect proxies — LLM-as-judge, trajectory evaluation, human spot-checks — and those generate examples that become your ground truth over time.

"There are so many metrics beyond just accuracy and each one of them will provide you some insight and you know what once you start doing evaluation you will have review once you start doing review you're actually going to accumulate more of the ground truth examples."
— Zheng Xue, JP Morgan Chase Private Bank

3. Choosing the Right Metric — Per Task, Not Per System

The most common mistake in evaluating multi-agent systems: using one metric (accuracy) for the whole system, when different sub-agents need different metrics entirely. Ask David makes this explicit — each sub-agent has its own evaluation dataset and metrics.

Structured Data Agent — SQL / API Queries

Metric: Query correctness (exact match for SQL) or semantic match for API responses.
Ground truth: SQL queries written by a data analyst + verified results. Can be generated from existing query logs.
Method: Automated — execute the query against a test database, compare results. No LLM needed for structured data.

Unstructured Doc Agent — RAG Retrieval

Metric: Precision@K (are relevant documents in the top K results?) + Recall@K (are all relevant documents in the top K?)
Ground truth: A human annotator marks which documents are relevant to each question. Starts small (20 questions), grows over time.
Method: Automated embedding similarity first pass → human spot-check on edge cases. Can also use LLM-as-judge to evaluate whether retrieved chunks actually support the generated answer.

Analytics Agent — Tool Use (ReAct)

Metric: Trajectory accuracy — did the agent call the right tools in the right order?
Ground truth: Expert traces: a data scientist works through 20 example queries and writes the ideal tool-call sequence.
Method: Automated comparison of the agent's tool-call sequence against the expert trace. Partial credit for partially correct trajectories.

Analytics Agent — Code Generation

Metric: Code correctness (does it execute without error?) + result accuracy (does the output match the expected answer?)
Ground truth: The expected computational result, computed independently by a data scientist.
Method: Automated execution in a sandbox — run the generated code, compare output. This is the only case where you can get near-perfect evaluation with zero human involvement.

Personalization Node — Role-Tailored Answer

Metric: Role-appropriate depth — does the answer match what the role expects?
Ground truth: SMEs for each role (advisor, due diligence specialist) rate whether the answer is appropriately detailed.
Method: Human review from SME panels. LLM-as-judge can approximate but cannot fully replace domain expertise for nuanced role-based judgments.

Reflection Node — Answer Quality Gate

Metric: Pass rate (what % of answers pass the reflection check?) + downstream accuracy (do answers that pass the check actually hold up to human review?)
Ground truth: Human review of a sample of passed answers.
Method: LLM-as-judge for pass/fail → human spot-check to calibrate. The goal is a reflection node that rarely passes bad answers and rarely fails good ones.

The calibration trap: LLM-as-judge can give consistent scores, but consistent scores aren't necessarily accurate scores. You must periodically calibrate: show the judge answers that humans have scored, and adjust the judge's prompt or threshold until its scores match human scores. Without calibration, you're measuring a proxy of a proxy.

4. The Evaluation Flywheel — From Zero to Robust

Zheng Xue's framing of the flywheel is the most practically useful insight from the entire Ask David talk. Here's how it works in practice:

Stage 1: Day 1 — No ground truth

LLM-as-judge → Auto-eval → Catches regressions

Use LLM-as-judge from day one. Imperfect, but gives signal. Catch obvious regressions immediately.

Stage 2: Month 1 — Human spot-checks begin

Human review → Ground truth seeds → Better auto-eval

10–20 human-reviewed examples per sub-agent per week. These become the ground truth seeds.

Stage 3: Month 3+ — Ground truth drives evaluation

Labeled dataset → Automated eval → CI-grade checks

50+ ground truth examples per sub-agent. Automated eval runs on every change. LLM-as-judge calibrated against ground truth.

The key discipline: evaluation runs on every change. A model update, a prompt change, a new tool added to a sub-agent — all of these trigger the evaluation suite. Without continuous evaluation, you can't distinguish improvement from regression.

5. Evaluating Sub-Agents Independently

Ask David makes one recommendation that most teams skip: evaluate each sub-agent independently before evaluating the full system.

Why? Because system-level accuracy hides weak links. If the full system gets 75% of answers right, you don't know if that's because the supervisor is routing well (90%) but one sub-agent is failing (40%) — or because everything is mediocre (75%). Identifying the 40% sub-agent and improving it from 40% to 70% moves the system score more than tuning a 75% component to 80%.

The weak-link principle: System accuracy is bottlenecked by the weakest sub-agent. Independent evaluation reveals the bottleneck. You improve the bottleneck, then remeasure. Repeat.

The practical workflow:

  1. Build and evaluate Sub-agent A independently. Get it to 80%+.
  2. Build and evaluate Sub-agent B independently. Get it to 80%+.
  3. Wire them under a supervisor.
  4. Evaluate the supervisor's routing accuracy separately.
  5. Evaluate the full system end-to-end.
  6. When system accuracy plateaus, go back to independent sub-agent eval to find the new bottleneck.
