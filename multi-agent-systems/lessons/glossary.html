Architecture

Terms related to the structural design of multi-agent systems, from Ask David.

Supervisor Agent L0001
The central orchestrating agent in a multi-agent system. Responsible for four jobs: understanding user intent, routing to the correct sub-agent(s), managing memory (both short-term and long-term), and composing the final answer from sub-agent outputs. The supervisor is the single entry point — no sub-agent is called directly by the user.
See also: routing, sub-agent
Sub-agent L0001
A specialized, narrowly-scoped agent that handles one type of task (e.g., SQL queries, RAG document retrieval, code generation). Sub-agents are never called directly by the user — always through the supervisor. Each sub-agent has its own evaluation dataset and metrics. Three canonical types: structured data agent, unstructured doc agent, and analytics agent.
See also: structured data agent, RAG, analytics agent
Planning Node L0001
A processing step that interprets the user's intent and decides which workflow path to follow. In Ask David: routes between the General QA Flow and the Specific Fund Flow. A pre-processing step before the supervisor handles the actual work.
See also: routing, intent classification
Reflection Node L0001
An LLM-as-judge quality gate that evaluates whether the assembled answer is good enough before delivery. The reflection node checks: does the answer address the user's question? Is it accurate? Is it appropriately detailed? If the answer fails the check, the workflow loops back to the relevant sub-agent for revision. Also called the reflection layer.
See also: LLM-as-judge, pass rate
Personalization Node L0001
A processing step that tailors the final answer to the user's role. The same underlying information is presented differently depending on who is asking: a due diligence specialist gets detailed technical answers; an advisor gets a concise summary. The supervisor determines the user's role from context or explicit profile data.
See also: role-tailored answer
General QA Flow L0001
One of two workflow subgraphs in Ask David. Handles general, non-fund-specific questions (e.g., "What are the top-performing funds this quarter?"). Kept separate from the specific fund flow so that changes to one don't risk degrading the other.
See also: subgraph, specific fund flow
Specific Fund Flow L0001
One of two workflow subgraphs in Ask David. Handles questions about specific funds (e.g., "Why was fund XYZ terminated?"). Kept separate from the general QA flow so that each can be independently tuned, evaluated, and deployed.
See also: subgraph, general QA flow
Memory L0001
State that the supervisor manages across interactions. Short-term memory: session context, the current conversation history, what sub-agents have already been called. Long-term memory: persistent facts about the user's preferences, past queries, and role. Together, they allow the system to maintain context and provide personalized responses over time.
See also: supervisor agent
Subgraph L0001, L0005
A self-contained workflow graph within a larger multi-agent system. Each subgraph handles a distinct query type and can be independently tuned, evaluated, and deployed. In Ask David: the General QA subgraph and the Specific Fund subgraph. Subgraph isolation prevents changes in one area from causing regressions in another.
See also: general QA flow, specific fund flow
RAG abbr Retrieval-Augmented Generation L0001
A pattern where the system retrieves relevant documents from a knowledge base (vector database) and includes them in the LLM's context window before generating an answer. For unstructured document agents: documents are vectorized and stored before queries arrive. RAG enables the system to answer questions about information that isn't in the model's training data.
See also: Precision@K, Recall@K, chunking strategy
ReAct abbr Reasoning + Acting L0001, L0003
A reasoning pattern where the agent alternates between thinking about the next step and taking an action (typically calling a tool). The agent sees the result of each action and uses it to decide the next action. Used in Ask David's analytics agent for multi-step data analysis tasks. Also called the ReAct loop.
See also: analytics agent, trajectory, tool-calling agent

Agents

The three canonical sub-agent types and related agent concepts.

Structured Data Agent L0001
A sub-agent that answers questions by querying structured data sources (databases, APIs). Converts the user's natural language question into a query (typically SQL or an API call), executes it, and returns the structured result. Example in Ask David: "What is the current AUM of fund XYZ?"
See also: sub-agent, query correctness
Unstructured Doc Agent L0001
A sub-agent that answers questions by retrieving relevant documents from a knowledge base using RAG. Converts the user's question into a vector search query, retrieves the top-K relevant chunks, and synthesizes an answer from the retrieved content. Example in Ask David: "What does the fund's risk disclosure say about derivatives?"
See also: RAG, Precision@K, Recall@K
Analytics Agent L0001
A sub-agent that performs multi-step data analysis using the ReAct pattern. Uses reasoning + tool-calling to execute a sequence of operations (query database, compute metrics, generate chart specs) to answer complex analytical questions. Often generates Python code that a human reviews before execution (approval gate).
See also: ReAct, trajectory, approval gate
Tool-calling Agent L0002
An agent that reasons about which tools to call and in what order to accomplish a task. The agent receives a list of available tools (with descriptions) and decides which to invoke based on the user's goal. Differs from simple prompt-based routing in that the agent can adapt its tool sequence dynamically based on intermediate results.
See also: ReAct, analytics agent
Routing L0001, L0002
The supervisor's decision about which sub-agent(s) to invoke for a given user question. Ask David routes at two levels: (1) which subgraph (general QA vs. specific fund), and (2) which sub-agent(s) within that subgraph. Routing accuracy is a key metric — a wrong route means the best sub-agent never gets called.
See also: supervisor agent, intent classification
Intent Classification L0002
The task of determining what the user is asking for — which subgraph, which sub-agent type, which workflow path. Can be done with an LLM, a classifier, or rules. The quality of intent classification directly affects routing accuracy.
See also: routing, planning node
Role-tailored Answer L0001, L0003
The same underlying information, presented differently based on the recipient's role and expertise level. A due diligence specialist gets detailed technical analysis; an advisor gets a concise summary with key takeaways. The personalization node applies role-appropriate depth, vocabulary, and formatting.
See also: personalization node

Evaluation

Terms related to measuring, comparing, and improving agent quality.

Ground Truth L0003
The correct answer or expected behavior for a given input, established by a domain expert or authoritative source. Ground truth is the reference point against which agent outputs are measured. In Ask David: accumulated from human review cycles over time, not present on day one. The evaluation flywheel generates ground truth through review, not the other way around.
See also: evaluation flywheel, LLM-as-judge
Evaluation Flywheel L0003
A self-reinforcing cycle: evaluation generates ground truth examples → ground truth improves evaluation accuracy → better evaluation reveals more weak links → weak links are fixed → system improves → fewer errors → less human review needed for the same quality. The flywheel is the mechanism that makes evaluation sustainable and makes human effort increasingly leveraged over time.
See also: ground truth, async review loop
LLM-as-judge L0003
Using an LLM to evaluate whether an agent's output is correct or good, by comparing it against ground truth or a rubric. The judge LLM scores the output (pass/fail, rating, or critique). Used at scale when human review is too expensive. Must be calibrated against human-scored examples to avoid the calibration trap.
See also: calibration trap, ground truth
Calibration Trap L0003
The risk that LLM-as-judge gives consistent scores without giving accurate scores. Consistent means the judge gives the same score for the same quality every time. Accurate means that score matches what a human would give. The trap: a judge can be perfectly consistent but systematically biased. Avoided by periodically calibrating: showing the judge examples humans have scored and adjusting the judge's prompt or threshold until its scores match human scores.
See also: LLM-as-judge
Trajectory Evaluation L0003
Evaluating whether the agent took the right sequence of steps, not just whether the final answer is correct. For ReAct agents: did the agent call the right tools in the right order? For routing: did the supervisor send the query to the right sub-agent? Trajectory evaluation catches errors that look correct in the output but came from the wrong process.
See also: trajectory, ReAct
Trajectory L0002, L0003
The sequence of steps (tool calls, sub-agent calls, reasoning outputs) an agent takes to complete a task. A trajectory is the process, not just the output. Trajectory accuracy measures whether the agent's process matches an expert-defined ideal process. Trajectory evaluation is critical for analytics agents using the ReAct pattern.
See also: trajectory evaluation, ReAct
Precision@K L0003
A retrieval metric: of the top K documents retrieved, what proportion are actually relevant to the query? Precision@K = (relevant documents in top K) / K. Used to evaluate RAG agents. High precision means the retrieval system isn't flooding the context with irrelevant documents.
See also: Recall@K, unstructured doc agent
Recall@K L0003
A retrieval metric: of all documents relevant to the query, what proportion appear in the top K results? Recall@K = (relevant documents retrieved in top K) / (total relevant documents). Used to evaluate RAG agents. High recall means the retrieval system isn't missing important documents, even if it also returns some irrelevant ones.
See also: Precision@K
Query Correctness L0003
An evaluation metric for structured data agents: does the generated SQL or API query return the correct result? For SQL: exact match of the query text, or semantic match of the query results against the expected results. For API calls: whether the response matches the expected data for the given question.
See also: structured data agent
Code Correctness L0003
An evaluation metric for analytics code-generation agents: does the generated code execute without error? Separate from result accuracy — code can be syntactically correct and still produce wrong answers. Evaluated by running the code in a sandbox and checking for errors.
See also: analytics agent, result accuracy
Result Accuracy L0003
An evaluation metric for analytics code-generation agents: does the output of the generated code match the expected computational result? Requires a pre-computed expected answer (ground truth). Evaluated by running the code and comparing its output to the expected result.
See also: code correctness, analytics agent
Pass Rate L0003
A metric for the reflection node: what proportion of answers pass the LLM-as-judge quality check? Tracked over time to detect quality drift. Also used as a proxy for system quality — if the reflection node is well-calibrated, a high pass rate means the system is generating good answers consistently.
See also: reflection node, LLM-as-judge
Routing Accuracy L0002
The proportion of user queries that the supervisor routes to the correct sub-agent(s). A key metric for evaluating the supervisor independently from the sub-agents. Poor routing accuracy means the best sub-agent is never called, regardless of how good individual sub-agents are.
See also: supervisor agent, routing
Few-shot Examples L0002
Examples included in an LLM's prompt to demonstrate the expected input/output behavior for a specific task. Unlike zero-shot prompting (no examples), few-shot prompting gives the model concrete instances to imitate. Used to improve SQL generation, routing classification, and synthesis quality. The quality and representativeness of the examples directly affects performance.
See also: structured data agent
Hallucination L0002, L0005
An LLM generating content that sounds plausible but is factually incorrect or not supported by the provided context. In RAG systems: the model answers based on knowledge from its training data rather than from the retrieved documents. The primary failure mode that the reflection node is designed to catch. More common when retrieved documents are irrelevant or insufficient.
See also: reflection node, Precision@K

Human-in-the-Loop

Terms related to keeping humans in the decision loop within automated systems.

HITL abbr Human-in-the-Loop L0004
A design approach where humans participate in AI workflows at defined points. Not binary — HITL exists on a spectrum from fully autonomous (Level 1) to human-led (Level 4). In high-stakes domains, HITL is permanent risk management, not a workaround for bad AI. The cost of reviewing errors is trivially small compared to the cost of errors going unchecked.
See also: approval gate, escalation trigger, async review loop
Approval Gate L0004
A HITL pattern where the system pauses before executing a high-stakes action and requires human approval. The human reviews the proposed action (e.g., generated code, computed trade) before it takes effect. In Ask David: between the analytics agent's code generation and its execution against live financial models. Appropriate for actions with irreversible consequences.
See also: HITL, analytics agent
Escalation Trigger L0004
A condition in the supervisor's routing logic that causes a query to be routed to a human reviewer instead of proceeding autonomously. Five canonical dimensions: confidence score (is the AI uncertain?), financial threshold (is the stakes above a dollar limit?), query category (is it in a flagged domain?), novelty (is it out-of-distribution?), and user signal (did the user request human review?).
See also: confidence score, financial threshold, HITL
Async Review Loop L0004
A HITL pattern where a sample of autonomous answers are reviewed by humans after delivery. Not every answer — a representative sample. Review feedback becomes ground truth examples for the evaluation dataset. This is the primary mechanism by which the evaluation flywheel generates labeled data. As the dataset grows, LLM-as-judge handles more cases, reducing human burden over time.
See also: evaluation flywheel, ground truth, LLM-as-judge
Confidence Score L0004
A signal used to trigger escalation. Measures how certain the AI system is about its answer. Sources: reflection node pass/fail, retrieval similarity scores (low similarity = low confidence), null query results, or explicit probability outputs from the model. A threshold is set; queries below the threshold escalate to human review.
See also: escalation trigger, reflection node
Financial Threshold L0004
A signal used to trigger escalation: if the implied dollar value of the query exceeds a set limit, human review is required. Examples: fund value > $1M, transaction size > $500K, portfolio change > 5%. The threshold is domain-specific and should be calibrated against the cost of error versus the cost of review.
See also: escalation trigger
Reviewer Fatigue L0004
The degradation of human review quality over time as the reviewer grows tired from evaluating AI-generated content for extended periods. Fatigue leads to inconsistent decisions — the exact problem HITL is meant to solve. Mitigated by: designing fast review interfaces (show question + answer + flagged reason; approve/revise/escalate in one click), randomizing case order, and tracking reviewer accuracy over time.
See also: async review loop
Triage Layer L0004
A pre-classification step in the review queue that separates incoming escalation requests by urgency and complexity before sending them to a human reviewer. High-confidence/low-stakes cases are fast-tracked; low-confidence/high-stakes cases get priority. Prevents the human reviewer from becoming a bottleneck on routine escalations.
See also: approval gate, escalation trigger

Accuracy Trajectory

Terms related to the phased improvement of AI system accuracy from 0% to 100%.

Accuracy Trajectory L0005
The empirically observed path that production AI systems follow as they improve. The trajectory has four phases with distinct accuracy ranges and requiring distinct techniques: Phase 1 (0–50%, base model baseline), Phase 2 (50–80%, engineering wins), Phase 3 (80–90%, subgraph isolation and fine-tuning), Phase 4 (90–100%, last-mile targeted fixes). Each phase requires different techniques and yields diminishing returns — phase order matters.
See also: chunking strategy, subgraph, last mile
Chunking Strategy L0005
How documents are split into retrievable chunks for RAG. Fixed-size chunking (e.g., 512 tokens) cuts across semantic boundaries. Semantic chunking (split on paragraphs or sections) preserves meaning. The optimal strategy is document-type-specific: annual reports chunked by section, emails by thread, compliance docs by regulatory reference. Chunking is the single highest-leverage engineering change for RAG agents — a pure engineering improvement, no model change required.
See also: RAG, Precision@K
Subgraph Isolation L0005
The practice of creating separate workflow subgraphs for different query types, so that each can be independently tuned, evaluated, and deployed. Prevents the most expensive type of regression: fixing one query type and accidentally degrading another. Each subgraph has its own evaluation dataset and can be improved without affecting other subgraphs.
See also: subgraph, routing accuracy
Last Mile L0005
The final phase of accuracy improvement, from 90% to 100%. Called "the hardest mile" because the remaining errors are rare edge cases — unusual fund structures, novel regulatory situations, atypical query phrasings. Requires per-error-category analysis and targeted ground truth expansion. No general technique applies; only relentless per-case analysis. HITL remains necessary even at 95%+ accuracy.
See also: accuracy trajectory, ground truth
Error Category Analysis L0005
A last-mile technique: classify each remaining error into one of four categories. (1) Retrieval failure: wrong chunks retrieved — fix chunking or embedding. (2) Synthesis failure: right chunks, wrong answer — fix synthesis prompt or add few-shot examples. (3) Routing failure: sent to wrong subgraph — improve supervisor's intent classification. (4) Personalization failure: right answer, wrong format for role — expand role-specific examples. Each category has a specific fix; trying to address all errors without categorization is inefficient.
See also: last mile
Fine-tuning L0005
Training a pre-trained model further on a domain-specific dataset to improve its performance on a specific task. In Phase 3: fine-tuning a smaller model for a specific subgraph's query distribution. More targeted and cheaper than fine-tuning a single model for everything. Only appropriate after engineering foundations (chunking, search) are optimized — a model fine-tuned on poorly chunked data learns chunking artifacts, not domain knowledge.
See also: subgraph isolation, chunking strategy
Semantic Chunking L0005
A document chunking approach that splits on semantic boundaries (paragraphs, sections, topic shifts) rather than fixed token counts. Preserves complete thoughts and arguments within chunks. For financial documents: chunks naturally align with sections (executive summary, risk disclosures, fund performance), making retrieval more precise. Preferred over fixed-size chunking for most document types.
See also: chunking strategy, RAG
Weak-link Principle L0003
The observation that system accuracy is bottlenecked by the weakest sub-agent, not the average. Independent sub-agent evaluation reveals which component is dragging down the overall score. Improving a 40% sub-agent to 70% moves the system score more than improving a 90% sub-agent to 95%. Identified bottlenecks are fixed in order; after fixing one, re-evaluate to find the new bottleneck.
See also: routing accuracy, evaluation flywheel
Hybrid Search L0005
A retrieval approach that combines dense vector search (semantic similarity) with sparse keyword search (BM25 or TF-IDF). Captures both semantic meaning and exact keyword matches. Often combined with a re-ranker that uses a cross-encoder to reorder the initial retrieval results by relevance to the specific query. A Phase 2 engineering improvement for RAG agents.
See also: Precision@K, Recall@K
Re-ranker L0005
A cross-encoder model that takes a query and a document and outputs a relevance score. Used after an initial retrieval pass to reorder the results. More accurate than vector similarity alone because it considers the full query-document interaction, not just embedding distance. Typically added as a post-processing step in Phase 2 retrieval improvements.
See also: hybrid search, Precision@K
Ground Truth Expansion L0005
The process of growing the evaluation dataset by adding new labeled examples, particularly edge cases and rare events. Required in the last mile — the remaining errors are rare events that aren't covered by the initial dataset. Expensive (requires domain expert labeling) but necessary for continued improvement above 90% accuracy. Each new labeled example is a potential fix for one error category.
See also: ground truth, last mile
