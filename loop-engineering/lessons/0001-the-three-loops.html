Lesson 0001
The Three Loops

The product loop, not the agent loop, is the unit of work in 0-to-1.

The pitch in one line

There is one loop that types the code, one loop that decides what the code is, and one loop that decides whether the code is worth shipping. They run at very different speeds, and confusing them is the most common mistake solo devs make with agents.

Andrew Ng published a short letter in The Batch, Issue 359 (Jun 26, 2026) — Loop Engineering: Three Key Loops for Building Great Software. It credits two viral quotes — from Boris Cherny (head of Claude Code) and Peter Steinberger (creator of OpenClaw) — but Andrew's contribution is a cleaner one: three loops, three time-scales, three different actors.

"These loops guide not just how I build software, but also how I decide what software to build."

— Andrew Ng, The Batch Issue 359

Why this sits above your other frameworks

You already have two adjacent courses in this workspace:

  • 4-Loop Ladder (LangChain) — Agent → Verification → Event-driven → Hill-climbing. Focus: agent infrastructure.
  • Agentic Engineering (Matt Pocock) — Procedure skills, queue workflow, AFK agents. Focus: dev harness.

Andrew's three loops are above both. They describe the product cadence, not the agent infrastructure. Loop 1 contains the 4-Loop Ladder — every time the agentic loop fires, the LangChain primitives (agent, rubric, trigger, hill-climb) are running inside it. Matt Pocock's decisions (which skills to install, what goes in CLAUDE.md, how to queue work) determine how fast Loop 1 runs.

The relationship: Andrew's framework is the clock. The 4-Loop Ladder is the engine. Matt Pocock's harness is the transmission. If you tune the engine and transmission without knowing the clock speed, you'll ship fast in the wrong direction.

The "context advantage" framing

Andrew rejects the word "taste" for the human contribution. He prefers "context advantage" — the gap between what the human knows and what the AI knows. The framing matters because it gives a path to fix the gap, not just admire it.

"So long as the human knows something the AI does not, human-in-the-loop is needed to inject that knowledge into the system."

— Andrew Ng, The Batch Issue 359

That single sentence is the course's thesis. Loop 1 can run autonomously because the spec encodes what the AI knows. Loop 2 cannot be fully automated because the spec doesn't encode your context advantage yet. Loop 3 cannot be automated at all because the users know things nobody in the room knows.

Where you probably are right now

Loop 1: You use Claude Code or Cursor. It runs, returns code, you paste it somewhere.

Loop 2: You spend time deciding what to build next. You re-write the spec more often than you re-paste code.

Loop 3: You haven't shipped to anyone yet. You're still arguing with yourself about feature priority.

The first step is not "make Loop 1 faster." It's "figure out where in the 3-loop cadence you're actually bottlenecked." Most solo devs over-invest in Loop 1 and under-invest in Loop 2 + 3.

What this course teaches

The next three lessons each cover one loop in depth, with a mapping onto Matt Pocock skills you already use. By the end, you'll know:

  • How to write a spec that Loop 1 can run against for an hour without you.
  • When to add evals, when not to bother.
  • Which Matt Pocock skill corresponds to which part of Loop 2.
  • How to set up Loop 3 cheaply, even with 0 users.
