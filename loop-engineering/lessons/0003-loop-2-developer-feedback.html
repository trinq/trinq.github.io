Lesson 0003
Loop 2: The Developer Feedback Loop

You are no longer the QA. You are the spec writer, the vision holder, and the editor. The hard part is not catching bugs anymore — it's translating what you know into a form the agent can act on.

From QA to spec writer

"Last year, a lot of developers (including me) were acting as the QA (quality assurance) function for our coding agents, manually finding bugs and then asking the agent to fix them. But with coding agents much more able to test their own code, the amount of time we need to spend on this function has decreased significantly."

— Andrew Ng, The Batch Issue 359

This is the single biggest shift Andrew describes. In 2024 you spent your Loop 2 time hunting bugs. In 2026 you spend it deciding what to build next. The agent has eaten the QA role; you have been promoted.

The promotion ladder:

2024 — you were a bug-finder (QA). Agent wrote sloppy code, you patched.

2025 — you were a code reviewer. Agent shipped better drafts, you polished.

2026 — you are a spec writer. Agent runs against the spec, you rewrite the spec.

What this loop actually does

Three concrete activities make up Loop 2:

1Examine the current build

Open the app. Click through the feature the agent just shipped. Look at the UI. Run the flow. Time-box this — 10 to 30 minutes max.

2Decide what to change

"Visual design is wrong" is not a decision. "The home button should be in the bottom-right with a 16pt label" is. Decisions need to be specific enough to encode in a spec.

3Re-author the spec, hand it back

Edit the PRD. Add new acceptance criteria. Remove obsolete ones. Hand the updated spec back to Loop 1.

Andrew's typing app example makes this concrete: "I changed my mind a few times about the visual design, what cat costumes she can unlock as she learns (she loves cats), and the user flow for a grown-up to log in and steer the child's learning experience." Three rounds of "look at app → decide → re-spec → re-run." Each round was tens of minutes.

The cadence is tens of minutes to hours

Andrew is explicit:

"The developer-feedback loop operates over time intervals between tens of minutes and hours — that's how frequently a developer might review a product and give feedback."

— Andrew Ng, The Batch Issue 359

Why not faster? Because your context advantage needs time to crystallise. You can't see that "the parent login flow is confusing" until you've actually tried it. Why not slower? Because the agent's output goes stale — by tomorrow, the spec has drifted from the build.

Why this loop can't be fully automated

Andrew rejects "taste" and offers "context advantage" as the better framing:

"For pretty much all the products I'm involved in, I see humans as having a significant context advantage over current AI systems — we know a lot more than the AI system about the users and the context the product has to operate in — and thus humans play a critical role."

— Andrew Ng, The Batch Issue 359

The framing matters because it gives you a path to fix the gap:

Framing What it implies
"Taste" Some things just feel right. You can't teach it. The AI either has it or doesn't.
"Context advantage" The human knows things the AI doesn't. Encode that knowledge into the spec, the eval, or the harness — and the gap shrinks.

The action loop for "context advantage": Every time you give the agent feedback that comes from context the AI didn't have, also update the spec so the AI has that context next time. Your job is to make your context advantage smaller, loop after loop, by encoding it.

Mapping to Matt Pocock skills

Loop 2 is where most of Matt Pocock's high-leverage skills live:

Andrew Ng's Loop 2 activity Matt Pocock skill What it does
Decide what to build next (vision) grill-me The agent plays senior PM and interrogates you on the design before any code is written. Forces you to clarify what you actually want.
Translate vision into a spec to-prd Turn a vague feature idea into a behaviourally-scoped PRD with acceptance criteria.
Re-author the spec after seeing a build to-issues Break the updated PRD into ordered, agent-runnable issues.
Long-term codebase health improve-codebase-architecture When the code is working but the structure is rotting, this skill has the agent refactor before more features land.
QA-style verification (legacy role) tdd You may still use this for tricky bugs, but it's mostly inside Loop 1 now.

The key insight: Matt's grill-me skill exists because Loop 2 is hard. It is a procedure for forcing yourself to extract your context advantage into a form the agent can use. Andrew's "context advantage" framing is the why; Matt's grill-me is the how.

For Matt's full harness setup, see Agentic Engineering course — specifically the grill-me and to-prd lessons.

When the loop is broken

Three failure modes for Loop 2:

A

The "decide what to change" step takes days

You're not bottlenecked on the agent — you're bottlenecked on your own clarity. Fix: run grill-me until you can state the change in one sentence.

B

The spec keeps growing without shipping

You keep adding features to the spec without re-running Loop 1. The spec and the build drift apart. Fix: cap each spec at one Loop 1 iteration (~30 minutes of agent work).

C

You keep pasting context into chat, not the spec

The next Loop 1 run will be just as confused as this one. Fix: every chat-injected context should become a spec line, an eval, or a CLAUDE.md entry.

Action Items

  1. Run grill-me on one feature decision20 min Pick one feature you're stuck on. Use Matt Pocock's grill-me skill. Stop when you can state the decision in one sentence.
  2. Re-author one spec after a recent build30 min Pick a recent Loop 1 build. What context did you paste into chat? Move each piece of context into the spec, an eval, or CLAUDE.md.
  3. Time-box one Loop 2 round45 min total Examine current build (≤15 min), decide change (≤15 min), update spec (≤15 min). If you can't finish in 45 min, the feature is too big — split it.
