Lesson 0002
Loop 1: The Agentic Coding Loop

Given a spec and optional evals, the agent writes, tests, iterates. Your job is to keep this loop running for as long as possible without you.

What the loop does

"Given a product specification and optionally a set of evals (that is, a dataset against which to measure performance), we can have an AI agent write code, test its work, and keep iterating until the code is bug-free and meets its specification."

— Andrew Ng, The Batch Issue 359

Three things go in:

  1. A product specification — what the software should do.
  2. An optional set of evals — a dataset the agent can score itself against.
  3. Tools the agent can call — file system, browser, tests, shell.

One thing comes out: a build that meets the spec. Andrew notes his daughter’s typing app had the agent working autonomously for about an hour, checking its own work via a browser multiple times before reporting back.

The cadence is minutes, not hours

"The engineering loop executes quickly. Every few minutes, the coding agent might build and test a new version of the software."

— Andrew Ng, The Batch Issue 359

This is the single most important fact about Loop 1: it is fast. If you find yourself babysitting a single agent run for 20 minutes, you are out of Loop 1 and into Loop 2.

Symptom What loop you're actually in
Agent runs, fails a test, fixes it, retries, passes — within ~5 minutes. Loop 1 — running as designed.
You sit watching the agent, occasionally pasting in extra context when it gets stuck. Loop 2 — you've become the spec.
Agent finishes, you spend 30 minutes deciding whether the result is what you wanted. Loop 2 — this is the developer feedback loop.
Agent finishes, you ship to a friend, wait a day for their reaction. Loop 3 — the external feedback loop.

What makes a good spec

Andrew is explicit that the hardest part of Loop 1 is upstream:

"When a developer has a clear vision for what to build, it is still a lot of work to translate that vision into a specification for a coding agent to implement."

— Andrew Ng, The Batch Issue 359

Three properties of a spec that lets Loop 1 run unattended:

1Behavioural, not architectural

Specify what the user does, not how the code is organised. "The parent enters a PIN, then sees the child's progress" beats "We need a ParentAuthController with a PIN field."

2Testable

Every requirement must be something you can write a check for. "Looks beautiful" is not a requirement; "the home screen fits within one viewport on a 6.1" phone" is.

3Scoped to one loop iteration

If the spec takes more than ~30 minutes of agent work to complete, it's two specs. Split it. Loop 1 should produce a running, testable build, not a half-feature.

When to add evals

"If you find that the system repeatedly runs into certain problems, building a set of evals for the agent becomes useful."

— Andrew Ng, The Batch Issue 359

This is a sharp, non-obvious line. Don't write evals preemptively. Wait until you see a failure mode repeat three or more times across Loop 1 runs, then encode that failure as an eval.

The trap: writing evals for things that already work. You will spend two days authoring evals for a feature that has never failed, and zero time on the feature that fails every third run. Evals encode failures, not aspirations.

A minimal eval has three parts:

  • Input — a concrete prompt / scenario (e.g., a JSON file, a screenshot, a CLI invocation).
  • Expected behaviour — what the agent should produce (a passing test, a structured output, a UI state).
  • Verifier — code that checks the output. Either deterministic (a test, a JSON schema check, a curl) or fuzzy (an LLM-as-judge for tone or UX).

Mapping to Matt Pocock skills

Loop 1 is where the agent infrastructure lives. Most of your Matt Pocock setup runs inside this loop:

Andrew Ng's Loop 1 Matt Pocock skill What it does in this loop
Spec authoring to-prd Turn a vague feature idea into a behaviourally-scoped PRD the agent can run against.
Agent verification tdd Write the failing test first, let the agent make it pass, then refactor. Classic red-green-refactor — for agents.
Evals to-issues (re-purposed) When the agent fails the same eval three times, the failure becomes a tracked issue that gets fed back into the spec.

The insight: Andrew's "spec + optional evals" is exactly what to-prd + tdd produces. Matt's framework already implements Loop 1's inputs; you just hadn't framed it that way.

For the agent infrastructure itself, see 4-Loop Ladder — Loop 1: The Agent Loop in the sibling course.

The "active area of invention"

Andrew closes the agentic-coding section with this:

"I hear frequently from developers who are finding new ways to engineer more effective engineering loops. This is an active area of invention!"

— Andrew Ng, The Batch Issue 359

This is a tell. The frameworks that will matter most for Loop 1 in 2026–2027 don't exist yet. Boris Cherny's dozens of /loop jobs — babysitting PRs, auto-rebasing CI, clustering Twitter feedback — are all Loop 1 automation. The on-ramp is one slash command. The hard part is everything you do before you type it.

Action Items

  1. Pick one feature for a current project10 min Write a behavioural spec: what does the user do, what does the system do, how do you verify it. Aim for ≤30 minutes of agent work.
  2. Run Loop 1 once30 min Hand the spec to Claude Code with permission to run tests, edit files, and use the browser. Time how long it runs without you.
  3. Track one failure modeongoing Keep a list. When the same failure shows up three times across runs, turn it into an eval — not before.
