The Impl Plan Phase — Decomposable, Not Perfect

A plan is not a blueprint. It's a series of bets on where complexity will appear. The decomposable contract makes each step independently auditable. This lesson covers step granularity, the checkpoint model, and why over-planning is as dangerous as under-planning.

Source: Matt Pocock — "The 7 phases of AI-driven development", AI Hero / YouTube, 8:26, 2026-03-03. Builds on lessons 0001–0006.

How to use this lesson. The Impl Plan is the phase most teams skip when they're excited about building. Read the "decomposable contract" section first — it's the core idea. Then read the step-granularity decision tree, because getting the step size right is the hardest part of writing an impl plan. The 7 failure modes at the end are what your team will hit if the plan isn't decomposable.

What Pocock Actually Said

The key phrase: "you get to iterate at the step level." That's what a decomposable plan enables. When the agent hits a wall in step 3, you don't have to re-plan the whole feature — you replan step 3. The rest of the plan holds. Without decomposition, a failure in step 3 means redoing the whole plan.

The Core Idea: Decomposable Contract

An impl plan is a decomposable contract: each step is independently auditable. The manager can check step 1 without reading step 2. The agent can execute step 1 without knowing step 3. When a step fails, only that step needs replanning.

Contrast with a monolithic plan: all steps are coupled. Changing step 1 ripples through step 7. The agent can't start until the entire plan is complete. A failure in step 3 means re-reading the whole plan. Most teams write monolithic plans.

The decomposability test

Read any step in the plan. Ask: "If this step changes, does it break any other step?"

  • If yes — the steps are coupled; the plan is monolithic.
  • If no — the steps are independent; the plan is decomposable.

Real-world analogy: IKEA furniture instructions. Each step is a card with a picture. You can do step 3 before step 2 if you want. Each step has a checkpoint: "did the panel click in?" If it didn't, you stop and fix step 3. You don't undo all the screws you've put in. An impl plan should work the same way.

Step Granularity

The hardest part of writing an impl plan is getting the step size right. Too big: the step has too much uncertainty, and the agent gets stuck. Too small: the plan is a todo list, and the agent spends more time tracking the plan than executing it.

Default to M-size steps. 200–800 lines, one component, multiple files. This is the sweet spot: the agent has enough room to make meaningful progress, and the manager has enough checkpoints to audit. XS and S are for maintenance tasks; L and XL should be decomposed.

The XL test: when to split

When a step has more than one answer to the question "what could go wrong here?", it's XL. Split it. Example: "Build the export API endpoint" — what's the answer to what could go wrong? "The database query is slow" and "The file size is too large for the response buffer." Two distinct failure modes. Split into two M-steps.

Anatomy of an Impl Plan

The artifact is plan-[feature].md in the feature folder. It lives alongside the PRD:

plan-[feature].md — minimum viable template

  • Feature: Name. Link to prd-[feature].md.
  • Dependencies: Which steps must complete before this step can start? (Most should be "none" — if a step has more than 2 dependencies, it may be XL.)
  • Step 1: [name]
    • PRD link: which requirement does this step satisfy?
    • Files touched: new + existing
    • Checkpoints: how does the manager verify this step is done?
    • If blocked: what's the unblock path?
  • Step 2–N: Same structure.
  • Step N+1: QA integration — link to the QA plan from phase 07.
  • Unknowns: What we're discovering as we go. This section grows.

Full step block example

Step 3 — M-size
Build CSV export background job
PRD: §4 Edge Cases — "Projects > 100K rows must use background job"
  • Files touched: src/jobs/export-job.ts, src/db/queries.ts (add export_status enum)
  • What to do:
    • Add job_status enum: pending | processing | done | failed
    • Create ExportJob class with enqueue() and process()
    • process() queries in batches of 1K rows, writes to temp file
    • On completion, writes file to /exports/[jobId].csv, updates DB to done
    • On failure, updates DB to failed, emits error event
Checkpoint
Run with a 5K-row project. Job enters processing, then done. File exists at /exports/[jobId].csv. Row count matches DB.

The checkpoint is not optional. Every step needs a checkpoint — a specific, observable outcome the manager can verify. Without a checkpoint, the step is complete when the agent says it's complete. With a checkpoint, the step is complete when the observable matches the description. The checkpoint is the decomposable part of the plan.

Step Dependencies: The Coupling Map

Most impl plans have too many dependencies. Each dependency is a coupling point — if the upstream step changes, the downstream step breaks. The rule:

Coupling

Keep, but write interface in PRD

A well-decomposed plan has most steps with zero dependencies. The only real dependencies are data dependencies: step B reads what step A wrote.

Impl Plan and the Phases Around It

Impl Plan ← PRD

The impl plan is a decomposition of the PRD. Each step in the plan maps to one or more requirements in the PRD. If a step can't map to a PRD requirement, it's either out of scope or missing from the PRD. The impl plan is what the PRD looks like when it's been decomposed for execution.

Impl Plan → Execution

The impl plan is what the agent reads during execution. Each step is a ticket. The agent executes step 1, verifies the checkpoint, then moves to step 2. The plan is the queue. The agent manages the queue; the manager audits the checkpoints.

Impl Plan → QA

Each step checkpoint becomes a QA test. Step 3's checkpoint ("Job enters processing, then done. File exists at /exports/[jobId].csv. Row count matches DB.") becomes three QA assertions. The impl plan is the QA test plan's ancestor.

Three Team Scenarios

Scenario A — The monolithic plan

The team writes a plan for a "CSV Export" feature:

  1. Set up the export API endpoint
  2. Query the database for project rows
  3. Format rows as CSV
  4. Return the CSV file
  5. Handle large projects with a background job

Steps 1–4 assume synchronous export. Step 5 says "handle large projects" but doesn't change steps 1–4. The agent builds steps 1–4 synchronously. When it gets to step 5, it realizes the endpoint from step 1 needs to be changed. Steps 1–4 need rewrites. The plan was monolithic.

What went wrong: Step 5 was a "handle large projects" wrapper around steps that assumed synchronous. The dependencies were sequential assumptions, not data dependencies. When the plan revealed that step 5 needed async, the whole plan broke.

Diagnostic: Sequential assumption dependency between steps 1–4 and step 5. Plan was not decomposable.

Scenario B — The over-engineered plan

Engineer U writes a 47-step impl plan for a 5-page feature. Each step is 20 lines. The plan includes: "Step 12: Create the type for the export job status enum." "Step 13: Create the database schema migration for export_jobs table." "Step 14: Add the database index for export_jobs.job_status."

The agent spends the first hour reading the plan before starting. The plan has more steps than the feature has files. When the team realizes the export needs a different job queue, 30 of the 47 steps need editing.

What went wrong: The plan was S-size for what should have been M-size steps. The plan was a todo list, not a decomposable contract. The agent couldn't see the forest for the trees.

Diagnostic: Step size too small. Steps are coupled by sequential assumption rather than by data dependency.

Scenario C — The plan that held

Engineer V writes a 7-step plan for the same CSV export feature. Following the template:

  • Step 1: Schema migration — add export_jobs table. Checkpoint: migration runs, table exists with correct columns.
  • Step 2: Job queue — create ExportJob class with enqueue. Checkpoint: calling enqueue creates a row in export_jobs with status pending.
  • Step 3: Sync export for < 10K rows — build POST /api/export/:id that calls the job synchronously. Checkpoint: curl the endpoint, receive CSV in 30 seconds for 5K rows.
  • Step 4: Batch query — paginate project rows in batches of 1K. Checkpoint: 10K rows exported with correct row count.
  • Step 5: Async worker for > 10K rows — move step 4 to background job. Checkpoint: 50K-row project returns job ID immediately; job reaches done in < 2 min; file exists.
  • Step 6: Email notification — on job done, email user with download link. Checkpoint: email received within 5 sec of job done.
  • Step 7: QA integration — write assertions for all checkpoints above.

Step 4 hits a snag: the pagination query is slow for projects with soft-deleted rows. Engineer V replans step 4 only. Steps 1–3 and 5–7 hold. The plan is decomposable.

Why it works: Each step has a specific checkpoint. Steps are independent by data dependency only. When step 4 failed, the rest of the plan held. Replanning was a one-step operation.

7 Failure Modes

SymptomWhat you actually missed
Changing step 3 breaks step 7 Steps are coupled by sequential assumption; plan is monolithic
Agent asks "what order should I do these in?" Sequential assumptions encoded as dependencies; remove them
Plan has 30+ steps for a 5-page feature Step size too small; plan is a todo list, not a contract
Agent gets stuck mid-step with no unblock path Step has no "if blocked" section; step is too big (XL)
Manager can't tell if a step is done Step has no checkpoint; manager relies on agent's word
Plan is out of date after step 2 Steps aren't decomposable; one failure broke the whole plan
Plan is written by the manager, not the agent Manager planning for the agent; impl plan should be co-authored

The co-authorship insight. The impl plan is the one phase where the agent should be involved in planning. The manager decomposes the PRD into steps; the agent reviews the decomposition and flags XL steps, suggests reordering, or identifies missing steps. The manager writes the plan; the agent approves it. Both sign off. This prevents the most common failure: a manager's plan that an agent can't execute.

The Manager Ritual for Impl Plan

Impl Plan ritual — 3 checks before execution

  1. "Is each step decomposable?" — read each step. Ask: "if this changes, does it break any other step?" If yes, the steps are coupled. Remove the coupling.
  2. "Does each step have a checkpoint?" — no checkpoint means no audit trail. The manager can't verify completion without one.
  3. "Did the agent review the plan?" — the agent who executes the plan should approve it. Manager plans; agent reviews. Co-authorship prevents re-planning mid-execution.

The manager's hardest moment after an impl plan. The moment the agent says "step 3 is harder than I thought." The manager's instinct is to re-plan everything. The manager's job is to ask: "is step 3 still decomposable? Can we split step 3 into 3a and 3b?" If the answer is yes, the plan holds — only step 3 needs replanning. If the answer is no, the plan is monolithic. That's the real failure.

Final note on this lesson. The impl plan is a decomposable contract between manager and agent. Each step is independently auditable. Each checkpoint is a verifiable outcome. When a step fails, only that step needs replanning — the rest of the plan holds. The test of a good impl plan is not whether it predicted the future perfectly, but whether it held when reality turned out differently than predicted. A plan that survives reality is better than a plan that predicted everything. The first is possible; the second is not.
