Agent Protocols — The Contract Between You and Your AI

The 7-phase system describes what to build. This lesson describes how to talk to the AI while building it — the instruction protocols, trust calibration levels, context windows, termination conditions, and the manager's real job: not writing code, but deciding what "done" means for each step.

Source: Matt Pocock — "The 7 phases of AI-driven development," AI Hero / YouTube, 2026. Combines execution phase content (lesson 0008) with the implicit communication protocols that Pocock demonstrates throughout the video series.

How to use this lesson. Read "The Manager's Real Job" first — it reframes the manager's role from code-writer to quality-definer. Then read the three trust calibration tiers. The instruction protocols (IP-01 through IP-08) are the most immediately actionable: they're five-sentence prompt templates for the eight most common situations in AI-driven execution. The context window management section is the most commonly ignored — and the most expensive.

The Manager's Real Job in AI-Driven Execution

In a traditional team, a manager reviews code, approves PRs, and signs off on deployments. In an AI-driven team, the AI writes the code. The manager's job hasn't disappeared — it's shifted upstream.

The manager's job in AI-driven execution is three things, in this order:

  1. Define "done" for each step — before the agent starts. "Done" means the acceptance criteria, not the implementation approach. The agent figures out the approach; the manager defines what success looks like.
  2. Calibrate trust based on evidence — start skeptical, elevate based on demonstrated reliability. Trust is not a personality trait; it's a performance metric.
  3. Terminate at the right moment — know when the agent has done enough and human review is necessary. The termination condition is a business decision, not a technical one.

The most common mistake managers make with AI agents: they give vague instructions ("make this work") and then spend more time correcting the agent than they would have spent writing the code themselves. The problem is never the agent. The problem is that "make this work" doesn't define done. The agent will interpret "make this work" in the way that maximizes its own sense of progress — which may not match the manager's intent.

Trust Calibration: Three Tiers

Trust in an AI agent is not binary. It's a calibration based on demonstrated performance in your codebase, with your stack, on your domain. The three tiers define the manager's supervision level, not the agent's capability.

Tier 1 — Supervised

Agent shows all output before execution.

Manager reviews every file change.

No autonomous commits.

Use for: unfamiliar stack, first time on a pattern, any step touching shared systems.

Tier 2 — AFK-able

Agent runs for 20–30 min without review.

Manager checks results at natural breakpoints.

Agent may commit on explicit instruction.

Use for: familiar stack, trusted patterns, well-scoped steps.

Tier 3 — Delegated

Agent owns a full M-sized step end-to-end.

Manager reviews only at the checkpoint.

Agent commits, runs tests, flags issues.

Use for: highly reliable agent, well-defined feature, established trust.

The trust tier is per-step, not per-session. An agent can be Tier 3 on a straightforward refactor and Tier 1 on a new payment integration in the same session. Moving from Tier 1 to Tier 2 requires demonstrated evidence. Moving from Tier 2 to Tier 3 requires a pattern of clean Tier 2 results.

Tier 1 — Supervised Tier 2 — AFK-able Tier 3 — Delegated

Trust elevation requires evidence accumulation. Every Tier 1 clean result = 1 step toward Tier 2. Every 5 Tier 2 clean checkpoints = 1 Tier 3 step. The agent earns trust, it is not given.

The 8 Instruction Protocols

Each instruction protocol (IP) is a five-sentence template for a specific execution situation. The templates are not scripts — they're structure. Fill in the blanks; the agent fills in the rest.

IP-01: The Step Brief

Start of any M-sized step

Use when: Starting a new M-sized step from the impl plan. The most common instruction. Sets the full context for the step.

We are in the [feature name] feature, step [N] of [total]. The goal of this step is [one sentence — the outcome, not the approach]. Success means: [criterion 1], [criterion 2], [criterion 3]. Do not proceed to the next step until all three criteria are met. After each sub-task, show me the diff before running any git commands. This is Tier [1/2] supervision.

Why it works: The five sentences cover: context, goal, success criteria, boundary (don't skip ahead), and supervision level. An agent that receives this instruction knows when to stop and report. An agent that receives "implement step 3" doesn't.

Common mistake: Replacing "The goal is to implement authentication" with "Add auth." The difference is precision. "Add auth" could mean 3 lines or 300.

IP-02: The Boundary Call

Out-of-scope item surfaces during execution

Use when: The agent encounters something that seems out of scope — a missing dependency, a design question, a new requirement. The agent will often either fix it silently (scope creep) or stop and ask (context switch). Neither is optimal.

I see that [observation — the thing that came up]. This is out of scope for this step. For now, do [the minimal safe fix — e.g., "stub it out", "log a TODO with a specific description", "use X as a temporary placeholder"]. Do not design the full solution. We will address this in [the phase it belongs to — PRD / impl plan revision / a new step].

Why it works: It answers the agent's implicit question ("should I fix this?") without derailing the current step. It documents the decision without making the agent guess.

Signal in retrospective: HF-06 (Scope Creep in Transit) from lesson 0010. Every Boundary Call is a documented decision that prevents a future loopback.

IP-03: The Deviation Report

High frequency

Use when: The agent reports a deviation — it tried to do something, it failed, it tried a different approach. This is the most common reason execution diverges from the impl plan.

Noted. Before you try the alternative approach: what was the specific failure? (Paste the error or describe the behavior.) What assumption in the impl plan was wrong? ([e.g., "I assumed the API was REST but it's gRPC."]) Based on that, is the fix a local adjustment to this step or does it require a revision to the impl plan? ([local / plan revision])

Why it works: It forces the agent to name the root cause before continuing. Agents often skip from "failed approach" to "new approach" without diagnosing why the first one failed. Naming the wrong assumption prevents the same failure from recurring in the next step.

Deviation types from lesson 0008: Type 1 (tool misfire) → local fix. Type 2 (context limit) → pause and clean context. Type 3 (wrong approach) → impl plan revision.

IP-04: The Termination Check

High frequency

Use when: The agent reports "step complete" or runs for longer than expected. The termination check prevents two failures: stopping too early (missing criteria) and running too long (wasted time).

Before we close this step: which of the three success criteria are you most uncertain about? ([criterion]) Walk me through how you verified it. ([answer]) If you cannot verify it to your own standard, treat it as open and tell me what you need to close it. Do not mark a criterion as verified if you are guessing.

Why it works: Agents tend to declare completion optimistically. The question "which criterion are you most uncertain about" surfaces the weakest point without making the agent admit failure. The agent self-triages.

Manager rule: If the agent cannot verify a criterion without guessing, it is not verified. An unverified step is not a closed step.

IP-05: The Context Window Alert

Context window > 70% full

Use when: The agent's context is filling up — visible in the token counter or in degraded output quality (repetition, forgetting earlier instructions, losing the thread). This is a Type 2 deviation (lesson 0008).

Your context is filling up. Before we continue: write a step summary as a comment at the top of [primary file]. Include: what is done, what is in progress, what is open. Then restart the context by running [specific command — e.g., "create a new session with the same instructions"]. I will paste the step summary as the first message in the new session.

Why it works: The step summary is the handoff document. Without it, the new session starts cold and repeats work. With it, the new session picks up where the last one left off with full context.

Proactive signal: If you see the agent re-reading the same files repeatedly without making progress, the context is degrading before it reports a problem.

IP-06: The Refactor Gate

Refactoring surfaces during a feature build

Use when: During a feature build, the agent encounters messy code and proposes "let me clean this up first." Refactors in the middle of a feature build are HF-06 in disguise — they are out-of-scope work that feels productive.

Noted that [file] needs refactoring. Do not refactor it now. This is a separate step. For this feature step, work around the messy code if possible. If it is not possible to work around it without breaking the feature, stop and report: [description of the specific problem]. I will add refactoring as a separate step to the impl plan after this feature is complete.

Why it works: Refactoring mid-feature is a scope change disguised as a quality improvement. The manager's job is to protect the feature scope, not to let the agent expand it — even for good reasons.

When to make an exception: If the messy code makes the feature literally unbuildable (not just inconvenient), the refactor is a prerequisite to the feature, not a separate concern. But document it explicitly: "Refactoring X as a prerequisite to feature step N."

IP-07: The Review Request

Mid-step human review needed

Use when: The agent reaches a decision point — an architectural choice, a naming convention, a design pattern — and needs a human judgment call. This is not a deviation; it's the agent correctly escalating.

I need a judgment call on [specific decision]. Option A: [description — pros/cons]. Option B: [description — pros/cons]. My recommendation is [A/B] because [one sentence]. If you disagree or want a third option, tell me before I continue. If neither option is acceptable, describe what "acceptable" means to you.

Why it works: It frames the escalation as a decision with options and a recommendation, not a blank question. The recommendation shows the agent has thought it through. The "neither is acceptable" clause gives the manager a way to redefine the problem without overriding the agent's analysis.

IP-08: The Session Close

End of a working session (manual or automatic)

Use when: The working session is ending — end of day, context is full, feature is done, or manager is stepping away. The session close ensures the next session can pick up without repeating work.

End of session. Summary: [what was completed in this session — be specific]. Next action: [the exact next step and why it comes next]. Open issues: [list with one sentence each — what is known but not resolved]. Checkpoint: [which success criterion from the impl plan we are at — e.g., "step 3 of 7, checkpoint 2 reached"]. Do not continue until the next session begins.

Why it works: The next session's first message should be this summary, pasted as context. The agent should never start a new session without knowing exactly where the last one stopped.

Context Window Management

The context window is the agent's working memory. When it's full, the agent forgets. The cost of a full context window depends on what was forgotten:

  • Best case: The agent forgets the impl plan. The manager re-explains. 5 minutes lost.
  • Worst case: The agent forgets which feature it's building and starts building the wrong one. Hours of rework.

Context management is the manager's responsibility, not the agent's. The manager sets the context at the start of each session and monitors it during execution.

The Context Hierarchy

Context has a priority order. The manager should maintain context at each level:

  1. Level 1 — Mission context (never drop): Feature name, phase, current step, success criteria. This fits in the first message of every session.
  2. Level 2 — Plan context (monitor): Impl plan, open steps, checkpoints. Dropped when context is 60% full.
  3. Level 3 — File context (rotate): Which files are currently being edited, which are recently read. Rotated based on what the agent is actively touching.
  4. Level 4 — Scratch context (discard freely): Agent's reasoning, failed attempts, exploration notes. Purge aggressively.

The 60% rule: When the context window reaches 60% full (not 70%, not 80%), start the IP-05 context window protocol. At 60%, you have room to write the step summary and restart cleanly. At 80%, you're making emergency decisions. At 90%, the agent is already degrading.

Session Continuity Protocol

When restarting a session (new terminal, context full, or resuming after a break), the continuity protocol prevents the agent from repeating work:

  1. New session first message: paste the IP-08 session close summary from the previous session.
  2. Agent reads it, confirms the checkpoint, and asks: "Ready to continue from [checkpoint]. Any changes to the plan since last session?"
  3. Manager confirms or adjusts the plan.
  4. Execution resumes from the confirmed checkpoint.

What happens without continuity protocol

Manager ends session. Agent writes no summary. Next session starts with "continue." Agent doesn't know the checkpoint. Agent re-reads the codebase, re-starts from step 1, and spends 20 minutes reproducing work that was already done. Manager either catches it (wastes 5 minutes) or doesn't (agent overwrites clean code with half-finished code).

The fix is always the IP-08 session close. It takes 2 minutes. The cost of skipping it is 20 minutes minimum.

When to Stop the Agent

The termination condition is the manager's most important decision. The agent will not stop itself unless told to. The signals for when to stop fall into three categories:

SignalCategoryAction
Context window approaching limit Caution Stop at next natural boundary (end of sub-task). Run IP-05 + IP-08.
Agent asks the same question twice Caution Stop. Run IP-03. The impl plan step is ambiguous — not the agent's memory.
Agent proposes a design change Caution Stop. Run IP-06 (Refactor Gate). Design changes are impl plan revisions.
Agent's output is repetitive or looping Stop Stop immediately. Context is degrading. Run IP-05 + IP-08 and restart.
Agent introduces code without showing diff first Stop Revert if possible. This violates the trust tier contract.
Agent bypasses a success criterion Stop Revert the step. Re-run with IP-01 including the missed criterion explicitly.
Agent makes a Type 3 deviation (wrong approach) Stop Stop execution on this step. Loop back to impl plan before continuing.

Example Exchange: IP-01 → IP-03 → IP-04

The following is a realistic execution exchange showing three protocols in sequence:

IP-01: Manager starts Step 4 of the impl plan
MANAGERWe are in the Payment Processing feature, step 4 of 7. The goal of this step is to add idempotency keys to the charge endpoint. Success means: (1) idempotency key is accepted as a request header, (2) duplicate charges return the cached response within the 24h window, (3) missing idempotency key returns HTTP 422. Do not proceed to step 5 until all three criteria are met. After each sub-task, show me the diff before running any git commands. This is Tier 1 supervision.
AGENTUnderstood. Step 4: idempotency keys on the charge endpoint. Three criteria: header accepted, duplicate returns cached response, missing key returns 422. Tier 1: I will show diffs before executing. Starting sub-task 1 — adding the idempotency key validation to the charge handler.
10 minutes later — IP-03: Agent hits a deviation
MANAGER(Agent reports: "The idempotency library I planned to use (`idempotent-oss`) requires Node 20+ but the project is on Node 18. I tried `node-idempotent` instead and it worked in my test.")
AGENTBefore I continue with `node-idempotent`: what was the specific failure? I assumed `idempotent-oss` was compatible with Node 18. The failure was a runtime `ERR_INVALID_PACKAGE_TARGET` when importing it. My assumption that it was Node 18-compatible was wrong. Is the fix a local adjustment (switch the library) or does it require a revision to the impl plan? (node-idempotent is a drop-in replacement for the same interface — local fix.)
5 minutes later — IP-04: Agent declares sub-task complete
MANAGERBefore we close sub-task 1: which of the three success criteria are you most uncertain about?
AGENTCriterion 1 (header accepted) and criterion 3 (422 on missing key) — I'm confident. Criterion 2 (duplicate returns cached response) — I tested it manually with curl and it worked, but I did not write an automated test for it. I can write the test now, or I can flag it as open for QA.
MANAGERFlag it as open for QA. Do not mark a criterion as verified if you haven't automated the verification. We're tracking test coverage in the QA plan. Sub-task 1 is open until an automated test is written.

The exchange above models the key discipline: The agent self-reported the deviation correctly (IP-03). The manager asked the termination check question (IP-04). The agent self-identified the gap. The manager closed the gap by refusing to let an unverified criterion pass. This is what "verification, not trust" (lesson 0009) looks like in practice during execution.

What the Manager Does All Day

The manager in an AI-driven team is not looking at code. The manager is looking at:

  1. The execution log — is the agent on track? Are the criteria being met?
  2. The context window — is it filling? Should we IP-05 soon?
  3. The impl plan — is the plan still valid? Are we deviating into HF-06 territory?
  4. The agent's questions — are they escalating appropriately (IP-07) or making silent assumptions (HF-02 risk)?
  5. The trust tier — are we ready to elevate? Has the agent earned a higher tier on this pattern?

The manager's single most valuable skill in AI-driven execution: Knowing when to say "stop." Not "stop and fix this" — just "stop" — to think about whether the plan is still correct. The agent will keep running if given the chance. The manager's job is to create pauses at the right moments.

What the Manager Never Does

  • Never give an instruction without a success criterion. "Do X" is not a brief. "Do X so that Y" is.
  • Never let an unverified criterion pass because "it looks right." Verification is not optional.
  • Never let the agent refactor mid-feature without a separate impl plan step.
  • Never let Tier 1 become Tier 3 because "it's been going well." Trust elevation requires evidence.
  • Never end a session without an IP-08 summary. The next session will pay for it.
  • Never let the agent introduce out-of-scope work without a Boundary Call (IP-02). The scope creep accumulates.
  • Never skip the termination check (IP-04). The agent will declare done optimistically.
