The Execution Phase — Calibrated Trust

Execution is not watching the agent code. It's checkpoint-first oversight: trust between checkpoints, audit at checkpoints, intervene only when the plan says to or the agent signals blocked. This lesson covers the supervision model, the intervention decision tree, and the three types of deviation from plan.

Source: Matt Pocock — "The 7 phases of AI-driven development", AI Hero / YouTube, 8:26, 2026-03-03. Builds on lessons 0001–0007.

How to use this lesson. The Execution phase is where most manager anxiety lives. Read the supervision model first — it's the calibration for how much to trust the agent. Then read the intervention decision tree — it's the operational rule for when to step in vs. let run. The three deviation types at the end are what your execution log should capture, and what determines whether you replan, revise the PRD, or accept the delay.

What Pocock Actually Said

The full quote, in context:

"So the sixth phase is implementation. This is where you actually implement the thing. The way that I've seen this work best is you give the agent the PRD, the plan, and you just let it go. And then at certain checkpoints, you come back and review it." — Matt Pocock

Two phrases matter: "you just let it go" and "at certain checkpoints." Together they define the supervision model. The manager's job is to set the context (PRD + plan), trust the agent between checkpoints, and audit at checkpoints. Not to write code alongside the agent. Not to watch the agent every minute.

The Core Idea: Supervised Autonomy

Execution is supervised autonomy: the agent has autonomy within the step, and the manager supervises at the checkpoint. This is neither micromanagement (supervise every line) nor disappearance (never check in). It's checkpoint-first oversight.

The Intervention Decision Tree

The question is never "should I intervene?" The question is: has a checkpoint been reached, or has the agent signaled blocked?

Intervention decision tree — run this before any action

Is a checkpoint in the impl plan reached?

→ YES: Audit the checkpoint.

Does the observable match the checkpoint description?

If yes → let agent proceed to next step.

If no → assess deviation type (see below).

Has the agent signaled blocked?

→ YES: Unblock.

Agent blocked types:

  • "I need a decision on X" → make the decision, give context
  • "This pattern doesn't work" → acknowledge, redirect to alternative
  • "I got an error I can't resolve" → diagnose, give the fix

Do NOT: re-explain the whole PRD, rewrite the step, or give low-level implementation instructions. Give the minimum context needed to unblock.

Neither?

→ Hands off. Trust the agent.

The agent is executing within the step. This is normal. The agent may be quiet for minutes or tens of minutes. This is not a signal to check in.

Exception: if the agent has been quiet for more than 2x the expected time for the step, gently ask for a status update. Use a question, not a command. "How's step 3 going?" — not "show me what you've done."

The Three Deviation Types

When the checkpoint doesn't match the description, the deviation is one of three types. Each has a different response:

Type 1: Replan the step

The PRD is right, but the plan was wrong. The step was XL, or the failure mode wasn't anticipated.

Response: Go back to impl plan. Replan this step only. Re-execute.

Signal: "The approach in step 4 doesn't work for the actual data shape."

Type 2: Revise the PRD

The plan was right, but the PRD was wrong. A requirement needs to change, or a new edge case was discovered.

Response: Go back to PRD. Update the requirement or edge case. Then replan the affected steps.

Signal: "The CSV export should include soft-deleted rows, but the PRD doesn't mention them."

Type 3: Accept the delay

Both PRD and plan were right, but the step took longer than estimated. Reality is slower than prediction.

Response: Note it in the execution log. Update the timeline. Do not change PRD or plan.

Signal: "Step 4 works correctly but took 3 hours instead of 1."

Not a deviation: scope creep

Something was built that isn't in the PRD. This is not a deviation from the plan — it's an addition.

Response: This is a PRD question. Does the new thing belong in v1? If yes → update PRD. If no → remove it from the build.

Signal: "The agent also built a PDF export, which isn't in the PRD."

The discipline: do not handle PRD changes in the execution phase. When execution reveals that the PRD is wrong, the right response is to flag it and return to the PRD phase — not to patch the PRD mid-execution. Executing with a wrong PRD means building the wrong thing. The loopback to PRD is not a failure; it's the process working correctly.

The Blocked Signal

Agents signal blocked in two ways: explicit and implicit. Both are valid, but the response differs.

Explicit blocked signals

  • "I need a decision on X before I can continue." — agent is asking for context. Give the decision, not the implementation.
  • "I'm not sure which approach to take here." — agent wants guidance on direction. Give the direction, not the code.
  • "I encountered an error I can't resolve." — agent has hit a technical wall. Diagnose, then give the fix.

Implicit blocked signals

  • Agent is repeating the same operation without progress. — implicit signal that something is stuck in a loop.
  • Agent asks the same clarifying question it asked 10 minutes ago. — implicit signal that the earlier answer wasn't sufficient.
  • Agent's output quality has degraded (incoherent, off-target). — implicit signal that the context window is degrading.

The unblock response rule. When an agent signals blocked, give the minimum context needed to unblock — not the maximum. The agent needs a decision, not a tutorial. The manager's job is to provide context, not to write code. If the manager finds themselves writing code to unblock the agent, the manager is micromanaging.

Trust Calibration

Trust is not binary and not static. It depends on two factors:

Agent experience with the task type

A new agent (fresh session, no prior work on this codebase) gets more oversight than an agent with established context. A task the agent has done before gets more trust than a novel task.

Step risk level

The impl plan's step sizes give you a risk signal. XS steps (well-understood, <50 LOC) warrant less oversight. L steps (multiple components, novel pattern) warrant more.

Trust but verify is wrong for AI agents. The aphorism "trust but verify" implies that verification is the exception. For AI execution, the rule is "calibrated trust": the calibration is the impl plan's checkpoints. Trust is the default between checkpoints. Verification is the rule at checkpoints. This is not a trust problem — it's a process design problem.

What Execution Reveals

Execution is the phase that tests the plan. A plan that survives execution is validated. A plan that doesn't survive execution is corrected. Both outcomes are correct.

What execution reveals about the PRD

  • Which requirements were architectural edge cases (specified in PRD) vs. discovered edge cases (not in PRD)
  • Which success metrics are actually measurable and which are aspirational
  • Whether the non-goals section was sufficient or scope crept anyway

What execution reveals about the impl plan

  • Which step sizes were XL (took too long, had too many failure modes)
  • Which dependency assumptions were wrong
  • Which checkpoints were well-specified vs. vague

What execution reveals about the team

  • Which agents do well on which types of tasks (capability map)
  • Which manager instincts are correct (intervention was helpful) vs. overcautious (intervention was unnecessary)
  • Which phase upstream had a gap (PRD missing a requirement → execution discovered it)

The Execution Log

The execution log is a living document created during execution. It captures what actually happened, not what was planned to happen. It is not a status report — it is the historical record of execution.

Step 1 — Schema migration — started 09:00, ended 09:12
Expected: 30 min. Actual: 12 min.
Checkpoint: passed. Table exists with correct columns.
Step 2 — Job queue — started 09:13, blocked 09:28
Agent asked: "Which job queue library should I use? We have BullMQ and Agenda in the codebase."
Type 1 deviation: BullMQ chosen (matches existing job infrastructure). Step 2 replanned to use BullMQ. Remaining steps unchanged.
Step 3 — Sync export endpoint — started 10:05, ended 11:48
Expected: 1 hour. Actual: 1h 43m. Agent hit a bug in the existing query optimizer with the large IN clause.
Checkpoint: passed after workaround (batch by 500 IDs instead of 5K). Workaround noted.
Step 4 — Batch query — started 11:49, blocked 12:30
Agent asked: "The CSV export should include soft-deleted rows, but the PRD doesn't mention soft-deleted rows. Should I include them?"
Type 2 deviation: Scope question — PRD needs to specify inclusion/exclusion of soft-deleted rows. Loopback to PRD phase. Execution paused.

The execution log's purpose. Three uses: (1) audit trail for what was built vs. what was planned, (2) input for the QA phase — deviations become test cases, (3) retrospective input for improving future impl plans and PRDs. The log is written during execution, not after. Every deviation, every block, every timing surprise gets a line.

Three Team Scenarios

Scenario A — The micromanager

Manager M watches the agent's every output. After each code block, M asks: "Is that right?" After each function, M asks: "Why did you name it that?" M changes variable names mid-session. M rewrites the error handling because M would have written it differently.

The agent makes progress but at half speed. Every 20 minutes, M interrupts to "align" on approach. The agent can't build momentum. The checkpoint at step 3 is a disaster: M has so many micro-changes that the agent has to reconcile conflicting instructions.

What went wrong: Manager treated the agent as a junior engineer who needs constant oversight. The supervision model wasn't applied. Interventions happened between checkpoints, not at checkpoints.

Scenario B — The ghost

Manager G sends the PRD and plan to the agent. G goes to a meeting and forgets about the agent for 4 hours. G comes back to find the agent has built the CSV export correctly — except it's synchronous for all project sizes, including 500K-row projects that will timeout.

The agent never signaled blocked because it didn't know that sync export for 500K rows was wrong — the PRD said "handle large projects" but didn't say what "large" meant in terms of row count. The agent made a reasonable assumption. G didn't audit at step 3.

What went wrong: Manager disappeared. The checkpoint at step 3 was missed. The PRD had a gap (threshold not specified), but the gap wasn't caught before execution built the wrong thing.

Scenario C — The calibrated supervisor

Manager V sends the PRD and plan. V trusts the agent. V checks in only at checkpoints.

Step 1 checkpoint: table exists. Passed. Step 2 checkpoint: job enqueues and creates a row with status pending. Blocked — agent asks which queue library. V answers: BullMQ (matches existing infrastructure). Agent continues.

Step 3 checkpoint: sync export returns CSV for 5K rows. Passed. Step 4: agent asks about soft-deleted rows — not in PRD. V recognizes this as a Type 2 deviation. Pauses execution. Loops back to PRD. Adds: "Include soft-deleted rows in export." Replans step 4. Continues.

Why it works: V trusted between checkpoints. V intervened at checkpoints. V recognized deviation types and routed them correctly: unblocked for Type 1, looped back to PRD for Type 2, noted and accepted for Type 3. The execution log captures every decision.

Execution and the Phases Around It

Execution ← Impl Plan

The impl plan's checkpoints are the manager's entry points. The plan is the queue; the agent manages it. The manager audits at each completion.

Execution → QA

The execution log feeds directly into QA. Every checkpoint becomes an assertion. Every Type 1 and Type 2 deviation becomes a new test case. "Soft-deleted rows must be included" — that's a QA test. "Batch query uses 500-ID batches" — that's a QA test. The execution log is the QA plan's source material.

Execution → Loopback to PRD

When execution reveals a PRD gap, the process is: pause execution, update PRD, replan affected steps, resume execution. Execution does not proceed with a wrong PRD. The loopback is not a failure — it's the process working correctly.

The natural next is QA (07) — the final phase in Pocock's sequence. The QA deep-dive will introduce the execution-log-to-test mapping: every checkpoint and deviation becomes a test case. Most teams QA agent-built code the same way they QA human-built code — but the failure modes are different. Agent-built code tends to have edge case gaps (the agent handles the happy path well but misses the hard edge cases). The QA phase needs to account for this pattern.

Final note on this lesson. The Execution phase is where the plan meets reality. The supervision model is the discipline that prevents two failures: micromanagement and disappearance. The intervention decision tree is the operational rule. The three deviation types are the routing logic. Type 1 → replan. Type 2 → loopback to PRD. Type 3 → accept and note. The execution log is the record of what actually happened. Execution is supervised autonomy: trust between checkpoints, audit at checkpoints, intervene only when signaled. The manager's job is not to write code — it's to maintain the context that lets the agent write code.
