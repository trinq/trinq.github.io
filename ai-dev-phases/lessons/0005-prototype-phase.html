The Prototype Phase — Proof Before Code

"I always prototype — isn't that just how you build?" No. A prototype proves exactly one thing. Without the proof, you have a half-built feature. This lesson covers the proof threshold, the throwaway test, and why most prototypes die by becoming the production system.

Source: Matt Pocock — "The 7 phases of AI-driven development", AI Hero / YouTube, 8:26, 2026-03-03. Builds on lessons 0001–0004.

How to use this lesson. The Prototype phase is where teams confuse three things: progress, exploration, and validation. Read the "proof threshold" section first — it's the test for whether what you're building is a prototype. Then read the throwaway test, because most "prototypes" in real teams quietly become production code. The 7 failure modes table at the end is the diagnostic you'll use in code review and sprint retro.

What Pocock Actually Said

The full quote, in context:

"The third phase is the prototype. So I want to build a small working version of the thing. Now, whether that's, like, a CLI tool, or, you know, an HTML page that fakes it, or a real implementation behind a feature flag — however the prototype looks — it's like, build a small working version of the thing." — Matt Pocock

He lists three prototype forms on purpose: "a CLI tool," "an HTML page that fakes it," "a real implementation behind a feature flag." The prototype is *whatever shape* serves the proof. The shape is not the point. The proof is the point. Most teams get this exactly backwards.

The Prototype Confusion

The most common objection: "I always prototype — that's just how software development works." It's a fair objection. The reason it's wrong:

  • Most "prototypes" don't have a proof threshold. Engineers build small versions of features every day. They're called spikes, POCs, scratch code, or just "writing the feature slowly." Those aren't prototypes in Pocock's sense — they're development.
  • Most "prototypes" are not throwaway. The code lives in the main branch. It's imported by main.ts. Test files cover it. Junior engineers "improve it" later. The prototype becomes the product — and the codebase inherits decisions that were made to prove a question, not to run a system.
  • Most "prototypes" never answer a question. They just exist. The team is uncertain whether the prototype proved the thing it was supposed to prove, and they've moved on to the next prototype.

Pocock's prototype is a specific thing: the smallest throwable piece of code that answers a specific, risky question. Three criteria. Drop any one and you have something else.

The Three Criteria

CriterionWhat it meansIf it's missing
Specific question The prototype proves one thing. The thing is named in 1 sentence. Vagueness — no one knows what success looks like
Risky assumption The answer is unknown. If you can predict it, no prototype is needed. Wasted work — the question could've been answered in Research.md
Throwaway-friendly Deleting the prototype doesn't break anything. The code is isolated. Mutation risk — prototype decisions leak into production logic

If you can name the question in one sentence, you have a prototype. If you can't, you have a half-built feature. The first test of a prototype is the 60-second test: can you write, in plain language, what this prototype is going to prove? If the answer takes more than one sentence, the prototype scope is wrong. Stop and rewrite the question before writing code.

The Three Questions a Prototype Can Answer

Most prototypes serve one of three purposes. Conflating them is a common mistake:

1. Approach prototype — "does this even work?"

You're unsure if the chosen approach is viable. The prototype tests whether the technology, library, or pattern does what research.md said it would. Example: "Can Redis Streams handle 10K writes/sec on a single node?" The proof is a benchmark, not a feature.

2. Interface prototype — "does this feel right?"

You're unsure if the API surface, UX pattern, or interaction model matches user expectations. The prototype tests shape, not implementation. Example: "Does the export-progress UI feel like progress, or does it freeze?" The proof is user feedback, not benchmarks.

3. Integration prototype — "do these two systems actually work together?"

You're unsure if two services, two libraries, or two clouds can talk. The prototype tests the integration edge. Example: "Does Stripe's webhook signature verify correctly when our reverse proxy adds an extra header?" The proof is end-to-end, not isolated.

One prototype, one question. Each prototype type answers a different question. If your prototype is trying to answer two of these, split it. Two parallel prototypes is cheaper than one prototype that proves nothing because it was spread too thin.

Anatomy of a Prototype

Like a research.md, every prototype has a minimum viable structure. The artifact is called prototype-N.md:

prototype-N.md — minimum viable template

  • Question: One sentence. "Does X do Y?" Pass the 60-second test.
  • Why this question matters: What decision gets made after the prototype?
  • Risky assumption: What you believe is unknown. (If nothing's unknown, why are you prototyping?)
  • Timebox: Hard deadline. No extensions without a manager call.
  • Throwaway boundary: Which files / branches / paths will be deleted after?
  • Location: prototypes/[question-slug]/ in the repo root. Never merged to main directly.
  • Success criteria: Specifically what you observe when the prototype proves the question (a number, a screen, a behavior).
  • Failure criteria: What you observe if the prototype disproves the question.
  • Decision after prototype: Proceed / Pivot / Abandon. Pre-filled, picked when the prototype concludes.

The pre-filled decision field is the secret. Every prototype concludes with one of three decisions: Proceed (the approach works), Pivot (we need a different approach), Abandon (the idea is wrong). Writing all three options in the prototype-N.md upfront prevents post-hoc rationalization. If the prototype disproves the question, the team doesn't get to retroactively define success.

The Throwaway Test

The most dangerous moment for a prototype is the moment it "kind of works." Engineers look at the code, think "it's already written, why delete it?" — and the prototype becomes the production system, complete with all the prototype-shaped decisions baked in.

The throwaway test: imagine deleting the prototype right now. What's the cost?

If the cost of deletion is…
Then it is…
And you should…
0 days of effort
A real prototype
Delete it after the decision. Don't merge.
1–3 days of effort
A prototype that grew up
Stop. Re-architect. Delete the throwaway parts; revisit the question.

Admit it's the production system. Add tests, error handling, observability. Move to phase 04 PRD retroactively.

The "already written" trap. Engineers who say "it's already written" about prototype code are conflating two separate decisions: "we built a thing" and "we shipped a thing." A prototype is not shipped. It's a model. Models can be torn down. If deleting the prototype scares the team, the team has accidentally shipped a system without shipping a feature.

Three Team Scenarios

Scenario A — The prototype that became the product

The team builds an AI-powered search feature. Engineer Q writes a quick prototype over 4 days. It works. Engineering manager says "ship it." The prototype is merged to main. Six months later, it's the search feature. The team discovers it doesn't handle trailing slashes correctly, has no observability, and the AI model isn't version-pinned.

What went wrong: The throwaway test was never applied. The "prototype" had no specific question; it just was the feature. When it shipped, it inherited research-stage decisions: no observability, no model version pinning, no error handling. The cost of those inherited decisions is showing up 6 months late.

Diagnostic: No prototype-N.md existed. No throwaway boundary was defined. Question wasn't named.

Scenario B — The prototype that proved nothing

Engineer R wants to know if their team can use WebSockets for real-time updates. The WebSocket prototype runs in a separate branch. Engineer R tests it manually, decides it works, and shares the result in standup: "WebSockets work." Three weeks later, under load, the connection drops every 30 seconds.

What went wrong: The prototype proved "WebSockets work in my laptop for 5 minutes," but the question that mattered was "can WebSockets work for our production load for 8 hours?" The question wasn't specific. The success criteria were subjective.

Diagnostic: Question was "does this work?" — too vague. Success criteria missing from prototype-N.md. Failed to define what counts as proving the question.

Scenario C — The proof artifact that compounds value

Engineer S wants to know if Turso's edge replicas give sub-50ms reads globally. She writes:

  • Question: "Do Turso edge replicas give p95 read latency < 50ms for users in 5 regions?"
  • Timebox: 5 days.
  • Success criteria: p95 latency < 50ms in 4/5 regions over 10K requests.
  • Failure criteria: p95 latency > 200ms in any region.
  • Throwaway boundary: Only the files in prototypes/turso-latency-test/. Test data deleted.
  • Post-prototype decision: Proceed with Turso / Pivot to Postgres / Abandon real-time feature.

Engineer S runs the prototype in 4 days. Result: p95 is 65ms in 3 regions, 180ms in 2 regions. Question is answered: no. The team pivots to a hybrid model. The prototype code is deleted. The prototype-N.md lives as a decision record: "we tried edge replicas, latency was too high, we're using Postgres + read replicas instead."

Why it works: Specific question. Numeric success criteria. Explicit failure criteria. Throwaway boundary enforced. Decision pre-filled. The prototype produced a *decision*, not code.

Prototype and the Phases Around It

Prototype ← Research

If research.md has a Decision section (from lesson 0004), the prototype is testing that decision. The Decision becomes the prototype's success criteria. If research.md doesn't have a decision, the prototype is doing research's job — and your team is paying for the same exploration twice.

Prototype → PRD

The PRD (phase 04) is much cheaper to write after the prototype. The prototype answered: "what's hard here? what's easy here? what are the gotchas?" The PRD can now be written from real data, not imagination. A PRD written before the prototype is mostly fiction.

Prototype → Execution (only after the decision)

The prototype produces a decision: Proceed / Pivot / Abandon. Execution (phase 06) only starts after Proceed. Pivot means another prototype or another research.md. Abandon means back to phase 01 Idea. Phase 06 should never start with a prototype as its foundation.

The PRD-after-prototype insight. Most teams write PRDs first, then build, then test. Pocock inverts the order: prototype first, then PRD, then build. The PRD captures what the prototype taught. A PRD written without a prototype is engineering fiction. A PRD written after a prototype is a buildable spec. Same template, different inputs.

Prototype runs but key behavior isn't tested Prototype not run end-to-end; only unit tests ran
Prototype answers the wrong question Question wasn't specific; team's prototype tested what was easy, not what was risky
Prototype cost more than the feature Gold-plating during exploration; no timebox enforced
Multiple parallel prototypes never reconciled No pre-decision rule; pre-prototype decisions weren't aligned

The Manager Ritual for Prototype

Prototype phase ritual — 2 questions × 2 moments

Before approving a prototype

  • "Can the engineer write the question in one sentence?" (60-second test)
  • "What's the throwaway boundary? Which files will be deleted when it's done?"

At end of prototype

  • "Did the prototype-N.md get written? Is the decision pre-filled with proceed/pivot/abandon?"
  • "Was the throwaway boundary enforced? If not, what does the codebase now carry from the prototype?"

The manager's hardest question after a prototype. It's not "did the prototype work?" It's "did we delete the prototype code?" If your team answers "no, we kept some of it," your team has shipped prototype-grade code into production. That's a phase 04 PRD problem that nobody noticed — because the code was already merged.

Recall exercise: audit your last 3 prototypes

For your last 3 features that began with a prototype, answer:

  1. Was the question specific and one-sentence?
  2. Did success criteria exist as numbers, screens, or behaviors?
  3. Was a throwaway boundary defined?
  4. Was the boundary enforced? (Is the prototype code still in main?)
  5. Did the prototype-N.md get written?
  6. Did the prototype produce a decision (proceed / pivot / abandon)?
  7. Did the prototype prove something you'd have to spend more time re-proving without it?

Score: each yes = 1 point. Below 4/7 → prototype phase is leaking into production. 4–5/7 → solid. 6+/7 → strong prototype discipline. Whatever the score, the value of measuring it is the same.

Final note on this lesson. The Prototype phase is the first phase where output is code, not documentation. That changes the failure modes — documentation failures are silent (no one reads it), prototype failures are visible (broken behavior). Use the 60-second test as the entry gate. Use the throwaway test as the exit gate. Between those two gates, the prototype's only job is to prove one specific thing. If it proves nothing, the team is paying prototype costs without prototype benefits — and the next phase will need to re-prove what this prototype was meant to prove.
