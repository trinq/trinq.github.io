The 7 Phases in Detail — What Each One Actually Does

You know the names. Now learn the verbs. Each phase has an action, an artifact, a team scenario, and a failure mode. By the end you can brief your team on what each phase is responsible for — without paraphrasing Pocock.

Source: Matt Pocock — "The 7 phases of AI-driven development", AI Hero / YouTube, 8:26, 2026-03-03. Builds on lesson 0001.

How to use this lesson. Read each phase in order. After each one, close the tab and explain the phase out loud in one sentence — action, artifact, failure mode. If you can't, the phase isn't learned yet. The 7 recall questions at the end of each section force the loop.

The 7 Phases at a Glance

# Phase Action (one verb) Artifact produced
01 Idea Name A written one-liner
02 Research Cache research.md (or scoped doc)
03 Prototype Impose Throwaway code, committed
04 PRD Specify End-state PRD doc
05 Impl planning Decompose Kanban board with dependencies
06 Execution Loop Merged PRs
07 QA Verify + feed back QA plan + new tickets

Memorize this table before going deeper. The rest of the lesson expands each row.

01

Idea — name the work

The action

Name the work. Before any code, you need a one-sentence description of what you're trying to ship. Not the implementation — the intent.

Pocock: "This idea can be as small and as big as you like." A bug fix, a feature, a refactor, a whole app — all are ideas. The size doesn't matter. The naming does.

The artifact

A written sentence (or two) somewhere durable. The medium doesn't matter — Notion, GitHub issue, Slack pin, a sticky note. What matters is that the work has been named in writing, not left in someone's head. If the team can't quote the idea back to you, the phase didn't run.

Anchor — team scenario

Your senior engineer says: "I'll add caching to the API." You ask: "What is the user-visible problem this fixes?" They say: "It's slow sometimes."

That's a half-formed idea. The phase exists to force the answer: "Users on the dashboard page wait 3+ seconds for the first chart to render. Adding Redis caching to the data layer brings p95 to under 800ms." Now the team has a target.

Failure mode

The team starts coding from a vague description. Six days in, you discover three engineers have three different ideas of what "caching" means. Sunk cost: you either re-scope the work or ship the wrong thing. Both are bad.

Recall: True or false — "The idea phase exists to pick the right tool (Redis, Memcached, etc.) before writing any code."
Reveal answer

False. The idea phase picks the user-visible outcome, not the tool. Tools get picked later — in prototype or implementation planning. The idea phase is "what does the user see / do / feel that they don't see / do / feel today?"

02

Research — cache hard explorations

The action

Identify the hard explorations the agent will need to do, do them once, write the result down. Things like: an external API's quirks, a new dependency's gotchas, the existing code's hidden assumptions, the production data's shape.

Pocock: "Every time your agent is doing work, it might need to explore the repo in a fresh context window. And if that exploration is difficult … you'll want to cache it in a research.md asset."

The artifact

A research.md (or scoped equivalent — docs/research/stripe-webhooks.md, a Notion page, a pinned GitHub issue) sitting somewhere the agent and team can find it. Aim for 1–3 pages. Longer than that and nobody reads it. Shorter than that and you haven't actually done the research.

Anchor — team scenario

Your team needs to integrate with a Vietnamese payment gateway. Nobody on the team has used it before. The agent will discover the quirks (signing algorithm, retry behavior, test-mode flag behavior, callback URL handling) every session, in a fresh context window. Caching this once into docs/research/vn-pay.md saves 20 minutes per session and prevents the same bug being re-introduced by a different engineer.

Failure mode

Each engineer — and the agent, every session — re-discovers the same external API quirk. They fix it locally. Next week, a different engineer hits the same wall. Knowledge lives in heads, not in the repo. The cost compounds: 6 engineers × 20 minutes × same discovery = 2 hours of duplicated work per session.

Recall: "If the exploration is easy (the agent can do it in 5 minutes), do you still need a research.md?"
Reveal answer

No. The research phase is only valuable when the exploration is expensive — i.e., hard enough that re-doing it costs more than writing it down. Reserve research.md for the truly hard stuff. If a junior engineer can answer it in 10 minutes, it's not research — it's a Slack thread.

03

Prototype — impose taste

The action

Write throwaway code to find the shape of the solution. Not to ship. To answer: when I build this for real, what does the user actually see? What does the data look like? What are the rough edges?

Pocock: "Doing this early is absolutely essential because then you can actually commit the prototype." The key word is commit. You don't lose it. You tag it as throwaway, but it stays in the repo as evidence of the design space.

The artifact

A prototype — usually a separate branch or folder, deliberately not polished. Often exploratory: a fake data layer, a stub of the UI, a sanity check on the API contract, a 50-line script that proves the hard part is possible. The output is a screenshot, a demo, or a 5-minute conversation with the PM — not production code.

Anchor — team scenario

You're building a new dashboard widget. Before writing the PRD, the engineer spends 2 hours building a fake version using mocked data. They show the PM: "Here's what it could look like." PM says: "No, the chart should be at the top, not the side. And the date range picker should be inline, not in a modal." That feedback goes into the PRD. Without the prototype, the PRD is abstract, and the engineer re-decides layout during implementation. Two weeks later: re-work cycle.

Failure mode

PRD is written before any code exists. Engineer reads it, makes their own taste call on layout, naming, structure. PM sees the result, says "No, I wanted X." Re-work. This is one of the most expensive failure modes in the entire framework — because the cost is paid in code, not in docs.

Recall: "Should the prototype use production-quality code?"
Reveal answer

No. Throwaway means throwaway. The artifact of this phase is taste validation, not working software. If you find yourself polishing the prototype — adding tests, refactoring names, chasing edge cases — you're building the wrong thing. The prototype's job is to provoke feedback, not to ship.

04

PRD — specify the end state

The action

Write down the end state — what the user will see, do, and feel, and what "done" looks like — without specifying how the code works. Implementation-agnostic. Use grill-me / the interview pattern to expand the decision tree.

Pocock: "We should now feel confident in ourselves that we can kind of like understand the end state … we won't know all of the implementation decisions yet."

The artifact

A PRD doc (Markdown, Notion, even a long GitHub issue) that another engineer could read and build a different implementation from, and you'd still get the same product. If the PRD says "use Redis" it's implementation-locked. If it says "p95 dashboard load under 800ms" it's outcome-locked. Only the second survives a tech-stack change.

Anchor — team scenario

Your team's PRD for the new feature reads: "Use the React Query library to fetch chart data and cache it client-side. Display a loading skeleton during fetches." That's an implementation decision masquerading as a spec. When the React Query upgrade goes badly in 6 months, the team is stuck — they have to rewrite the PRD. A good PRD says: "On dashboard load, the user sees their data within 800ms. Refreshes don't require a full page reload. While data is loading, the user sees a skeleton, not a blank page." Both are valid specs. Only one survives tech-stack changes.

Failure mode

PRD reads like a tech spec. The team implements the spec literally. When business needs change, the spec changes, and the code has to be rewritten. The PRD is supposed to be the stable doc across implementation changes. If it's not stable, the framework collapses — every phase downstream rebuilds from scratch every time.

Recall: "The PRD should describe: (A) the implementation, or (B) the end state."
Reveal answer

B — always the end state. If your PRD contains class names, library choices, framework decisions, or "use X for Y," you're writing the wrong doc. Push those into prototype or impl planning. The PRD is for non-engineers to read and recognize "yes, that's what we wanted."

05

Implementation planning — decompose into tickets

The action

Break the PRD into small, independently-scoped tickets. Identify dependencies between them. Some tickets block others; some can run in parallel. The point is to convert a doc into a queue.

Pocock: "A Kanban board is just a list of tickets that have blocking relationships between them."

The artifact

A Kanban board (Linear, GitHub Projects, Jira, Notion) where each ticket has:

  • One sentence title
  • 2–3 sentence description
  • Acceptance criteria (a checklist of "done" conditions)
  • Explicit "blocked by" / "blocks" relationships

A ticket without acceptance criteria is a wish, not a ticket.

Anchor — team scenario

Your team is building the new dashboard. Tickets include: "API endpoint for chart data," "Frontend chart component," "Cache layer for chart data," "E2E test for dashboard load."

Without dependencies: the frontend engineer picks up the chart component ticket first, realizes the API doesn't exist, asks the backend engineer, who is busy on a different ticket. Stalled.

With dependencies: API first → cache layer in parallel → frontend waits for API → E2E waits for both. Sequential where needed, parallel where possible. Predictable throughput.

Failure mode

Tickets exist but without dependency order. Engineers grab tickets randomly. Either they block each other, or they build things that don't fit together. Integration hell at the end: three components, none of which composes, two days of "why doesn't this work" debugging.

Recall: "Can tickets ever be parallelized across multiple AFK agents?"
Reveal answer

Yes — but only if the dependencies are explicit and the dependencies are met. Two agents on two independent tickets = good. Two agents on two tickets where one's output is the other's input = chaos. The Kanban board is not a free-for-all; it's a directed graph, and the dependencies are the edges.

06

Execution — loop the agent through tickets

The action

Loop the coding agent (or the team) through each ticket. The agent picks up a ticket, reads the description and acceptance criteria, does the work, opens a PR, moves on. Repeat until the Kanban board is empty.

Pocock: "You can totally run this execution loop AFK, and the results will be really good." The conditional: with research, prototype, PRD, and Kanban board in place.

The artifact

Merged PRs. Each ticket → one PR → review → merge. Or: an agent working through a queue, producing commits per ticket. The output is in version control, reviewable, deployable.

Anchor — team scenario

Your team has 8 well-scoped tickets from the planning phase. One engineer + one AFK agent pair: the engineer reviews PRs and handles scope questions, the agent picks up tickets one at a time. After 2 days, 6 of 8 are merged.

Without the upstream phases, the same engineer spends 2 days on a single ticket, blocked by ambiguity. The bottleneck is rarely the agent. It's the ticket.

Failure mode

Tickets are vague (no acceptance criteria, no description). The agent asks a clarifying question, gets an answer, asks another, gets another answer, and after 30 minutes of back-and-forth, has produced nothing. The engineer ends up writing the code by hand. The AFK promise breaks down — not because the agent is bad, but because the ticket was bad.

Recall: "Execution can run AFK if: (A) the agent is good enough, (B) the upstream phases are done well, (C) the tickets are well-scoped, or (D) both B and C."
Reveal answer

D — both B and C. A is almost always a non-issue (modern agents are good enough for scoped work). B and C are the levers. If the AFK agent keeps asking questions or producing garbage, the fix is upstream — better research, better prototype, better PRD, better tickets. Not a better model.

07

QA — verify, then feed back

The action

The agent writes a QA plan (a list of scenarios to test, edge cases, user flows). The human executes the plan against the merged code. Anything that fails becomes a new ticket. Loop back into execution.

Pocock: "A human actually walks through and QAs the completed work and then produces more tickets for the Kanban board, which then goes and executed."

The artifact

Two things:

  • A QA plan doc — reusable for regression, scoped to the feature
  • A set of follow-up tickets on the Kanban board — the actual findings

The QA plan lives; the tickets drive the next loop.

Anchor — team scenario

Your team has shipped the new dashboard. The agent generates a QA plan: "Test with empty data, test with 1k records, test with 10k records, test slow network, test concurrent users, test the error state, test mobile viewport, test with expired session." Your QA engineer runs through these. The slow-network test reveals the dashboard doesn't show a loading state — it shows a blank page for 4 seconds. That's a new ticket. The agent fixes it. The dashboard gets re-tested. The loop continues.

Failure mode

QA = "engineer tests their own code." No separate lens. The engineer is too close to the implementation to spot user-experience issues. Bugs that a fresh tester would catch in 30 seconds ship to production. Worse: there is no feedback loop back into execution, so the same class of bug gets re-introduced next sprint.

Recall: True or false — "The QA phase is only for finding bugs."
Reveal answer

False. The QA phase is a loop that produces new tickets. Some are bugs, some are "this is fine but I noticed X while testing," some are documentation gaps, some are refactoring opportunities. The output is not a binary pass/fail — it's a queue of improvements that feeds back into phase 06.

Synthesis — The Order, the Loop, the Cost of Skipping

Phases 01–05 are linear. Phases 06–07 are a loop, not a sequence. The loop is the point.

01 Idea ↓ 02 Research ↓ 03 Prototype ↓ 04 PRD ↓ 05 Implementation planning ↓ 06 Execution ──────┐ ↓ │ 07 QA ─────────────┤ ↓ │ (new tickets) ─────┘

Expect to iterate phases 05–07 two to four times per feature. A good team runs the loop, not the waterfall. The first pass through 01–05 might take a sprint. The loop on 05–07 might run twice in the next sprint.

The cost of skipping each phase

Skip What breaks
01 Idea Team builds the wrong thing. Re-scope or re-ship, both expensive.
02 Research Agent and team re-discover the same hard exploration 10+ times. Token cost compounds.
03 Prototype PRD is abstract. Engineer re-decides taste during implementation. Re-work cycle.
04 PRD Spec is implementation-locked. Brittle to tech-stack changes. Rewrite every 6 months.
05 Impl plan Engineers grab tickets randomly. Integration hell at the end.
06 Execution No harness. Engineer writes code without the framework. AFK promise breaks.
07 QA Bugs ship. No regression catching. Same class of bug re-introduced each sprint.

Manager heuristic. When your team is producing low-quality work, don't ask "how do we work harder?" Walk the table above. Find the phase they're skipping. That's where the fix is — not in the execution phase. The execution phase is where the leak becomes visible. The cause is upstream.

How to Brief Your Team

When you sit down with the team to introduce this framework, use this exact 3-sentence brief:

"We're adopting Matt Pocock's 7-phase AI-engineering framework. Every feature, from now on, traverses all seven phases: idea, research, prototype, PRD, impl planning, execution, QA. Execution is the loop, not the goal — the goal is that the team can run phases 5–7 without me in the room. I'll score you against the framework in retro. If I find a phase is being skipped, we fix the upstream phase, not the execution phase."

That's the brief. Anything longer loses the team. Anything shorter is unmoored from the framework.

Manager commitment. If you brief this and don't follow through, you've made the framework worse than not having it. The team will learn that phrases don't have consequences. Pick one feature next sprint, walk all seven phases with them end-to-end, and score the result publicly. One real run is worth ten slides.

I asked these at the end of lesson 0001. You didn't answer them yet. They still drive where lesson 0003 goes. Re-asking them now that you have a deeper grip on the framework.

  1. Which phase is your team currently strongest at? Signal: the place where you already have a real ritual. Treat it as a temporary asset — and as a reference example for the lesson on that phase.
  2. Which phase is your team currently weakest at? Signal: the phase where we will focus lesson 0003. Pick from research, prototype, PRD, impl plan, execution, or QA.
  3. Is there a phase your team is entirely skipping? Signal: a place where a new ritual needs to be designed. I can draft a research.md template, a Kanban template, or a QA plan template for whichever phase you name.
  4. Does your PM / CTO know these 7 phases? Signal: do we need to prioritize a leadership memo this week, or defer it? The answer also tells me whether you have an ally in leadership, or whether you are pushing this alone.

Reminder. You don't have to answer all four. Answer whichever ones feel true. The ones you skip, I'll ask again in lesson 0003.
