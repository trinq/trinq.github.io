The QA Phase — Verification, Not Trust

The QA phase answers one question: did the build match the PRD? Most QA on agent-built code is over-trusting or under-trusting. The QA phase uses the PRD as the contract and the execution log as the test plan. Three verdicts determine what happens next: ship, fix, or rollback.

Source: Matt Pocock — "The 7 phases of AI-driven development", AI Hero / YouTube, 8:26, 2026-03-03. Builds on lessons 0001–0008. This is the final phase in Pocock's 7-phase sequence.

How to use this lesson. The QA phase is the last gate before the user's code goes to production. Read the "verification, not trust" section first — it reframes how to test agent-built code. Then read the test taxonomy — it categorizes tests by source (PRD requirement / impl-plan checkpoint / execution deviation / stability). The three verdicts at the end determine what happens next in the process: ship it, fix it, or roll back. The loopback section at the end shows that QA is not always the end — sometimes it loops back to Idea.

What Pocock Actually Said

The full quote, in context:

"The seventh and final phase is QA. So this is where you actually test the thing. The trick here is that a lot of people get really excited about this and they spend a lot of time on testing, but actually QA is really just checking that you built the right thing." — Matt Pocock

Two phrases matter: "checking that you built the right thing" and the warning about getting "really excited about this." QA tests whether the build matches the PRD — not whether the code is perfect. There's a difference: the code could be perfectly clean and still have built the wrong thing. That's the test case QA must catch.

The Core Idea: Verification, Not Trust

QA for agent-built code differs from QA for human-built code in one fundamental way. For human code, the assumption is: "the human understood the spec — QA catches the surface-level mistakes." For agent code, the assumption is: "the agent built what the plan said, but the plan may have been wrong." QA verifies the build against the PRD, not against the plan.

The trust problem

When an agent builds a feature, the manager saw the agent execute. They watched several checkpoints. They have a felt impression of how the build went. They feel like the build is good. This is the trust problem: feeling isn't verification. The build passed the agent's steps; that's not the same as the build satisfying the PRD's requirements.

The rare bug pattern for agent code. Human code tends to fail on the happy path (it's undertested because "obviously" it works). Agent code tends to fail on edge cases (it's good at the happy path because the plan specifies it well, and weak on edge cases because edge cases aren't always in the plan). This means QA should over-test edge cases and under-test the happy path. Reverse the human intuition.

Test Taxonomy

Tests fall into four buckets by source. Each source answers a different question:

Source
Question it answers
Where it comes from
A. PRD requirement test
Did we build the thing the PRD said to build?
PRD §2 user outcomes, §3 interface contract
B. Impl plan checkpoint test
Did each step do what the plan said?
Impl plan step checkpoints
C. Execution deviation test
Did the deviation get resolved correctly?
Execution log Type 1 / Type 2 / Type 3 entries
D. Stability test
Does the build hold under load, time, weird inputs?
Edge cases, performance, security

Most teams only write (D) and skip (A), (B), (C). That's the failure. (D) is necessary but not sufficient. The PRD's success metrics become (A) tests. The impl plan's checkpoints become (B) tests. The execution log's deviations become (C) tests.

Anatomy of a Test Case

Every test case has five parts. Skipping any one turns the test into something weaker than it should be:

A.001 — PRD §2 User Outcome #1
User can export CSV from project page in < 30 sec for < 10K rows
Source: PRD §2 / success metric #1
Given: Project with 5,000 rows
When: User clicks "Export" button on project page
Then: CSV file is downloaded within 30 seconds, with row count equal to source
B.003 — Impl plan step 5 checkpoint
50K-row project returns job ID immediately; reaches done within 2 min; file exists
Source: plan-step-5.md checkpoint
Given: Project with 50,000 rows
When: User calls POST /api/projects/:id/export
Then: Response within 200ms with jobId; after job runs, file at /exports/[jobId].csv exists with correct rows
C.002 — Type 2 deviation: soft-deleted rows
Soft-deleted rows must appear in CSV export
Source: execution log step 4 (Type 2 deviation)
Given: Project with 1,000 active + 100 soft-deleted rows
When: User exports project
Then: CSV contains 1,100 rows; both active and soft-deleted rows present
D.007 — Stability: 100K rows under 60 sec
100K-row project export completes within success metric window
Source: PRD §5 success metric / edge case lower bound
Given: Project with 100,000 rows
When: User triggers export
Then: Job completes within the documented window; file row count matches DB

Tests A, B, C come from documents, not from the engineer's imagination. This is the structural discipline: if a test isn't traceable to a PRD requirement, an impl plan checkpoint, an execution deviation, or a stability concern, it's a wandering test. Wandering tests are easy to write but not auditable. Auditable tests are the QA's output.

The Three Verdicts

Once testing is complete, one of three verdicts. Each verdict has a defined next step:

Verdict 1: Ship

SHIP — All PRD requirements met; impl plan checkpoints passed; deviations resolved; stability under thresholds

Action: Release the build. Move the PRD to v1-shipped state. Archive the execution log. Update the team retrospective template with what was learned. The feature is in production.

Next phase: Loop back to idea for v2 — or wait for new feature requests.

Verdict 2: Fix

FIX — Build has gaps but is salvageable in the same iteration

Action: Create a new impl plan section called qa-fixes.md. List each failing test. The agent fixes the tests. Re-run QA against the failing subset. Tests that pass move to ship; tests that fail move to either fix or rollback.

Next phase: Back to execution with the qa-fixes plan.

Verdict 3: Rollback

ROLLBACK — Build is fundamentally wrong; PRD or architecture needs revisit

Action: Revert the merge. Don't try to fix forward. The build is wrong at the requirement level, the architecture level, or both. Fixing forward risks shipping something the team can't reason about.

Next phase: Loop back to PRD for v2 — or, if the idea itself is wrong, loop back to idea.

Notice the verdict flows: ship ends the loop in success. fix stays in execution with a focused plan. rollback loops back to PRD or idea. The QA phase is not always the end of the loop. Sometimes it's the longest feedback path.

The Verification Ladder

Not all tests are equal. Some run fast and cheaply; others run slow but thoroughly. Verification builds from cheap to expensive:

LevelWhat it testsSpeedCostWhen
L1 — Unit Individual functions behave as the plan specified Seconds Free Every commit
L2 — Integration Components cooperate; data flows Minutes Low Every step checkpoint
L3 — End-to-end User outcome from PRD §2 works Minutes–hours Medium After execution completes
L4 — Performance Success metrics in PRD §5 met under load Hours High (load infra) Before ship
L5 — Security Permissions, secrets, audit trails Hours High (security team) Before ship

L1–L3 run during execution; L4–L5 run after execution. The impl plan's checkpoints are L1–L2 tests in disguise. Every step's checkpoint becomes an automated test. By the time execution finishes, most of L1–L3 coverage exists. The QA phase adds L4–L5 (performance and security) and runs the L3 end-to-end tests.

Three Team Scenarios

Scenario A — The QA that over-trusted

Manager M oversaw execution. The agent built the CSV export. The impl plan's checkpoints all passed at runtime. M feels good — the build went well. M approves ship without further testing.

Two weeks later, a back-office user reports: "The export is missing the soft-deleted rows we're supposed to audit." The team discovers this was a Type 2 deviation resolved mid-execution — but the test that would have caught it was never written. The PRD's success metric was "0 silent failures," but the failure mode wasn't even tested, so it wasn't a "silent" failure at all — it was a missed test.

What went wrong: QA was "did the agent do what the plan said." It should have been "did the agent do what the PRD said." The Type 2 deviation didn't get a (C) test. QA over-trusted.

Diagnostic: Tests were (B) impl plan checkpoints only. No (A) PRD requirement tests. No (C) execution deviation tests.

Scenario B — The QA that under-trusted

Manager U oversaw a careful execution. The agent built the feature. U runs a full manual QA: 6 hours of clicking through every edge case U can think of. U writes 80 tests in JIRA. By the time U is done, the team has shipped 3 features total this quarter.

The problem: 60 of U's 80 tests are wandering tests — not traceable to a PRD requirement or a PRD success metric. The team can't keep them updated when the PRD changes. Half of them are skipped after the first month. The other half become a maintenance burden.

What went wrong: QA generated more test debt than insight. Tests weren't tied to upstream documents, so they drifted over time.

Diagnostic: Tests had no (A) / (B) / (C) source trail. Tests were written from the QA's imagination, not from documents.

Scenario C — The QA that verified

Manager V oversaw a careful execution with a full execution log. After execution completes, V runs QA:

  1. A-tests (PRD requirements): 8 tests derived from PRD §2 user outcomes and §3 interface contract. All pass.
  2. B-tests (impl plan checkpoints): 7 tests derived from impl plan step checkpoints. All pass.
  3. C-tests (execution deviations): 4 tests derived from execution log deviations. All pass — including "soft-deleted rows included" (Type 2 deviation from step 4).
  4. D-tests (stability): 5 tests for edge cases and performance. 4 pass; 1 fails — performance drops for 80K-row projects.

The 1 failing D-test triggers a check: is 80K a documented boundary? PRD §4 said "projects > 100K rows must use background job." 80K is in the sync-export zone, but it's at the boundary. Verdict: FIX — the impl plan didn't specify batch sizing for 80K-row projects, and the agent assumed 1K batches but timed out at 80K. Manager V returns to execution with qa-fixes.md: "For sync exports, batch size should be tuned to the project size OR bring the 100K threshold down to 50K."

The fix is small (4 hours). The build ships 1 week after QA begins. The retrospective captures the lesson: "Sync export boundary should be more conservative; check PRD thresholds against realistic DB shapes."

Why it works: Tests trace to documents. Verdict is mechanical (passing vs. failing tests). Loop back is the FIX path, not the rollback path. The retrospective input is high-signal because the failures are documented.

Test debt grows faster than coverage Wandering tests; no (A)/(B)/(C) trail; tests without owners
QA takes longer than execution L4/L5 tests run before L1/L3; ladder sequence broken
"100% coverage" but real bugs slip through Line coverage without PRD requirement coverage
PRD changes in v2, tests break Tests not linked to PRD ids; can't be updated automatically
QA approval is a stamp, not a check Manager over-trusts; doesn't read the test outputs

The Manager Ritual for QA

QA ritual — 3 tests before any verdict

Question 1: Did every PRD requirement become at least one A-test?

Question 2: Did every execution deviation become at least one C-test?

Question 3: Did every impl plan checkpoint become at least one B-test?

→ All yes: Read test outputs. Apply verdict.

Ship / Fix / Rollback — depending on test results.

→ Any no: Add missing tests before verdict.

Tests without sources are wandering tests.

The manager's hardest moment after QA. The verdict is ship but one test is borderline — say, 28 seconds when the threshold is 30 seconds. The instinct: ship anyway; 2 seconds is "close enough." The discipline: send it back to FIX with a tightened threshold or a documented variance. The PRD's success metric is the contract. If you ship below the threshold, you changed the contract without a new PRD. Future builds will look at the precedent. Hold the line.

The Loopback: When QA Returns to Idea

QA is the last phase in Pocock's sequence, but it's not always the last phase the feature sees. Three loopbacks are valid:

Loopback A — QA → PRD (v2)

Build passed QA but stakeholders want changes. The next iteration starts with a new PRD, not a PR to the existing one. The previous PRD is moved to prd-archive/ per lesson 0006's versioning rule.

Loopback B — QA → PRD (rollback)

Build failed QA at the requirement level. The fix isn't in the code — it's in the requirements. Loopback to PRD to revise, then back to impl plan.

Loopback C — QA → Idea (kill)

The rare but real case: the build technically meets the PRD, but the feature itself doesn't solve the user's problem. The build is correct; the idea is wrong. The team has discovered that the user's pain is something else. Loopback to idea. The 4 prior phases' worth of work is archived as research. The new idea starts fresh.

Loopback C is the rarest and most valuable. It happens when the team has shipped a perfect feature that nobody uses. The idea was wrong, not the execution. The instinct is to blame the team. The discipline is to record what was learned and move on. Most products that succeed have several Loopback C events in their history — features built perfectly that didn't solve the problem. The team that gets good at recognizing Loopback C early saves months of work.

The QA Phase as the System's Memory

The QA phase produces three artifacts that persist beyond the feature:

ArtifactLives whereUsed for
Test suite traceable to PRDTests folder, with PRD ids as test idsRegression prevention
Execution logFeature folder, archived on shipRetrospective, future impl plans
Verdict recordFeature folder, status: shipped/fixed/rolled-backAudit trail for the team
Retrospective notesRetrospective templateImproves PRD, research, impl plan patterns over time

The QA phase's discipline produces organizational learning, not just feature validation. Every (A) test that gets linked to a PRD id is a tiny improvement to the spec-test-contract chain. Every retrospective note that captures what the next impl plan should do better is an improvement to the team's pattern library. The QA phase is where the team becomes the team it will be on the next feature.

Final note on the 7-phase sequence. The seven phases — Idea, Research, Prototype, PRD, Impl Plan, Execution, QA — are not a checklist. They are a system. Each phase's output is the next phase's input. Each phase's failure mode is the next phase's hidden cost. The phases exist to catch errors at the cheapest moment: a missed requirement caught at Idea saves 100x the cost of a missed requirement caught at QA. The team that runs all 7 phases routinely produces better features faster than the team that runs only Execution and QA. The cost of the phases is the budget for catching errors early. That budget is always worth it.
