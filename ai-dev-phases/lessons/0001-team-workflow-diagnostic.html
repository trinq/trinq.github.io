Diagnose Your Team's Workflow Against Pocock's 7 Phases

Close the tab. Recall the seven phases from memory. Then map your team's actual workflow against them. The gap you find is your lesson 0002 anchor.

Source: Matt Pocock — "The 7 phases of AI-driven development", AI Hero / YouTube, 8:26, 2026-03-03.

Manager link. This lesson assumes you are an engineering manager / team lead setting up a reproducible AI-driven workflow for your team. Every exercise is filtered through that lens: not "what's interesting in the video," but "what does my team actually do today, and where does it leak?"

1. Hook — The 7 Phases as a Diagnostic Lens, Not a Checklist

Matt Pocock opens with a claim that sounds simple: "These are the seven phases you should be thinking about for shipping great work." But his point is not "follow seven sequential steps." His point is that any serious AI-engineering workflow traverses all seven phases — whether the implementation is Ralph, GSD, Spec Kit, or a raw loop.

In eight minutes and twenty-six seconds, Pocock names the seven phases every AI-driven development loop passes through. The names are implementation-agnostic — he pins them to no tool. These are the seven phases you need to know cold, and which you need to score your team against:

01
Idea App idea, bug fix, feature, refactor. Any size.
02
Research Cache hard explorations into research.md (external APIs, new dependencies).
03
Prototype Throwaway code to impose taste. Commit it before writing the PRD.
04
PRD Describe the end state — not the implementation. Prompt the agent to grill your decision tree.
05
Implementation planning PRD → Kanban board. Blocking relationships between tickets. Sequential or parallel.
06
Execution Loop a coding agent through each ticket. Ralph loop or human-in-loop. Can run AFK.
07
QA Agent writes a QA plan → human executes → new tickets → loop back into execution.

2. Retrieval — Close the Tab, Recall the 7 Phases

The goal is to force working memory. When you recall the seven phases without the video open, your ability to wield them in next week's retro will be much higher than if you read them five times. That's storage strength, not fluency strength.

Steps

  1. Open paper / Notion / Excalidraw / a whiteboard. Turn the video off.
  2. Draw a left-to-right (or top-to-bottom) flow with seven boxes.
  3. Name each box. Don't peek. Don't search.
  4. Under each box, write one line: "What artifact appears at this phase?" — PR file? Research doc? Throwaway repo? Merged code? QA plan?
  5. Set a timer for 10 minutes.

Check yourself (after drawing)

For each question, write your answer first, then reveal the discussion below.

1. What is the first phase?
A. Idea
B. Research
C. PRD
D. Execution
Reveal answer
A. Idea. Pocock: "This idea can be as small and as big as you like." A bug fix, a feature, a refactor, a whole app — all are ideas.
2. If your team is integrating with a poorly-documented external API (Stripe, an obscure VN vendor), which phase must run before the PRD?
A. Prototype
B. Research
C. Implementation planning
D. Execution
Reveal answer
B. Research. Pocock: "If that exploration is difficult … you'll want to cache it in a research.md asset." Cache the hard exploration so every fresh context window can reuse it.
3. Which order is correct per Pocock?
A. PRD → Prototype → Research → Implementation planning → Execution
B. Idea → Research → Prototype → PRD → Implementation planning → Execution → QA
C. Idea → PRD → Prototype → Research → Execution → QA
D. Idea → Research → PRD → Prototype → Implementation planning → Execution
Reveal answer
B. Pocock is explicit: prototype must happen before PRD so you have concrete feedback to ground the spec. PRD without a prototype is abstract; you end up refining the PRD in code.
4. According to Pocock, the execution phase can run AFK (no human in the loop) when…?
A. Always — agents are good enough now
B. Only when using Ralph, not GSD or Spec Kit
C. Only for bug fixes, never for new features
D. When research + prototype + PRD + Kanban board are already prepared
Reveal answer
D. Pocock: "With all this setup… you can totally run this execution loop AFK, and the results will be really good." The setup is the precondition; without it, execution is hand-holding.
5. How is Pocock's QA phase different from a normal automated test run?
A. QA is just running unit tests in CI
B. QA is reserved for security review
C. Agent writes a QA plan, human executes it, results generate new tickets that loop back into execution
D. QA is owned by PM, not engineers
Reveal answer
C. QA is a loop, not a one-shot gate. Each QA pass produces new Kanban tickets that flow back into execution. Expect to iterate the last three phases several times before the product converges.

3. Anchor — Map Your Team's Workflow Onto the 7 Phases

This is the most important part of the lesson. Don't let the file stop at "Idea → understand the brief." Anchor it to your actual team:

For each phase, answer four questions

  1. Does this phase exist in your team's current workflow? (A real ritual, artifact, or process — or is it happening informally / not at all?)
  2. Who owns it at this phase? (Which engineer? PM? You?)
  3. What artifact gets produced? (Name it specifically: research.md, a prototype repo, a PRD doc, a Kanban ticket, a merged PR, a QA plan.)
  4. Leak point: is this phase running well, running half-heartedly, or being skipped / jumped over?

QA … … … …

Heuristic to spot leaks fast: if an engineer says "I just start coding from the ticket," phases 2, 3, and 4 are being skipped. If a PM says "QA is whatever the engineer does," phase 7 does not exist as a separate ritual. If nobody can name the owner of a phase, it doesn't exist.

5. Ask Back — 4 Questions That Drive Lesson 0002

When you send the file back, include short answers to these four questions. They decide the direction of the entire next batch of lessons.

  1. Which phase is your team currently strongest at? Signal: a place where you already have a real ritual — treat it as a temporary asset, and as a reference example for the lesson on that phase.
  2. Which phase is your team currently weakest at? Signal: the phase where we will focus lesson 0002 — research, prototype, PRD, impl plan, execution, or QA.
  3. Is there a phase your team is entirely skipping? Signal: a place where a new ritual needs to be designed — I can help you draft a research.md template, a Kanban template, or a QA plan template.
  4. Does your PM / CTO know these 7 phases? Signal: do we need to prioritize a leadership memo this week, or defer it? The answer also tells me whether you have an ally in leadership, or whether you are pushing this alone.
