The PRD Phase — Buildable, Not Long

A PRD fails in two opposite directions: too vague to build, or too long to read. The buildable spec lives between them. This lesson covers the granularity ladder, what goes in vs. what stays out, and the non-goals section most teams forget.

Source: Matt Pocock — "The 7 phases of AI-driven development", AI Hero / YouTube, 8:26, 2026-03-03. Builds on lessons 0001–0005.

How to use this lesson. The PRD phase is the first place where vague documentation becomes buildable code. The lesson teaches a precision test for every requirement: does an agent need to know this to build it? If yes, write it. If no, leave it out. Read the granularity ladder first — it answers the question "how long should my PRD be?" Then read the failure modes at the end. Those are the traps your team will hit if the precision test isn't applied.

What Pocock Actually Said

The full quote, in context:

"The fourth phase is the PRD — a product requirements document. So this is where you're actually writing down what the thing is going to do. So this is the moment where you write down what the thing actually does." — Matt Pocock

He says "writing down what the thing is going to do" and "what the thing actually does" on purpose. The PRD captures behavior, not architecture, not implementation, not motivation. The behavior is the contract between the team and the build agent.

The Two Failure Modes

Most PRDs fail in two opposite directions. Recognizing them is the first skill:

Mode 1: Too vague

Symptoms: "the system should be fast and user-friendly." An agent reads the PRD and asks 30 clarifying questions. The build phase becomes a series of guesses. Engineers and agents interpret the same sentence differently. The result is something built, but not what was meant.

Mode 2: Too long

Symptoms: 50-page PRD with edge cases for every conceivable input. The agent drowns in noise. The team stops reading. The PRD becomes a wishlist with no priority. The build phase has too much to consider and ends up implementing the most recent thing the team read.

The buildable spec is between these two modes. "Buildable" means: an agent who knows nothing about your project can read the PRD and start coding without asking clarifying questions. Test: hand the PRD to a fresh agent session. If they ask more than 3 questions before starting, the PRD is too vague. If they can't find the buildable parts in 5 minutes of reading, the PRD is too long.

What Goes In vs. What Stays Out

The hardest part of writing a PRD is deciding what to include. The precision test is the rule:

Include in PRDExclude from PRD
User outcomes — what the user getsImplementation details — how the engineer codes it
Interface contracts — API surface, schema, UX flowInternal refactoring — what stays the same
Edge cases that change the architectureEdge cases that don't change the design
Success metrics — how we know it worksCode style, lint rules, formatting
Non-goals — what we're NOT doingFuture considerations — what might come later
Out-of-scope behaviorsDetailed migration plans for non-related systems

The user-outcome test

Every requirement should be traceable to a user outcome. "The system processes payments." — user outcome. "The system uses a payment processor abstraction layer." — implementation. Implementation goes in the impl plan (phase 05), not the PRD.

The agent-input test

Every requirement should be something an agent needs to know to build it. If an agent doesn't need it, it's noise. "Users will see a 3-second loading spinner." — agent needs to know. "The codebase uses TypeScript strict mode." — agent can read tsconfig.json.

Two Mistakes in Two Places

Most PRD failures are two mistakes in different places:

Premature precision

Specifying things that should be discovered in execution. Example: "The database connection pool size will be 50." The agent doesn't need this upfront; the agent can pick a reasonable default and tune later. Premature precision makes the PRD longer without making the build better.

Late precision

Leaving things vague that should be specified upfront. Example: "The system will authenticate users." The agent needs to know: which method (OAuth? magic link? password?)? What providers? What's the session lifetime? Late precision forces the build phase to make decisions the PRD was supposed to make.

Rule of thumb: precision belongs in the PRD when it changes the architecture, and in the impl plan when it doesn't. If changing the answer would force a rewrite of the implementation, specify it in the PRD. If changing the answer is a simple constant tweak, leave it for the impl plan.

Anatomy of a PRD

Every PRD has a minimum viable structure. The artifact is prd-[feature].md in the feature folder:

1. Summary

One paragraph. What the feature does, who it's for, why it matters. Plain language.

"This feature lets users export their project data to a CSV file. It serves the back-office team who currently copy-paste from the dashboard. It reduces the export workflow from 20 minutes to 30 seconds."

2. User Outcomes

  • The user can trigger an export from the project page.
  • The user receives a CSV file within 30 seconds for projects < 10K rows.
  • The user gets an email link for projects > 10K rows.

3. Interface Contract

  • API: POST /api/projects/:id/export — returns { jobId }
  • UI: an "Export" button on the project toolbar
  • Schema: the CSV must match the existing dashboard columns

4. Edge Cases That Change Architecture

  • Projects > 100K rows must not be exported synchronously (background job required)
  • Concurrent exports from the same project share a job (no duplicate work)

5. Success Metrics

  • 95% of exports < 10K rows complete in < 30 seconds
  • 0 silent failures (user always gets a result or a clear error)
  • Back-office team reports export workflow < 1 minute (subjective, surveyed)

6. Non-Goals

  • We are NOT supporting Excel (.xlsx) exports in v1
  • We are NOT supporting scheduled / recurring exports
  • We are NOT exporting audit logs or user data outside the project

7. Links to Upstream

  • Idea doc: idea-export-feature.md
  • Research: research-csv-export.md
  • Prototype: prototype-csv-export.md

The non-goals section is the most skipped part of a PRD. Without explicit non-goals, scope creep is inevitable. A PRD without non-goals is a wishlist. The non-goals are not "future considerations" — they're active constraints on v1. When stakeholders ask for one of them, the answer is: "in v2, not v1."

When teams write a "user story" but actually mean a PRD, the story grows. When they write a "PRD" but actually mean a spec, the PRD becomes a 50-page document.

Three Team Scenarios

Scenario A — The PRD that was a wishlist

The team writes a PRD for a "User Onboarding" feature. The PRD is 12 pages, with sections on every conceivable aspect: email verification, profile completion, social sign-in, SSO, password recovery, account deletion, GDPR compliance, 2FA, passwordless login, social graph, profile photos, friend lists, group onboarding, cohort tracking, A/B test hooks…

The agent reads the PRD. After 30 minutes, it starts implementing. It picks social sign-in first, because that was the section the team wrote most recently. Six weeks later, the build is half-done, and the team has implemented 30% of the wishlist, including things no one asked for.

What went wrong: The PRD had no non-goals. It had no priority order. It was a wishlist, not a buildable spec. The agent had to pick; the agent picked the most-recently-written section.

Diagnostic: No non-goals section. The PRD included "future considerations" instead of explicit out-of-scope items. Total feature surface was too large for one PRD; should have been split into multiple PRDs.

Scenario B — The PRD that was a user story

The team writes a one-paragraph "PRD" for a "Login" feature: "User can log in with email and password." The agent starts building. The first question: "What's the session lifetime?" Second question: "What happens on failed login — lockout, throttle, or none?" Third question: "Do we support password reset?" Fourth question: "What's the email verification flow?"

Each question forces a sync with the manager. The build takes 3 weeks because the spec wasn't a spec.

What went wrong: The PRD was a user story dressed up as a PRD. The user outcome (log in) was specified, but the interface contract (what the API returns, what the UI shows) wasn't. The agent had to make decisions that the PRD was supposed to make.

Diagnostic: No interface contract section. No edge case section. The PRD was the wrong level on the granularity ladder — L1 (one paragraph) for what should've been L3 (2–3 pages).

Scenario C — The PRD that built

Engineer T writes a PRD for "CSV Export." Following the template:

  • Summary: 1 paragraph, names the user (back-office team), the current pain (20-minute copy-paste), and the goal (30 seconds).
  • User outcomes: 3 bulleted, each testable.
  • Interface contract: API endpoint, UI button, CSV schema. Specific.
  • Edge cases that change architecture: 2 bullets — 100K-row threshold forces async, concurrent exports share jobs.
  • Success metrics: 3 numbers — p95 < 30s, 0 silent failures, subjective survey.
  • Non-goals: 3 bullets — no Excel, no scheduled exports, no audit log export.
  • Links: idea, research, prototype docs.

Engineer T hands the PRD to a fresh agent. The agent reads it, asks 1 question (a clarification on CSV delimiter), and starts building. The build takes 1 week. The result matches the PRD's success metrics. The team can review the diff and the PRD side by side.

Why it works: Specific outcomes, specific contracts, specific edge cases that mattered, specific success metrics, specific non-goals. The PRD is 3 pages. The agent builds in 1 week. The team reviews in 1 day.

PRD and the Phases Around It

PRD ← Prototype

The prototype (phase 03) was the model. The PRD captures what the prototype taught: what's hard, what's easy, what the gotchas are. Without the prototype, the PRD is mostly fiction. The prototype-N.md is the prototype's "decision artifact"; the PRD is the buildable translation of that decision.

PRD → Impl Plan

The impl plan (phase 05) decomposes the PRD into steps. The PRD is what the plan decomposes. The plan is what the agent reads in execution. If the PRD is vague, the plan has to fill in the gaps, and the plan gets too long to verify.

PRD → Execution

The execution (phase 06) is the agent reading the PRD and the plan, then writing code. The PRD is the source of truth. If a contradiction appears between the PRD and the plan, the PRD wins — plans are decomposable, requirements aren't.

PRD → QA

The success metrics in the PRD become test cases in QA (phase 07). "95% of exports < 10K rows complete in < 30 seconds" — that's a test. "0 silent failures" — that's a test. The PRD's success metrics are the QA's acceptance criteria.

The PRD is the contract. Between manager and engineer, between engineer and agent, between build and QA. Every phase reads the PRD. If the PRD is wrong, every phase inherits the error. If the PRD is vague, every phase invents its own interpretation. The PRD is the most-read document in the whole pipeline — so its quality matters more than any other single artifact.

The Lifetime of a PRD

PRDs decay. The PRD for v1 isn't the PRD for v2. The lesson includes a versioning rule:

VersionStatusAction
v1 (initial)ActiveSource of truth for build
v1.1 (post-build patches)ActivePatch the PRD when scope changes mid-build
v2 (post-launch iterations)PendingWrite a new PRD; don't edit v1 in place
v1 (after v2 ships)ArchivedMove to prd-archive/; never edit

Never edit v1 in place once v2 is written. The PRD is a historical record. If you edit v1 after v2 ships, you lose the ability to ask "what did the build actually do?" A year from now, someone wants to know why a feature works the way it does — the answer is in the v1 PRD. Once v2 ships, v1 is a museum exhibit.

The Manager Ritual for PRD

PRD phase ritual — 3 questions before approving

  1. "Is it buildable by an agent?" — hand the PRD to a fresh agent session. If they ask more than 3 clarifying questions, the PRD is too vague.
  2. "Does it have non-goals?" — read the non-goals section. If it's empty or full of "future considerations," the PRD is a wishlist.
  3. "Does it link to the prototype that justified it?" — find the links-to-upstream section. If the prototype is missing, the PRD is fiction.

The manager's hardest question after a PRD. It's not "is it complete?" — completeness is vague. It's "can I read this PRD and tell a junior engineer to build it, and have the result match my expectations?" If the answer is no, the PRD is too vague, too long, or written in the wrong voice. The PRD is the most-read document in the pipeline. Manager review is the highest-leverage use of the manager's time.

Recall exercise: audit your last 3 PRDs

For your last 3 PRDs, answer:

  1. Could a fresh agent build from it without more than 3 clarifying questions?
  2. Did it have an interface contract section?
  3. Did it have a non-goals section with explicit out-of-scope items?
  4. Did it have success metrics that map to test cases?
  5. Did it link back to the prototype / research / idea?
  6. Was it 2–3 pages (L3) or 10+ pages (L5)?
  7. Did anyone read it from start to finish before building?
  8. Did the build match the PRD's success metrics?

Score: each yes = 1 point. Below 4/8 → PRD is leaking into wishlist territory. 4–6/8 → solid. 7+/8 → strong PRD discipline. Whatever the score, the value of measuring is the same.

Most impl plans fail by being either too vague (the agent has to re-decompose) or too prescriptive (the agent can't adapt to discovery). The same precision test applies, but to steps instead of requirements.

Final note on this lesson. The PRD is the first place where vague intent becomes buildable contract. The buildable spec is rare — most PRDs are either too vague (the agent re-asks) or too long (the agent drowns). The 5-level granularity ladder gives the team a calibration. The 8 failure modes give the team a diagnostic. The 3-question manager ritual gives the manager a forcing function. Together, they make the PRD a useful artifact instead of a ceremonial one. The PRD is the most-read document in the pipeline. Make it count.
