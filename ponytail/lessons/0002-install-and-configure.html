Install & Configure Ponytail

Get ponytail into your GitHub Copilot CLI, choose your intensity, and run it on a real task.

Ponytail is not a CLI tool you run. It's a skill that rides inside your AI agent. Once installed, you call it with slash commands. The agent applies the ladder every time it touches code.

Step 1 — Add the Ponytail Marketplace to Copilot

1
Open Copilot CLI

Run copilot in your terminal to start the interactive CLI session.

2
Add the marketplace
/plugin marketplace add DietrichGebert/ponytail

This registers the ponytail repo as a plugin source.

3
Install the plugin
/plugin install ponytail@ponytail

Reload the CLI if prompted. You should now see /ponytail and /ponytail-review in your command list.

If you don't see the commands: Run /help and search for "ponytail". If still missing, restart the CLI and try again — sometimes the marketplace doesn't refresh until a new session.

Step 2 — Pick Your Intensity

Ponytail is not binary. You choose how aggressive the ladder is. Match the intensity to the task and your familiarity with the codebase:

/ponytail lite    # rung 1 (YAGNI) + rung 6 (one-line) only
/ponytail full    # all 7 rungs — the default
/ponytail ultra   # all 7 + extra skepticism on abstractions
/ponytail off     # turn it off; agent behaves normally

Recommended starting point: Begin with full on a small, low-risk task. If the agent refuses to do something you'd normally do, drop to lite. If you find the codebase already has a history of over-engineering, escalate to ultra.

Intensity can be changed mid-session. /ponytail off is your escape hatch if the agent gets stuck on a rung that doesn't apply.

Step 3 — Run a Real Task

Pick a small piece of work from your queue. The first time you use ponytail, the task should be:

  • Bounded — one file or one component, not a whole feature
  • Low-risk — if you need to throw away the result, you won't lose sleep
  • Concretely spec'd — the user stories are clear, no ambiguity about "done"

Why these constraints? Because the first run is mostly about calibrating your trust in the skill, not testing the skill itself. You need a win you can inspect.

The pattern that works best: Combine ponytail with `/prd` and `/issues` from Matt Pocock's skills directory. The flow:

  1. Write a PRD (captures the destination)
  2. Break into vertical-slice issues (plans the journey)
  3. Implement each issue with ponytail active (writes minimal code)
  4. Review with /ponytail-review before merge (catches over-engineering)

Step 4 — Review the Diff

Once the agent finishes writing, do not merge yet. Run the review command:

/ponytail-review

Ponytail returns a delete list — specific lines it would cut, with the rung each deletion satisfies. You accept or reject each item. This is the moment to apply your judgment:

  • If the deletion violates a "never cut" rule (validation, error handling, security, accessibility), reject it.
  • If the deletion is genuine over-engineering the agent wrote, accept it.
  • If you're unsure, leave it for now — review the diff yourself before deciding.

The review is interactive. Treat it as a conversation, not a CI gate.

The Five Commands You Will Actually Use

Most of the time, four commands cover 95% of work. The fifth is for audits:

Command When to use
/ponytail full Start of session — turn the ladder on
/ponytail-review Before merging any change
/ponytail off When the agent is stuck on a rung that doesn't apply
/ponytail-debt When you accept a shortcut ponytail flags — log it
/ponytail-audit One-time scan of an unfamiliar repo to see if the codebase has an over-engineering pattern

What This Lesson Did Not Cover

The README has more commands (/ponytail-gain for scoreboards, /ponytail-help for inline reference). You will discover those once you have run a few real cycles. The five above are the ones that compound.

Also not covered: using ponytail on Cursor. The skill is documented for Claude Code and Copilot CLI. Cursor support is possible via rules but unverified at time of writing.
