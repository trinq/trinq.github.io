Apply the Ladder

Three real before/after diffs and what each rung catches.

You installed ponytail. You set the intensity. Now: what does it actually change in the code?

This lesson shows three realistic examples from feature tickets. Each one illustrates a different rung catching something the baseline agent wrote. Your job is to feel the pattern — not memorize rules.

Critical reminder: The ladder runs after the agent understands the problem. It does not replace reading the PRD, the codebase, or thinking. It says: "Now that you know what to build, write only what is necessary."

The 7 Rungs

1. Does this need to exist? YAGNI
2. Already in this codebase?
3. Stdlib does it?
4. Native platform feature?
5. Installed dependency?
6. One line?
7. Only then: the minimum that works

Example 1 — Rung 4: Native Platform Feature

"Add a date picker to the form."

Rung 4 Date picker component
Baseline
npm install flatpickr
import Flatpickr from 'flatpickr';
import 'flatpickr/dist/flatpickr.css';

const DatePicker = ({ value, onChange }) => {
  useEffect(() => {
    fp = new Flatpickr(domNode, {
      dateFormat: 'Y-m-d',
      altInput: true,
      altFormat: 'F j, Y',
    });
  }, []);
  return <input ref={register} />;
};

54 lines including config, lifecycle, and CSS import

With ponytail
<input type="date" />

Rung 4 fires: the browser already ships a date picker. The HTML spec solved this problem in 2012.

Example 2 — Rung 1: YAGNI

"Add user roles to the auth system."

Rung 1 User roles feature
Baseline
enum Role { Admin, Editor, Viewer, Guest,
  Moderator, Owner, Contributor, Subscriber };

type Permission = 'read' | 'write' | 'delete' |
  'manage_users' | 'manage_billing';

const rolePermissions: Record<Role, Permission[]> = {
  Admin: ['read', 'write', 'delete', 'manage_users',
          'manage_billing'],
  // ... 7 more entries
};

export function canAccess(role: Role, perm: Permission) {
  return rolePermissions[role].includes(perm);
}

Baseline enumerated 8 roles and 5 permissions before the ticket even mentioned what the app does

With ponytail
// Only admin vs non-admin for now
const isAdmin = (user) => user.role === 'admin';

Rung 1 fires: "Only admin and non-admin were mentioned in the ticket. Guest, Subscriber, Moderator, Owner — none of these appear. Skip them."

Example 3 — Rung 5: Installed Dependency

"Validate email input."

Rung 5 Email validation
Baseline
npm install validator
import validator from 'validator';

export const validateEmail = (email: string) => {
  if (!email) return false;
  if (!validator.isEmail(email)) return false;
  return true;
};

Dependency added for a single regex

With ponytail
// Checked: validator is installed
import { isEmail } from 'validator';
export const validateEmail = (email: string) =>
  isEmail(email);

Rung 5 fires: validator is already in package.json. Reuse the import instead of checking if it exists first.

What the Ladder Never Cuts

The ladder reduces code by cutting unnecessary abstractions, redundant dependencies, and features that aren't specified. It does not reduce safety.

Ponytail never removes:

  • Trust-boundary validation — any code that guards against external input or untrusted sources
  • Data-loss handling — error states, undo paths, confirmation dialogs for destructive actions
  • Security controls — auth checks, input sanitization, rate limiting, RBAC enforcement
  • Accessibility — ARIA labels, keyboard navigation, focus management

The agent with ponytail is not a lazy agent. It is a disciplined one. Discipline means writing exactly what the task requires — no more, no less.

Exercise: Find Over-Engineering in Your Own Work

Find one piece of code in your current project that you wrote or that an agent wrote recently. Apply the ladder mentally:

  1. Read the task that produced this code (PRD, issue, or your own description). Did the code do exactly what was asked?
  2. Scan the imports. Are they all used? Is there a package.json dependency that does one thing you could replace with stdlib?
  3. Count the functions. Are there abstractions that aren't called from anywhere except themselves?
  4. Run /ponytail-review on the diff. How many of the flagged deletions would you accept?

Expected result: Most over-engineering comes from two sources — agents (or developers) anticipating requirements that were never in the spec, and installing libraries for single-use cases. The ladder catches both.

What You Are Calibrating

The first few times you run ponytail, you will feel uneasy. The agent writes less code than you expect. You will wonder: "is this really enough?"

This is the calibration. The uneasy feeling is you noticing the gap between "what I usually write" and "what the task requires." That gap is the cost of over-engineering — and it compounds across every feature, every ticket, every month.

The ladder shrinks the gap. Trust the process for 10 tickets before you judge it.
