Combined Workflow — Full Pipeline

Grill → PRD → Issues → Implement → Ponytail → Review. Step-by-step with real commands and outputs.

This lesson walks through one complete feature request using the full combined workflow. Each tool fires at a specific point, produces a specific output, and hands off to the next tool. You see exactly what enters and exits every phase.

The example task: "Add a dark mode toggle to the nav bar."

Grill → PRD → Issues → Implement → Ponytail → Review

Phase 1 — Grill

The user says: "Add a dark mode toggle to the nav bar."

Grill fires. The first question is never "how" — it's "why" and "what exactly."

1
Grill — Clarify the problem
/grill-me

Q: Dark mode toggle. For who? Always needed?

A: For users who prefer dark. Most apps auto-detect from OS. Manual toggle exists in settings, buried 3 clicks deep. This is about surfacing it.

Q: What's the goal — accessibility or preference?

A: Preference. But auto-detect + manual override is the standard pattern. We should match that.

Q: Does the codebase already have dark mode?

A: Yes — Tailwind uses class="dark" on the root element. System preference is already read via prefers-color-scheme.

Q: Is the toggle only in nav, or across the app?

A: Nav only. The theme applies app-wide.

Q: What happens to the user's choice?

A: Stored in localStorage. Overrides system preference until user toggles again or clears storage.

Grill output — decision tree resolved
  • Toggle is a shortcut to what already exists in settings
  • Codebase already has theming infrastructure (Tailwind dark mode)
  • User choice persists in localStorage
  • Not accessibility-driven — no WCAG constraint
  • Nav only — no cross-app changes

Grill prevents the most expensive mistake: building the wrong feature. Here, it reveals the destination is not "build dark mode" — it's "add a shortcut to existing dark mode in settings."

Phase 2 — PRD

Grill output feeds directly into PRD. No extra conversation needed — the decisions are already made.

2
PRD — Document the destination
/write-a-prd
PRD Output
4 sections, under 30 lines total

Problem

Users who prefer dark mode must navigate to Settings → Appearance → Theme to switch. The current path is 3 clicks. A quick-toggle in the nav bar reduces this to 1 click.

Solution

Add a sun/moon icon toggle in the top nav bar. Clicking it cycles between light and dark. The icon changes to reflect current state. The choice persists in localStorage and overrides the system preference.

User stories

  • As a user, I see a toggle in the nav bar, so I can switch theme in one click.
  • As a user, the toggle icon reflects my current theme, so I know what state I'm in.
  • As a user, my choice persists across sessions, so I don't have to toggle every visit.

Constraints

  • Use existing Tailwind dark mode setup — do not modify theme config.
  • Use existing localStorage pattern for persistence — do not add a new state library.
  • Do not add any new dependency.

PRD is 30 lines. Three constraints are explicit:

  1. Use existing Tailwind dark mode — ponytail rung 2 (reuse from codebase) will fire if agent tries to reconfigure
  2. Use existing localStorage pattern — rung 4 (native platform) fires if agent adds a state library
  3. No new dependency — rung 5 fires if agent proposes installing something

The PRD is the contract. Ponytail enforces it. Grill made the decisions. Now the implementation can be lean because the destination is clear and the constraints are explicit.

Only 2 slices. Issue #46 is the UI. Issue #47 is the persistence. Both are small enough to finish in one session. Both are independent — the toggle works with hardcoded theme in #46, persistence is a separate concern added in #47.

Phase 4 — Implement

Pick up issue #46. Run with ponytail active.

4
Implement — Agent with ladder active
/ponytail full

Prompt: "Implement issue #46: add theme toggle icon to top nav bar. Use existing Tailwind dark mode. Toggle cycles light ↔ dark."

Agent proposes → Ladder intercepts

- const [theme, setTheme] = useState('light');
- useEffect(() => { document.documentElement.classList.toggle('dark'); }, [theme]);

Agent proposed a state variable and effect to manage theme. Rung 2 fires — the codebase already has a useTheme() hook in hooks/useTheme.ts:

// Already exists in codebase — rung 2
import { useTheme } from '@/hooks/useTheme';

// Replace 2 files of state management with 1 hook call
const { theme, toggle } = useTheme();
- // TODO: install react-icons

Agent suggested installing react-icons for sun/moon icons. Rung 5 fires — Tailwind has built-in icon utilities. Or use inline SVG:

// Rung 5 — use built-in or inline SVG, not a library
// Tailwind uses: Heroicons, Lucide, or inline SVG
// Inline SVG is 1 line — rung 6
const SunIcon = () => (
  <svg className="w-5 h-5"><circle cx="12" cy="12" r="4" fill="currentColor"/></svg>
);
- // Create ThemeToggle.tsx component - // Add ThemeToggle.css stylesheet - // Export from index.ts

Agent proposed creating a separate component file. Rung 7 fires — keep it inline in Nav.tsx for now. It is a detail of the nav, not a reusable component. "Do you want to promote it later? Yes. But not now."

Rungs that fired in this session:
Rung 2 useTheme hook from codebase Rung 5 react-icons rejected — inline SVG Rung 7 separate component rejected — keep inline

Final output — 9 lines added to Nav.tsx

// Inside the nav, next to the user avatar:
const { theme, toggle } = useTheme();

<button
  onClick={toggle}
  aria-label={`Switch to ${theme === 'dark' ? 'light' : 'dark'} mode`}
  className="p-2 rounded hover:bg-gray-100 dark:hover:bg-gray-800"
>
  {theme === 'dark' ? <SunIcon /> : <MoonIcon />}
</button>

Before ponytail: Agent would have created ThemeToggle.tsx + useThemeState.ts + ThemeToggle.css + installed react-icons. ~80 lines, 1 new dep.

After ponytail: 9 lines inline in Nav.tsx, zero new deps, zero new files.

Where Each Tool Fits

Tool Question it answers Output Cost of skipping
Grill Are we building the right thing? Decisions made, assumptions tested High — wrong feature built
PRD What exactly is the destination? Problem + Solution + Constraints written High — scope creep, missing constraints
Issues How do we slice this? Vertical slices, each deployable Medium — horizontal layers hard to cut
Ponytail (active) Is this line necessary? Rejected proposals, ladder applied Medium — over-engineered code
/ponytail-review Did anything slip through? Delete list Low — catches misses, but prevention is better

Exercise

  1. Pick one feature from your current sprint.
  2. Run /grill-me before writing the PRD. Write down every question it asks. Answer honestly.
  3. Write the PRD. Make the constraints explicit — what should the agent not touch?
  4. Break into vertical slices. Each slice should be deployable on its own.
  5. Run /ponytail full during implementation.
  6. Run /ponytail-review before opening the PR.
  7. Record: How many lines saved? Which rungs fired? Did the PRD change?

The first ticket will feel like a lot of steps. By ticket 5, the workflow compresses. The grill questions become instinct. The PRD constraints become automatic. Ponytail becomes a reflex.
