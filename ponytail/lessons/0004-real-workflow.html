Ponytail in a Real Workflow

From PRD to merged PR — the complete cycle with ladder logs and review results.

This lesson walks through a complete feature ticket from start to finish. You will see the PRD, how it breaks into issues, how the agent applies the ladder during implementation, and what /ponytail-review catches before merge.

The task: add a notification badge to the user avatar in the nav bar. A small feature. A realistic one.

Step 1 — The PRD

Before writing any code, the team writes a PRD. This captures the destination, not the implementation:

PRD Add notification badge to user avatar

Problem

Users miss notifications because there is no visual indicator on the nav bar. They have to click the avatar to check.

Solution

Add a red dot badge to the user avatar in the top nav. The badge shows the unread count (1–9, or "9+" for more). It disappears when the count is zero.

User stories

  • As a logged-in user, I see a badge on my avatar when I have unread notifications, so I know to check them.
  • As a logged-in user, the badge shows the exact count (1–9) or "9+" when there are more.
  • As a logged-in user, the badge disappears when I have no unread notifications.

Constraints

  • Do not change the notification API — use the existing /api/notifications/count endpoint.
  • Do not add a new dependency.
  • Badge must be accessible: announce count changes to screen readers.

Notice what the PRD does not say: what CSS framework to use, what animation library, whether to store the count in local state or a global store, how many files to create.

Step 2 — Break Into Vertical Slices

Using the prd-to-issues pattern from Agent Loop Skills, the PRD breaks into 4 slices. Each slice is end-to-end working code — not a horizontal layer:

#42 Show badge with unread count Do first

Badge component. Shows 1–9 or "9+". Calls /api/notifications/count on mount. Accessible. Hidden when count is 0.

#43 Place badge on avatar in nav Parallel to #42

Position the badge on the existing avatar in the top nav. Use CSS absolute positioning. No new component files.

#44 Wire to notification API Blocked by #42

Replace mock data in #42 with real /api/notifications/count call. Add loading and error states.

#45 Poll for new notifications Blocked by #44

Add polling (every 60s) to keep badge in sync. Stop polling when tab is hidden.

Step 3 — Implement with Ponytail Active

Each issue is picked up by the agent with /ponytail full active. Here are the ladder logs for issue #42 — the badge component:

Rung 4 — Native platform feature
Proposed: install a badge library (react-hot-toast, react-badge)
Fired: position: relative on avatar container + position: absolute badge is native CSS. No library needed.
Rung 3 — Stdlib
Proposed: custom number-clamping utility for "9+" display
Caught: Math.min(count, 9) + (count > 9 ? '+' : '') — pure JS, no utility file needed.
const display = count > 9 ? '9+' : String(count);
Rung 1 — YAGNI
Proposed: add animation when count changes (bounce effect)
Cut: "bounce on change" was not in the PRD. Rung 1 fires.
Rung 7 — Minimum that works
Proposed: new file components/NotificationBadge/Badge.tsx with separate stylesheet
Accepted: kept inline in Nav.tsx for now — the badge is a detail of the nav, not a standalone component.

The Result

Issue #42 — badge with count — delivered in 18 lines total. Baseline agent (from benchmark): 94 lines across 4 files, 2 dependencies.

// Rung 7 — kept inline in Nav.tsx
const display = count > 9 ? '9+' : String(count);

// Rung 3 — stdlib Math, no utility file
// Rung 4 — native CSS position, no badge library

{count > 0 && (
  <span aria-label={`${count} unread notification${count > 1 ? 's' : ''}`}>
    {display}
  </span>
)}
// Total: 18 lines including the conditional render

What Never Changed

  • The PRD destination stayed the same. Ponytail trims the implementation, not the goal.
  • Accessibility code stayed. The aria-label is in — it was never flagged because it's a "never cut" item.
  • The /api/notifications/count endpoint was used. Constraint from the PRD — respected by the agent.
  • Tests stayed. Ponytail does not touch tests. Tests are the safety net.

Exercise: Run It on Your Next Ticket

  1. Pick one feature ticket from your current sprint. It should be small enough to finish in one session.
  2. Write the PRD first — not for your boss, just for yourself. 4 sections: Problem, Solution, User stories, Constraints. Under 20 lines.
  3. Run /ponytail full and implement the ticket.
  4. Before opening the PR, run /ponytail-review. Accept the deletions. Reject anything touching accessibility, validation, or error handling.
  5. Record: how many lines did you save? Which rung fired most? Did the PRD stay the same?

The first 3 tickets will feel slow. You are calibrating. By ticket 10, the ladder will feel like a reflex — the agent stops proposing things the ladder would cut before you even notice.
