ponytail Lesson 0007 2026-06-28

The Matt Pocock Formula

Five ideas from a one-hour talk that reframe how a solo developer should think about AI tools — and which one to act on first.

Source: Matt Pocock's Agentic Engineering Workflow (just copy him), David Ondrej podcast, 1h02m, 224K views, 2026-06-18.

Mission link. This lesson assumes your mission is ship faster solo. Every idea is filtered through that lens: not "what's interesting," but "what changes how much you ship per week."

Part 1 — Stop Optimizing the Engine

Matt opens the talk with a framing that runs through everything else. He says it twice — once at the very start, once when David asks about his setup — almost word-for-word:

"Everyone's obsessed with the engine of the Formula 1 car whereas in fact the engine is really only a part of the whole system. You've got the entire chassis. You've got how it moves through the air. Everyone's obsessed with the model and I think they should be more interested in the harness, what you can do to get the most out of the harness. Giving it the right prompts, giving it the right skills to work with and improving the environment in which the model runs, improving the codebase and all that stuff. So yeah, the model is useful, but I think the harness has an equal amount of work and you have much more control of the harness than you do the model."

Four levers in that quote, in order of how much you control them:

  1. Prompts — fully yours.
  2. Skills — fully yours (you write them, you install them).
  3. Harness — partially yours (sandbox, CI, GitHub Actions, parallel agents).
  4. Model — least yours (you pick from what labs ship).

Matt's reasoning is brutal: "if you overoptimize around a model, if you get too focused on the model, you will lose focus on the fundamentals." He names this trap explicitly — it's the bitter lesson fallacy. The bitter lesson says compute will always win, so just wait. Matt's counter: waiting is stupid, but the inverse is also stupid. Don't model-hop. Don't believe the one-shots on Twitter. Don't patch your prompts because Fable behaves differently from Opus 4.8.

What he actually does every day, in his words: "figure out, okay, should I be using tmux here, should I be using a VPS, should I be using Tailscale here — trying to actively improve everything else except for the model."

Why this matters for shipping faster

Every hour you spend chasing "is Opus 4.9 better than Fable 1.2" is an hour not spent making your codebase easier to change. Matt's claim is testable: "if your codebase architecture is better, then you can get a cheaper model to do the same work because your guard rails are better, it's easier to explore, it needs to spend fewer tokens banging its head against the wall. If you're hamstringing your model from day one, then you will need a smart model to get the most out of it."

Translation for a solo dev:

  • Don't change tools this week. Whatever you have — keep it.
  • Change one lever you control. Pick one: prompts, skills, harness, or codebase. Improve it for a week.
  • Measure with a real ticket. Not vibes. Pick a real ticket from your backlog, ship it, count lines/tokens/PRs.

The trap to avoid. David calls it "the quintessential vibe coder who is switching tools every single week." Replit agent → Lovable → this → that. Constantly switching and never learning any programming principles. That path looks productive but produces nothing. The opposite path — boring tools, good codebase, AI running faster — compounds.

Exercise: One lever, one week

  1. Pick the lever you currently neglect most. For most solo devs, it's codebase.
  2. Pick ONE change that takes 30 minutes. Examples:
    • Add a single file docs/agents/architecture.md describing module boundaries.
    • Extract one tiny pure function out of a 200-line component.
    • Add one missing test on a public interface.
    • Install or update one skill in your harness.
  3. Ship a real ticket using that change. Note: how many fewer clarifying questions did the agent ask? How many fewer tokens spent?
  4. Don't switch models. Don't switch editors. Just measure that one lever.

Pass criterion: if the change made no measurable difference, the change was wrong, not the strategy. Try another change in the same lever. If the strategy made no difference after 4 changes, switch levers.

Part 2 — Tactical Programming Is Eaten

The single most important reframe in the talk. Matt sets it up via John Ousterhout's A Philosophy of Software Design distinction between tactical and strategic programming. Direct quote:

"Tactical programming is all about the on-the-ground day-to-day stuff — the actual writing of the code, the actual messing about with the syntax, figuring out bugs as they come up, and actually creating the code, creating the commits. Strategic programming is winning the war, not the battle. It's longer-term thinking. It's the general sitting right at the top. How does the codebase need to look? What strategies can I use to improve our velocity?"

Then the punchline Matt is famous for in this talk:

"AI has basically eaten tactical programming. It's gone, right? It's all gone. AI is just better at doing tactical programming than you are because it can do it for cheaper. And so you need to be great at strategic programming in order to get the most out of this infinite fleet of tactical programmers that you now have access to."

"Infinite fleet of tactical programmers." That's the frame. Not one assistant — a fleet. Each one is a junior that you, the senior, scope, brief, and review. Your job stopped being "writing code" on the day you wired up an agent. Your job is now "designing the conditions under which the fleet writes code that doesn't make me regret it."

What strategic programming looks like in 2026

Matt lists what strategic programming means now:

"You need to design the hard parts up front. You need to make sure those tasks are really, really well scoped. You need to be thinking about the interfaces between all of the modules in your codebase. You need to be thinking about test scenes and good tests. You need to essentially design a codebase that's easy to work in and have just enough documentation that can point AI to the right places where it's going to make those changes and make them effectively."

Notice what he doesn't list: TypeScript tricks, regex wizardry, knowing which Redis method to call. Those are tactical. The strategic list:

  • Design the hard parts up front — PRD work, not coding work.
  • Scope tasks well — /to-issues work, not coding work.
  • Think about interfaces between modules — architectural work, not coding work.
  • Design good tests — specification work, not coding work.
  • Document enough for AI to find the right places — harness work, not coding work.

Ship-faster implication. The five things above are what multiplies your output. The agent can write code 24/7. The bottleneck is no longer code — it's decisions about scope, interfaces, and tests. If you're a solo dev, your job is to spend less time coding and more time on these five levers.

Exercise: Audit your last week

  1. Pull up your git log from the last 7 days.
  2. For each commit, tag it T (tactical: bug fix, syntax tweak, type errors) or S (strategic: new module, refactor, test scaffolding, docs).
  3. Count: what percentage of your time went to T?
  4. If T > 50%: those tasks should be going to the agent. Hand them the next time they come up.
  5. If S < 30%: you're not actually programming strategically yet. Pick one of the five levers above and ship something for it this week.

Part 3 — Skills Are Procedures, Not Abilities

Around the 18-minute mark, David asks: "What separates a good agent skill from a bad one?" Matt's answer reshapes how you should think about every skill you install.

"You can think of there as being two types of skills. There are skills that are procedures — skills that you intend to run yourself — and there are skills that are more like abilities — things that you intend the model to invoke itself. [...] A good ability for instance might be your coding standards. So let's say your agent is doing its own thing, working along, and it needs to check how you like your React code written. So it's going to pull in the ability, great React coding standards, and then it reads it and it understands: okay, I shouldn't use useEffect, I should use something different. A procedure is more like something you invoke yourself to get the model to behave a certain way."

Then Matt reveals his preference:

"I tend to prefer my skills as procedures. I like to be the one in control. I like to go, okay, we'll do grill-me and then we'll go, let's write a product requirements document. So we use two — PRD for instance. Then let's take that PRD and turn it into individual issues so that we can work through them. That's just personally how I like to do it."

And a critical warning about abilities:

"Skills are really hard to write because every single skill that you write leaks a description into the context window. And you can disable this. So there are some skills in here — I think in my engineering zoom out — which has disable model invocation true. This skill can only be invoked by the user, and this means its description is not leaked into context. Every single ability — let's say we have 100 different skills — you're going to be leaking 100 descriptions into the context window."

The context window is expensive real estate. Every ability you install without disable_model_invocation takes up space whether it fires or not. Matt's fix: "I prefer to hide most of these descriptions from the AI itself and keep all of that knowledge inside the human, right, inside the developer."

Three dimensions of a skill

Dimension Procedure Ability
Who invokes it You (the human) The model itself
Context window cost Zero until invoked Always present
You control the flow Yes — you're driving No — model decides when
Matt's recommendation Preferred Use sparingly

There's a third dimension: stateless vs stateful. A stateless skill doesn't need to remember anything between runs. A stateful skill — like /teach — saves state locally so it can track your progress across sessions.

Applying this to your setup. The skills you installed because someone said "this is amazing" — go check their descriptions. Are they leaking into your context window right now? Matt's action at the end of the talk: "Delete every single skill, every single plugin, every single MCP server. Go back to absolutely nothing, and then observe the agent. See what it does. Once you're seeing what the agent does in that basic mode, then layer things on top of it — and make sure those things are procedures, procedure skills, not ability skills."

Exercise: Audit your skills for context leak

  1. List every skill currently active in your editor.
  2. For each one, check: does it have modelInvocation: false (or equivalent)? If not, it leaks into your context window always.
  3. Count: how many descriptions are currently in your context window without you invoking them?
  4. Delete the ones you haven't explicitly used in the last 7 days.
  5. For each remaining skill, tag it P (procedure) or A (ability). Aim for zero A skills unless you can justify the context cost.

Part 4 — The 10 Consequential Decisions

David proposes a pattern he's discovered for working with AI. Matt validates it immediately. This is the most directly applicable idea in the entire conversation for a solo dev building something real:

"The biggest difference instead of saying one-shot this app — I describe my vision for this app and say: list out the 10 most consequential decisions — the software design decisions, architectural decisions, product decisions that will shape this project. And ask — interview me — until you understand 98% about it."

Matt's response: "Absolutely." No qualification.

This is the grill-me pattern in practice. Matt describes it:

"The grill-me skill — this is under productivity under grill-me — it's incredibly short. I think this skill maybe five sentences. And it's unreasonably effective because it just turns the agent into an adversarial interviewer asking you questions, interviewing you, and popping up with ideas that you might not have considered — until you reach a shared understanding."

Five sentences. That's what it takes to turn any agent into a first-principles interviewer. The goal isn't to build faster — it's to not build the wrong thing. Because once you start building, the cost of changing a consequential decision is high. The cost of an adversarial question before you write a line of code is zero.

What "10 most consequential decisions" actually means

In a real project, the list looks something like this — not code decisions, but commitment decisions:

  • Who is this for, really? (Not "users" — a specific person with a specific problem.)
  • What does done look like? (Not "shipped" — a specific outcome the user notices.)
  • Monolith or separate services? (For a solo dev: probably monolith.)
  • Database now or later? (And: which one, and why?)
  • Auth from day zero or gate it? (Skip auth until you need it.)
  • How many environments? (Dev + prod = minimum viable.)
  • Public or private? (Private = fewer security constraints.)
  • Who pays for infrastructure? (Solo dev = one place, minimize options.)
  • What breaks if I'm wrong? (Measure: reversibility.)
  • What does version 2 look like? (If you can't answer this, you haven't thought hard enough about v1.)

Most of these aren't code. They're conversations. And conversations are what /grill-me does.

Why this compounds for solo devs. You don't have a co-founder to push back. You don't have a CTO to say "wait, have you thought about X?" The agent running /grill-me is the pushback. It asks the uncomfortable question you would have gotten in a code review — before the code exists. The earlier the pushback, the cheaper the fix.

Exercise: Run grill-me on your next feature

  1. Before writing any code for your next feature, invoke /grill-me.
  2. Paste your feature description. Let the agent interview you for 5–15 minutes.
  3. After the interview, write down: what did you change in your mental model? What decision did you revise?
  4. Then: did you feel "interviewed" or "interrogated"? If interrogated — the skill needs a better prompt from you. Personalize it.

The metric. Track how many consequential decisions changed after the grill-me session vs how many changed after implementation started. If grill-me caught decisions before coding, the cost was ~0. If changes happened after coding, the cost was tokens + time + context switching.

Part 5 — Queues, Not Loops

The conversation shifts to agentic loops. Matt cuts through it with characteristic clarity:

"What we're essentially talking about here is the difference between human in the loop work and AFK work. Human in the loop — the human is there with the agent, talking together, figuring something out. Really useful for planning, really useful for some kind of more complicated implementations, really useful for unscoped work. And then we're talking about AFK stuff — away from keyboard — you ping off the agent and it goes and does something."

The moment Matt discovered AFK was, in his words, "the moment I really got into AI coding and the moment I was really able to massively increase my output because then instead of me having to sit in the loop, handle all the permissions requests, handle anything the agent needs to ask me — the moment I can just remove myself from the equation, I've paralyzed myself. Suddenly there are two of me. Or three of me. Four of me. Five of me. Able to go and produce so much more code that I then go and review."

Why loops are the wrong mental model

The hype uses "loop" language: "run your agent in a loop", "while loop with claude code", "the Ralph loop." Matt's reframe:

"I don't really need to run this as a loop, right? The only thing I need out of this is the AFK agent to take on a specific task and do that task. The way I mostly think about these things is as queues. Okay? Cues — not loops. The Q is really the backlog of tasks that I need to complete."

Queues are GitHub issues, Linear tickets, Notion tasks. Each item:

  • Has a clear scope (one thing, not "run forever")
  • Can be picked up by anyone (human or agent)
  • Produces a done/not-done result
  • Can be parallelized across multiple agents
"This isn't a loop really — it's just a queue that eventually gets resolved. This will come off the queue once the pull request gets merged. And that's all development is really. You just have a queue of tasks that you need to get done. Project managers add more stuff to the queue. You complete the tasks in the queue. And there are multiple nodes picking stuff off the queue, multiple developers. And so an idea that there's a single loop that just sort of goes and completes all the tasks doesn't really match with how developer teams generally work."

The queue-based workflow in practice

Backlog (your task list)
├── Feature ideas (scoped enough to be an issue)
├── Bugs (triage: critical / can-wait / wont-fix)
├── Experiments (bounded time box: "try X, report back")
└── Reviews (PRs, architecture decisions)

When you pick something up:
  - Needs a human in the loop → do it with agent, together
  - Scoped, done-able, no judgment needed → send AFK
  - Blocked → don't pick it up

Implication for shipping faster. Your bottleneck isn't code generation. It's scoping. If you can't write a GitHub issue that an AFK agent can pick up and complete without asking you follow-up questions, the issue isn't ready. The skill of shipping faster is the skill of writing good issues, not the skill of writing good code.

Exercise: Convert your task list to a queue

  1. Take your current task list (Notion, text file, your head).
  2. Write each item as a GitHub issue: one sentence title, 2–3 sentence description, acceptance criteria.
  3. For each issue: can an AFK agent complete this without asking you questions? If no — what's the blocker? (Usually: not scoped enough, or requires judgment you haven't encoded.)
  4. Send one AFK agent on one well-scoped issue. Track: how long did it take? How many clarifying questions? What was the result?

Pass criterion. If the AFK agent came back with more questions than answers, the issue wasn't scoped. Rescope it. If it came back with done work, the queue model is working — scale it up.

Part 6 — Knowledge, Skills, and Wisdom

Matt introduces a framework that quietly explains why some skills are learnable and some aren't:

"I think of there as being three things that you need to be good at anything. You need knowledge — you need the fundamental, what is that thing, understanding it in your head. You need the skills — you need to be able to have done it a bunch of times, in muscle memory. And then you need wisdom — you need to know when to do it. You need to know how it fits in the real world. And wisdom is almost impossible to obtain without actually having done the thing in the exact context where you need to do it."

This is the most clarifying thing Matt says about what can and cannot be delegated to AI. Knowledge and skills can be bundled into procedures. Wisdom? Wisdom requires context. It requires having been wrong before in exactly that situation.

His example is sharp: "if you want to be like someone at Anthropic, sure you can gain the knowledge, you can gain the skills, but then how are you going to gain the wisdom? Like you need to probably go to Anthropic to gain the wisdom to actually understand how to do the thing."

Triad for the solo dev

Type Can AI help? Must you build it?
Knowledge Yes — instantly No
Skills Yes — with practice Yes
Wisdom Only from real context Yes — through shipping

The implication: don't spend time acquiring knowledge. The agent has it. Spend time building skills and, over time, accumulating wisdom. The fastest path to wisdom is shipping things and watching what breaks.

Ship-faster implication. Every ticket you ship is a wisdom unit. Not code — wisdom units. The agent does the knowledge work and the skill work. You accumulate the wisdom: knowing when to trust the agent, when to push back, when a decision will cost you later, when to say no to a feature. That's the ceiling that Matt talks about.

Exercise: Map your triad

  1. Pick your current project.
  2. List 3 things you know (knowledge) — you could explain them to someone.
  3. List 3 things you can do without thinking (skills) — muscle memory.
  4. List 3 decisions you made that turned out right or wrong — and why (wisdom).
  5. For each item: where did it come from? Book/course/video? Practice? Real experience?
  6. Your goal: convert knowledge to skills through deliberate practice. Convert skills to wisdom through shipping. The agent handles the knowledge layer — make sure you're spending your time on the layers that compound.

Part 7 — Build Self-Improving Systems

David shares a story about Fable finding a security bug he had missed for months. His instinct: "that means AI is more involved." Matt's response is the most quietly radical thing in the talk:

"What you've actually learned? Sure, you've learned that Fable is good definitely, but you've also learned that there are security issues in your code, right? And you should probably have something that runs and checks for more security issues in the future. We need to build loops into our systems. You need to figure out why it happened. Why did that even get to this place? You know, it's like if someone keeps stealing your bike, maybe buy a lock."

The question isn't "is Fable better than Opus?" The question is: "why did I have this bug for so long and not notice it?" That question has a systematic answer — and the fix is systematic too. Not a new model. A new process.

"We need to be designing systems that are self-improving over time, right? And this is something that we've been doing as software engineers for a long time. We write test suites so that we can test our own code. We do human reviews so that we can make sure things are looking the way they need to. We refactor so that we can change code better in the future. And sure, a model has uncovered that we need to do a bit more of that. So let's do a bit more of it. But we don't need to use the fancy model in order to get those insights."

Matt's rule of thumb: if a model found something, you should be able to find it again with a cheaper model + the right harness. The goal is not to depend on the fancy model. The goal is to build the system that catches the bug without needing the fancy model to be there.

The three-system view

  • The agent — executes tasks, writes code, proposes solutions. Powered by whatever model you choose.
  • The harness — skills, prompts, sandbox, parallelization. Fully in your control.
  • The system — the process that catches what the agent misses. Tests, reviews, linting, security scanning. This is the loop that improves over time.

Most devs focus entirely on the agent. Matt's point: the system is where the compounding happens. A better agent writes code faster. A better system catches more bugs before they ship, forever, without you being there.

Ship-faster implication. The fastest devs aren't the ones with the best model. They're the ones who, after every significant bug or mistake, ask: how do I build a system so this doesn't happen again? Not "how do I be more careful?" — "how do I make the system catch this next time?" The answer is usually: a test, a lint rule, a checklist, or a skill.

Exercise: The bug retrospective

  1. Pick one bug you shipped in the last 3 months.
  2. Answer: how did the bug get past you? (No test? No staging? Didn't read the error? Made an assumption?)
  3. Design a system that would have caught it. Options: a test, a lint rule, a staging check, a skill that runs before every deploy, a code review checklist item.
  4. Build that system. Delegate to an AFK agent if it's a coding task.
  5. Verify: does the new system catch the bug on a sample of old code?

One retrospective per month. By month 6, your codebase has started catching its own mistakes.

Part 8 — DX vs AX

The hiring discussion leads Matt to articulate the connection between developer experience and agent experience. He names it explicitly:

"I think of there as being a difference between DX — developer experience — and AX, right? Agent experience. And so agent experience is the experience that the agent has working in the codebase. And anything you can do — whether that's better skills, increasing the power of the model, improving the harness, improving the codebase — improving the codebase for better AX is something people forget about. They forget about all the edges you can get with good software fundamentals. And so that's where the senior will be useful — because the senior knows how to build good DX. If they're a good senior, they know how to build a codebase that can work well with humans. And there's a huge overlap between good DX and good AX, but they're just coming at it from different angles."

This is the technical core of why senior devs get 10x and juniors don't. A senior with 10 years of DX knowledge doesn't just know how to write code — they know how to structure a codebase so that the next person (or the next agent) can understand it, change it, and extend it without breaking it. That's AX thinking.

And the converse is equally true: a codebase that's painful for humans to work in is painful for agents too. Good interfaces, clear module boundaries, minimal hidden state, explicit dependencies — these make the agent's job easier the same way they make a new human developer's job easier.

The overlap

    DX (humans)     AX (agents)
         \            /
          \          /
           \________/
          Codebase quality
          Module boundaries
          Explicit interfaces
          Minimal hidden state
          Good tests at public APIs
          Enough documentation
          Consistent patterns

  "Anything good for DX is good for AX"

Matt's point earlier in the talk is confirmed here: "if your codebase architecture is better, then you can get a cheaper model to do the same work." You are building AX every time you improve DX.

Ship-faster implication. If you have been focused on "shipping features" without investing in codebase quality, you have been simultaneously hurting DX and AX. The good news: everything you do to improve your codebase — cleaner interfaces, better tests, extracting shared logic, documenting module purpose — helps both you and your agent. One investment, double return.

Exercise: Check your AX/DX gap

  1. Pick a module in your codebase that an agent (or a new human) would struggle to modify safely.
  2. List the DX problems: unclear what it does, hidden dependencies, implicit assumptions, no tests on public interface.
  3. Each DX problem is also an AX problem. Fix one — refactor the interface, add a test, extract the hidden dependency.
  4. Ship the fix with your agent. Note: does the agent make fewer mistakes in that module after the refactor?

Part 9 — One Thing to Do Today

David's closing question: "If you could take the average AI enthusiast and give them one or two action steps to do today, what would they be?"

Matt's answer is a single instruction followed by a single permission:

"First thing I would do is I would delete every single skill, every single plugin, every single MCP server. I would go back. Delete your claude.md, delete your agents.md, go back to absolutely nothing, and then observe the agent. See what it does. In my experience, everyone bloats up their context window with too much stuff, with too many instructions."

Then:

"Go back to a blank slate and see what the agent does. Once you're seeing what the agent does in that basic mode, then layer things on top of it — and make sure those things are procedures, procedure skills, not ability skills. Layer things on that you yourself decide. If you're noticing problems, then try to find solutions to fix those problems. And try as much as you can to delegate the implementation to an AFK agent. AFK is just incredible way to work. It's just takes a little bit of setup, but once it's set up, it just goes crazy."

The two-step protocol

  1. Subtract. Delete everything. Start from zero. Watch what the agent does with nothing. This is your baseline.
  2. Layer back. Add one procedure at a time. Each layer should solve a specific problem you've observed, not a hypothetical one. No layer before you have the problem.

The discipline is to never add a layer you haven't earned — that you haven't noticed the agent failing at first. The skill comes from watching the agent fail, not from reading about what skills other people use.

The one action. Right now, before your next coding session: open your editor's skills folder. Delete everything you installed because a YouTuber said it was amazing. Keep nothing. Start with a blank slate. Watch your agent for one session. Then decide what you actually need.

Exercise: Start from zero

  1. Uninstall or disable every skill, plugin, MCP server, and custom instruction file.
  2. Run your agent on a real task — not a demo, something you're actually trying to ship.
  3. Log what it did well. Log what it struggled with.
  4. For each struggle: is it a skill problem? A harness problem? A codebase problem? A scoping problem? Identify the right lever.
  5. Add exactly one layer to address one specific struggle. No more.
  6. Repeat.

That's the protocol. Not: install 20 skills and wonder why your context window is bloated. But: observe, diagnose, fix, layer, repeat. Forever.
