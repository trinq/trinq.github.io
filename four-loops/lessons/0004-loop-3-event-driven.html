Lesson 0004
Loop 3: The Event-Driven Loop

Here's where it stops waiting for you to open a terminal.

What this loop does

Loops 1 and 2 fire when you tell them to. Loop 3 fires on its own — when something happens. A message in Slack. A webhook from GitHub. A cron at 3am. A new row in a database. A comment on a doc.

event
slack msg, cron, webhook, db row → trigger fires → agent runs
(Loops 1 + 2) → write result back to system

The key property: no human invocation. The agent becomes a participant in the system, not a tool you reach for.

Two properties that matter

  • No human invocation. The agent runs because something fired, not because someone opened a terminal.
  • Lives where you already work. The trigger happens inside a tool you're already in all day — Slack, GitHub, your CRM, your data warehouse.

When both are true, the agent isn't "an app you go visit." It's a coworker who's always on and never files an invoice.

Concrete examples:

  • A "heartbeat" cron: every morning at 8am, your docs agent checks for broken links across your docs site.
  • A Slack channel like #docs-plz: any message in that channel triggers the docs agent to answer.
  • A GitHub webhook on issues labeled good-first-task: the agent drafts a response.
  • A row arriving in a "support tickets" table: the agent triages the ticket and assigns priority.

The canonical primitives

From the LangChain blog:

"LangSmith Deployment supports the trigger infrastructure, including support for cron schedules and webhooks. One popular example of crons in action is 'heartbeats' in openclaw, which turn your agent into an always-on, proactive assistant."

Two flavors in the LangChain stack:

  • LangSmith Deployment with cron / webhooks — for triggers you control directly (schedules, HTTP endpoints).
  • Fleet channels — for triggers that live inside chat-style tools (Slack, Teams, etc.).

Other frameworks have equivalents. The pattern matters; the brand name doesn't.

Why this rung unlocks the next one

Loop 3 generates traces at scale. Every event-driven run produces a trace — the system prompt, the inputs, the tool calls, the rubric verdict, the output. When you have hundreds or thousands of these, you have the data Loop 4 needs to actually improve the agent.

Loop 3 is the data-collection layer for Loop 4.

This is why "no human invocation" matters. A human-run agent produces a handful of traces per day. An event-driven agent produces hundreds. Loop 4 needs volume to find patterns.

Common mistakes

Triggering too often

If your cron runs every minute and most runs are no-ops, you're burning tokens. Set the cadence to match when the work actually shows up.

No output channel

If the agent runs but has nowhere to write the result, the work dies in the trace log. Loop 3 must end with a write — to Slack, to a file, to a DB column.

No idempotency

If the same event fires twice (webhook retries are common), the agent runs twice. Idempotency keys or dedup at the trigger layer save you.

No human override

Always-on agents go off the rails. Have a kill switch — a way to pause the trigger, or a rate limit that fails closed.

Exercise: design one event-driven agent

  1. Pick a recurring annoyance Something that happens every day or week and currently needs a human to notice. New signups with no first action. Docs pages with broken links. Customers waiting too long for a reply.
  2. Define the trigger What's the event? Be specific. "New row in users table where first_action IS NULL" is a trigger. "User signups" is not.
  3. Define the output channel Where does the agent write when it's done? Be specific. Slack DM to the user. Email. Update a column. Post in a channel.
  4. Identify the rubric What does "done well" look like? At least 3 criteria, mixed deterministic and fuzzy. This is your Loop 2 — the verification inside Loop 3.
