Loop 4: The Hill-Climbing Loop

The one that took the author a while to actually believe. The return arrow doesn't go back to the top. It reaches inside and edits the agent itself.

What this loop does

Every Loop 3 run leaves a trace. Those traces feed an analysis agent. The analysis agent reads them, spots the failures that keep happening, and rewrites the prompt and tool config of Loop 1.

traces
from Loop 3 → analysis agent
cluster failures → propose patch
prompt / tool / rubric → Loop 1 runs on new config
tomorrow's agent is better than today's

The three properties

  • It notices where it keeps screwing up. Failures cluster. The agent reads clusters.
  • It patches its own setup. Not just the prompt — also the tools, the rubric, the model choice.
  • You wake up to a better agent. The improvement happens while you slept. You see the diff in the morning.

Compound effect: if Loop 4 finds even a 1% improvement each cycle, after 365 cycles you're at 1.01365 ≈ 37.78× the original quality. That's not a metaphor — it's the math of geometric compounding applied to agent improvement.

The canonical primitive: LangSmith Engine

From the LangChain engineering blog, How We Built LangSmith Engine, Our Agent for Improving Agents:

"Engine is itself an agent: an orchestrator that uses specialized components to run the improvement loop end to end. It pulls traces, reads code when a repository is connected, groups failures into issues, proposes evaluators and dataset examples, and updates its understanding of your agent over time."

The mechanics:

  1. Pull traces. Engine reads the LangSmith trace store for the target agent.
  2. Group failures. Similar failures cluster into recurring issues.
  3. Open an issue. Each recurring failure becomes a GitHub-style issue: "Agent fails when input contains Korean characters."
  4. Propose a fix. Engine proposes a patch — usually a prompt edit, sometimes a tool change, sometimes a new rubric criterion.
  5. Apply and verify. The patch goes in. Engine re-runs the failing cases. If they now pass, the patch sticks.

Engine is built on top of Deep Agents and connects to a sandbox where it can write files, inspect traces, execute code, and work with a checked-out repository. The core piece is the agent's memory: an Agent Overview document that gets updated as Engine learns about your system.

Why this is the rarest loop

Three reasons people stop short of Loop 4:

  • It needs trace volume. You can't find patterns in three runs. You need hundreds. That means Loop 3 must be running first, and for long enough.
  • It needs a safe place to patch. Letting an agent rewrite your prompt in production is scary. You need a sandbox, a reviewer (human or LLM), and a rollback path.
  • It needs clear signals of "better." If your rubric is fuzzy or your traces don't capture failures cleanly, Engine can't tell which patches help.

The companies that win next year won't be the ones with the best model. They'll be the ones whose agent got 1% better every single night for a year, on its own, while the competition was still typing prompts by hand.

Manual Loop 4

You don't need LangSmith Engine to do Loop 4. You can do it by hand, just slower:

  1. After a week of Loop 3 runs, dump the failed traces into a single doc.
  2. Read them. Cluster them. Notice the patterns.
  3. For each cluster, write a one-paragraph hypothesis: "the agent fails on X because Y."
  4. For each hypothesis, propose a patch: an addition to the prompt, a new tool, a rubric criterion.
  5. Apply the patches. Re-run on the failing cases.
  6. Commit the working patches. Roll back the ones that don't help.

That's Loop 4 with you as the analysis agent. Engine just automates steps 1-5. The discipline is the same.

The unlock: once you've done this once, you see that "the analysis agent" is just another agent. The pattern composes. Loops inside loops.

The whole picture

Loop 1

Agent runs. create_agent.

Loop 2

Grader retries. RubricMiddleware.

Loop 3

Triggers fire. LangSmith Deployment / Fleet.

Loop 4

Patches the agent. LangSmith Engine.

Each loop wraps the one before it. The cost goes up with each rung. The leverage goes up faster.

Exercise: do one manual Loop 4 cycle

  1. Wait for traces Either let Loop 3 collect them, or run your Loop 1/Loop 2 agent manually 20-30 times. Save every failure trace.
  2. Cluster the failures Group by what's wrong, not by the input. "Hallucinated links" is one cluster. "Missed the tone requirement" is another.
  3. Pick one cluster Pick the cluster that hurts the most. Don't try to fix everything at once.
  4. Write a patch Add a rubric criterion, or a sentence to the prompt, or a tool. One small change.
  5. Test it Re-run the failing cases. Did the patch help? If yes, keep it. If no, revert.

After one cycle, you understand what Loop 4 is. After ten cycles, you understand why 1% a night compounds.
