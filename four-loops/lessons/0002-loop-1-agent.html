Lesson 0002
Loop 1: The Agent Loop

The floor of the ladder. If you're here, you have an agent that runs. This is the rung everyone starts on.

What this loop does

The model picks a tool. Calls it. Reads what came back. Picks another tool. Calls it. Reads. Picks another. Keeps going until the task says "done."

task → model picks tool → tool returns → model reads + decides
loop until "done"

That's it. The whole loop. No verifier. No trigger. No self-improvement. Just the agent doing the work.

What you give it

  • Context: the system prompt, the conversation history, the data the model needs to reason about
  • Tools: the actions it can take — file system, search, HTTP calls, databases, custom functions
  • A termination condition: something that says "stop now," either a final answer or a budget

This is the floor, not the ceiling. Stop here and what you've actually got is a fancier autocomplete. It runs, but it doesn't grade itself, it doesn't fire on its own, and it doesn't get better over time.

The canonical primitive: create_agent

In LangChain, the canonical name for this pattern is create_agent. It's the entry point — pick a model, plug in tools, get back something you can invoke. From the source:

"This is what LangChain's create_agent gives you. Pick any model, plug in tools, and you have a working agent loop. Tools are what give the agent the power to take action in the real world."

The exact API lives in the LangChain reference. The shape is the same across frameworks:

agent = create_agent(
    model="anthropic:claude-sonnet-4-6",
    tools=[search_docs, read_file, write_file],
    system_prompt="You are a docs editor. ...<instructions>...",
)

result = agent.invoke({"messages": [HumanMessage(content="...")]})

That's Loop 1. The pattern repeats across Claude SDK, LangGraph, Cursor, and pretty much every agent framework now. Once you see it once, you see it everywhere.

The Three Things That Matter Most

Most Loop 1 failures aren't model failures. They're these three things:

1. Wrong context

If the system prompt is vague, the model improvises. Improvisation looks like work but isn't durable. Be specific about who the agent is, what success looks like, and what to do when it doesn't know.

2. Wrong tools

Too many tools = model confusion. Too few = agent can't finish. The sweet spot: as few tools as possible, each one well-named and documented in its docstring. The model reads those docs.

3. No clear termination

If the agent can't tell when it's done, it loops until it runs out of tokens. Define a "done" state — a final tool call, a structured output, a max-step budget.

Why this is the floor, not the ceiling

Loop 1 produces output. Output is the easy part. The hard parts come next:

  • Is the output good? → Loop 2 (verification)
  • Does the agent run without you being there? → Loop 3 (event-driven)
  • Does the agent improve without you editing its prompt? → Loop 4 (hill-climbing)

Each of those questions unlocks a new rung. None of them are answered by Loop 1 alone.

The trap: you can spend months on Loop 1. Better prompts, better tool design, better context. None of that is wasted — but none of it gets you above the second rung.

Exercise: Audit Your Current Loop 1

Pick one agent you run by hand. Answer these questions:

  1. What's your termination condition? How does the agent know it's done? If the answer is "I read the output and decide," that's a Loop 1 bug — the agent should be able to terminate itself.
  2. How many tools does it have? If more than 10, suspect the model is confused. If fewer than 3, suspect the agent can't actually finish complex tasks.
  3. How long is your system prompt? If under 200 words, you probably haven't been specific enough. If over 2000 words, you probably have conflicting instructions. Both are common.

You don't need to fix anything yet. Just measure. Lesson 0003 will add the verifier that makes Loop 1's output quality measurable.
