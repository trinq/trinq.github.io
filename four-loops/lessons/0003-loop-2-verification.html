Lesson 0003
Loop 2: The Verification Loop

The rung where 90% of people stop. It's also, annoyingly, exactly where the money was the whole time.

What this loop does

The agent finishes its work. Instead of you eyeballing the output and clicking "retry," a grader scores the output against a rubric. If it's under the bar, the feedback goes straight back into the agent and it tries again. No human in the loop.

Loop 1 output → grader scores
rubric → pass? → return
if fail: feedback → agent → retry (max N iterations)

Two kinds of checks

The grader is usually a mix of two things:

Deterministic checks

For the boring stuff. Do the links resolve? Does the code compile? Did the output match the requested schema? Is the file written?

These run as code, not as an LLM. They're cheap, fast, and unambiguous.

LLM-as-judge

For the fuzzy stuff. Did it actually answer the question? Is the tone right? Does the explanation make sense? Did it cover the edge cases?

An LLM grades the output against the rubric. Slower and more expensive, but the only way to grade non-deterministic output.

The canonical primitive: RubricMiddleware

LangChain ships this as a middleware you attach to a deep agent. From the docs:

from deepagents import RubricMiddleware, create_deep_agent

rubric_middleware = RubricMiddleware(
    model="anthropic:claude-haiku-4-5",   # the grader model
    system_prompt="You are a code reviewer grading generated code against a rubric.",
    tools=[run_test_suite],                # optional: tools the grader can call
    max_iterations=5,                      # cap retry loops
)

agent = create_deep_agent(
    model="anthropic:claude-sonnet-4-6",   # the agent model (often bigger)
    system_prompt="You are a careful Python engineer...",
    middleware=[rubric_middleware],
)

result = agent.invoke({
    "messages": [HumanMessage(content="Write a function...")],
    "rubric": "- Criterion 1\n- Criterion 2",
})

Key facts from the docs:

  • If rubric is absent on invocation, the middleware does nothing — it's safe to attach unconditionally.
  • The grader can be a smaller, cheaper model than the agent (e.g., Haiku judges Sonnet).
  • The grader can use tools itself — to run tests, count words, lint output, etc.
  • max_iterations defaults to 3 and is hard-capped at 20. After the cap, the run terminates with max_iterations_reached — a failure, not a pass.

What this gets you

The honest answer: it costs more tokens per task — roughly 2-3× — because you're running the agent plus the grader, and sometimes running the agent again. But what you buy is:

  • The agent never hands a customer a wrong answer without you checking first.
  • Failures get caught immediately and re-attempted with feedback, instead of being discovered in production.
  • The rubric is a contract — you can change it without rewriting the agent.

One wrong answer in prod costs you more than a thousand retries ever will. Verification is the cheapest insurance you can buy.

Why most people stop here

Loop 2 is comfortable. The output is good. The agent retries when it fails. The harness runs without you. There's no obvious next step.

But here's what Loop 2 can't do:

  • It can't run on its own. You still have to invoke the agent.
  • It can't improve the rubric. You write the rubric, the rubric stays as-is.
  • It can't patch the agent when the model itself drifts. You're stuck with whatever prompt you wrote.

That's where Loops 3 and 4 come in.

The trap: you stop optimizing because Loop 2 is "good enough." Meanwhile, your competitor is on Loop 4, and their agent has been improving itself for six months while yours hasn't moved.

Writing a good rubric

A bad rubric is vague. "Make it good." A good rubric is concrete. "Every link must resolve to a 200. The opening sentence must contain the word 'because.'"

Rules of thumb:

  1. Each criterion is one thing. If a criterion has "and," split it.
  2. Each criterion is testable. If you can't write a sentence saying what "passing" looks like, the criterion isn't ready.
  3. Deterministic first. Anything that can be checked by code should be. Save LLM-as-judge for things only a human-shaped judgment can score.
  4. 3 to 8 criteria. Fewer and you're not specific enough. More and the grader loses the plot.

Exercise: write one rubric

Pick one of the agents from Lesson 0002's exercise. Write a 3-5 item rubric. Mix deterministic and fuzzy. Then:

  1. Run it against your last 3 outputs How many would have passed? If all 3 pass, your rubric is too loose. If 0 pass, it's too strict.
  2. Count criteria If you wrote more than 8, cut the weakest. If you wrote fewer than 3, add at least one more.
