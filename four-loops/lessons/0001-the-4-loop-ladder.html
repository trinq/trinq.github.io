Lesson 0001
The 4-Loop Ladder

Stop typing prompts by hand. Build the loops that prompt for you.

The pitch in one line

Stop being the thing that prompts the agent. Go build the thing that prompts the agent for you.

Most people using AI agents are stuck on rung 2 of a 4-rung ladder. The other two rungs are basically empty — and that's where the leverage is.

The four loops

Loop 1 — Agent

Model calls a tool, reads what comes back, calls another, keeps going until the task is done. You already have this.

LangChain primitive: create_agent

Loop 2 — Verification

After the agent finishes, a grader scores it against a rubric. If it's under the bar, feedback goes back in and it retries. No human clicking retry.

LangChain primitive: RubricMiddleware

Loop 3 — Event-driven

A message in Slack kicks it off. A webhook kicks it off. A 3am cron job kicks it off. Nobody invokes it. It just runs, at scale, inside the tools you already live in.

LangChain primitive: LangSmith Deployment (cron/webhooks) or Fleet channels

Loop 4 — Hill-climbing

Every run leaves a trace. An analysis agent reads them, spots the failures that keep happening, and rewrites Loop 1's prompt and tools. You wake up to a better agent than the one you shut the laptop on.

LangChain primitive: LangSmith Engine

Why most people plateau at rung 2

The honest version: loops 1 and 2 are where everyone is elbowing each other. Better prompts, better models, better graders. Packed.

Loops 3 and 4 are basically empty. That's the whole edge, sitting there.

The wrong takeaway: "I need a better model." The model is no longer the bottleneck. Everyone rents the same model anyway, same weights, same price. The bottleneck is the harness around it.

The right takeaway: "I need a better harness." A harness is just loops inside loops.

The math

If something improves by 1% every day:

1.01365 ≈ 37.78

That's 37.8× better in a year. Loop 4 is the loop that makes this compounding happen automatically. Every run leaves a trace. The trace gets analyzed. The agent gets patched. Tomorrow's run is on a slightly better agent.

Loop 4 is hill-climbing because the agent keeps walking up the slope of its own failures.

The four loops, as a picture

Loop 4
hill-climb ↑ patches Loop 1
agent runs → Loop 2
grade + retry → Loop 3
trigger
all wrapped in: harness, traces, observability

The arrows matter. Loop 4 doesn't just restart the agent — it reaches inside and edits the agent itself. The system prompt. The tools. The rubric. Each patch is small. Over a year, those small patches compound.

Where you probably are right now

Loop 1: You're using an agent in Cursor, Claude Code, or similar. It runs, it returns text, you copy-paste it somewhere.

Loop 2: You've added some kind of test or check. You eyeball the output. Maybe you retry when it's bad.

Loop 3: You haven't wired it into a cron or webhook. You still open the terminal.

Loop 4: Your agent doesn't improve on its own. You patch the prompt by hand when it goes off the rails.

That's the median state. The good news: the median is the easiest place to escape from, because the next rung up is well-defined.

What this course teaches

Each of the next four lessons is one rung. By the end, you'll know:

  • What each loop does and the canonical primitive that implements it
  • Where you're currently on the ladder (and what climbing one rung costs)
  • Which rung gives you the biggest leverage for the kind of work you do

Action Items

  1. Identify your current rung Pick one agent you use regularly. Which loop does it stop at? Be honest — most agents you've built by hand stop at Loop 1.
  2. Estimate the cost of climbing one rung If you're on Loop 1, what's it take to add a verifier (Loop 2)? If you're on Loop 2, what's it take to add a cron trigger (Loop 3)?
  3. Read the source Open Resources and read LangChain's "The Art of Loop Engineering" blog post. The course is a simplified walkthrough of that ladder.
