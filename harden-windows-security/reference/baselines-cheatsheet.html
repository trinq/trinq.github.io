Baselines Cheatsheet

One-page printable. The session-prep card for the next rollout. Every layer covered in the Harden Windows Security on-prem series, consolidated.
How to use this card
Print this page (or save as PDF). Pin it next to the keyboard during a rollout, an audit, or an incident review. The cards below cover, in order, the L4 floor / choice / disable taxonomy, L5’s 16 ASR rules grouped by attack class, L6’s recommended driver block list, L7’s HVCI and Credential Guard GPO locations, and the cross-references back to each lesson for full context.

L4 — The floor / choice / disable taxonomy

Every setting in every baseline maps to one of three postures. Walk every setting you change through this decision and record the choice somewhere search-able.

FLOOR
Apply by default to every machine in the OU. No documented exception. Examples: SmartScreen filter, disable SMBv1, require Secure Boot, audit-Logon subcategory.
CHOICE
Apply per environment context. Choice rules require per-OU documentation. Examples: PSExec / WMI command launch (IT automation depends), unsigned processes from USB (offline patch flow).
DISABLE
Document, opt out, and revisit later. Recording the disabling matters more than the disabling itself. Example: “Credential Guard: disabled on the X OU pending hardware refresh.”
The seven-day rule
For any change — at any layer — allow seven days between Audit and Enforced. The seven-day window is the buffer for legitimate frequency to surface in the telemetry.

L5 — The 16 ASR rules, grouped by attack class

Configured at Computer Configuration > Administrative Templates > Microsoft Defender Antivirus > Windows Defender Exploit Guard > Attack Surface Reduction. Telemetry events: 1121 (audit), 1122 (block / warn).

Office
5 rules. FLOOR. Office app spawning child processes, injecting code into other processes, loading unsigned add-ins, using macros to call Win32. Default: Block all 5.
Credential
1 rule: Block LSASS credential stealing. FLOOR. Stops Mimikatz-class tools at the process layer. One of L2 boundary 3’s defenses.
Script
3 rules covering script abuse and obfuscated script execution. FLOOR for obfuscation; CHOICE for the “persisted through PSExec” rule.
Email
1 rule: block executable content from email and webmail clients. FLOOR with disable-by-exception-OU if a documented workflow needs signed EXEs by email.
Lateral
2 rules: PSExec / WMI command launch; unsigned processes from USB. CHOICE. Both depend on whether IT uses PSExec and whether OT / IT uses USB for offline patch flows.
Persistence
2 rules: WMI event subscription persistence; Adobe Reader child processes. FLOOR for both. Legitimate software does not need to subscribe WMI events or spawn Reader children.

Staging pattern: Audit → Warn → Block, per OU, seven days between each flip. The 1121 / 1122 ratio across the staging period is your leading indicator.

L6 — WDAC + Microsoft recommended driver block list

Configured at Computer Configuration > Administrative Templates > System > Device Guard > Deploy Windows Defender Application Control. Telemetry events: 3076, 3077, 3099, 3091 in Microsoft-Windows-CodeIntegrity/Operational.

Recommended list
Pre-built, signed, kernel-mode WDAC policy. Delivered as a .p7b file. Source: Microsoft Documentation (auto-updates via Windows Update).
Authoring workflow
New-CIPolicy (enumerate) → Merge-CIPolicy (combine) → ConvertFrom-CIPolicy (XML → .p7b) → sign with internal code-signing cert → deploy via GPO → verify with gpresult.
Modes
Audit / Enforced / Disabled. No Warn — kernel-mode enforcement happens before user-mode UI exists. Staging: Audit → Enforced (skip Warn).
Multi-policy
Base policy (the recommended block list) at higher priority; supplemental policy (your LOB apps, IT utilities) at lower. Base blocks known-bad drivers anywhere; supplement loosens per-OU.
Known breakages
Unsigned LOB apps (sign them or supplemental carve-out); single-policy-for-forest (use multi-policy); DCs in stage one (stage on member servers first); old hardware drivers (audit full seven-day cycle); no signing pipeline (deploy Audit only, document as DISABLE per L4).

L7 — HVCI + Credential Guard (L2 boundary row 7 made practical)

Configured at Computer Configuration > Administrative Templates > System > Device Guard > Turn On Virtualization Based Security. Verify with msinfo32 under Virtualization-based security.

HVCI
Platform Security Level: Secure Boot and DMA Protection. Blocks kernel-code modification regardless of driver identity — backstops the recommended block list for unknown-bad drivers.
Credential Guard
Credential Guard Options: Enabled with UEFI lock. Moves Kerberos TGTs and NTLM hashes out of LSASS into the VBS-isolated kernel. Even BYOVD-affected hosts cannot read them.
Hardware floor
All four required: UEFI with Secure Boot enabled; IOMMU virtualization (Intel VT-d / AMD AMD-Vi); SLAT; TPM 2.0. Hardware floor — older machines are below the threshold; document as DISABLE per L4.
Documented breakages
Legacy NTLM authentication paths; Kerberos delegation; SSO providers that pull credentials from LSASS directly. Test in a pilot OU first; carve out per-OU as needed.

Series-wide defense posture (the capstone claim)

This is the single sentence a junior admin should be able to recall to defend the investment in the seven layers above:

No local admin + standard user + signed kernel drivers + HVCI on supported hardware + Credential Guard on supported hardware + Microsoft recommended driver block list + ASR audit-then-block is the series-wide defense posture. Removing any one of them has a measurable effect on the Hardening Score.
