App Control for Business (WDAC) On-Prem

L6 Kernel-mode identity · ~16 min · Lesson 6 of the Harden Windows Security series

L5 was about behavior: ASR rules that ask “is this process trying to do something bad?” This lesson is about identity: a kernel-mode enforcement that asks “is this executable allowed to start at all?” Together they are defense in depth at the process boundary — and the identity layer is the one attackers cannot talk their way past, because the check happens before user-mode code is even loaded.

The mental model: WDAC is policy-driven executable-identity enforcement, not behavior blocking. Where ASR rules decide what an allowed executable is allowed to do, WDAC decides who is allowed to run. The two are complementary, not redundant. A defense-in-depth posture runs both; an attacker has to defeat identity (kernel-level) and behavior (user-level) before getting a single instruction executed.

The load-bearing concept
ASR + WDAC together = a process boundary that requires both an allowed identity (kernel-mode WDAC) and acceptable behavior (user-mode ASR) before user code runs. Without WDAC, ASR is your only line at the process boundary; without ASR, WDAC allows anything any signed binary wants to do.

How WDAC differs from ASR

Three differences that matter for an admin. Each of them is the answer to a question you would have asked during a Tuesday-morning ticket.

ASR (L5) WDAC (L6)
Targets behaviors — what an already-running process tries to do. Targets identities — which executables are allowed to start at all.
User-mode enforcement; runs in Defender; the policy is in the Defender workload. Kernel-mode enforcement; runs in the Windows kernel; the policy is in CodeIntegrity.
Four modes — Not configured / Audit / Warn / Block. Three modes — Audit / Enforced / Disabled. No Warn mode.

The no Warn mode row is not a gap; it is a deliberate consequence of where WDAC runs. WDAC is a kernel-mode enforcement. The kernel decides whether to load an executable before any user-mode UI is even available. There is no screen to draw a prompt onto. So the modes are: Audit (let it through and log), Enforced (block it), Disabled (no rule check runs). The transition is binary.

  • Identity — signer, hash, file path, or family of an executable. The check answers “is this file on the allow-list?”.
  • Mode — what to do when the answer is “no”. Audit logs it. Enforced blocks it silently at launch. Disabled ignores the policy entirely.
  • Scope — kernel-mode (drivers, boot components) or user-mode (applications, scripts). Two separate policies.
Enforced is silent
When WDAC is Enforced and blocks an executable, the user sees the OS “this app has been blocked by your system administrator” dialog. There is no Warn-style “ask the user whether they really want to run it.” Make sure staging covers your line-of-business apps before flipping to Enforced.

The two policy scopes: kernel-mode vs. user-mode

A WDAC policy is either kernel-mode or user-mode. They are separate artifacts; each has its own deployment target. Junior admins conflate them; the distinction matters.

Scope What it enforces Where it runs Real-world target
Kernel-mode Drivers and early-start boot components. Before the user session exists; in the kernel itself. BYOVD (Bring Your Own Vulnerable Driver) attacks.
User-mode Applications, DLLs, scripts launched after the user signs in. In the user session, in concert with CodeIntegrity. Unsigned line-of-business apps, malware-as-signed-binary.
Compared with ASR rules, what does a WDAC policy additionally enforce?

The two policy types: signed vs. unsigned

WDAC policies come in two flavors, distinguished by whether the policy itself is cryptographically signed.

Property Signed WDAC policy Unsigned WDAC policy
Trust root Code-signing certificate embedded in the policy. No trust root. Whoever can write the policy can rewrite it.
Deployment Deployable from any source, including a read-only network share. Must be deployed from the local machine’s own CiPolicy folder.
Mutability A user with administrator rights cannot stop or modify it. A user with administrator rights can disable it via registry.
HotCakeX recommendation Yes — this is the recommended path. Initial policy authoring only — never the production target.

HotCakeX’s framing: unsigned policy is for the first draft (you build a policy on a test box, audit it, see what breaks, then re-issue as a signed policy for production). The default Windows policy and the Microsoft recommended driver block list are both signed. Your final production policy should be signed too — otherwise, you have shipped a kernel-mode enforcement that any local administrator can disable.

Why signed matters on-prem
If your WDAC policy is unsigned, then any user with local administrator can disable it via a single registry key — which means anyone who escalates to local admin (your L1 row-3 adversary from L2) has defeated the kernel-mode layer. Signed policy closes the registry door. It is the difference between “policy” and “policy that attackers cannot turn off.”

The three modes — audit → enforce, no warn step

WDAC has three modes per policy. The staging sequence mirrors ASR except it skips Warn.

Mode What happens on a disallowed launch What you see
Audit Rule check runs. The file is allowed, but the event is logged. Event Log entry 3076 (would-be-blocked). No user-visible impact.
Enforced Rule check runs. The file is blocked from launching. “This app has been blocked by your system administrator.” Event 3076 + 3077.
Disabled No rule check runs. Nothing. Default Windows behavior.

Notice the absence of Warn. The reason is operational, not pedagogical. A user-mode Warn prompt requires Windows to draw a UI — and the kernel-mode decision happens before the user session exists. There is no place to put a Warn prompt during a boot-time driver check, and no time to draw one during a fast app launch. So Warn is omitted by design.

The staging sequence reuses L4’s seven-day rule:

  1. Build an unsigned policy in Audit on a test box

    Use the New-CIPolicy PowerShell cmdlet (part of the WDAC policy authoring toolset) to enumerate signed apps that should run in your environment. Save the policy to C:\Windows\System32\CodeIntegrity\CiPolicy\Active\<GUID>\CiPolicy.{p7b}. Do not deploy yet.

  2. Convert to signed, Audit, deploy to one OU

    Sign the policy with a code-signing cert from your internal CA. Deploy to a single pilot OU. Wait seven days. Pull CodeIntegrity event log 3076 — you will see what would-have-been-blocked.

  3. Pull the audit events; classify by app

    For each event, ask: is this an attacker trying to launch a known-bad executable, or is it a legitimate line-of-business app we forgot to enumerate? Each legitimate hit becomes a supplemental policy rule, or a carve-out OU per L4.

  4. When audit is clean for seven days, flip to Enforced

    Switch the policy mode from Audit to Enforced. The same policy — now the would-be-blocks become real blocks. Telemetry now records real blocks (events 3076/3077) instead of would-have-blocked events.

  5. Roll forward to the next OU

    Repeat. The seven-day rule per OU is the buffer; do not skip it. A wrong Enforced policy on a production OU is a domain-wide outage waiting to happen — blue screens on devices that needed an app the policy didn’t allow.
No patch = no rollback
Unlike ASR rules, a wrong WDAC Enforced policy cannot be undone remotely in real time if it blocks a critical boot component on a domain controller. There is no “tell GPO to ignore” button. Plan for the rollback path before flipping: a WinPE USB, an offline editor, a Safe Mode boot, or a known-good supplemental policy you can deploy to undo the bad one. Three of these in your pocket is the recommended posture.
ASR has audit → warn → block staging. What is the WDAC equivalent?

Where WDAC sits in the L2 boundary map

Walk L2’s seven boundaries again. WDAC is the kernel-side enforcement for two of them, and the perfect complement to ASR’s user-side role.

L2 row Boundary ASR — user side WDAC — kernel side
1 CPU ring No (user-mode only). Yes — signed kernel-mode policy blocks unsigned drivers at load.
2 Process Yes — ASR blocks behaviors of allowed processes. Yes — WDAC blocks unsigned user-mode binaries at launch.
4 Network (lateral via PsExec) Yes — the “Block PsExec” ASR rule. Indirectly — if PsExec itself is unsigned, WDAC kernel-mode blocks the driver.

Rows 3, 5, 6, 7 are not WDAC’s job (WDAC does not move boundaries — it configures them, per the L2 decision rule). Boundary 7 (VBS) is covered in L7 alongside Credential Guard and HVCI.

The Microsoft recommended driver block list

The single highest-leverage on-prem win in this lesson ships as a pre-built, signed, kernel-mode WDAC policy — the Microsoft Recommended Driver Block List. It blocks drivers known to be vulnerable to BYOVD attacks — the same class of risk covered in L7. You do not author it; you deploy it.

Property Value
What A pre-built WDAC .p7b file maintained by Microsoft.
Scope Kernel-mode only; blocks specific known-bad drivers.
Signed Yes — signed by a Microsoft-controlled code-signing cert.
Update cadence Microsoft updates the list as new BYOVD-discoverable drivers are found; published through Windows Update and a public GitHub repository.
Staging method Deploy in Audit first (count would-be blocks); flip to Enforced once the count stabilizes.

Why it matters: the alternative is hand-curating your own block list of ~80 known-vulnerable drivers, which means a junior admin must (a) know about every CVE-disclosed driver, (b) map each CVE to a specific file hash, (c) keep the list current as Microsoft publishes new ones. The Microsoft list is the same set, curated by Microsoft, updated automatically. Use theirs. It is the L4 “floor” decision applied to driver-block rules.

Updated 2026 baseline
As of the Windows 11 24H2 baseline, the recommended driver block list covers roughly 80 active block rules for drivers with disclosed exploitation in the wild. The list does not enumerate “unused” drivers or historical CVEs — it is targeted at the BYOVD-class attackers L7 covers. Audit logs for the list are dominated by security-research and EDR tooling, not production workloads.

Authoring a custom WDAC policy on-prem

The recommended driver block list is the floor. Everything else (your line-of-business apps, your IT automation tools, your internal signed binaries) lives in your custom policy, layered on top. Authoring on-prem is a PowerShell workflow.

  1. Enumerate the executables you want to allow

    Run New-CIPolicy -Audit -FilePath C:\Policy\p_baseline.xml -Level Publisher on a representative endpoint. This generates an Audit-mode policy in XML listing everything currently signed that the machine runs.

  2. Merge multiple reference machines into one policy

    Different OUs run different apps. Use Merge-CIPolicy -PolicyPaths .\p1.xml,.\p2.xml,.\p3.xml -OutputFilePath .\merged.xml to combine per-machine policy snapshots into a domain-wide allow-list.

  3. Convert the XML policy to a binary .p7b

    ConvertFrom-CIPolicy .\merged.xml .\SignedPolicy.p7b produces the binary, signed policy artifact. Sign with your internal code-signing cert before the next step.

  4. Deploy via GPO

    Computer Configuration > Administrative Templates > System > Device Guard > Deploy Windows Defender Application Control. Point at the path where the signed .p7b lives (UNC path or local file). Link the GPO at the target OU in Audit first.

  5. Verify via gpresult and the CodeIntegrity log

    On a target machine, gpupdate /force, then gpresult /h gp.html. Confirm the WDAC policy is listed. Pull Microsoft-Windows-CodeIntegrity/Operational — 3076 events are the would-have-blocked trail during Audit.
Multi-policy stacking
Real AD domains rarely have a single WDAC policy. The recommended driver block list (signed, kernel-mode) is the base; your custom LOB policy (signed, kernel- and/or user-mode) is supplemental. Windows merges them by priority: the more specific policy wins. The supplemental custom policy can be looser (allow your LOB apps) without re-allowing the vulnerable drivers — because the recommended block list blocks them at base priority.

Telemetry without Defender for Endpoint

WDAC events land in Microsoft-Windows-CodeIntegrity/Operational. Unlike ASR (Defender Operational log), this log is owned by the kernel. It is on by default. The events to watch:

Event ID Mode Meaning
3076 Audit or Enforced A binary was allowed in Audit, or would have been blocked.
3077 Enforced A binary was actually blocked from launching.
3099 Audit An “allow” event was triggered — an allowed binary launched.
3091 Audit Policy load summary — shows which base policies applied.

The 3076 / 3077 ratio across staging periods is your WDAC analogue of the L5 1121/1122 ratio. If 3076 (Audit would-have-been-blocked) is high and 3077 (Enforced actually-blocked) is low, the rule is hitting real attacks or legitimate outliers. If 3077 is much larger than 3076 was, you have stopped normal work and need more supplemental rules.

Minimum-viable telemetry without Defender for Endpoint: forward the Microsoft-Windows-CodeIntegrity/Operational log via Windows Event Forwarding to a central collector, alongside the Defender Operational log from L5. Same WEF subscription infrastructure; both event channels require forwarding. The log wraps at 1 MB by default — schedule a weekly export before staging at scale.

ETW tracing for kernel depth
Forensics during a BYOVD-class incident requires the process tree at kernel level — not just “this driver was blocked” but “this driver tried to load with this stack and was blocked at this kernel module.” That depth requires tracelog -start with the CodeIntegrity provider enabled. Out of scope for L6; the audit-and-block trail from 3076/3077 is your operational baseline.

Common breakages and how to avoid them

Five failure modes dominate the “I deployed WDAC and now everyone is broken” tickets. Knowing them in advance saves a week of rollback planning.

Unsigned line-of-business apps — the biggest on-prem pitfall

Most enterprises with a 10-year-old line-of-business app have at least one or two executables that have never been code-signed. WDAC Enforced silently blocks them. Mitigation: sign the binaries with your internal code-signing cert (the cost is real, not enormous); or carve out a supplemental policy rule per binary (path-based, not recommended; or hash-based, which is brittle). The Microsoft recommended block list does not cover your LOB apps — only known-bad drivers.

The “single policy for the whole forest” trap

Authoring a single policy for the entire domain is the wrong shape. The IT OU runs different apps than the Finance OU; the production line OU runs different apps than the call-center OU. Mitigation: use multi-policy stacking — one base policy (the Microsoft block list) and multiple supplemental policies per OU. The base blocks known-bad drivers everywhere; the supplements allow LOB apps per function.

Staging on domain controllers

A WDAC Enforced policy that blocks a critical boot component on a DC blue-screens the DC. DCs do not recover from kernel-mode errors during boot the way member servers do. Mitigation: stage on member servers first; DCs last; have a rollback plan (WinPE USB, Safe Mode with the policy disabled, a known-good supplemental policy).

Itanium-era drivers and old hardware

The recommended driver block list sometimes includes drivers needed by old hardware (printers, plotters, ancient USB-to-serial adapters). Audit mode catches this before Enforced — but only if Audit was actually run. Mitigation: audit for at least the full seven-day rule across all your hardware profiles before flipping. Hardware that’s only used on Tuesdays is the classic miss.

The “no patch = no rollout” pressure

Sometimes the correct answer is “do not enforce WDAC today.” If your line-of-business apps are unsigned and cannot be signed in the rollout window, the floor of the recommended driver block list (Audit) is your realistic posture. Mitigation: deploy Audit only; record this as a floor/choice/disable decision per L4; revisit when signing has shipped.

What this lesson gives you

  1. WDAC is identity enforcement; ASR is behavior enforcement. They are complementary, not redundant. Together they form defense in depth at the process boundary (L2 row 2 + boundary 1 support).
  2. Two scopes: kernel-mode (drivers, the BYOVD surface) and user-mode (applications, the unsigned-binary surface). Separate policies for each.
  3. Two types: signed (production) vs. unsigned (initial authoring). HotCakeX recommends signed for production; the recommended driver block list is signed.
  4. Three modes, no Warn: Audit / Enforced / Disabled. Warn is omitted because kernel-mode enforcement happens before user-mode UI exists. Staging is audit → enforce, not audit → warn → block.
  5. The Microsoft recommended driver block list is the canonical on-prem win — pre-built, signed, kernel-mode. Deploy this before authoring custom policies.
  6. Authoring workflow: New-CIPolicy → Merge-CIPolicy → ConvertFrom-CIPolicy → .p7b → GPO at target OU.
  7. Telemetry in Microsoft-Windows-CodeIntegrity/Operational. Events 3076 (would-have-blocked), 3077 (actually-blocked), 3099 (allowed audit), 3091 (policy load). The 3076/3077 ratio is your leading indicator.
  8. Known breakages: unsigned LOB apps, single-policy-forest, DCs in stage one, old hardware drivers, “no signing pipeline yet.” Each has a documented mitigation.
