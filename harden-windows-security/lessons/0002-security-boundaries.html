Security Boundaries in Windows

L2 Architecture · ~14 min · Lesson 2 of the Harden Windows Security series

L1 gave you the 2×2 of defense in depth and the rule that local admin collapses the rest of the stack. This lesson explains why that rule is true. The answer is a single Windows-architecture fact, restated in three layers, then translated into where GPO can and cannot enforce.

The fact: Group Policy does not create security boundaries. Group Policy configures them. If the boundary itself is bypassed, the GPO configuration is bypassed with it. This is the single most useful thing to internalize when debugging a deployment that “isn’t working.”

The boundary hierarchy, top down

A security boundary in Windows is a line that an attacker at a lower privilege level cannot cross to reach a higher privilege level, without first breaking the boundary itself. The interesting ones for an enterprise admin are seven. Each is enforced by a different mechanism. Each can be bypassed in a different way. Each is configured by a different set of GPO settings — or by no GPO at all.

# Boundary What it separates What enforces it What bypasses it
1 CPU ring Ring 0 (kernel) from Ring 3 (user mode) Hardware + Windows kernel Local Administrator (no boundary — by design)
2 Process One process from another’s memory Windows memory manager Code running with the target process’s privilege
3 User (Standard vs Admin) Standard user from local Administrator UAC + token filter User who is local Administrator
4 Network One machine from another over the wire Windows firewall, SMB signing, NTLM/Kerberos, WinRM, RDP Credential theft, BYOVD, exposed management port
5 AD domain One AD realm from another Kerberos, ACLs on AD objects, DCOM/RPC auth KRBTGT compromise, DC compromise, ACL misconfiguration
6 AppContainer (sandbox) UWP / Store app from rest of system AppContainer capability profile Capability grant, broker escape (rare)
7 Virtualization (VBS) Normal kernel from VBS-isolated kernel (Credential Guard, HVCI) Hyper-V hypervisor on supported hardware Hypervisor exploit (extremely rare, Tier 4 territory)

Read this table once. The lesson that follows is just a restatement of it. Boundaries 1–3 are OS-level on a single machine. Boundaries 4–5 are network and AD. Boundary 6 is for Store / MSIX apps. Boundary 7 is the new hypervisor-enforced layer that arrived with Windows 10 and is the foundation Credential Guard and HVCI sit on.

The load-bearing row
Row 1: there is no security boundary between Administrator and Kernel. That sentence appears in Microsoft’s own Security Servicing Criteria for Windows. It is not a quirk. It is by design — Administrators are expected to be able to install drivers, modify system files, and load kernel software. The cost is that any code running as Administrator can, too.

Why “no local admin” matters — in boundary terms

Read the table again with this question: at which row does a phishing attachment stop being dangerous?

  • User as Admin. The attachment runs in Ring 3 (boundary 1), but Admin tokens can load kernel drivers, so boundary 1 is advisory. It can read other processes’ memory (boundary 2 bypass). It can impersonate other users on the box (boundary 3 absent). It can pivot via SMB (boundary 4 depends on credentials). It can request Kerberos tickets (boundary 5 depends on ACLs). The attacker has effectively crossed every row except 5 (AD) and 7 (VBS, if enabled).
  • User as Standard. The attachment runs in Ring 3 but cannot load drivers. It cannot read LSASS without an elevation prompt. It can still run, but every protection below it — ASR rules, WDAC, Defender real-time, SmartScreen — gets a chance to fire because the attachment has not bypassed boundary 1.

The 98% from L1 is the gap between those two paragraphs. Same attachment, same OS, same domain. The only difference is whether boundary 3 exists.

A phishing attachment runs as a Standard user. Which row of the boundary table is doing the heavy lifting?

Where GPO can and cannot enforce

Now the part that matters for your day job. GPO configures boundaries. It does not create them. This distinction shows up in two failure modes that look identical but aren’t:

  • “My GPO isn’t applying.” Almost always a GPO problem: scope, security filtering, WMI filter, replication. Fixable with gpresult /h on the target machine.
  • “My GPO applies but the attack still works.” Usually a boundary problem: the user is Admin (boundary 3 absent), so the GPO setting that “blocks” Office from spawning executables is bypassable. Fixable only by fixing the boundary (remove local admin), not by writing another GPO.

The second failure is the more dangerous one because it looks like GPO is broken when actually GPO is doing exactly what it was told. The boundary was already gone before the GPO ran.

Per-boundary GPO action map

Boundary What GPO can configure What GPO cannot do
1 — CPU ring Driver block rules (WDAC), HVCI policy. Stop an Admin from loading a driver it didn’t block.
2 — Process ASR rules (Office child process, etc.), Attack Surface Reduction. Stop a process running with that process’s own privileges.
3 — User / Admin LAPS rotation, “Add administrator” rights removal, UAC level. Prevent an account that is in the local Administrators group from acting as Administrator.
4 — Network Windows Firewall rules, SMB signing, NTLM restrictions, RDP NLA, WinRM hardening. Stop a stolen credential from being used by an attacker who has it.
5 — AD domain Restricted Groups, ACL templates on OUs, GPO delegation, Protected Users group. Recover from KRBTGT hash theft; reset trust with a compromised domain.
6 — AppContainer AppLocker / WDAC user-mode policy targeting MSIX apps. Force arbitrary Win32 binaries into a sandbox they were not designed for.
7 — VBS Credential Guard policy, HVCI opt-in, kernel DMA protection. Run on hardware without a hypervisor; recover a leaked VBS-protected secret.
Failure mode to memorize
GPO applies correctly, attack still succeeds. The fix is not a stronger GPO. The fix is restoring the missing boundary — which usually means removing local admin, isolating tier 0, or enabling Credential Guard. If you respond to “GPO isn’t blocking this” by writing a new GPO, you are debugging the wrong layer.
An attacker dumps LSASS memory despite your WDAC and ASR rules. Which boundary has been bypassed?

Three scenarios from your domain

Three realistic incidents and what boundary reasoning says about each.

Scenario A — “Phishing got an admin.”

A user in Domain Admins clicks a phishing link. The attachment runs in their elevated context. Every boundary except AD (row 5) collapses. Fix is not GPO. Fix is tier-0 isolation: separate Domain Admins from everyday accounts (a PAW, Privileged Access Workstation), require separate credentials for tier 0. HotCakeX recommends this for the same reason — GPO cannot enforce “do not browse the web as a domain admin.” Only an architectural separation can.

Scenario B — “Workstation compromised, attacker moves laterally.”

A workstation user runs a malicious attachment as Standard. ASR blocks one exploit chain; the attacker pivots to a different chain that ASR did not cover. They reuse a credential to access a file server via SMB. The fix is at boundary 4. Require SMB signing (GPO: Microsoft network server: Digitally sign communications (always)), block NTLM where possible, rotate the local admin password with LAPS so the credential reuse fails. GPO is the right tool here.

Scenario C — “Service account is Domain Admin.”

A SQL service account has been a member of Domain Admins since 2017 because someone needed it to “just work.” An attacker who SQL-injects the web app inherits those rights. Fix is at boundary 5. GPO can use the Protected Users group (no NTLM, no DES, no Kerberos delegation) for the human admin account, but the service account needs a separate fix: remove from Domain Admins, give it only the rights it needs (least privilege), and audit with BloodHound or equivalent. GPO cannot demote a service account; that is an AD object ACL edit.

Which scenario is most directly fixed by a GPO setting?

What this lesson gives you

  1. The seven boundaries. CPU ring, process, user, network, AD, AppContainer, VBS. Each is enforced differently and bypassed differently.
  2. The decision rule. GPO configures boundaries; it does not create them. If the boundary is gone, GPO cannot bring it back.
  3. The failure-mode vocabulary. “GPO isn’t applying” is a GPO problem. “GPO applies but the attack works” is a boundary problem. The second is the dangerous one to misdiagnose.
  4. The per-row GPO action map. When you get a hardening request, identify which row it lives on before you start writing GPO.
