Microsoft Security Baselines & LGPO.exe

L4 Tooling · ~15 min · Lesson 4 of the Harden Windows Security series

L1 gave you the 2×2 of defense in depth. L2 gave you the boundary map — seven places where Windows can stop an attacker, and which GPO configurations configure each. L3 gave you the policy tree — how AD expresses placement. This lesson is the bridge from those frameworks to an actual, deployable hardening policy, by way of the official Microsoft toolchain.

The mental model: a baseline is a starting point you adapt, not a checklist you install. Microsoft ships the Windows 11 security baseline with ~300 settings. Deploying all of them unmodified gives you a “Microsoft said so” answer to every audit question — and that answer doesn’t survive a real review. The lesson is how to ingest the baseline, audit it category by category, and decide which settings stay, which get modified, and which get disabled because your environment cannot support them.

The Microsoft Security Compliance Toolkit

The MSCT is a free download from Microsoft. It is the source of truth for every Windows hardening baseline Microsoft itself recommends. The box has four tools:

Tool What it does When you reach for it
Baselines ADMX/ADML + GPO backups + PolicyRules for Windows, Office, Edge, etc. When you want the source artifact. The baseline is updated quarterly.
Policy Analyzer Compares a machine’s effective policy against a baseline and reports drift. When you want to know what the machine actually has set. The audit tool.
LGPO.exe Command-line tool to import/export local GPO and apply registry-based policy. When you want to test a baseline on a single machine before deploying via GPO. Also used in build images.
Set-RegistryPolicy PowerShell module to write registry-backed policy outside of GPO. When you want to script a baseline into an OS image or a non-domain-joined host.

For an enterprise AD admin, the workflow is roughly: download the MSCT, take the Windows 11 baseline’s GPO backup, restore it into a new GPO in your domain, audit it, link it where L3 says it should live (domain root for the floor, child OU for the rest), verify with Policy Analyzer on a sample machine.

Why a baseline GPO is not a destination
Microsoft’s baseline is opinionated. Some settings assume an enterprise context (TPM 2.0, Secure Boot, Pro/Enterprise edition). Some assume a particular threat model (Azure-AD joined). Some are policy choice rather than floor — Microsoft picks a value, but the value is yours to set. The baseline is a starting point, not a destination. L4 is about adapting it, not applying it.

The two workflows: ingest and verify

Workflow A — Ingest the baseline into a domain GPO

  1. Download the MSCT

    From microsoft.com/download/details.aspx?id=55319. Extract the ZIP. Inside: Windows 11\Baseline\BaselineLocalInstall.admx and the matching GPO backup folder.

  2. Stage a new GPO in your domain

    In GPMC, create a GPO named MS Windows 11 Baseline (Staging). Do not link it yet. Restore the baseline GPO backup into it.

  3. Inspect the contents category by category

    Open the staged GPO. Walk through Account, Audit, Network, Kernel, App Execution sections. Mark each setting as floor, choice, or disable (incompatible with my environment).

  4. Apply the decisions

    Disable or modify the settings you marked. The result is your baseline, not Microsoft’s, on top of Microsoft’s recommendations.

  5. Link per L3

    Floor settings live at the domain root (L3 strategy 1). Per-OU overrides live at child OUs (strategy 2). Hardware-class variations live behind WMI filters (strategy 3).

Workflow B — Verify the deployed baseline

  1. Pick a sample machine

    One per OU you care about. For a 500–5000 endpoint fleet, that means at minimum: one Tier 0 server, one file server, one Tier 1 workstation, one kiosk, one Domain Controller.

  2. Run gpresult /h gp-report.html

    Confirm the GPO is in the Applied GPOs list with the right precedence.

  3. Run Policy Analyzer against that machine

    Point Policy Analyzer at the same baseline GPO backup you ingested. It will report every drift — settings where the machine’s effective state differs from the baseline. Investigate each drift. Some are legitimate overrides (per-OU exceptions); others are bugs.

  4. Repeat on the next sample machine

    Until you have coverage. Then schedule the same audit weekly and alert on new drift.
The audit is the lesson
The most important step in Workflow B is not the gpresult — it is the Policy Analyzer diff. A gpresult tells you which GPOs applied. Policy Analyzer tells you what effective setting each policy produced, and whether it matches the baseline. These are different questions. Use both.

The five baseline categories

The Windows 11 baseline organizes ~300 settings into roughly five categories. Each maps to a row in L2’s boundary table.

Category What it covers L2 boundary Example settings
Account Password policy, account lockout, Kerberos, Protected Users. Row 5 (AD domain) + Row 3 (user/admin) Min password length, lockout threshold, NTLM restrictions.
Audit Advanced audit policy subcategories, Event Log size, retention. Row 5 (AD domain) + the detection row of the 2×2 Audit logon, audit privilege use, audit process creation.
Network SMB, RDP, WinRM, Windows Firewall, NTLM, Kerberos, Wi-Fi. Row 4 (network) Disable SMBv1, require SMB signing, RDP NLA, WinRM hardening.
Kernel Secure Boot, HVCI, Credential Guard, driver block list, VBS. Row 1 (CPU ring) + Row 7 (VBS) Require Secure Boot, enable HVCI, opt into Credential Guard.
App execution ASR rules, SmartScreen, AppLocker, WDAC, MOTW enforcement. Row 2 (process) + Row 6 (AppContainer) Enable SmartScreen, configure ASR rules (L5 territory), block Office child processes.

The mapping is the load-bearing concept. When you audit a baseline setting and ask “do I need this on a 500–5000 endpoint on-prem domain?” the answer starts with “which boundary row does it touch, and what’s my threat posture there?” If you can’t answer that, you don’t have the audit trail to keep the setting — or to disable it.

Floor vs choice vs disable

This is the single most useful distinction for adapting a baseline. Every setting is one of three types:

Floor

Microsoft-recommended across all environments. You should not disable this without a documented business reason and compensating controls. These are the L3 domain-root policies:

  • Disable SMBv1.
  • Disable the guest account.
  • Require Secure Boot (when hardware supports it).
  • Require SMB signing at the server side.
  • Disable autorun.
  • Enable LSA Protection (PPL).

If you find yourself considering disabling any of these, you should be writing a document explaining why and what compensates. “Microsoft said so” is not a floor; Microsoft’s recommendation is well-justified by these three CVEs is a floor.

Choice

Microsoft recommends a value, but the value is yours to set. The baseline gives you Microsoft’s opinion; you set yours. These are policy choices:

  • Minimum password length (Microsoft picks 14; you may pick 12 or 16).
  • Account lockout threshold (Microsoft picks 10; you may pick 5).
  • Audit retention size and rollover behavior.
  • BitLocker recovery key rotation period.
  • ASR rule audit/warn/block posture (Microsoft ships in audit; you decide when to flip to block).

These are documented in your hardening policy, not in the baseline. They get applied via child-OU GPOs (L3 strategy 2) where they need to vary by business unit or role.

Disable (incompatible with environment)

The setting cannot apply in your environment — missing hardware, missing OS feature, business application breaks. Examples:

  • Credential Guard requires VBS-capable hardware. If 30% of your fleet is five-year-old desktops, you can’t enforce it everywhere; you enforce it where supported and document the gap elsewhere.
  • HVCI requires recent drivers; a line-of-business app from 2019 may not have HVCI-compatible drivers. Document the exception.
  • Some ASR rules (L5) break specific Office workflows; in audit mode, you see which machines need exceptions, then disable per-OU.

Disable is not a permanent state. It is a documented exception with a re-evaluation date. Treat it like an architectural decision record.

Audit-trail discipline
Every floor setting you disable, every choice you change from Microsoft’s value, every environment-incompatibility you accept — all three must be in a written document with a date, an owner, and a re-evaluation trigger. This is the document that survives a CISO audit, an incident, or a new hire’s first week.

Why LGPO.exe is still load-bearing even on AD-joined machines

LGPO.exe is Microsoft’s command-line tool for managing local GPO and registry-based policy. It is not a replacement for GPO — it is a verification and build-image tool.

Three uses for an on-prem AD admin:

  1. Pre-deployment testing. Before linking a baseline GPO to a production OU, run LGPO.exe /r baseline-backup on a test machine. Verify the resulting registry state with secedit /export. If the test machine still works, the baseline is safe to link.
  2. Build image authoring. When you build a Windows image (MDT, WDS, or hand-rolled), LGPO.exe writes the baseline into the image so that a freshly-imaged machine has the floor even before it joins the domain. This buys you security during the pre-join window.
  3. Audit & diff. LGPO.exe /b snapshot-folder captures the local GPO’s effective state as a backup. Diff this against a baseline backup with Policy Analyzer to find what the machine actually has.

LGPO.exe is the most-misunderstood tool in the MSCT. Junior admins treat it as “the script that deploys the baseline.” It is not. It is the tool that tells you what’s deployed. Use it accordingly.

The first-time-baseline checklist

The first time you bring a baseline into a domain, the workflow that works:

  1. Stage and audit

    Create the GPO, restore the baseline, walk every category, classify each setting as floor/choice/disable. Document the classification in a spreadsheet.

  2. Test on five machines

    Run LGPO.exe /r on a Tier 0 server, a file server, a workstation, a kiosk, and a Domain Controller. Verify each.

  3. Link at domain root, audit only

    Link the floor subset at the domain root in audit-equivalent mode. Wait two weeks. Check gpresult and Policy Analyzer drift reports on the test machines.

  4. Move floor subset to enforce mode

    If no L1/L2/L3 breakage during the audit period, flip the floor subset to enforce mode. The choice and disable settings still live in their respective OUs.

  5. Hand the choice subsets to business owners

    Each business unit owns its choice subset. The IT team is not in the business of deciding password length for HR; HR is. Provide them the baseline category and ask for the value with rationale.

  6. Schedule weekly drift reports

    Run Policy Analyzer weekly against a sample of machines per OU. Alert on new drift. Drift is the leading indicator of either compromise or unauthorized change.
The seven-day rule
After you link a new baseline GPO at the domain root, do not flip it to Enforced for at least seven days. The audit period is your buffer against the line-of-business app that breaks every Tuesday at 09:00.

What this lesson gives you

  1. The MSCT is a tool, not a checklist. Baselines, Policy Analyzer, LGPO.exe. The baseline is starting material; the toolchain is the verification apparatus.
  2. Two workflows. Ingest (restore the baseline into a GPO, audit per category, link per L3) and verify (gpresult + Policy Analyzer on a sample of machines, drift reports weekly).
  3. Five baseline categories mapped to L2’s boundary table. Account (rows 5/3), Audit (row 5 + detection), Network (row 4), Kernel (rows 1/7), App execution (rows 2/6). The mapping is the audit trail.
  4. Floor vs choice vs disable. Floor applies everywhere with rationale; choice is documented per-OU; disable is an exception with a re-evaluation trigger.
  5. The seven-day rule. Audit, then enforce. Always. The audit window is your buffer against silent breakage.
