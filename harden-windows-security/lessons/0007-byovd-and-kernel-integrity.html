BYOVD & Kernel Integrity

L7 Capstone · ~18 min · Lesson 7 of the Harden Windows Security series

Six lessons ago you accepted that L1’s 2×2 is incomplete without every row populated; four lessons ago you learned that GPO configures boundaries rather than creating them; two lessons ago you added an identity enforcement layer (WDAC) that runs in the kernel. This final lesson ties the series together: what the kernel-side attacker looks like, what stops them, and what to do in the first hour when one gets through.

The mental model: kernel integrity is the layer below identity and behavior, where the OS itself trusts the running code. Where L5 blocked behaviors of allowed user-mode executables and L6 blocked executable identities that were not on the allow-list, L7 covers the attack class that uses allowed identities doing exactly what they were signed to do. This is the BYOVD class — Bring Your Own Vulnerable Driver — and it is the only attack family in the series that bypasses L5 and L6 simultaneously by design.

The capstone claim
HotCakeX’s “98% / 99%” claim from L1 is anchored on no local admin + standard user + App Control + hardening baseline. L7 makes that claim concrete: no local admin + WDAC identity layer + recommended driver block list + HVCI + Credential Guard is the series-wide defense posture. Each component is the load-bearing piece of one slice of the 2×2; remove any of them and the 98% number collapses by a measurable amount.

What BYOVD actually is

A Bring Your Own Vulnerable Driver attack uses a real, signed, Microsoft-accepted kernel-mode driver that has a publicly known vulnerability. The attacker does not need to write a driver; they need to find one. Microsoft distributes known-vulnerable driver hashes in a regularly updated public list — LOLDrivers, maintained by the security research community. The attacker picks a driver whose vulnerable function does something useful (read arbitrary kernel memory, terminate a protected process, allocate executable kernel memory) and they load it.

Why L5 does not help: ASR rules are a user-mode behavior policy. They run in Defender in Ring 3. A kernel-mode driver loads below the layer ASR inspects. The driver never trips an ASR check.

Why L6 partially helps: WDAC kernel-mode is exactly the layer that should block these drivers. The catch is that the signed-but-vulnerable driver is legitimately signed; the kernel-mode policy must include a block rule for that specific hash or signer to stop it. The default policy does not enumerate ~80 vulnerable drivers; the Microsoft recommended driver block list does.

The reminder L1 set up
Every layer in this lesson assumes the running user is not local Administrator. A BYOVD attack from a Standard user is a tier 2 incident. A BYOVD attack from a logged-in Administrator is a tier 3 or tier 4 incident because the attacker inherits every driver they can name. The no-local-admin rule from L1 is the assumption behind everything that follows.

Why BYOVD is tier 3–4 territory

The attacker who lands BYOVD on your network has done one of three things: bought a subscription to an access broker that delivers BYOVD payloads as part of the initial-access toolchain; tricked a Standard user into running a loader that pulls a known-vulnerable driver from disk; or escalated from a Standard user to a local Administrator first, and is now leveraging admin to load the driver. The first is organized crime (tier 2). The second is an opportunistic tier 1 with a public BYOVD wrapper. The third is the path most APT playbooks describe — tier 3 or 4 depending on the post-exploitation objective.

The published BYOVD history is short and instructive. Three reference examples every defender should be able to name:

Example Driver What it gave the attacker Why it matters here
Capcom.sys class (capcom) A legitimate-looking “Capcom” driver shipped as a test loader; signed by a real cert at the time. zwterminatestub — ability to terminate any process, including protected-antimalware processes. The canonical example used to teach the attack pattern. Demonstrates that any legitimately signed driver with an execute-anything IOCTL is enough.
Realtek audio driver class Signed audio drivers shipped by OEMs; vulnerable IOCTLs in the wildcard. Kernel read/write primitives from a driver that is on most Windows machines. Demonstrates that drivers you have already deployed are the attack surface. Removing them is not an option; blocking them via WDAC is.
MSI / Gdrv class AfterBurner / MSI’s Gdrv; signed; vulnerable. Read/write kernel memory, allocate executable pages. Demonstrates that gaming / utility software is a frequent BYOVD vector. The recommended driver block list includes it for a reason.

The common thread: each driver was legitimately signed, used in production for years, and had a vulnerable IOCTL that the security research community disclosed publicly. The fix was not a Microsoft Windows patch; the fix was adding the driver hash to a block list. That is what the Microsoft recommended driver block list is for.

LOLDrivers catalog
The public repository LOLDrivers tracks every disclosed vulnerable driver. It is the reference catalog for any team operating a WDAC kernel-mode block-list policy. The Microsoft recommended driver block list (covered in L6) is the curated subset of LOLDrivers that Microsoft deems actively exploitable. The two are related: Microsoft’s list is automatically updated; LOLDrivers is the exhaustive set you would audit manually.

HVCI — the second enforcement layer beneath WDAC

WDAC kernel-mode blocks drivers whose identity is on the block list. Hypervisor-protected Code Integrity (HVCI) blocks kernel code that does not satisfy Microsoft’s WHQL signature plus memory-integrity checks. The two are complementary.

Layer What it stops Where it runs Failure when bypassed
WDAC kernel-mode Specific driver identities on the block list (signed-but-vulnerable). CodeIntegrity module; consulted at driver load. Attacker uses a vulnerable driver not yet on the block list.
HVCI Any kernel code that tries to allocate executable memory, write to executable pages, or inject code into running kernel modules. Inside the VBS-isolated hypervisor (L2 row 7). Attacker uses a class-of-attack, not a specific known-bad driver.

HVCI is what stops the class of BYOVD attacks: the attacker may land a fresh, not-yet-blocked driver, but the moment that driver tries to modify kernel code or allocate executable memory, HVCI trips. Blocked behavior, even from a passed-identity driver.

Hardware prerequisites

HVCI requires the same hardware building block that Credential Guard does: the VBS hypervisor. Which means:

  • UEFI with Secure Boot enabled (firmware-level root of trust).
  • IOMMU virtualization support in the CPU — on Intel this is VT-d, on AMD it is AMD-Vi. The IOMMU is what the VBS hypervisor uses to enforce memory isolation between the normal kernel and the VBS-isolated kernel.
  • Second-Level Address Translation (SLAT) support.
  • TPM 2.0 — required by Credential Guard, not strictly by HVCI alone, but the two ship together as a posture.

The GPO that flips HVCI on lives at: Computer Configuration > Administrative Templates > System > Device Guard > Turn On Virtualization Based Security > Select Platform Security Level: Secure Boot and DMA Protection. The setting requires a reboot. After the reboot, msinfo32 under Virtualization-based security shows whether the kernel is running HVCI-eligible memory.

Hardware floor
If your fleet includes machines without virtualization support — older desktops, low-end laptops, OT shop-floor hardware — HVCI will silently not engage on those machines. The GPO does not error; the layer is just absent. This is the same “no patch = no rollout” pattern from L5 and L6: document the machine floor in your L4 floor/choice /disable decision, and accept that the layer is unavailable on that subset.

Credential Guard — L2 row 7 made practical

L2’s row 7 is the VBS boundary: a hypervisor-isolated kernel that keeps secrets out of reach from the normal kernel. Credential Guard is the version of that boundary specifically for logon credentials. Where the L1 attack class was a phishing attachment that dumps LSASS memory from user space, Credential Guard moves the credential material out of LSASS entirely — into the VBS-isolated kernel.

The attacker who lands Mimikatz against a Credential Guard-protected host gets nothing useful: the LSASS handle does not actually contain the Kerberos TGTs or NTLM hashes anymore. They are sealed inside the hypervisor. Even kernel-mode attackers (with or without BYOVD) cannot read them, because the VBS isolation is enforced by the hypervisor itself — which is below the kernel that BYOVD lands in.

Property Value
What it isolates Kerberos TGTs, NTLM hashes, Kerberos service tickets.
Where the secrets live Inside the VBS-isolated kernel — not accessible to the running OS kernel.
What breaks if you enable it Legacy NTLM authentication in many configurations; Kerberos delegation; any SSO provider that pulls credentials from LSASS directly. Test in a pilot OU before broad rollout.
Hardware prerequisite Same as HVCI: UEFI Secure Boot, IOMMU virtualization (VT-d or AMD-Vi), SLAT, TPM 2.0.
GPO location Computer Configuration > Administrative Templates > System > Device Guard > Turn On Virtualization Based Security > Credential Guard Options: Enabled with UEFI lock.
L2 row 7 — the rare boundary GPO enables
L2 row 7 is the only boundary in the table that does not require administrative separation (tier 0 isolation) to enforce. The other rows are administrative (remove local admin, isolate tier 0) or network-level (SMB signing, NTLM restrictions). Row 7 is the one boundary where a single GPO toggle plus supported hardware turns on a layer that earlier generations of Windows could not provide. Use it.

The recommended driver block list, from a BYOVD angle

L6 introduced the Microsoft recommended driver block list as the load-bearing pre-built, signed, kernel-mode WDAC policy. This lesson revisits the same artifact from a different angle: it is the specific BYOVD mitigation. Each rule in the list corresponds to a driver whose vulnerability has been publicly disclosed and likely exploited. The list is the operational answer to the LOLDrivers catalog; rather than audit every LOLDriver yourself, deploy Microsoft’s curated subset.

The deployment shape is the L6 staging pattern verbatim: audit for seven days, look at the would-have-been-blocked entries, classify by app (genuine BYOVD attack vs. legacy hardware driver an OEM requires), flip to enforced after the audit stabilizes. The staging cadence is the same as L5 and L6 because the pattern is the same — audit, classify, enforce — only the rule surface differs.

Attack Stops it Does not stop it
Driver on the recommended block list Yes — WDAC denies the load at boot. —
Newly discovered driver not yet on the block list No — WDAC accepts the load; HVCI is the backstop. Yes — until Microsoft updates the list.
Driver class with no known vulnerability No — driver loads; HVCI backstops behavior. Yes — HVCI applies to all kernel code.
Signed driver with novel vulnerability not in LOLDrivers No — until disclosure, the driver is a hidden attack surface. Yes — the BYOVD-attack window is the disclosure-to-patch window.

The takeaway: the recommended driver block list is the floor for known BYOVD. HVCI is the floor for unknown BYOVD. L1’s no-local-admin rule is the floor for Standard users running a loader that pulls a driver from disk. Three independent layers — none of them is alone sufficient.

Reading kernel-mode alerts — first-hour IR playbook

Defenders reading this series want to know what to do when EDR fires on the things L6 and L7 prevent. The first hour is the operational window. These are the habits that separate a good IR outcome from a six-week rebuild.

  1. Do not reboot the affected host

    A kernel-mode driver that loaded is still loaded in volatile memory. Rebooting clears it — but also clears every forensic artifact that would have told you what the driver did once it ran. The driver’s payload is the question; clearing it removes the answer.

  2. Preserve volatile memory

    Use a memory acquisition tool (WinPmem, FTK Imager, or Magnet RAM Capture) to image the host’s RAM before touching disk. Memory holds the kernel driver’s payload, decrypted C2 configuration, and any in-process LSASS / Kerberos material the driver touched. Disk forensics recover file state; memory captures driver state.

  3. Pull an ETW trace of the CodeIntegrity provider

    tracelog -start with the CodeIntegrity provider enabled gives you the would-have-blocked and was-actually-blocked log around the alert. ETW is the only source for the kernel-load order if the driver blocked silently. The L6 ETW-trace note applies here in force.

  4. Pull the Microsoft-Windows-CodeIntegrity/Operational log

    Event IDs 3076 (would-have-blocked), 3077 (actually-blocked), 3099 (audit allow), 3091 (policy load). Cross-reference against the recommended driver block list version to identify the rule that fired.

  5. Identify the driver and the loader process

    From the ETW trace and the CodeIntegrity log, find: which driver hash was attempted, which signed it, what executable loaded it, which user it ran as. This tells you whether the loader was a user-mode phishing payload (Standard user) or a privilege-escalation post-exploitation step (Admin).

  6. Pull the audit log for the same host, same time window

    Look for the user-mode chain that led to the load: process creation, logon events, scheduled-task creation. The kernel-mode alert is the symptom; the user-mode chain is the cause.

  7. Treat the loader host as compromised regardless of block outcome

    A blocked attempt is evidence someone on that machine tried to load BYOVD. That is enough to consider the host compromised and start a parallel investigation of how the loader got there — regardless of whether the kernel-mode block succeeded.
What not to do in the first hour
Three habits that produce bad outcomes: reimage the host first (you destroy evidence before analyzing it), broadcast the alert before you know the loader chain (you tip off operators), or trust the WDAC block as “fixed” (the block stopped the driver; it did not stop the user-mode loader that brought it). Block is not containment.

What this lesson gives you

  1. BYOVD is a tier 2–4 attack class that L5 and L6 do not fully cover by themselves. It uses signed identities doing exactly what they were signed to do. WDAC blocks known identities; HVCI blocks unknown-bad kernel code.
  2. The Microsoft recommended driver block list is the BYOVD floor. Same artifact as L6, viewed from the attack-class angle. Same audit-then-enforce discipline as L5 and L6.
  3. HVCI is the second kernel-mode enforcement. It blocks kernel-code modification regardless of identity. Requires UEFI Secure Boot + IOMMU + SLAT + (recommended) TPM 2.0.
  4. Credential Guard is L2 row 7 made practical. It moves credentials out of LSASS into the VBS-isolated kernel; even an admin who lands on the box via BYOVD cannot read them.
  5. The first hour of a kernel-mode alert is a memory-preservation window. Do not reboot. Capture volatile memory. Pull ETW. Cross-reference CodeIntegrity log. The loader process is evidence of compromise even when the block succeeded.
  6. The series-wide posture (no local admin + standard user + signed kernel drivers + HVCI + Credential Guard + recommended driver block list + ASR audit-then-block) is the capstone answer to the L1 98% claim.
