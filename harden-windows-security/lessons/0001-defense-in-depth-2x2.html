The Defense-in-Depth 2×2

L1 Mental model · ~12 min · Lesson 1 of the Harden Windows Security series

You administer a Windows domain with 500–5000 endpoints. There are roughly thousands of security settings you could flip: GPO objects, registry keys, ASR rules, WDAC policies, BitLocker policies, LAPS settings, SmartScreen toggles, and on. Nobody can deploy all of them at once. The question this lesson answers is: where do you start, and what mental model tells you which layer a given setting belongs to?

HotCakeX's anchor text for this series is built around one idea: no single setting makes you secure. Defense in depth is not a slogan — it's an architectural claim. If one layer fails, the next layer catches the attacker. If the next layer also fails, the next one catches. And so on, until either the attacker runs out of layers or you do.

HotCakeX's claim
Use the Harden System Security app to apply the hardening measures it offers, and your system is secure against at least ~98% of the threats when you use a Standard (non-Privileged) account for everyday work. Then use the AppControl Manager to deploy an App Control policy and have even more control. Together, you get ~99% protection. [HotCakeX / Harden-Windows-Security — How Do You Make the Right Choice?]

That 98% / 99% claim sounds like marketing. It's not. It's a statement about who you are defending against. To use it correctly, you need to first name the threat actors you actually face. That's the first axis of the 2×2.

The threat-actor hierarchy

Not all attackers are equal. The defenses you need against a script kiddie running a downloaded tool are not the defenses you need against a motivated nation-state actor with a zero-day. Most enterprise compromises — roughly 70–80% per Microsoft's annual Digital Defense Report — involve one of three lower tiers, not the top tier. HotCakeX's "98%" is calibrated against the lower tiers.

Tier Actor What they bring Defense that stops them
Tier 1 Script kiddie Public tools, known CVEs, social engineering scripts. Patched OS + SmartScreen + Standard user + Defender.
Tier 2 Organized cybercrime Ransomware affiliates, access brokers, commodity malware. ASR rules + WDAC + LAPS + credential hygiene.
Tier 3 Hacktivist / insider Credential abuse, lateral movement, data theft. PAW tiering + tier-0 isolation + audit logging.
Tier 4 Motivated nation-state Zero-days, supply-chain attacks, BYOVD, long-dwell intrusions. Kernel-mode WDAC + HVCI + Confidential Computing + DRTM.

Calibrate your hardening to tiers 1–3 first. Tier 4 is a different problem with a different budget (think Microsoft DART / GHOST engagements and Secured-Core PC fleet, not GPO). HotCakeX's 98% holds for tiers 1–3. Pushing for tier 4 without first solidifying tier 1–3 is a common mistake.

The 2×2: prevention vs detection, baseline vs ongoing

Every Windows hardening setting falls into one of four buckets. Two axes:

  • Vertical axis: prevention vs detection. Does this setting stop an attack, or does it notice one?
  • Horizontal axis: baseline vs ongoing. Is this a one-time configuration (a "baseline") or something that needs to keep running (a "control")?
Baseline (one-time config) Ongoing (must keep running)
Prevention Hardened config. Disable SMBv1, disable guest account, require SMB signing, disable autorun, set UAC to highest, require Secure Boot. Set once via GPO; verify with Policy Analyzer. Active enforcement. ASR rules, WDAC policies, attack-surface-reduction telemetry, LAPS password rotation, account lockout, EDR in real-time. These must run, log, and react.
Detection Audit policy. Enable advanced audit policy subcategories, configure Event Log size, ship logs to a SIEM. Set once; verify the channel exists and is populated. Monitoring. SIEM correlation rules, Defender alerts, scheduled task to detect GPO drift, honey-token files, periodic policy compliance scans.

Most junior admins default to the top-left cell (hardened config) because it's the easiest to deploy: download a Microsoft baseline, import it, link the GPO, walk away. But that cell alone is insufficient — it doesn't notice when an attacker compromises a machine after the baseline is applied. The right answer is at least one cell from each row.

Common drift
Deploying a Microsoft Security Compliance Toolkit baseline and considering the domain "hardened." The baseline configures the prevention cell but leaves the detection row empty. When an attacker compromises a workstation six months later, there is no log of how they got in. Always pair prevention with detection.

The one rule that buys you the most: no local admin

Of every setting in every Microsoft baseline, the single highest-leverage change is also the simplest: domain users do not have local administrator on their workstations. This is the rule HotCakeX explicitly anchors its 98% claim to.

Why this rule matters so much is a Windows-architecture fact, not a policy choice. Read it carefully — it's the load-bearing mental model for the rest of the series.

The Windows security boundary
There is no security boundary between Administrator and Kernel. Any process running with local Administrator privileges on a Windows machine can load a kernel driver, modify system files, dump LSASS memory, and disable security software. Once a user has local admin, every other security layer below them is bypassable from user space. This is by design — it's how Microsoft expects Administrators to manage their machines. [Microsoft Security Servicing Criteria for Windows]

In other words: every other protection layer in Windows — ASR rules, WDAC policies, Defender real-time protection, credential guard, BitLocker — was designed assuming the running user is not a local administrator. The moment an attacker gets local admin, those protections stop being trustworthy.

That's why HotCakeX says "when you use a Standard (non-Privileged) account for everyday work" — the 98% claim is conditional on the user not being admin. Without that condition, the 98% collapses to something much lower. The single most leveraged change you can make in your domain is to remove local administrators from your standard users' workstations.

What this lesson gives you

Three things to take away:

  1. Calibrate to tiers 1–3 first. Tier 4 is a different problem with different tools. HotCakeX's 98% holds for the bottom three tiers; tier 4 needs DRTM, kernel-mode WDAC, and Secured-Core hardware.
  2. Every setting lives in one cell of the 2×2. Prevention vs detection, baseline vs ongoing. If your deployment only has settings from one row or one column, you have a hole.
  3. No local admin is the single highest-leverage rule. Every other protection in Windows assumes the user is not Administrator. The moment you give that up, the rest of the stack becomes advisory.
