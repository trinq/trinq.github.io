Attack Surface Reduction (ASR) Rules

L5 Process behavior · ~15 min · Lesson 5 of the Harden Windows Security series

L4 ended with the MSCT toolchain and the floor / choice / disable taxonomy. This lesson is a deep dive on the largest single category of that baseline — App execution — using the only setting class in Windows that can block a malicious behavior without a signature, without an EDR agent, and without ever seeing the malware before.

The mental model: ASR rules are policy-driven process-blockers, not signatures. They target behaviors — Office spawning child processes, scripts downloading executables, USB running unsigned code — rather than specific malware. A signature-based AV waits until it has seen the file; an ASR rule blocks the trick, regardless of which file is trying to use it. That makes ASR rules the single highest-leverage setting on the baseline against opportunistic attack.

How ASR rules differ from antivirus

Three differences that matter for an admin:

Antivirus / EDR ASR rules
Targets files — hashes, signatures, behaviors per-machine. Targets process behaviors — what Office, scripts, USB, WMI are allowed to do.
Needs to have seen the file (or run an ML model on it) to block. Blocks the trick before the file is even fully written.
Configured per-machine (Defender preferences) or per-tenant (Defender for Endpoint). Configured centrally via GPO, applied to Ring 0–Ring 3 user and kernel processes.

ASR rules are part of Microsoft Defender, but the rules engine ships with Windows 10/11 Pro+ regardless of Defender for Endpoint licensing. You do not need an EDR product to deploy them. You need:

  • Real-time protection enabled in Defender (this is the engine that runs the rules).
  • A GPO with the ASR rule ADMX imported (Microsoft ships the ADMX in the MSCT).
  • Telemetry to confirm the rules are firing (L4’s verification discipline).
The cost of “Defender is off”
If real-time protection is off on a machine, ASR rules silently do nothing. That is the most common “I deployed ASR rules but they aren’t firing” failure mode. Check Defender’s real-time status first.

The 16 ASR rules, grouped by attack class

There are 16 rules as of the Windows 11 2025 baseline. Trying to memorize all 16 is the wrong shape of effort. The right shape is group them by attack class and know the pattern each class targets.

Attack class What it covers Rules Example
Office abuse Office apps spawning processes, injecting code, loading unsigned add-ins, using macros to call Win32. 5 rules Word document macro spawns powershell.exe — the classic phishing payload.
Script abuse JavaScript / VBScript launching executables, obfuscated script execution, copied/impersonated system tools. 3 rules Obfuscated PowerShell from a phishing site lands and runs.
Email-based delivery Executable content from email and webmail clients. 1 rule Outlook HTML attachment tries to launch an EXE.
Credential theft LSASS memory dumping from non-Microsoft processes. 1 rule Mimikatz tries to read LSASS — L2 row 3 bypassed.
Lateral movement PSExec / WMI command launch; unsigned processes from USB. 2 rules Attacker pivots via PsExec.exe or via a USB drive with a renamed mimikatz.exe.
Persistence & ransomware WMI event subscription; Adobe Reader child processes. 2 rules Macro installs a WMI event subscription for re-entry; ransomware spawns Adobe child process.

Notice the map: every attack class corresponds to a row in L2’s boundary table.

  • Office abuse, script abuse, email-delivery → L2 row 2 (process boundary).
  • Credential theft → L2 row 3 (user/admin boundary — LSASS is a per-user boundary).
  • Lateral movement via PSExec/WMI/USB → L2 row 4 (network / removable media boundary).
  • Persistence via WMI subscription → L2 row 1 (CPU ring — WMI is a privileged surface).

ASR rules are the policy configuration of L2’s row 2 (and partly rows 1, 3, 4). They are exactly what the L2 per-row GPO action map promised for “process-level” enforcement.

The four rule modes

Each rule has four modes. The mode is set per GPO, per OU. You stage deployment by changing modes over time, not by adding rules all at once.

Mode What happens when a process tries the action What you see
Not configured No rule check runs. Default Windows behavior. Nothing.
Audit Rule check runs. Action is allowed, but the event is logged. Event Log entry (1121), no user-visible impact.
Warn Action is blocked, but the user gets a dismissal option. User sees a Defender prompt; Event Log 1122 entry.
Block Action is blocked; no user option to dismiss. Silent block from the user’s perspective; Event Log 1122 entry.
Audit does not block
The most common junior-admin mistake: deploy ASR rules in Audit mode, see no telemetry, conclude the rules aren’t firing. They are firing — they are just not blocking. Audit means “let it through and tell us.” Block means “stop it.” The two are entirely different operational states.

The on-prem deployment pattern: audit → warn → block

The deployment pattern reuses the seven-day rule from L4, applied per OU and per rule. The sequence:

  1. Stage a single GPO with the full ASR rule set in Audit mode

    Create a GPO named ASR Baseline - Audit. Import the ASR ADMX from the MSCT. Set all 16 rules to Audit (mode 2). Do not link yet.

  2. Link per OU, starting with the lowest-risk

    Link at one OU first — say, a non-production pilot OU or an IT test OU. Wait seven days. Run Get-WinEvent -LogName "Microsoft-Windows-Windows Defender/Operational" on a sample machine. Confirm Event IDs 1121 are arriving.

  3. Classify each rule per L4

    For each rule, walk through the L4 floor / choice / disable taxonomy. Block Office child processes is almost always floor. Block executable content from email depends on whether your org receives signed EXEs by email (rare; usually floor). Block LSASS credential stealing is floor on any machine that uses Windows logon.

  4. Document the exceptions

    For each rule that broke a legitimate workflow during audit, create an exception OU per L4’s discipline. Move the affected machine(s) to the exception OU with written documentation.

  5. Flip to Warn per OU

    Once audit is clean, link a ASR Baseline - Warn GPO at one OU. Wait seven days. The Warn mode produces user-visible Defender prompts. Track the dismissal rate — high dismissal means the rule is blocking normal work, not malicious work.

  6. Flip to Block per OU

    Once Warn is stable for seven days, link ASR Baseline - Block at the same OU. The rule now silently blocks. Track Event 1122 (block events). Compare 1122 count to 1121 (audit count) from earlier — if 1122 is much smaller, you stopped real attacks; if it’s similar, you stopped normal work and need more exceptions.

  7. Roll forward to the next OU

    Repeat the cycle for the next OU. The seven-day rule per mode is the buffer; do not skip it.
1121 vs 1122
Event ID 1121 = “ASR rule fired in Audit mode.” Event ID 1122 = “ASR rule fired in Block or Warn mode.” The ratio of these two across the staging period is your leading indicator of whether the rule is hitting real attacks or hitting normal work.

Telemetry without Defender for Endpoint

Defender for Endpoint gives you a centralized dashboard for ASR events. On pure on-prem without DfE, you can still get the data:

Source How to query What it gives you
Microsoft-Windows-Windows Defender/Operational Get-WinEvent -LogName "Microsoft-Windows-Windows Defender/Operational" -FilterXPath "*[System[EventID=1121 or EventID=1122]]" The full ASR audit / block trail.
Microsoft-Windows-Windows Defender/WHC Same approach; this log has higher-fidelity details for block events. Block-mode events with process tree.
ETW tracing (tracelog -start) Higher-effort; used when you need process-tree details that the Defender log doesn’t capture. Full process genealogy and module loads.
Forwarded events Windows Event Forwarding (WEF) collects from machines and lands in a central collector. Centralized archive of Defender events without SIEM.

The minimum-viable on-prem telemetry for ASR is: turn on Windows Event Forwarding, subscribe to the Defender Operational log, and ship the events to a central collector. This gives you a single query path to answer “how many times did this rule fire last week?” without DfE.

Audit events roll off
The Defender Operational log rolls over at 1 MB by default. Without WEF or a SIEM, your audit events disappear when the log wraps. Schedule a weekly export of the Defender logs to a network share before staging “ASR - Audit” at scale.

The floor / choice / disable classification per rule

L4’s taxonomy applied to the 16 ASR rules. This is the working classification that a junior admin can audit and defend.

Rule class Default classification Why
Office abuse (5 rules) Floor Phishing-as-entry-point is the dominant attack pattern. Office child-process spawning is the most-abused behavior.
LSASS credential stealing (1 rule) Floor Blocks Mimikatz-class attacks at the process level. The L2 row 3 boundary needs every layer of defense.
WMI event subscription persistence (1 rule) Floor Persistence via WMI is rare for legitimate software. Block-by-default with rare exception.
Script obfuscation (1 rule) Floor Legitimate scripts are not obfuscated. Block is reasonable for almost every environment.
PSExec / WMI command launch (1 rule) Choice IT departments legitimately use PSExec. Floor if your IT team uses only Intune / WinRM; choice otherwise.
Executable from email (1 rule) Floor — with disable exceptions for IT scripts. Almost no legitimate org receives EXE attachments. Disable per L4 only if a documented workflow needs it.
Unsigned processes from USB (1 rule) Choice If your org uses USB drives for offline patch deployment, the rule breaks it. Audit first.
Adobe Reader child processes Floor Acrobat Reader should not be spawning child processes. Block.

The exact per-rule classification depends on your environment — the table is the starting point, not the destination. The principle: classify every rule, document every choice, record every disable, exactly per L4’s discipline.

An ASR rule is in Audit mode for a week. The telemetry shows it firing on a legitimate line-of-business script daily. What next?

Breakage patterns to expect per rule

Each rule has a known set of legitimate apps that it will break. Knowing these ahead of time saves a week of “why is Finance complaining” tickets.

Office child-process rules — breaks software updaters inside Office

Some line-of-business apps install a small updater inside Word or Excel that spawns a child process to check for new versions. Block mode breaks the updater. Mitigation: audit first; identify the updaters; move those machines to an exception OU; or fix the updater to use a scheduled task instead of an Office child process.

PSExec / WMI command launch — breaks IT automation

If your IT team uses PSExec or WMI command launch as part of standard remediation scripts, Block mode breaks the entire script. Mitigation: rewrite the script to use WinRM / PowerShell Remoting, which is on a different ASR rule surface; or carve out an exception OU for IT machines.

Executable content from email — breaks IT distribution

If your IT team distributes signed EXEs by email to staff (rare but happens), Block mode breaks that channel. Mitigation: move distribution off email and onto a download portal; the Block rule becomes floor.

Unsigned processes from USB — breaks offline patch deployment

Some OT / IT shop floors use USB drives for offline patch deployment to air-gapped machines. The unsigned-process rule breaks that. Mitigation: sign the patch payloads with a code-signing cert; or carve out an exception OU for the air-gapped machines.

What this lesson gives you

  1. ASR rules are behavior-blockers, not signatures. They target tricks, not files. They run on real-time protection in Defender, configured centrally via GPO.
  2. The 16 rules grouped by attack class. Office abuse, script abuse, credential theft, email delivery, lateral movement, persistence. Each maps to a row in L2.
  3. Four modes: Not configured / Audit / Warn / Block. Audit means “let it through and tell us.” The staging sequence is Audit → Warn → Block, with the L4 seven-day rule between each.
  4. Floor / choice / disable classification per rule. Reuse L4’s taxonomy. Most Office, LSASS, WMI, and script rules are floor. PSExec and USB rules are usually choice. Document every exception.
  5. Telemetry without Defender for Endpoint. Defender Operational log, Event IDs 1121 (audit) and 1122 (block). Forward via WEF to a central collector. The ratio of 1122 to 1121 is your leading indicator.
  6. Known breakage patterns per rule. Office child processes break software updaters. PSExec breaks IT automation. USB breaks offline patch deployment. Each has a documented mitigation.
