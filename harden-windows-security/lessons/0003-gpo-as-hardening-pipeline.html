Group Policy as the Hardening Pipeline

L3 GPO design · ~14 min · Lesson 3 of the Harden Windows Security series

L2 ended with a map: seven boundaries in Windows, each with its own GPO action column. L1 ended with a single rule: no local admin. This lesson is the bridge between the two — how to take a boundary-layer policy you want to deploy and actually place it in Active Directory so it lands on the right machines, in the right order, without breaking everything else.

The mental model for the lesson: Active Directory is a policy tree, not just an object tree. The OU hierarchy is the spine; the GPOs are nodes that cascade down the spine in a specific order; the user objects at the bottom inherit whatever the cascade says. If you treat the tree as only an object tree, you will misplace policies. Treat it as a policy tree, and placement decisions become obvious.

LSDOU: the four-layer cascade

When a user logs in, GPOs are evaluated in this order, and the last one that applies wins on any setting that conflicts:

  1. Local GPO

    Each machine has one. Applies first. Use it for “this machine is special” overrides, never for fleet-wide hardening.

  2. Site GPO

    Linked to an AD Site (a subnet grouping). Use it rarely — only for site-bound things like WAN-friendly SMB defaults.

  3. Domain GPO

    Linked at the domain root. Applies to every machine and user in the domain. The home of “must apply everywhere” hardening.

  4. OU GPO

    Linked to an OU. Applies to everything in that OU (and below, until something blocks it). Where most of your hardening lives.
The order matters
“Last writer wins” is the rule. If a domain-root GPO disables SMBv1 and a child OU GPO enables it for legacy file servers, the child OU setting wins for machines in that OU. Layer your policies from broad (domain root) to narrow (deep OU), then test by running gpresult /h gp-report.html on a target machine.

Note what is not in the list: site OUs, group OUs, dynamic OUs. Group Policy doesn’t care about group membership as a primary targeting mechanism — it cares about where the object lives in the tree. Targeting who in a GPO is done by security filtering on the GPO, not by where the user object sits.

The four placement strategies (mapped to L2’s boundaries)

For each boundary-level policy you want to deploy, you have four placement options. Each option has a different reach, a different debugging path, and a different L2-boundary implication.

1. Domain-root placement — “must apply everywhere”

Linked at the domain root. Inherits down every OU in the domain. Use this for policies that every machine and user must follow — the L1 baseline “floor” of your hardening. Typical contents:

  • Disable SMBv1.
  • Disable guest account.
  • Require SMB signing (at the server side).
  • Require Secure Boot.
  • Disable autorun.

These all target boundary 4 (network) and boundary 1 (CPU ring via Secure Boot). They are reversible per-OU if a child OU needs an exception, but should not need to be.

2. Single-OU placement — “this business unit only”

Linked to one OU and its descendants. Use this for policies that vary by business unit — finance has different software from engineering, so finance’s smart-screen and ASR-default-app rules differ. Also used for site-specific things like DNS suffix or NTP server.

3. WMI-filtered placement — “this hardware class only”

You can filter a GPO by hardware: laptops only, desktops only, machines with TPM 2.0 only, machines with at least 8 GB of RAM. This is the right tool for hardware-class targeting — if your laptops and desktops need different BitLocker policies, use WMI filtering at the same link target, not separate OUs. Useful because you do not have to maintain parallel OU trees for “laptops” and “desktops.”

The trade-off: WMI evaluation runs at every gpupdate and on every boot. A WMI query that touches the registry or WMI repository can be slow. Keep WMI filters to a half-dozen classes per filter, and pre-test them with Get-WmiObject from a sample machine.

4. Loopback-replaced placement — “this machine owns this user”

By default, User Configuration in a GPO applies to the user at logon, no matter which machine they log onto. Loopback processing replaces that: User Configuration from the GPO applies to whichever user logs onto the machine the GPO is linked to. The use case is kiosk or terminal server — the workstation, not the user, decides what the user can do. This is how a call-center OU can lock down what a call-center employee sees only when they log onto a call-center machine; their normal-account GPO still applies when they log in elsewhere.

Don’t loopback by accident
Loopback Processing is set in the GPO under Computer Configuration » Administrative Templates » System » Group Policy » User Group Policy loopback processing mode. The default is Not configured. Leave it that way unless you have an explicit kiosk / RDS requirement — misconfigured loopback has caused every junior admin in history to chase their tail for a week.

Security filtering vs WMI filtering

Two mechanisms narrow a GPO’s effective scope. They are not interchangeable.

Security filtering WMI filtering
Targets Who the GPO applies to (users, computers, groups) What the machine looks like (hardware class, OS, features)
Configured by The Security tab on the GPO The WMI Filter dropdown on the GPO
Default state Only Authenticated Users — effectively “everyone” No filter — effectively “all machines”
Typical use Narrow to a security group, e.g. “HR-Workstations” Narrow to laptops, TPM-2.0 machines, etc.
Footgun Leaving Authenticated Users on the GPO and adding a deny elsewhere; the GPO will still apply to most users. WMI query that returns no rows on some machines — silent non-application.

A pattern I have seen work at 500–5000 endpoints: create a security group per OU (GG_Finance_Workstations_Apply, etc.), put the computer objects in the group, set the GPO’s security filter to that group, and never touch Authenticated Users. The GPO has explicit, auditable targeting.

OU hierarchy at 500–5000 endpoints

At your scale, the OU tree stops being decorative. The pattern most admins converge on is a 2×2 of axes: tier (security isolation) and function (what the machines are for). 

domain.local
·
·— Domain Controllers           (Tier 0, has GPO: DCSecBaseline)
··   ·— DC01
··   ·— DC02
·
·— Tier 0 Servers               (PAW, identity, secrets)
··   ·— ...
·
·— Servers
··   ·— FilePrintServers
··   ·— AppServers
··   ·— DatabaseServers
·
·— Workstations
··   ·— BYOD
··   ·— Standard
··   ·— PrivilegedAccess (loopback policy here)
·
·— Kiosks                       (loopback policy here)

The two axes:

  • Tier expresses trust — a Domain Controller in tier 0 only ever talks to other tier 0 systems. The GPO at that tier enforces the trust boundary; not the network firewall alone.
  • Function expresses role — a file server, a workstation, a kiosk. The GPOs at each function express what that role is allowed to do.

Three rules for keeping this tree healthy at scale:

  1. Never nest by both tier and function at the same depth. Workstations / PrivilegedAccess is fine because the privileged-access workstation is still a workstation. Tier0 / Workstations / PrivilegedAccess is bad because it has two truths about trust.
  2. Move objects, don’t edit GPOs. If FINANCE01 needs a different policy, move the computer object to a different OU. Editing GPO filters every time is how GPOs drift.
  3. Block inheritance is a strong statement. Use it on tier 0 OUs (so workstation GPOs cannot accidentally reach DCs). Never on user OUs. Always pair it with Enforced on the GPOs you do want.
The Enforced flag
When a GPO is set to Enforced on its link target, no child OU can block inheritance from stopping it. Enforced is the one-way valve: it guarantees a policy makes it through. Use it sparingly on tier 0 policies; never on user-facing ones.

The four things that go wrong at scale

At 500–5000 endpoints, these four failure modes cause roughly 90% of GPO debugging tickets. Knowing them saves hours.

  1. SYSVOL replication lag

    You edit a GPO on DC01 at 14:00. DC02 may not have it until the DFS-R sync finishes — five minutes later on healthy links, never on broken ones. Machines serviced by DC02 will run with the old GPO until the next sync. Check Get-DfsrState if a policy “isn’t applying” on a subset of machines.

  2. Block Inheritance without audit

    Someone enabled block inheritance on an OU for a real reason. Six months later, a domain-root GPO silently stops applying to that OU and nobody notices. Treat block inheritance as a documented event — if you cannot defend it in a review, remove it.

  3. Security-filter sprawl

    A GPO has 14 groups in its security filter, six deny entries, two WMI filters, and is enforced on three OUs. Diagnosing why one machine is mis-applying it is archaeology. Force the security filter down to one or two groups; force WMI to one filter; force linkage to one OU. Make the GPO boring.

  4. Computer vs User configuration confusion

    Settings placed under User Configuration apply at logon. Settings placed under Computer Configuration apply at boot. If you put “disable autorun” under User Configuration when the computer boots before any user logs in, the setting won’t apply during boot. Match the section to the moment you need enforcement.

What this lesson gives you

  1. AD is a policy tree. OU hierarchy is the spine; GPOs cascade down it; last writer wins. Plan placement from broad (domain root) to narrow (deep OU), and verify with gpresult /h on a target.
  2. Four placement strategies. Domain-root for “must apply everywhere.” Single-OU for business-unit or site variation. WMI-filtered for hardware-class targeting. Loopback-replaced for kiosk / RDS scenarios.
  3. Two filtering mechanisms. Security filtering narrows by who; WMI filtering narrows by what the machine looks like. Keep both simple. Aim for one target.
  4. The tier × function OU pattern. Trust plus role. Tier 0 isolated, workstations split by privilege, block-inheritance audited.
  5. The four things that go wrong. SYSVOL lag, block inheritance without audit, security-filter sprawl, computer-vs-user confusion. Memorize these.
