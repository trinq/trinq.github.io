When to Auto-Merge

Reserve human review for changes that actually need a human. Auto-merge the rest.

This is lesson 5 of engineering-at-scale. Lessons 0001–0004 gave you a workflow and a way to measure it. This lesson makes the workflow fast: how to spend your human review time on changes that actually need it, and how to trust everything else.

Your takeaway: human review is your most expensive resource. Spend it where it pays off. At Spotify, they auto-merge 85% of migrations because the migration script was reviewed once. You can auto-merge most of your PRs the same way — once you've built the trust.

The Spotify Threshold

From the podcast (~26:00):

"We end up at somewhere around 85% of these migrations that are auto-mergeable. So basically we have a tool that looks at the diff and... it can confidently auto-merge it." — Niklas Gustavsson, ~26:00

The remaining 15% need a human because they touch load-bearing code, change behavior, or modify things outside the migration's scope. Spotify's threshold-setting is implicit, but the principle is explicit: auto-merge anything where the generator is trusted and the diff is in scope.

The Three Tiers

Every PR falls into one of three tiers. The tier determines who reviews it (if anyone).

Tier

Reviewer
Auto-merge Generated deterministically. Confidently safe.

How to Set the Threshold

The threshold is trust + scope. Two questions. If both answers are yes, auto-merge.

Question 1: Is the generator trusted?

"Trusted" means: you've reviewed the prompt + the example PR + the exclusion list (lesson 0003), AND you've seen at least 3 successful runs of this generator with correct outputs. Until you've seen 3 runs, the generator is untrusted and all outputs go to skim review at minimum.

This is important. Trust is earned, not assumed. The first run is sketchy. The third run is boring. The tenth run is invisible.

Question 2: Is the diff in scope?

"In scope" means: the file list matches what the prompt asked for, and the exclusion list was respected. Grep for any file in the exclusion list — if the diff touched it, it's out of scope. Grep for any pattern the prompt said to preserve — if it's been modified, it's out of scope.

If both answers are yes, auto-merge. If either is no, the tier drops to the next level down.

The threshold-setting rule: when in doubt, drop the PR down a tier. A PR that gets skim-reviewed when it needed full review causes a regression. A PR that gets full-reviewed when it could've been skim-reviewed wastes 15 minutes of your time. Time is recoverable. Landmines aren't.

The Small-Scale Version: GitHub Branch Protections

You can encode this tier system directly in your branch protections. Here's how:

Tier 1 (Auto-merge)

  • CI must pass (lint, type check, tests)
  • Label auto-merge applied — only a tooling bot can apply this label
  • No required reviewers
  • Branch protection rule: require status checks only, no require pull request reviews

Tier 2 (Skim review)

  • CI must pass
  • One required reviewer
  • PR template asks: "Did this PR come from a reviewed generator? (yes/no)"
  • If yes: reviewer's job is the lesson 0003 generator-check (sample 3 files, grep exclusions, check stats)
  • If no: reviewer's job is full review

Tier 3 (Full review)

  • CI must pass
  • One required reviewer, optionally two for security-sensitive changes
  • PR template asks: "What's the regression plan?" — if blank, block merge
  • Reviewer does a real review
The trap: most teams only have one tier. Either everything requires review (which creates a bottleneck) or nothing does (which creates regressions). Two-tier is the minimum. Three-tier is worth the setup time.

The Exercise: Set Up Your Tiers (25 min)

Pick a single repo. Set up three tiers in branch protections + PR template. This is mostly configuration, not coding.

Step 1 — Create a PR template (~5 min)

Add .github/pull_request_template.md:

## Tier

- [ ] Tier 1: Auto-merge (codemod, generated, dependency update)
- [ ] Tier 2: Skim review (AI-authored, reviewed generator in scope)
- [ ] Tier 3: Full review (manual, behavior-changing, security-sensitive)

## Generator (if Tier 1 or Tier 2)

- Generator prompt/PR: #___
- Generator runs to date: ___
- Files in exclusion list respected: yes / no

## Regression plan (if Tier 3)

How will you know this PR didn't break something?
- [ ] Test added that exercises the changed behavior
- [ ] Feature flag with rollback
- [ ] Monitor metric X for 24h
- [ ] Other: ___

Force the author to declare the tier. The declaration is the gate. If they checked Tier 3 but the change looks like Tier 1, the reviewer downgrades it.

Step 2 — Configure branch protection (~5 min)

You can't gate by tier programmatically (GitHub branch protections don't read PR bodies). But you can:

  • Require status checks (CI) — applies to all tiers
  • Require 1 reviewer — applies only to Tier 2 and Tier 3
  • Set up CODEOWNERS files for sensitive paths — auto-requests review from the right person for auth/, db/, security/, etc.
  • Use the auto-merge bot token for Tier 1 PRs (only the bot can apply the label, and the bot only applies it to known codemod PRs)

Step 3 — Set the "trust threshold" (~10 min)

For each generator you plan to use, write a one-line entry in trust-registry.md:

# Generator Trust Registry

## type-wrapper-codemod (prompt v2)
- First run: 2026-06-15 (PR #142)
- Successful runs: 4 of 4 (PRs #142, #148, #151, #155)
- Status: trusted → Tier 1 eligible
- Last regression: none

## notification-deduplicator refactor
- First run: 2026-06-22 (PR #160)
- Successful runs: 1 of 2 (PR #160 had a scope bug)
- Status: untrusted → Tier 2 maximum until 3 successful runs
- Last regression: PR #160, root cause: scope text didn't say `src/notif/*` explicitly

The threshold is mechanical: 3 successful runs = trusted. Less than 3 = untrusted. Update the registry after every generator run.

Step 4 — Connect to your scorecard (5 min)

Add a fourth metric to your fleet-scorecard.md:

## This week
- Landmines caught: 2
- Generator merges: 3 of 5 refactors (60%)
- Post-deploy regressions: 0
- Tier 1 / Tier 2 / Tier 3 split: 1 / 3 / 2
  (good ratio: Tier 1 should grow over time)

The split tells you whether you're earning the trust. If Tier 3 ratio stays at 80% for a month, you're not generating enough cheap PRs to auto-merge. If Tier 1 jumps to 90%, you've stopped reviewing generators carefully.

Why the 3-Run Threshold?

One run is anecdote. Two runs is coincidence. Three runs is pattern. After three successful generator runs with no regressions, you have evidence that the generator is sound.

You can adjust the number based on the change's blast radius. A YAML config rename might only need 1 run to be trusted. A security-sensitive refactor might need 10. The principle is the same: count successful runs in your risk-weighted score, not just calendar weeks.

What You Should NOT Do

  • Don't make everything Tier 3. That's the bottleneck. Tier 3 should be the exception, not the default. If 80% of your PRs are full-review, you haven't built enough trust in your generators yet. Go back to lesson 0003.
  • Don't auto-merge on the first run of a generator. Always skim-review the first 3 runs. After 3 clean runs, the generator earns Tier 1.
  • Don't auto-merge if the exclusion list wasn't checked. Auto-merge requires evidence that the generator stayed in scope. The PR template's "files in exclusion list respected" question is the gate.
  • Don't trust AI-generated tests. "I'm sure the test passed" is not a regression plan. The lesson 0003 scope check doesn't apply to test changes the agent also wrote. Tier 2 at minimum for AI-touched tests, Tier 3 if you can't verify the test exercises the failure mode.
  • Don't skip the regression plan on Tier 3. "How will you know this didn't break something?" must be answered before merge, not after.

The Connection to Lesson 0004's Anti-Metrics

Lesson 0004 warned against measuring "AI lines written." Auto-merge Tier 1 PRs are an even sneakier false metric. If you count "PRs auto-merged" as a win without counting regressions, you'll game yourself into shipping landmines at scale. The scorecard's regression counter is what catches this.

A team that auto-merges 50 PRs a week and has 5 regressions is doing worse than a team that manual-reviews 10 PRs and has 0 regressions. The aggregate is the only honest measure.
