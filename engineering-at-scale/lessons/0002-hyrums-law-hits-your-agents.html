engineering-at-scale Lesson 0002 2026-06-30 ~10 min

Hyrum's Law Hits Your Agents

Why AI migration tools fail — and how to find the parts of your codebase they will break.

This is lesson 2 of engineering-at-scale. Lesson 0001 had you skip a PRD for one feature ticket. This lesson teaches you to spot the code your agents will silently corrupt.

Your first lesson takeaway: you have some feature shape that you want to validate quickly with a prototype. Lesson 2 takeaway: before you let an agent touch your existing code, you need to know which behaviors of that code are load-bearing even though you never documented them.

The Law

From hyrumslaw.com, originally observed by Hyrum Wright at Google:

"With a sufficient number of users of an API, it does not matter what you promised in the contract — all observable behaviors of your system will be depended on by somebody."

At Spotify scale, this stops being theoretical. From the podcast (~27:00):

"Imagine you have this well-defined API, but there are quirks of the implementation of that and every one of those quirks, if you have large enough scale, someone is going to have a dependency on that quirk." — Niklas Gustavsson, ~27:00

This is why deterministic migrations hit a glass ceiling. You write a script that rewrites every use of the old API to the new one. The script catches 99% of cases. Then someone files a bug because their code called the old API in a way your script didn't recognize, and now their code is broken — and they never read your docs.

LLMs are the workaround. They pattern-match on the call site in context. They get more cases. But they also hallucinate fixes and miss edge cases. There's no free lunch.

Why This Matters to You (At Small Scale)

You don't have 100M users. You don't have a "fleet" of services. So why does this matter?

Because you are now the customer of your own APIs. Your agents are. Every codebase you're working in has internal APIs — functions, modules, classes — that you've used in places you don't remember. When an agent rewrites one of those APIs, the rewrite will look correct. The agent will have done its job. But some other module — perhaps one you wrote six months ago and forgot — depended on a quirk that's now gone.

A common case: sort order

You wrote getRecentPosts() last year. The function returns posts sorted by created_at. You never documented the tiebreaker — it happens to be insertion order because that's what your SQL query gave you.

// Documented: "Returns posts sorted by creation date, newest first"
SELECT * FROM posts ORDER BY created_at DESC;

An agent decides to "optimize" this and adds an explicit id-based tiebreaker for determinism:

// Agent's "improvement"
SELECT * FROM posts ORDER BY created_at DESC, id DESC;

Looks better. Is faster. Is more deterministic. But your notification-deduplicator module relied on the old ordering — same timestamp meant "earlier insertion wins." Two users who should have received one notification each now receive two. Bug filed three weeks later.

This is Hyrum's Law in miniature. No customer-facing API was violated. The contract was honored. A behavior was changed, and someone downstream depended on it.

The Pattern at Spotify Scale

Spotify ran into this so often they had to build Honk — an LLM-mediated migration runner that uses one LLM to write the change and another LLM to verify the change matches the prompt. Even with both, edge cases slip through. That's why they keep humans in the loop for AI-authored PRs.

Their lesson for you: the more an agent writes, the more you need to verify what it changed beyond the diff itself. "This PR looks correct" is not enough. "This PR preserves every observable behavior" is the bar.

The Exercise: Audit One Function (15 min)

Pick one function in a codebase you actively maintain. Criteria:

  • It's been in the codebase for at least 3 months.
  • It's called from at least 3 different places.
  • It's small enough to read in one sitting (under 30 lines is ideal).
  • It has at least one "implementation detail" you didn't document.
Not a public API. Internal is fine — better, in fact. The point is to find the load-bearing behaviors in code you control, before an agent decides to "improve" them.

Step 1 — List the documented behaviors

Open the function. What does its docstring (or comments, or README) say it does? Write those down.

// Documented:
// 1. Returns posts sorted by created_at DESC
// 2. Limits to N results
// 3. Skips deleted posts

Step 2 — List the undocumented behaviors

Read the implementation. What does it actually do? List everything that's not in the docstring:

// Implementation quirks:
// - Tiebreaker is implicit (insertion order / row id)
// - Caches results for 60 seconds
// - Throws on DB connection failure, returns [] on query timeout
// - Includes draft posts where author_id = current_user
// - Excludes private posts where viewer_id != author_id
//   (yes, even though they're not deleted)

Step 3 — Classify each quirk

For each undocumented behavior, fill in this table:

Quirk Caller depends on it? Safe to change?
Tiebreaker = insertion order Probably yes (deduplicator) No — silent data corruption
60s cache Probably yes (perf) No — latency regression
Returns [] on query timeout Maybe (caller might check) Rename to make explicit first
Drafts visible to author Yes (UI assumes it) No — feature regression
Private post exclusion Yes (privacy invariant) Never — security

Step 4 — Mark the agent-hostile ones

Anything marked "No — silent data corruption" or "Never" is a landmine. An agent rewriting this function will almost certainly break it without realizing it. Two options:

  • Document the quirk so the agent has a chance of preserving it. Move it from "implementation detail" to "documented contract."
  • Lock the function — don't let agents touch it. Add a comment like // AGENT-DO-NOT-MODIFY: tiebreaker behavior is load-bearing, see #1234. This is ugly but works.

For now, do whichever is faster. Document later.

Self-Check

You should be able to tick all of these:

  • I picked one function (or method, or module) for the audit
  • I listed its documented behaviors
  • I listed at least 3 undocumented behaviors
  • I classified each quirk by "caller depends on it" and "safe to change"
  • I identified at least one "landmine" the agent would break
  • I either documented the landmine or marked the function as agent-hostile

What You Should NOT Do

Common mistakes:

  • Don't audit your whole codebase. Pick one function. The exercise scales to ten later, but you have to do one first.
  • Don't try to fix the landmines. The goal is to find them, not to rewrite the function today. Fixing happens in a follow-up.
  • Don't pick a function you wrote last week. You'll only know the behaviors you intended. Pick something old enough that you've forgotten what you did.
  • Don't pick a function with no callers. If nothing calls it, Hyrum's Law doesn't apply. Pick something with real dependents.

Why This Isn't Just Theory

From the podcast (~28:00):

"We have our backend is mostly Java and we have many of these code migrations that try to migrate some way of using Java to some other way... and we find so many edge cases to these all the time. It is fascinating how challenging it is to modify code." — Niklas Gustavsson, ~28:00

Spotify's fleet shifts do migrations that affect millions of lines of Java. Your agent-driven code changes affect dozens or hundreds of lines. The ratio is different; the principle is identical.

The good news: at your scale, you can audit every important function in a week. Spotify can't audit Java's entire standard library. Use your size as an advantage.
