Code Ownership at Scale

Make ownership explicit before you grow. "Everybody owns everything" is the worst answer.

This is lesson 7 of engineering-at-scale. Lessons 0001–0006 gave you a workflow that protects a single repo. This lesson is about what happens when the workflow meets multiple repos, multiple engineers, and a component graph too big for one person to hold.

Your takeaway: ownership is implicit at one person, explicit at two, and political at three. Encode it in CODEOWNERS and runbooks before you're at three. Encode it before you grow — not after you collapse.

The Spotify Ownership Model

From the podcast (~22:00):

"We basically have one engineer per squad that's the component owner for that component. And what that means is they're responsible for keeping the lights on — they get paged when things break — but they also have to approve changes to that component." — Niklas Gustavsson, ~22:00

Spotify's pattern: every component has exactly one named owner. The owner approves changes (lesson 0005's full-review tier), responds to incidents (lesson 0006's on-call), and maintains the runbook. One name, one pager, one runbook per component.

This is the polar opposite of the "everybody owns everything" model that emerges by accident when nobody makes ownership explicit. That model has no owner, so any change is everyone's problem, so no change gets attention, so the codebase rots.

The Three Ownership Patterns

Three patterns exist. They have very different failure modes.

Pattern
Description
Failure mode
Nobody owns it "Everybody owns it."
No assigned owner. Anyone can change anything. Nobody is on call.
The code rots. PRs get stuck in review. Incidents have no responders. The team grows to "fix the ownership problem."
Committee owns it "The team owns it. Three of us review."
Multiple owners. Approval requires N of M. Review by committee.
Slow decisions. Diffuse responsibility. "I thought YOU were on call." Decision paralysis at the smallest change.
One person owns it "X is the component owner for auth."
Exactly one owner. The owner approves, responds, maintains the runbook. Backup owner named for vacations.
Bus factor of 1. Single point of failure. Mitigated by runbook + backup owner.

Spotify picked the third pattern deliberately. A bus factor of 1 sounds scary, but the alternative is "nobody" or "committee," both of which fail worse. The mitigations are runbook (the code is documented) and backup owner (the pager is shared).

The Small-Scale Translation

You work solo. Today, you own everything. That's fine. But you can still apply the pattern:

  1. CODEOWNERS — declare yourself the owner of every path. Future-you joins the repo, sees the pattern, knows where to start.
  2. Runbook per critical module — auth, billing, data, anything that breaks the business. One page each.
  3. Backup owner as a role — name a friend, an outside dev, or "future-hire" as the backup. Whoever is on the runbook when you can't be.

When you grow to two engineers, the pattern is already in place. Cut components in half. Assign one to each. Update CODEOWNERS. The pattern is now load-bearing.

When you grow to three, the pattern is already in place. Split out the third component. The pattern still works because it scales by component, not by engineer.

When you grow to ten, you'll be glad the pattern existed at three.

The CODEOWNERS File

CODEOWNERS is the operational encoding of ownership. One file, one rule per component. Located at .github/CODEOWNERS, docs/CODEOWNERS, or CODEOWNERS at the repo root.

Format

# CODEOWNERS — last updated 2026-06-30
# One owner per component. Backup owners listed with @.

# Default owner for everything not listed below
* @your-handle

# Critical paths require backup owner review
/auth/          @your-handle @backup-handle
/billing/       @your-handle @backup-handle
/db/migrations/ @your-handle @backup-handle
/security/      @your-handle @backup-handle

# Component-level ownership
/src/notifications/  @your-handle
/src/feed/           @your-handle
/src/profile/        @your-handle
/src/api-gateway/    @your-handle
/web/                @your-handle
/mobile/             @your-handle

# Generated code — bot owns it, not humans
/src/generated/      @auto-merge-bot
/codemod-output/     @auto-merge-bot

# Documentation — anyone can review
/docs/               @your-handle

The rules

  • Default owner at the top. * @your-handle first. Then exceptions. The default is what catches paths you forgot to enumerate.
  • One owner per path. Not two, not three. One. Code review is unblocked by one approval.
  • Backup owner for critical paths only. Don't enumerate backups for everything. The backup role is for components where a missed alarm hurts. Auth, billing, schema, security.
  • Bot owners for generated code. The auto-merge bot from lesson 0005 owns /src/generated/. Humans don't review Tier 1 codemod output unless the bot flags it.
  • Last-owner-wins. GitHub matches the deepest pattern. So * at the top, then specific overrides. The override wins for that path.

The Runbook

Code ownership without a runbook is "person on call who doesn't know the system." A runbook is one page per critical module. It tells the on-call (you, today; future-you or a teammate tomorrow) what to do when something breaks.

# Runbook — notifications

## Component
Push notifications: queue, delivery, deduplication.

## Owner
@your-handle (primary), @backup-handle (backup)

## What this does
Receives notification requests from upstream services, deduplicates
them, queues to delivery, retries on failure. Drop targets: 99.5%
delivery within 60s.

## Common failure modes

### Delivery rate drops
**Symptom:** scorecard shows < 95% delivery over 1h
**First check:** `kubectl logs -n notif deploy/delivery | grep ERROR`
**Likely cause:** upstream queue full
**Fix:** scale deployment: `kubectl scale deploy/delivery --replicas=10`
**Rollback:** if scale doesn't help, revert last PR via `git revert HEAD`

### Duplicate notifications
**Symptom:** users report getting the same notification twice
**First check:** scorecard dedup-hit-rate metric
**Likely cause:** dedup cache TTL expired
**Fix:** increase TTL via config flag `notif.dedup_ttl_seconds`
**Rollback:** revert TTL change via `git revert HEAD`

### Notification spam (single user)
**Symptom:** one user reports getting > 50 notifications in 1h
**First check:** check that user's notification settings page
**Likely cause:** upstream rate-limiter not honoring user pref
**Fix:** user-side: throttle that user's notifications
**No rollback needed** — this is a per-user config issue

## Recent incidents
- 2026-06-30: dedup sort-key flip (Sev-2, see postmortems/2026-06-30-notif-dedup.md)

## Related docs
- Architecture: docs/notifications-architecture.md
- Scorecard: tools/fleet-scorecard.md
- Trust registry: trust-registry.md (notification-deduplicator entry)

One page. Five sections. The "common failure modes" section is the load-bearing part — it tells the on-call what to do in the first 5 minutes without thinking.

The Exercise: Set Up Ownership (30 min)

Pick one repo. Do this even though you own everything. The exercise is about establishing the pattern, not about scaling.

Step 1 — Write CODEOWNERS (~10 min)

Start with * @your-handle. Add 3-5 critical paths with backup owners (even if the backup is "future-hire"). Add a bot owner for any /generated/ or codemod output directories.

Critical paths to list explicitly: - /auth/ (or wherever login lives) - /billing/ (or wherever payments live) - /db/migrations/ (schema changes) - /security/ (anything touching secrets) - Plus 2-3 of your largest non-critical components

Step 2 — Write 3 runbooks (~20 min)

Pick the three most-likely-to-break components. Use the template above. Five sections each:

  • Component (one-line description)
  • Owner (primary + backup)
  • What this does (1-2 sentences)
  • Common failure modes (with symptom → first check → fix → rollback)
  • Recent incidents (link to postmortems)

Each runbook: 5-10 minutes to write. Don't over-engineer. The point is "exists," not "perfect."

Step 3 — Connect CODEOWNERS to lesson 0005 (~5 min)

Branch protection: enable "require review from Code Owners." This makes the lesson 0005 tier question partially automatic:

  • Tier 1 (auto-merge) — bot owner, no human review
  • Tier 2 (skim review) — bot may label, but owner gets ping
  • Tier 3 (full review) — owner must approve

The CODEOWNERS file is the load-bearing artifact. Branch protection reads it. Runbooks reference it. PR templates (lesson 0005) reference it. Postmortems (lesson 0006) reference it. **One file, many consumers.**

What CODEOWNERS Does Well

  • Auto-requests review — git platform handles the routing. No "I forgot to tag the right person."
  • Makes ownership visible — anyone joining the repo sees who owns what before they make a change.
  • Scales with the codebase — same file at 100 lines and 100,000 lines.
  • Encodes the bus factor — backup owners document the second-line responder.
  • Plays well with lesson 0005 — Tier 3 PRs auto-route to the right reviewer.

What CODEOWNERS Does Poorly

  • Doesn't tell you what the owner *does* — it tells you who reviews. Not who maintains, who docs, who runs tests.
  • Single-file blizzard — large repos get hundreds of owners. The file becomes unreadable.
  • Ownership ≠ expertise — the named owner might not know that part of the code anymore. Don't trust ownership without a runbook.
  • Drifts — owners leave the team, change roles, go on vacation. The file gets stale. Audit it quarterly.
  • Doesn't handle shared code well — utility libraries, framework code. These need "shared owner" patterns.
The trap: treating CODEOWNERS as the full ownership answer. It's the routing layer. The substance is the runbook + the trust (from lessons 0003 and 0005) that the owner knows what they're doing. CODEOWNERS without runbooks is "we know who to page, but they don't know what to do."

Why This Lesson Matters Even If You're Solo

The most common reason ownership becomes a problem is "we're a small team, we'll add ownership when we grow." That never happens. By the time you grow to 3 engineers, the ownership question is load-bearing and you have no practice answering it. The pattern is harder to retrofit than to install at scale.

Writing CODEOWNERS + 3 runbooks today, while you're solo, means the pattern exists when you grow. The pattern is the answer. The specific owners (just you today) can change.

What You Should NOT Do

  • Don't have an empty default. * at the top must have an owner, or undeclared paths become "nobody owns it."
  • Don't list too many owners per path. Two is the max. Three is a committee. One is best. The point is unblocking reviews, not requiring consensus.
  • Don't skip the runbook. CODEOWNERS tells you who. Runbooks tell them what. CODEOWNERS without runbooks is "we know who to page, but they don't know what to do."
  • Don't let CODEOWNERS drift. Owners leave the team. The file gets stale. Review it every quarter. Tag people who left as `@former-eng` for context, not as new owners.
  • Don't have ownership without an escalation path. Every owner has a backup or the runbook says "if I'm unresponsive in 15 min, page @backup." A pager that doesn't escalate is a black hole.
  • Don't over-document at solo scale. Three runbooks, not thirty. The point is "the pattern exists," not "every file documented."

The Connection to Lessons 0005 and 0006

Lesson 0005's tier system works better with CODEOWNERS. Tier 3 PRs auto-route to the right reviewer. The CODEOWNERS file encodes "who reviews what" so the tier declaration is just metadata.

Lesson 0006's on-call rotation works only with CODEOWNERS + runbook. The pager needs a name (CODEOWNERS) and a thing to do (runbook). Without both, the incident has no responder.

Lessons 0005 and 0006 are operational. Lesson 0007 is structural. They're three views of the same answer: ownership is explicit, routed, and documented before it's needed.
