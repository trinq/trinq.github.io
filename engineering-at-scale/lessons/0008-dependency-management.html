Dependency Management

Every dep is a security surface, a maintenance burden, and a learning curve. The threshold for adding one is higher than most teams realize.

This is lesson 8 of engineering-at-scale. Lessons 0001–0007 gave you a workflow for building, reviewing, measuring, and owning code. This lesson is about the code you don't write — the code you pull in from outside. Every dependency is a trust decision. This lesson is about making that decision consciously.

Your takeaway: the cost of a dependency is paid over years, not at install time. The install is free. The CVE at 2am on a weekend is not. The abandoned package two years from now is not. The decision to add a dep should weight those costs.

The Spotify Policy

From the podcast (~29:00):

"We have a fairly conservative dependency policy. We don't want to take in dependencies lightly, because once you're taking in a dependency, that team is effectively in your codebase. And if they do something bad, that's your problem." — Niklas Gustavsson, ~29:00

The key insight: a dependency is not a library — it's a team. The moment you add a dep, that team's decisions are your decisions. Their security posture is your security posture. Their maintenance velocity is your maintenance velocity. Their deprecation is your migration. Install is free. Ownership is not.

The Three Costs of Every Dependency

Cost 1: Security surface

Every dependency is an attack vector. The 2021 Log4Shell vulnerability affected millions of Java deployments. The 2022 node-ipc protestware corrupted user files. The 2023 xz-utils backdoor was caught before widespread deployment — this time. The average Node.js project has 80+ transitive dependencies. Each one is a potential CVE.

Security cost is not "will this package get a CVE" — it's "when this package gets a CVE, what is my response time?" The answer depends on how fast you can update, how well-tested your upgrade path is, and whether the package has a maintained replacement.

Cost 2: Maintenance burden

Dependencies need updating. Major version upgrades. API changes. Deprecation removals. Breaking changes in transitive deps. Each update is a potential regression. The more deps you have, the more time you spend updating instead of building.

The average JS project spends 15-20% of its engineering time on dependency maintenance. Most of it is invisible — it doesn't show up in sprint planning, it doesn't produce features, it just keeps the lights on.

Cost 3: Lock-in and learning curve

Every dependency has an API. The API is a contract. When the dep changes its API, you change your code. When the dep is abandoned, you own its API forever or migrate it. When the dep does something unexpected, you need to understand its internals to debug it.

Learning curve cost is paid by every engineer who touches the code. A dep with 5,000 users has StackOverflow answers and blog posts. A dep with 5 users has nothing. The niche dep that saves you two weeks of work might cost you two weeks of debugging when it breaks.

The Four-Tier Dependency Policy

Tier every dep before adding it. The tier determines the review depth, the monitoring requirements, and the upgrade cadence.

Tier Description Examples Review depth Upgrade cadence
Tier 1 Language runtime or foundational tooling Node.js, Python stdlib, webpack, npm Security watchlist only LTS schedule
Tier 2 Well-maintained, widely-used, narrow-purpose library lodash, date-fns, zod, axios Security audit + maintainer check Monthly
Tier 3 Newer, less-widely-used, or broader-purpose library Specific utility libraries, UI component libs, new tooling Full: maintainer history, GitHub activity, CVE history, transitive deps Bi-weekly
Tier 4 Single-author, unmaintained, or beta library npm packages with no commits in 6mo, experimental tools Not allowed without explicit exception from owner —
The tier question: if the answer to "who maintains this?" is "I don't know" or "one person who might stop tomorrow," it's at least Tier 3. If the last commit was more than 6 months ago, it's Tier 4 unless you have a written migration plan.

The Decision Framework: Should I Add This Dep?

Before running npm install, answer these five questions. If you can't answer all five, the dep doesn't belong in your project.

Question 1: Is this solving a real problem or a hypothetical one?

Don't add a dep to solve a problem you don't have yet. The most common dep mistake: "this library looks useful, let's add it" → the problem it solves never comes → the dep accumulates security updates → the dep gets a CVE → you spend 4 hours migrating off it.

Only add deps for problems you have right now. "I'll need this later" is not a valid reason.

Question 2: What is the maintenance track record?

Look at: last commit date, release frequency, open issue count, response time on issues, security vulnerability history, maintainer diversity (single vs team).

Red flags: no commits in 3+ months, open critical issues without response in 2+ weeks, history of unaddressed CVEs, maintainer has abandoned other popular packages.

Question 3: What does the transitive dependency tree look like?

npm ls --depth=2 (Node) or pipdeptree (Python). You're not just adding one dep — you're adding its entire dep tree. A dep that looks clean at the top might have 40 transitive deps, some of which are Tier 4.

If the transitive tree is too deep or too large, find a narrower alternative. A 5kb dep with no transitive deps is better than a 200kb dep with 30 transitive deps.

Question 4: What is the migration path?

If this dep is abandoned tomorrow, what's the cost to remove it? Calculate:

  • How many files import it?
  • How many API calls are used?
  • Is there a maintained alternative with a compatible API?
  • Is the API thin enough to wrap in a local abstraction?

If you can't answer these, add a local wrapper around the dep's interface. The wrapper isolates the API contract. When the dep needs to change, you change one file, not twenty.

Question 5: Does this belong in the language ecosystem or the platform?

Some problems are solved better by the language or platform than by a library. Before adding a dep, check:

  • Node: does this exist in the built-in modules (fs, path, stream, crypto)?
  • Python: does this exist in the stdlib?
  • Browser: does this exist in the Web APIs?

If yes, use the platform. No dep to update. No CVE surface. No migration path to maintain.

Dependabot and the Automation Layer

The decision to add a dep is manual. The decision to update it should be automated. Use Dependabot (GitHub) or Renovate (self-hosted) to auto-open PRs for dep updates. Configure the update frequency and merge policy.

# .github/dependabot.yml
version: 2
updates:
  - package-ecosystem: "npm"
    directory: "/"
    schedule:
      interval: "weekly"
      day: "monday"
    open-pull-request-limit: 10
    groups:
      production-dependencies:
        dependency-type: "production"
        update-types:
          - "minor"
          - "patch"
      development-dependencies:
        dependency-type: "development"
        update-types:
          - "minor"
          - "patch"
    # Major version updates always get their own PR — too risky to group
    ignore:
      - dependency-name: "*"
        update-types: ["version-update:semver-major"]
    labels:
      - "dependencies"
      - "automated"

  - package-ecosystem: "pip"
    directory: "/"
    schedule:
      interval: "weekly"
      day: "monday"
    open-pull-request-limit: 5
    labels:
      - "dependencies"
      - "automated"

The grouping policy matters: group minor/patch updates in production, but always split major updates. Major version updates are where breaking changes live. Grouping them masks what changed.

The automation rule: every dep update that can be automated should be automated. Your time is for major version migrations and Tier 3+ review — the decisions that actually need human judgment. Routine minor updates are a waste of human attention.

The Exercise: Audit Your Deps (40 min)

Step 1 — List all deps and their tiers (~10 min)

Run npm ls --all 2>/dev/null | head -200 or equivalent for your language. Every dep goes into one of four tiers. If you're not sure, the dep is Tier 3.

# deps-audit.md — updated 2026-06-30

## Tier 1 — Runtime & Tooling

|---|---|---|---|

| npm | 10.x | Package manager | 2026-06-30 |
| typescript | 5.x | Type checker | 2026-06-30 |

| Package | Version | Purpose | Last audit |
|---|---|---|---|

| Package | Version | Purpose | Last commit | Issue response | Transitive deps |

| Package | Version | Reason | Action |

| experimental-tool | 0.0.x | Experimental, no stability guarantee | Evaluate after 1.0 |

Step 2 — Check transitive dep explosion (~5 min)

For each Tier 2 and Tier 3 dep, run the transitive tree check. Flag anything that pulls in more than 20 transitive deps without a good reason. The "good reason" is: the dep is a framework (Next.js, Django) or the dep is unavoidable (React). Everything else should be lean.

Step 3 — Check for abandoned deps (~5 min)

Look at last commit date for every Tier 3 and Tier 4 dep. Anything with no commits in 6+ months goes to Tier 4. File an issue: "should we remove this dep? Last commit: [date]. Migration path: [what would we do]."

Step 4 — Add a local wrapper for one broad dep (~10 min)

Pick a Tier 2 dep that you use in more than 3 files. Write a local wrapper that isolates its interface. This is the migration insurance: when the dep needs to change, you change one file.

# src/lib/date-utils.ts
// Wrapper around date-fns to isolate the interface.
// If we ever need to switch date libraries, change only this file.

import { format, parseISO, addDays, subDays, differenceInDays } from 'date-fns';

export function formatDate(date: string | Date, pattern: string): string {
  const d = typeof date === 'string' ? parseISO(date) : date;
  return format(d, pattern);
}

export function addDaysToDate(date: string | Date, days: number): Date {
  const d = typeof date === 'string' ? parseISO(date) : date;
  return addDays(d, days);
}

export function daysBetween(start: string | Date, end: string | Date): number {
  const s = typeof start === 'string' ? parseISO(start) : start;
  const e = typeof end === 'string' ? parseISO(end) : end;
  return differenceInDays(e, s);
}

// Re-exported for consumers that need direct access
export { parseISO, subDays } from 'date-fns';

Step 5 — Enable Dependabot (~10 min)

Add .github/dependabot.yml to the repo. Set weekly schedule, group minor/patch, split major. Enable the PR limit (10 is a good starting point). Label all PRs with dependencies + automated for easy filtering.

The Rule: Prefer Narrow Over Broad

The most consistent pattern in good dependency management is: prefer narrow APIs over broad frameworks. A 5kb dep with one purpose is better than a 200kb framework with fifty purposes you don't need.

Instead of Use Why
Moment.js (200kb) date-fns (5kb) or Temporal Immutable, tree-shakeable, no mutation surprises
Lodash (full, 70kb) Native JS methods + lodash-es (tree-shakeable) Modern JS covers most lodash use cases natively
jQuery Native DOM APIs Native is universally supported, no migration needed
Class-transformer + class-validator Zod (narrower, typed end-to-end) Single dep, schema + validation in one, inferred types
axios Native fetch No new dep for a use case browsers handle natively
The trap: reaching for a dep instead of the language. Lodash's _.groupBy is Object.groupBy in modern JS. _.uniqBy is [...new Map(...)] or Array.from(new Set(...)). Before adding a dep, check if the problem is solved by a method you haven't learned yet.

Why This Lesson Matters for the Prior Lessons

Dependencies are invisible to the workflow. Lessons 0001–0007 describe how you work with code you write. This lesson is about code you pull in — and it has the same risk surface as the code you write, plus more:

  • Lesson 0002 (audit): a new dep should be audited before it's added. The audit should include the five questions above.
  • Lesson 0003 (generator): AI-generated code that imports a dep is adding a dependency. The lesson 0003 review should catch dep additions and apply the five-question framework.
  • Lesson 0004 (scorecard): add a dep health metric: count of outdated deps, count of Tier 3+ deps, count of open critical CVEs.
  • Lesson 0005 (auto-merge): Dependabot PRs are Tier 1 (auto-merge for minor/patch, auto-reject for major unless reviewed). Add a rule: "Dependabot dep bumps get auto-merged on the weekly schedule unless flagged."
  • Lesson 0006 (incident response): a dep CVE is a potential SEV-1. Runbook should include "dep CVE response": assess impact, patch version, rollback if needed.
  • Lesson 0007 (ownership): the dep owner is not you — it's the library maintainer. Document in the runbook: "if notification-deduplicator's dep x has a CVE, who is responsible?" The answer: you, even though you don't maintain x.

What You Should NOT Do

  • Don't add a dep to solve a problem you don't have yet. "This looks useful" is not a reason. Wait until the problem exists, then evaluate the dep against the five questions.
  • Don't skip the transitive dep check. A dep with a clean top-level surface might have a nasty transitive dep tree. Check npm ls --all before adding.
  • Don't use abandoned deps. If the last commit is more than 6 months ago and the maintainer hasn't posted a deprecation notice, treat it as abandoned. File a migration issue today.
  • Don't wrap broad deps without a migration plan. The wrapper buys you time, not safety. If the dep is abandoned, the wrapper is a migration liability. Remove the dep or maintain it yourself.
  • Don't ignore Dependabot PRs. "We can update later" is how you end up with 200 outdated deps and a migration that takes a week. Weekly update cadence is the right pace.
  • Don't add deps without a local owner. Every dep needs a human who cares about its maintenance. If no one owns the dep, no one will update it, and no one will notice when it has a CVE.
