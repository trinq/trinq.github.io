engineering-at-scale Lesson 0004 2026-06-30 ~10 min

Measure the Aggregate

The metrics that survive scale are the metrics worth using now.

This is lesson 4 of engineering-at-scale. Lessons 0001–0003 gave you a workflow: prototype to validate shape, audit to find landmines, generator-review to ship changes. This lesson gives you the measurement layer — how to know if the workflow is actually working, without optimizing for the wrong thing.

Your takeaway: measure what moves when the system improves, not what moves when people try to look good. At Spotify, that means aggregate deploy frequency, not individual lines of code. At your scale, it means the same.

The Spotify Policy

From the podcast (~22:00):

"We don't measure individuals. We never have. We measure teams, we measure aggregates. But individuals? Never." — Niklas Gustavsson, ~22:00

The reason is not philosophical — it's functional. At scale, any individual metric becomes gameable. If you measure lines of code, you get verbose code. If you measure PRs merged, you get small PRs that split one logical change into ten. If you measure test coverage, you get tests that exercise the happy path. Every individual metric has an adversarial response.

The aggregate doesn't lie like that. If the team's deploy frequency went up 3x after adopting a new workflow, the workflow worked — regardless of whether any individual developer looks more or less productive. The aggregate is robust to gaming.

The Small-Scale Translation

You work alone or on a tiny team. You might think this doesn't apply to you. You're wrong.

The moment you start using AI agents, you'll feel the pull to measure the agent: "how many lines did the agent write this week?" That number is meaningless. Here's why:

  • The agent wrote 500 lines in one PR. 490 were landmines it introduced. The other 10 fixed a real bug. Net: -480 lines.
  • The agent wrote 50 lines in another PR. All 50 were correct. Net: +50 lines.
  • Total: 550 lines. You look like you're shipping fast. You're actually accumulating debt.

Lines written is a bad metric because it measures activity, not outcome. At scale, Spotify learned to measure outcomes: did the change make it to production? Did it stay there? Did it break anything?

The Three Metrics That Actually Matter

For a small-scale developer using AI agents, here are the three metrics worth tracking. Track them weekly. Don't track anything else.

Everything else — lines written by agent, PRs merged, files changed, time-to-review — is noise. If landmines caught goes up and post-deploy regressions goes down, the workflow is working. If generator merges goes up and post-deploy regressions stays flat, the workflow is working. The metrics converge on the right answer.

The Exercise: Set Up Your Weekly Scorecard (15 min)

Create a single file. No dashboards. No spreadsheets. One markdown file you update every Friday.

# Fleet Scorecard — Week of 2026-06-30

## This week's summary
- Landmines caught: 2 (function: getRecentPosts tiebreaker, notification-deduplicator sort)
- Generator merges: 3 of 5 refactors (60%)
- Post-deploy regressions: 0

## Trends (4-week rolling)
- Landmines caught: 2 → 1 → 3 → 2 (avg 2.0)
- Generator merges %: — → 20% → 40% → 60% (improving)
- Post-deploy regressions: 0 → 0 → 1 → 0 (watch the one)

## Notes
- Week 3's regression was the getRecentPosts tiebreaker change.
  Root cause: exclusion list didn't include deduplicator caller.
  Fix: added caller-path audit to lesson 0002 exclusion-list step.
- Week 4 improvement: landmines caught up because I started doing
  the audit BEFORE running the generator, not after.

## Next week goals
- [ ] Get generator merges to 80%
- [ ] Catch the auth module landmine (identified, not yet documented)
- [ ] Write the auth module exclusion entry

That's the entire tracking system. Update it weekly. Look at it monthly to see if the trends are moving the right direction.

Step 1 — Create the file

Name it fleet-scorecard.md in your project root. Start with this week's baseline.

Step 2 — Fill in landmines caught

Go back to lesson 0002. Did you audit any functions this week? How many landmines did you find? Write the number. If zero, write zero and note why you didn't audit.

Step 3 — Fill in generator merges

How many refactors did you ship? How many used generator-review (lesson 0003)? Calculate the percentage. If none used generator-review, note why.

Step 4 — Fill in regressions

Did anything break this week? Write the count. For any regression, write a brief root cause. The root cause feeds the "Notes" section and informs next week's lesson 0002 audit priorities.

Step 5 — Set next week's goals

One or two concrete goals, not a list of ten. If you missed a goal this week, carry it forward. Don't punish yourself; just track.

Why This Is the Hardest Lesson

Lessons 0001–0003 gave you techniques. This lesson asks you to stop optimizing for the thing that feels productive and start optimizing for the thing that actually is.

The pull to measure "agent lines written" will be constant. It feels like progress. It isn't. The scorecard is designed to make the right thing easy and the wrong thing visible: when lines written goes up but regressions also goes up, you'll see the correlation. When landmines caught goes up and regressions goes down, you'll see that too.

The aggregate scorecard is a forcing function. It makes you do the lesson 0002 audit every week, because you need the number for the scorecard. It makes you use generator-review, because you need the percentage. It makes you track regressions, because the number tells you if the other two are working.

The trap: weekly scorecard feels like busywork until it isn't. The first four weeks feel pointless. By week 8, you'll have a trend line that tells you whether your AI-assisted workflow is actually working or just feeling like it is. That's the value. The first eight weeks are the investment.

What You Should NOT Do

  • Don't track agent lines written. It will make you feel good when the agent is producing lots of code and feeling bad when it isn't, regardless of whether the code is correct. It measures activity, not outcome.
  • Don't track individual PRs merged. Small PRs inflate this number without reflecting real velocity. A 1-line PR and a 100-line PR can both be "one merge."
  • Don't make the scorecard complex. If it takes more than 5 minutes to update, you're tracking too much. One file, three metrics, a few notes.
  • Don't skip weeks. Two missing weeks break the trend line. Even if you do nothing else that week, update the scorecard. Zero for everything is still data.
  • Don't optimize to the metric. If you catch 20 landmines in one week by auditing frantically, that number is meaningless. The goal is a stable weekly average, not a spike.
