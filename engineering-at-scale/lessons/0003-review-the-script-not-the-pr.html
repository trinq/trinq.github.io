Review the Script, Not the PR

The Spotify fleet-management trick that works even when your "fleet" is one repo.

This is lesson 3 of engineering-at-scale. Lesson 0001 had you skip a PRD for a prototype. Lesson 0002 had you audit one function for agent-hostile landmines. This lesson teaches you to stop reviewing changes line-by-line and start reviewing the thing that made the changes.

Your takeaway: code review is not the bottleneck. Generator review is. If you trust the script, you can trust the output. If you don't, no amount of reading the diff will save you.

The Spotify Pattern

From the podcast (~26:00), Niklas describes how Spotify runs fleet shifts across thousands of services:

"We try to automate 100% of these migrations. Then we end up at somewhere around 85%, I think, of these migrations that are auto-mergeable. So basically we have a tool that looks at the diff... and it can confidently auto-merge it." — Niklas Gustavsson, ~26:00

The remaining 15% need a human. But here's the key: Spotify doesn't review 85% of the diffs line-by-line. They review the migration script once, then trust its output thousands of times.

That's the entire trick. Review the generator, not the artifacts. At Spotify scale, the generator might be a custom transformation tool that touches 2,000 services. At your scale, the generator is your agent's prompt + a single example PR.

Why Line-by-Line Review Doesn't Scale

When an agent produces a PR that touches 40 files, you have three options:

  1. Read all 40 diffs. Costs an hour. Easy to miss a landmine. Exhausting.
  2. Skim the diffs. Costs 10 minutes. Catches obvious mistakes. Misses subtle ones. This is what most people actually do.
  3. Read the prompt that generated the PR. Costs 5 minutes. Catches everything the prompt would cause. Lets you re-run and re-generate.

Option 3 is what Spotify does. Option 3 is what you should do.

The shift in mental model: stop asking "does this diff look right?" Start asking "does this process produce correct diffs?" If the process is correct, every diff is correct (modulo randomness, which lesson 0002's audit addresses).

The Small-Scale Version

You don't have a fleet. You have one repo, maybe two. So the Spotify pattern needs translation:

Original (Spotify scale)

// Migration script
rename_method(oldName: String, newName: String, scope: FilePattern) {
  for file in find(scope):
    replace_all(file, oldName, newName)
}

// One review of the script → 2,000 trusted diffs

Translation (one-repo scale)

// Agent prompt
"Add a typed error wrapper around every function in src/api/ that
throws an unhandled exception. Use the example PR #42 as your
template. Do not modify any function marked AGENT-DO-NOT-MODIFY."

// One review of the prompt + example → 1 trusted PR
// If the PR is wrong, fix the prompt, re-run, get a correct PR

The shape is identical: review the generator, trust the output, iterate on the generator when the output is wrong.

The Exercise: Write Your First Generator-Review (20 min)

You will do one of these two options:

Option A — Have an agent generate a small batch PR

Pick a refactor that's small enough to attempt but not trivial. Good candidates:

  • Rename a method across one directory (5–15 files)
  • Add a typed wrapper around a set of similar functions
  • Convert one type of error handling to another (e.g., throw new Error(msg) → typed errors)
  • Update a dependency call pattern across callers (e.g., switch from fetch() to a wrapper)

Write a prompt that includes:

  1. One example PR (or file) showing the desired style of the change
  2. An explicit list of files or patterns to skip (AGENT-DO-NOT-MODIFY, generated files, vendor code)
  3. The success criterion: "all files in scope are modified the same way the example shows"

Don't run the prompt yet. Read the prompt as if you were reviewing it.

Step 1 — Pretend you're reviewing the prompt

Ask yourself these questions:

  • Is the scope unambiguous? ("all files in src/api/" or "every function matching throw new Error" — pick one.)
  • Is the example PR a real example, or a fantasy? (If you wrote it 5 minutes ago, it's a fantasy.)
  • Are the exclusion rules explicit, or assumed? (Explicit. Always.)
  • If this prompt produced 1,000 diffs, would you trust all 1,000? (If no, what's the first one you'd reject and why?)

Step 2 — Run the prompt on a small slice first

Before letting the agent touch the whole scope, narrow the scope to 1–2 files and run it there. Check:

  • Did the agent follow the example's style?
  • Did it touch the excluded files?
  • Did it hallucinate new patterns?

If yes-yes-no, expand to the full scope. If no, fix the prompt and re-run on the slice. Don't re-run on the full scope until the slice works.

Step 3 — Review the full PR as a generator check

When the full PR comes back, do not read every diff. Instead:

  1. Sample 3 random files. Did they match the example's style?
  2. Grep for any file the prompt said to exclude. Did the agent touch any of them?
  3. Grep for any pattern the prompt said to preserve. Did it preserve them?
  4. Look at the diff stats: number of files changed, lines changed. Does it match the expected scope size?

If all four checks pass, the generator worked. Merge.

If any check fails, do not fix the PR. Fix the prompt. Re-run. The whole point is that you don't hand-edit the output.

The trap: you'll be tempted to "just fix that one file" in the PR. Don't. Every manual fix in the output is a sign the generator is wrong, and you'll have to make that fix again next time. Always fix the prompt.

Option B — Extract a pattern from past manual work

Skip the agent step. Look at the last 3 refactors you've done by hand. They're probably similar:

  • Same rename pattern
  • Same import-update pattern
  • Same wrapper style

Write down the generator that could have produced all three. It doesn't have to be code — it can be a paragraph in a markdown file. The point is to make the pattern explicit so the next refactor can use it (and so an agent can use it on the next-but-one refactor).

This is the "review the script" discipline applied retrospectively: notice you already have a script, you just haven't written it down yet.

The Spotify-Scale Generators You'll Eventually Want

You don't need these today. But for context, here's what the pattern looks like when it grows:

Generator type Example Output scale
Code-mod tool jq, sed, codemod scripts 100s–1000s of files
Agent prompt + example PR "Refactor X like #42, but in src/api/" 10s of files
Migration framework OpenRewrite, ts-morph 100s–1000s of files
Honk-style LLM + judge One LLM writes, one LLM verifies 10s–100s of files
Hand-edit (no generator) You, today, opening 30 files ≤10 files before exhaustion

The "hand-edit" row is what most people default to. It works for ≤10 files. It doesn't scale because humans get tired and skip things.

Self-Check

You should be able to tick all of these:

  • I picked one refactor and wrote the prompt for it (Option A) OR identified the pattern in 3 past refactors (Option B)
  • The prompt has an explicit scope, an explicit exclusion list, and a real example
  • I narrowed the scope to 1–2 files for a first run
  • I reviewed the prompt as if reviewing a PR — and answered the 4 questions honestly
  • I committed to fixing the prompt, not the output, if the run is wrong
  • I merged a generator-produced PR this week (or scheduled one)

What You Should NOT Do

  • Don't hand-edit the agent's output. Every hand-edit is a sign the prompt is wrong. Fix the prompt.
  • Don't run the full scope first. Always run a slice. The slice catches prompt bugs cheaply.
  • Don't write a fake example PR. A real example from your own history is 10× more useful than one you invented.
  • Don't pick a refactor that requires design judgment. The generator pattern is for mechanical changes. Renames, type wrappers, import updates. Not "redesign the module."
  • Don't skip the exclusion list. Always tell the agent what not to touch. The lesson 0002 audit tells you what to put in that list.

Why This Connects to Lesson 0002

Lesson 0002 had you find landmines — behaviors the agent will silently break. Lesson 0003's exclusion list is where those landmines go:

// In your prompt to the agent
"DO NOT MODIFY:
  - src/api/legacy/getRecentPosts.ts  (tiebreaker is load-bearing, see #1234)
  - src/api/auth/*  (security-sensitive, manual review required)
  - src/generated/*  (regenerated by toolchain)"

The audit feeds the generator review. The two lessons compose into a single workflow:

  1. Audit: find landmines (lesson 0002)
  2. Encode: put landmines in the exclusion list
  3. Generate: run the prompt on a slice
  4. Verify: check that the generator respected the exclusions (this lesson)
  5. Expand: run on full scope, merge

That's the small-scale fleet-management loop. It's what Spotify does at 2,000 services. You do it at 1.
