Prototype, Don't PRD

The first lesson from Spotify's VP of Engineering: throwaway prototypes replace static specs.

This is lesson 1 of engineering-at-scale. The series is anchored to Beyond Code #12 with Niklas Gustavsson, VP Engineering at Spotify.

Your first tangible win: replace one PRD with a working prototype this week.

The Insight

At ~31:00 and again at ~54:00 in the conversation, Niklas describes a shift happening inside Spotify right now:

"It used to be that we did fairly static designs for something that we were going to build. Now because we can prototype that — and we can even prototype it within our apps — that means that we can touch and feel and use a product with our own data within our app experience compared to looking at a static image. So that's what we're seeing very rapidly — that's the way teams choose to prove out ideas." — Niklas Gustavsson, ~54:00
"A lot of the ways that we work and like the whole process for how we build products is just fundamentally changing as we speak." — Niklas Gustavsson, ~31:00

Prototypes are not new. What is new: prototypes are now cheaper to build than specs are to write. AI agents make this possible. When you can ship a clickable prototype in an afternoon, the cost-benefit of writing a 2-page PRD inverts.

Old vs New

Static PRD

  • 2–8 pages of prose
  • Wireframes attached as PNG
  • Reviews happen asynchronously
  • User feedback deferred to beta
  • Engineering starts from interpretation
  • Changes to scope require doc rewrites

Working Prototype

  • Runnable code, throwaway
  • Real interaction, real data
  • Feedback collected in the same session
  • User tests with their own data
  • Engineering starts from a working baseline
  • Changes are just commits

This is not "PRD bad." The PRD still wins for: cross-team coordination, regulatory compliance, irreversible decisions, anything where the cost of being wrong is high. The prototype wins for: feature shape, interaction feel, user delight.

When to Prototype vs When to PRD

Use this decision rule. It is the one Spotify is operating under, paraphrased:

  • Prototype when you can answer "is this the right shape?" by clicking through it.
  • PRD when you need to answer "is this the right thing for the company to bet on for the next 18 months?"

If the question is about shape, skip the spec. If the question is about commitment, write the spec.

The Exercise: One Prototype This Week

Pick one feature ticket from your current sprint. The criteria:

  • It's a new feature, not a bug fix.
  • It involves UI or user-visible behavior (not pure backend).
  • You have the time to build a rough version in one session.
  • You can show it to at least one other human (user, teammate, designer).
Pick small. If the prototype would take more than 4 hours to build, you picked the wrong ticket. The whole point is that it's cheaper than the PRD.

The 5 Steps

1
Skip the PRD. Write 5 bullets instead — what the prototype needs to demonstrate, not what it needs to do. Keep these on a sticky note.
2
Build the throwaway. Use your AI agent (Cursor, Copilot — whatever you already use). Prioritize the happy path. Hard-code data. Skip edge cases. Skip accessibility. Skip tests. The prototype is allowed to be ugly.
3
Deploy or screen-share it. If it has UI, get it in front of at least one real person using their real data. If it's a backend change, demo it. Static screenshots don't count — the point is touch-and-feel.
4
Capture reactions. Write down exactly what the user said, in their words. The 3 most surprising comments are the most valuable. These will become the real PRD — or the kill-switch.
5
Decide. Three outcomes:
  • Kill it. The prototype showed the idea was wrong. Good — you saved the PRD.
  • Refine it. The shape is right, but small changes. Iterate the prototype 1–2 more times.
  • Spec it. The shape is locked. NOW write the PRD for the engineering team to implement properly.

What You Should NOT Do

This lesson is not a license to skip all planning. Things you should still do:

  • Set a hard time budget (1–4 hours max). If it blows past that, stop and write the spec instead.
  • Label the prototype as throwaway. Do not let it become the implementation by default. "It works, ship it" is not the goal.
  • Capture what surprised you. The point is to learn something you couldn't have written into a PRD.
The trap: the prototype becomes the implementation because writing more code is easier than reverting. Force yourself to delete the prototype repo / branch before you start the real one. Yes, really.

What Spotify Doesn't Tell You

Niklas glosses over a non-trivial point: Spotify has internal prototyping tools that live inside their apps. You probably don't. So you'll be using whichever framework you already use, and that's fine. The point is the workflow, not the tooling.

What you do need that Spotify has and you don't: a teammate willing to look at rough work. If you don't have one, find a user, a designer, or a Discord community where you'll be embarrassed to ship something ugly. Embarrassment is a forcing function.

Self-Check Before You Start

Tick these. If any are missing, fix them before you start the exercise:

  • I have a feature ticket that's small enough (under 4 hours of work)
  • I have access to my usual AI agent
  • I have at least one human who'll look at the prototype with me
  • I've set a hard time budget
  • I've committed to deleting the prototype after the experiment
