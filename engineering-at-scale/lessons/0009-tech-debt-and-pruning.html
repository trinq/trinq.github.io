Tech Debt and Pruning

Every line you keep is a line you have to maintain. Regular pruning is cheaper than a rewrite. Keep the codebase lean, or it becomes a museum.

This is lesson 9 of engineering-at-scale. Lessons 0001–0008 built a workflow for the code you write, the code you pull in, and the code you own. This lesson is about the code you have but shouldn't. Tech debt isn't bad code — it's code whose cost exceeds its value. Pruning is the discipline of making that judgment while you can still see clearly.

Your takeaway: every line you keep is a line you have to maintain. Every file you don't delete is a file that needs to be understood, audited, updated, and migrated. The cost is paid over years. The decision to delete is paid once.

The Spotify Philosophy

From the podcast (~34:00):

"I think it's pretty important to keep pruning the codebase. Both the codebase you're building and the dependencies you're using. If you're not pruning, your code becomes a museum. And once it's a museum, you're not allowed to touch anything because you don't know what will break." — Niklas Gustavsson, ~34:00

The museum metaphor is precise. A museum preserves things because they have value. A codebase preserves things because no one wants to make the call to remove them. The result is the same — nothing changes — but the motivation is very different. Museums are intentional. Codebases drift into museum status by default.

The Four Verdict Categories

Every piece of code — file, function, feature, dep — has exactly one verdict. The verdict determines the action.

Verdict Question Action Cost of keeping
KEEP Is this code doing useful work today? Document, monitor, maintain Standard maintenance overhead
PRUNE Is the cost of keeping this higher than the value of replacing it? Delete. Schedule the cleanup commit. ~0 (gone)
HOLD Is this used by someone but I can't tell who? Investigate. Add telemetry. Re-evaluate in 30 days. Maintenance overhead, plus investigation cost
REPLACE Is this still useful but the implementation is bad? Schedule a rewrite. Or wrap it. Don't keep the old version. Migration cost + maintenance of the old version

The categories are mutually exclusive. Every piece of code gets exactly one. The hard part is not the categories — it's being honest about which category something belongs in.

The Cost of Keeping Code

Every line you keep is paid for in five ways:

  1. Read time. Every future engineer (including future-you) reads this code at some point and decides whether to keep or change it. That's not free.
  2. Test time. Every test suite that covers this code is run on every PR. CI time is not free.
  3. Update time. Every framework upgrade, every dep update, every API change — this code has to be checked, migrated, or rewritten.
  4. Documentation time. If the code is documented, that doc has to be maintained. If it's not documented, future engineers pay the discovery cost.
  5. Mental model time. Every piece of code that exists is part of the codebase's mental model. If a feature has 5 implementations, the mental model has 5 features even if only 1 is used.

The cost is real even if the code "works fine." A working, unused feature still takes up mental space. An undocumented working feature takes up even more.

The Three Question Test

Before keeping any non-trivial piece of code, answer these three questions honestly. If you can't answer all three with evidence, the verdict is HOLD or PRUNE.

Question 1: Who uses this and when was the last use?

Don't guess. Check the telemetry. Look at the last log entry, the last user action, the last metric sample. If you don't have telemetry, that's already a sign — code that matters is instrumented.

Red flag: "it's important, I just can't find anyone who actually uses it." That code is PRUNE.

Question 2: What's the cost if this breaks tomorrow?

Calculate the user impact, the recovery time, the on-call burden. Compare that to the value the code provides.

Example: a logging utility that 3 internal tools depend on. Last used: 6 months ago. Cost of keeping: quarterly updates, possible breakage on framework upgrade. Cost of deleting: 3 internal tools need to be updated to use the platform logger. Net: probably worth pruning, even though it "works."

Question 3: What would I tell a new engineer about this code?

If the answer is "I'd tell them it's legacy, don't touch it," that's PRUNE. Code that "shouldn't be touched" is code that's already a museum exhibit. The cost of not deleting is paid on every future change.

If the answer is "I'd tell them how it works and why," that's KEEP. Code that has a story to tell has value.

The Pruning Ritual

Pruning is not a one-time event. It's a recurring ritual. The Spotify team prunes continuously. You should too. Three rituals, three cadences:

Weekly: The "delete what you added" ritual (5 min)

Every Friday, before logging off: look at what you merged this week. Is any of it now unused because you changed something else? Delete it in the same commit or the next PR. Delete the code in the same PR that made it dead. Not next quarter. Now.

The rule: the PR that makes code dead also deletes the dead code. Not a separate cleanup PR. Not "we'll get to it." Same PR.

Monthly: The "feature flag sunset" ritual (15 min)

For every feature flag that's been at 100% rollout for 30+ days, file a deletion PR. Remove the flag, remove the code paths it gated, remove the tests. Most feature flags are forgotten — they're added for a launch and never removed because no one is responsible for sunset.

# Feature flag cleanup — June 2026

## Flags at 100% for > 30 days
- `flag:new-notification-center` (rolled out 2026-05-12) → DELETE by 2026-07-15
- `flag:profile-redesign` (rolled out 2026-04-22) → DELETE by 2026-07-15
- `flag:dark-mode-beta` (rolled out 2026-03-01) → DELETE by 2026-07-15

## Flags at < 100% (still in progress)
- `flag:search-v2` (currently 50%) → keep, re-evaluate 2026-08-01
- `flag:onboarding-flow-v3` (currently 80%) → keep, re-evaluate 2026-08-15

## Action items
- [ ] Delete `new-notification-center` flag + dead code paths (PR #1234)
- [ ] Delete `profile-redesign` flag + dead code paths (PR #1235)
- [ ] Delete `dark-mode-beta` flag + dead code paths (PR #1236)

Quarterly: The "museum audit" (1 hour)

Every quarter, audit the codebase for museum exhibits. Files that no one wants to touch, features no one can explain, code that's documented as "legacy." Make a verdict on each. The audit produces a list of deletion PRs that get scheduled across the next quarter.

The "Code is Dead When..." Heuristics

Specific signals that code is ready to prune:

  • Last commit on the file is > 12 months ago and no recent imports.
  • Zero telemetry hits in production logs for 6+ months.
  • No tests cover this code and no engineer can explain what it does.
  • Documentation says "deprecated" or "legacy" — at that point it's just debt with a label.
  • The feature has a replacement that's been at 100% rollout for 30+ days.
  • The dep is abandoned and only this code uses it.
  • The code imports from a directory that's been deleted — the import will fail at runtime.

If any two of these are true, the verdict is PRUNE. If three or more are true, the verdict is PRUNE urgently — every day you keep it is a day you might be paged for it.

The Exercise: Run Your First Pruning (45 min)

Step 1 — Pick a repo (~1 min)

Pick the repo where you spend the most time. The one with the most code, the most history, or the most uncertainty about what it does.

Step 2 — Find files with no recent activity (~10 min)

# Find files not modified in 12+ months
find . -type f \( -name "*.ts" -o -name "*.js" -o -name "*.py" \) \
  -mtime +365 \
  -not -path "./node_modules/*" \
  -not -path "./.git/*" \
  | head -50

# Find functions never imported (rough heuristic — count import statements)
for file in $(find . -name "*.ts" -not -path "./node_modules/*"); do
  basename=$(basename "$file" .ts)
  count=$(grep -r "from.*$basename" --include="*.ts" . 2>/dev/null | grep -v "$file" | wc -l)
  if [ "$count" -eq 0 ]; then
    echo "Potentially unused: $file"
  fi
done

Step 3 — Investigate each candidate (~20 min)

For each candidate file, answer the three questions:

  • Who uses this and when was the last use? (check telemetry, logs)
  • What's the cost if this breaks tomorrow?
  • What would I tell a new engineer about this code?

Make a verdict: KEEP / PRUNE / HOLD / REPLACE. Document the reasoning in a markdown file.

Step 4 — Delete the PRUNE candidates (~10 min)

For each PRUNE verdict, write a deletion PR. Don't combine deletion with feature work. The PR is "delete [thing]" and the diff is mostly negative lines. Review it like any other PR — by what it removed, not what it added.

git checkout -b prune/remove-unused-utils-2026-06
git rm src/utils/legacy-format.ts
git rm src/utils/legacy-date-helpers.ts
git rm src/services/unused-analytics.ts
git commit -m "prune: remove legacy utils unused since 2024-08

Per museum audit 2026-06-30. All three files had no imports
since Q3 2024, no telemetry, no documentation explaining
their purpose. Cost of keeping exceeded value."
git push origin prune/remove-unused-utils-2026-06

Step 5 — Set up the weekly ritual (~5 min)

Add a Friday afternoon recurring task: "prune what you merged this week." Make it visible. 5 minutes. The point is the ritual exists, not the duration.

The "But What If We Need It Again" Trap

The most common excuse: "we might need this code someday." The answer: git has history. git log --all --source --remotes --oneline -- path/to/file.ts shows every commit that touched the file. The code isn't lost when you delete it — it's archived in git history.

The cost of keeping "just in case" code is paid every day. The benefit of having it "just in case" is paid maybe once in five years, and even then git can resurrect it in 5 minutes.

The asymmetry kills the "just in case" argument. Pay the deletion cost once. Don't pay the keeping cost every day.

The "But It's Documented" Trap

Documentation is not a reason to keep code. Documentation is a reason the code is expensive to remove. The cost of removing a documented feature is "we have to update the docs." The cost of keeping it is "the docs have to be maintained alongside the code." Both are real, but the keeping cost compounds. The removal cost is paid once.

Why This Lesson Matters Even If You Just Wrote the Code

The most common reason tech debt accumulates is "I'll clean it up later." That later never comes. By the time the code is 6 months old and 5 engineers have touched it, the cost of deletion is "we don't know what'll break." That's the museum problem.

Pruning works best when the code is fresh. The PR that makes code dead also deletes the dead code. The ritual exists at solo scale because the cost of deletion is lowest at solo scale.

When you grow, the pattern is already there: every PR carries the deletion that goes with it. The codebase doesn't accumulate museums. Engineers learn the rhythm: build, prune, build, prune.

What You Should NOT Do

  • Don't keep code "just in case." Git has history. The cost of keeping is paid every day; the benefit is paid maybe once.
  • Don't combine deletion with feature work. A deletion PR should be a deletion PR. Reviewing deletion + feature is harder than reviewing either alone.
  • Don't prune what you don't understand. If you can't tell who uses the code, the verdict is HOLD, not PRUNE. Add telemetry, re-evaluate in 30 days.
  • Don't document "deprecated" without a deletion date. Deprecation is a label, not a destination. Either delete the code or document the migration plan.
  • Don't wait for a "pruning sprint." Sprints dedicated to pruning are a sign the codebase is already a museum. Pruning is daily, weekly, monthly, quarterly — not a quarterly event.
  • Don't prune without telemetry. If you can't measure who uses the code, you can't make the call. Add telemetry first, then prune.
