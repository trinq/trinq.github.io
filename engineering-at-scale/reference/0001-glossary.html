Glossary

Terms from the Beyond Code #12 conversation, defined for quick reference.

This glossary is the source of truth for terminology across all lessons in this series. When you encounter one of these terms in a lesson, link here first.

People & Roles

Squad
A small, autonomous cross-functional team at Spotify. Owns a set of components end-to-end — from ideation through on-call. Squads are the unit of autonomy.
Mentioned ~02:00 and ~17:00 in the podcast. The "Spotify Model" coined the term.
Program Manager
Coordinates work across multiple squads for a single big "bet" (e.g., a product launch). Does not own the squads' engineering decisions — owns the timing and coherence.
~21:00 — Niklas describes company bets crossing many squads.

Architecture

Service decomposition
Breaking a large application into many small services that communicate over the network. Spotify started with "a handful of backend services" and now runs ~3,000. Each service is owned by one squad.
~03:00 — the founding team picked this up from Unix principles.
Fleet
The totality of all production services + their config + their repos. At Spotify, ~3,000 services in the fleet. Operations on the fleet (upgrades, migrations) run across the whole set in parallel.
~25:00.

The AI Era

Honk
Spotify's internal AI agent. Wraps an LLM + an agent loop + an LLM-as-judge verifier. Runs on top of the fleet-management infrastructure. Can be invoked from Slack or GitHub. On its 6th iteration as of the podcast.
~29:00. The name is internal; the architecture is the interesting part.
Fleet Shift
A packaged migration script (Docker container) + a manifest (Kubernetes CRD describing how to run it) + a Backstage UI for monitoring. One shift can open thousands of PRs across the fleet.
~25:00, ~52:00.
Fleet Management
The discipline of running fleet shifts at scale. Originally deterministic (code mods). Increasingly LLM-mediated via Honk for anything past a certain complexity threshold.
~25:00. Spotify blog has deeper posts.
LLM-as-judge
Using one LLM to verify the output of another. Honk does this: the first LLM writes the change, a second LLM checks the change matches the original prompt. Catches "weird unintentional changes" the first LLM makes.
~28:00.
Background agent
An agent you kick off and walk away from. Returns minutes later with a result (usually a PR). Honk supports this; Claude Code supports this locally; the pattern is the same.
~33:00 — Niklas describes asking Honk from Slack and getting coffee while it works.

Engineering Principles

Hyrum's Law
"With a sufficient number of users of an API, it does not matter what you promised in the contract — all observable behaviors of your system will be depended on by somebody." At scale, every quirk of your implementation has a customer. This is why deterministic migrations are hard and why LLMs become necessary.
~27:00. Originated at Google by Hyrum Wright. hyrumslaw.com.
Single source of truth
Source code remains authoritative. Spotify is NOT storing intent (prompts, specs) alongside the code as the canonical reference. They ARE investing in capturing insights and decisions made by agents at runtime.
~57:00. This is a deliberate choice — not an oversight.
Squad autonomy + company synchronization
The two forces Spotify balances. Squads own their components. But "company bets" (large features) require cross-squad programs and coordination. Less autonomy now than the original 2015 model implied — and that's a feature, not a bug.
~17:00, ~62:00.

Tools & Platforms

Backstage
An internal developer portal. Plugin-based. Started as a service catalog at Spotify; now an open-source CNCF project. Functions as the "single pane of glass" for engineers — service ownership, deployments, on-call, schemas, logs, all in one place.
~42:00. backstage.io.
Single pane of glass
One UI for everything related to a service. The opposite of the "12 different shitty tools" pattern Backstage replaced.
~45:00.
Service catalog
A registry of every service in the fleet. Required fields: owner, production state, reliability tier. Spotify got 100% coverage by tying capacity provisioning to registration — a clever incentive design.
~43:00. Called "system set" originally.

Operations

Auto-merge
Merging a PR without human review. Spotify does this for ~85% of deterministic fleet shifts — because the *script* is reviewed, not each individual PR. For human-authored or AI-authored changes (Honk output), manual review is still required.
~36:00.
PR attribution
Identifying whether a PR was authored by a human, an AI agent, or a fleet shift. Used to compare code-quality metrics across categories. APIs for this are "rapidly improving" per Niklas.
~01:00:00.
Time to change
A Spotify productivity metric: time from starting to code on something to it shipping to users. One of several contributing metrics alongside CI time, deployment frequency, and self-reported survey data.
~01:01:00.
Aggregate-level measurement
Spotify measures engineering metrics at the team or company level — NEVER individual engineer level. Explicitly stated philosophy. Avoids Goodhart's law and preserves trust.
~01:00:30.
