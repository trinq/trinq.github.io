Building a Routing Strategy

L4 Judgment skill · ~15 min · Lesson 4 of the Multi-Model Orchestration series

Lesson 1 gave you the four-model map. Lesson 3 gave you a measurement protocol. This lesson gives you something harder to acquire: the ability to look at a task and make a routing call in under thirty seconds — and be right most of the time.

Routing is a judgment skill, not a rule you look up. The way you build judgment is not by reading about it — it's by seeing it done, then doing it yourself, then checking whether you were right.

The heuristic that replaces "reasoning vs. mechanical"

Lesson 1 called it "reasoning-heavy" vs. "mechanical." That's right but vague. Here's a sharper frame:

A task is worth routing to the expensive model (Opus) when the valuable output is expensive to produce — meaning the next step is uncertain, a wrong turn wastes significant effort, or the output has to be right the first time.

A task belongs on the cheap model (Sonnet) when the valuable output is cheap to produce — meaning the pattern is clear, the quality gate is the existing test suite, or you're regenerating something structurally similar to code that already exists.

In other words: expensive reasoning earns its cost when the output is expensive to get wrong.

The "expensive output" test
Ask: what's the most expensive thing this task could produce? If the answer is a long, detailed, high-value artifact — an architecture doc, a debug trace, a novel implementation — route to Opus. If the answer is 200 lines of familiar pattern — CRUD endpoints, test coverage, boilerplate — route to Sonnet.

Four worked case studies

Each case shows how a practitioner applies the heuristic. Read each one as a decision process, not just a conclusion. Ask: would I have routed the same way?

Case A — Database selection

The task: "We need to store time-series sensor data. Should we use Postgres + TimescaleDB, ClickHouse, or DynamoDB?"

The routing: Opus. Every time.

The reasoning: The valuable output is a decision that constrains the next six months of development. The downside of a wrong choice is high — a migration from ClickHouse to Postgres is not a weekend project. The expensive model's cost is trivial relative to the cost of a wrong database. Sonnet could generate a comparison table of all three, but it would be shallow and would not hold up to scrutiny from a DBA. The routing decision here is not about tokens — it's about the cost of being wrong.

Case B — CRUD endpoint generation

The task: "Add a POST /users endpoint and a GET /users/:id endpoint to our Express app. Use the existing User model."

The routing: Sonnet. Every time.

The reasoning: The valuable output is 80 lines of familiar Express boilerplate. The pattern is established — the app already has three similar endpoints. The quality gate is the existing test suite: if the tests pass, the implementation is correct. Routing this to Opus pays for a senior engineer to write the same thing a junior engineer would write. Sonnet will do it correctly and cheaply. The only reason to involve Opus here is if the existing endpoints have subtle inconsistencies that require deep reasoning about the codebase — and if that's the case, the routing decision should have been made earlier, at the "should these endpoints follow the same pattern?" question.

Case C — Debugging a race condition

The task: "Our queue consumer sometimes processes the same message twice under high load. It's intermittent and only happens in production."

The routing: Opus. But the session should include Codex in parallel for the high-stakes phase.

The reasoning: Race conditions are high-uncertainty problems. The next step is genuinely unclear — it could be the queue ack timing, the database transaction isolation level, a shared mutable variable, or a misconfigured consumer group. A wrong theory leads to hours of debugging in the wrong direction. The expensive model is worth it here. The routing decision: Opus for the root-cause analysis phase. Sonnet for implementing the fix once the root cause is known (implementing a idempotency key is not architecturally hard — it just requires knowing where to put it). Codex in parallel during the analysis phase: a race condition has a second, equally valid explanation that the first-pass analysis might miss. The cost of Codex's time is negligible relative to the cost of shipping the wrong fix.

Case D — Greenfield feature design

The task: "Design the data model and API shape for a new notification system we're building from scratch."

The routing: Opus for the initial design. Sonnet for implementation.

The reasoning: The design phase has high output value and high cost of being wrong — a bad data model requires a migration to fix; a bad API shape requires a breaking change. Route to Opus for the design. Then, once the design is agreed, the implementation is mechanical — translate the design doc into code. Sonnet handles that. The routing decision is time-phased: the split is not "which model" but "which model at which stage."

The routing decision tree

A reusable tree for the thirty-second call. Run it in order. Stop when you hit a decision.

  1. Is the output expensive to get wrong?

    If wrong output requires a significant rewrite, a breaking change, or a costly rollback: Opus. Otherwise, continue.

  2. Is the output pattern clear and familiar?

    If there's an existing similar implementation and the quality gate is "tests pass": Sonnet. Otherwise, continue.

  3. Is this a high-stakes decision with a second valid perspective likely?

    Architecture, debugging, novel implementation: Opus + Codex in parallel. Otherwise, Opus.

  4. Is the task multi-phase?

    If the task has a design phase followed by an implementation phase: split across time. Opus for design, Sonnet for implementation.

Most tasks stop at step 2. The routing decision happens fast once you build the habit.

The three signals that override the tree

The tree handles routine cases. These three signals override it every time:

Signal What it means Override
You're stuck You've been on the same problem for more than 30 minutes with no clear next step Send to Codex (/codex:rescue --background). A different perspective is worth its cost.
You don't know what you don't know The task touches a domain you haven't worked in (a new database, a new protocol, a new framework) Opus. The cost of the expensive model is lower than the cost of discovering the gaps at the Sonnet level.
The team is watching The output will be seen by stakeholders who need confidence in the work Opus + Codex. The synthesis has to hold up to external scrutiny.

What's next — the measurement loop

Lesson 3 gave you the measurement protocol. This lesson gave you the judgment heuristic. Together, they form a loop: you make a routing call, you measure the outcome, you update the heuristic.

The loop has three states:

  • Confident: The routing decision felt right, the quality gate passed, the cost was what you expected. Keep routing the same way.
  • Cost surprised: The routing felt right but the cost was much higher than expected. Look at the routing counts. Did Sonnet produce the output cheaply? If not, the task was more complex than it looked — note it for next time.
  • Quality surprised: The routing felt right but the quality gate failed. Did Opus miss something? Did Sonnet produce something subtly wrong? This is the most valuable data point — it tells you where the heuristic is miscalibrated.

The goal is not a perfect heuristic — it's a heuristic that's calibrated to your codebase. The measurements from Lesson 3 are what calibrate it. Run the experiment, log the worksheet, and the routing tree starts reflecting your reality, not a general principle.
