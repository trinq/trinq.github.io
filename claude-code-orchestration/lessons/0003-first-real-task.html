First Real Task

L3 Measurement · ~20 min active + a real run · Lesson 3 of the Multi-Model Orchestration series

Lesson 1 gave you the pattern. Lesson 2 set it up on your machine. This lesson does the actual measurement: run a real task both ways, log the numbers, and turn "orchestration saves money" from a claim into a data point.

What we're measuring

We are measuring the routing effect — the difference orchestration makes when the orchestrator (Fable 5) is the same in both arms. We are not measuring whether Fable 5 beats Opus, or whether Sonnet is good enough on its own. Those are different experiments.

Arm Active model Subagents CLAUDE.md snippet
A — control Fable 5 (orchestrator, but solo) disabled / not invoked absent
B — orchestrated Fable 5 (orchestrator) deep-reasoner + fast-worker snippet from Lesson 2 present

Both arms use Fable 5. The only thing that changes is whether Fable 5 is allowed to delegate. This isolates the effect we're claiming: that routing the right work to the right model reduces cost while keeping quality stable.

Why this design
A simpler A/B — "Fable 5 vs Opus" — confounds two effects: model capability and routing strategy. We'd never know whether any cost difference came from the model itself or from the delegation. By holding the model constant and varying only the routing, we get a clean answer to the question the mission actually asks: does orchestration pay for itself?

The protocol

  1. Pick a task

    A real feature in a real codebase. Small enough to do in one Claude Code session (10–60 minutes of work). Big enough to require both reasoning and implementation. The worked example below uses a React class-to-hooks refactor; substitute your own.

  2. Define the quality gate

    A binary check that says "this work is done." For the worked example it's the existing test suite passing. For a new feature it's whatever acceptance criterion you can phrase as "X passes." Without this, both arms will be judged by vibes.

  3. Run arm A — control

    Temporarily rename or comment out the orchestration snippet in CLAUDE.md so Fable 5 doesn't know about the subagents. Use the same task prompt as arm B. Log: total tokens, wall-clock time, whether the quality gate passed.

  4. Run arm B — orchestrated

    Restore the orchestration snippet. Revert arm A's code changes. Use the same task prompt. Log: total tokens, wall-clock time, whether the quality gate passed. Also note how many times Fable 5 routed to each subagent.

  5. Fill the worksheet

    Copy the worksheet at the end of this lesson into learning-records/0004-cost-experiment-first-real-task.md. Fill it in. Don't skip the quality assessment.

Worked example — React class-to-hooks refactor

Suppose you have a React codebase with a 200-line class component (UserProfile) that handles user data, lifecycle effects, and form submission. The task: convert it to a hooks-based component with the same behavior.

Why this task fits the experiment:

  • Real — class-to-hooks is the kind of refactor you do constantly.
  • Multi-file — the conversion will likely produce several small helper files plus the main component.
  • Has a quality gate — the test suite passes, or it doesn't.
  • Splits naturally — "what's the right decomposition into hooks?" is Opus-grade reasoning; "do the mechanical conversion" is Sonnet-grade implementation; "did we lose any behavior?" is Codex-grade peer review.

The task prompt (use the same text for both arms)

Goal: Convert src/components/UserProfile.jsx from a React class component
      to a function component using hooks, preserving identical behavior.
Context: React 18, TypeScript. The existing test suite at
         src/components/__tests__/UserProfile.test.tsx must continue to
         pass without modification.
You're the lead. Plan first, then execute.

Running arm A (control)

Comment out the orchestration snippet in CLAUDE.md for this run: 

<!--
## Orchestration workflow
You (Fable) are the orchestrator. Plan, decompose, synthesize.
Reasoning-heavy phases → deep-reasoner
Mechanical work → fast-worker
...
-->

Fable 5 still loads the subagent definitions (you can't fully un-register them without deleting the files), but with the snippet absent, it won't think of itself as an orchestrator and won't delegate. Run the task. When it finishes, capture:

  • Total tokens from the session footer (Claude Code shows this at the end).
  • Wall-clock time from when you sent the prompt to when the work was done.
  • Did the test suite pass? (npm test -- UserProfile)

Revert the changes when you're done — you want a clean repo before arm B.

Running arm B (orchestrated)

Restore the CLAUDE.md snippet. Send the same prompt. Capture the same numbers, plus:

  • How many times did Fable 5 route to deep-reasoner?
  • How many times did it route to fast-worker?
  • Did it ever call /codex:rescue?
  • Did the test suite pass?
Finding the routing counts
The session log shows subagent invocations. In Claude Code, scroll up after the session — you'll see lines like [subagent: deep-reasoner] or [tool: Agent] each time Fable 5 spawned a subagent. Count them. If Fable 5 didn't delegate at all, that's a finding too — either the task was too simple for orchestration to add value, or the snippet isn't being read.

What to look for in the numbers

Three findings are worth watching for. None of them is a "win" or "loss" on its own — the point is to surface what orchestration actually does in your codebase.

1. Token cost ratio

Compute the dollar cost of each arm using the table above. The interesting question is not "did arm B cost less?" — arm B might cost more if Fable 5 does a lot of planning. The interesting question is cost per unit of useful work, which is why the quality gate matters.

A clean win looks like: arm B costs 40–70% of arm A for the same passing test suite. A wash looks like: arm B costs 90–110% of arm A. A loss looks like: arm B costs more than arm A and the test suite still passes in arm A.

2. Where the subagents went

If Fable 5 routed to deep-reasoner for the architectural decomposition (which hooks, where do state and effects live) and to fast-worker for the per-file rewrite, you have evidence the routing rules are working. If Fable 5 routed to deep-reasoner for everything, the snippet isn't giving it permission to use the cheap model for mechanical work — you'll want to revise the CLAUDE.md snippet to be more explicit about when to use which.

3. Quality delta

If both arms pass the quality gate, orchestration's job is done — cost ratio is the only metric. If arm A passes and arm B doesn't, something in the routing caused the work to degrade; check whether a subagent returned a wrong conclusion that Fable 5 trusted. If arm B passes and arm A doesn't, orchestration genuinely improved quality and the cost question becomes "is the improvement worth the extra spend?"

When NOT to trust the data

One run is not a benchmark. Before you write a learning record that says "orchestration cuts cost by 50%," watch for these confounders:

  • One-shot variance. A single run of either arm is noisy. The right move is to run both arms twice and average, or to pick a task with enough tokens that the noise is small relative to the signal.
  • Prompt cache effects. The second arm reuses cached context from the first. This can understate arm B's tokens. If you're using the same project, log the cache-hit ratio from the session footer.
  • Model version drift. Sonnet 5 may be in introductory pricing until August 31, 2026, then jump to the standard rate. If you re-run the experiment across that boundary, log the price change as a footnote.
  • Quality gate too loose. "Tests pass" is a clean gate. "Looks fine" is not. If your gate is subjective, run both arms past a teammate who doesn't know which is which.
  • Fable 5 didn't actually orchestrate. If the routing counts are zero in arm B, your snippet isn't being read or Fable 5 decided the task was too simple to delegate. The experiment then compares "Fable 5 alone" in both arms, not "orchestrated vs un-orchestrated."

Knowledge check

Why does this experiment hold Fable 5 constant across both arms, instead of comparing Fable 5 vs Opus?
Arm B costs less than arm A, but both pass the quality gate. What's the right interpretation?
In arm B, Fable 5 routed everything to deep-reasoner (zero invocations of fast-worker). What's the most likely cause?
