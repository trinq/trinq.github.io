Advanced Orchestration Patterns

L5 Power-user techniques · ~15 min · Lesson 5 of the Multi-Model Orchestration series

Lesson 4 gave you the judgment: which model for which task. This lesson gives you the technique that makes the orchestrator genuinely more powerful in daily use. It's the pattern most orchestrator users miss.

The pattern most users get wrong

The naive use of an orchestrator is full delegation: give the subagent the whole task and wait for the result. That works, but it wastes the orchestrator's context on managing the handoff, and it makes the orchestrator dependent on a single subagent's output.

The power-user technique is partial delegation: keep the orchestrator actively in the loop and send it targeted, specific questions. The orchestrator is not a dispatcher that hands off work and waits — it's a synthesizer that uses subagents as consultants for pieces of the problem.

The orchestrator's job is not to delegate tasks. It's to hold the problem and use subagents as consultants — not contractors.

The difference: a contractor gets the whole job and returns a finished product. A consultant answers a specific question and hands the answer back. You decide what to do with it.

Pattern 1 — Partial delegation

When: Any task where you want the orchestrator to stay in control of synthesis, not hand off the whole problem.

The naive version:

@deep-reasoner: Our auth system needs to handle refresh token rotation.
The app is Next.js with a Postgres database. Design the full token
management strategy.

The subagent returns a full design document. Fable 5's job is to implement it. If the design has a flaw, Fable 5 either fixes it silently or propagates it into the implementation.

The partial delegation version:

@deep-reasoner: We're considering two approaches to refresh token rotation:
(A) sliding window with one active token per user, or (B) rotation with a
refresh token family. Our app is Next.js + Postgres. What are the security
tradeoffs of each, specifically: which is harder to implement incorrectly?
Which is more resistant to token theft?

The subagent returns a focused comparison. Fable 5 reads it, picks an approach, and implements. Fable 5 made the decision — the subagent informed it. That's the difference.

When to use partial delegation
Use it whenever the subagent's output would be a decision that Fable 5 would then act on anyway. A design doc from a subagent is a decision the subagent made on Fable 5's behalf. A focused comparison is a set of facts Fable 5 can use to make the decision itself.

How to phrase the partial delegation prompt

The art of partial delegation is in the question, not the answer. Four types of partial delegation questions that work well:

Type Signal phrase Why it works
Tradeoff comparison "What are the tradeoffs of X vs Y?" Returns structured pros/cons, not a recommendation
Failure mode analysis "If we did X, what would break first?" Returns the most fragile assumption — useful for Fable to probe
Constraint extraction "What does this require that we haven't built yet?" Returns the dependency list — useful for sequencing
Analogy / precedent "Has a pattern like this caused problems in systems like X?" Returns hard-won knowledge without asking for a design

Pattern 2 — Parallel background synthesis

When: An important architectural decision where you'd normally ask a teammate to review independently, but you only have one teammate and Claude Code can run two in parallel.

The classic use case from Lesson 1: Fable 5 asks Opus and Codex to work on the same problem independently, then synthesizes the best of both — without showing either the other's answer. This is the "two perspectives" routing.

The difference from partial delegation: Opus and Codex are running simultaneously, not sequentially. Both get the same problem statement. Neither sees the other's output until Fable 5 synthesizes. 

/codex:rescue --background "Design the event sourcing strategy for our
order management system. The system handles 5000 orders/day, needs
event replay for analytics, and must support idempotent event
handlers. Focus on the event schema design."

@deep-reasoner: Design the event sourcing strategy for our order
management system. The system handles 5000 orders/day, needs event
replay for analytics, and must support idempotent event handlers.
Focus on the aggregate root boundaries and consistency model.
Do not read the Codex output.

Fable 5 now has two independent designs to synthesize. If they agree, confidence is high. If they diverge on a key architectural point, Fable 5 makes the call explicitly, with both perspectives in mind.

When not to use this
Don't parallel synthesize routine tasks. Running two expensive models simultaneously defeats the purpose of orchestration if the task doesn't require two independent perspectives. Reserve this pattern for architectural decisions where the cost of a wrong call is high — database selection, API boundary design, consistency model.

Pattern 3 — Sequential escalation

When: A task where Sonnet's first output is not quite right, but sending it back to Sonnet hasn't worked either.

Sequential escalation is simple: Sonnet attempts the task, reports back. If the attempt is wrong or incomplete, send the specific failure to Opus instead of back to Sonnet. Opus handles the hard part. Sonnet then implements the fix. 

@fast-worker: Add a rate-limiting middleware to our Express app using the
existing token bucket implementation in lib/ratelimit.ts.

# Sonnet returns the implementation, but the test fails with:
# "Error: token bucket not initialized before use"

@deep-reasoner: Our rate-limiting middleware has a test failure:
"token bucket not initialized before use." The error occurs in
the middleware when it calls getTokenCount() before the Redis
connection is established. The token bucket implementation is
at lib/ratelimit.ts. What's the right initialization sequence
given that Redis is async and Express middleware is synchronous?

The key is that Sonnet's failure report goes to Opus — not back to Sonnet. Sonnet already failed; a second attempt at the same task at the same model level rarely succeeds. Escalating to Opus gets the architectural insight Sonnet couldn't produce. Sonnet then implements the fix.

Pattern 4 — Context window management

When: The orchestrator's context is filling up and you need to keep a long session productive.

Long Claude Code sessions accumulate context. At some point the orchestrator starts losing the thread. The orchestrator pattern has a built-in solution: offload completed work to a subagent for a clean summary, then continue with a lean context. 

@deep-reasoner: Our refactoring session has covered:
- Component extraction (completed)
- Hook migration (completed)
- TypeScript conversion (completed)
- Still in progress: test migration
Summarize the completed work in 5 bullet points, noting any
unresolved edge cases we need to watch for in the test migration.
Keep it to 150 words.

The subagent produces a compact summary. Fable 5 keeps the bullet points and discards the full context from the completed phases. The session stays lean for the remaining work.

When to compress context
Watch the token count in the session footer. When the orchestrator has been running for more than 30 minutes, or when the token count is above 70% of context, trigger a compression. Don't wait until it's full — by then the orchestrator is already struggling to hold the thread.

The partial delegation prompt builder

Use this template to convert any full-delegation prompt into a partial delegation prompt. The structure is always the same: context, the question type, the specific question, and a constraint on the output shape. 

@[subagent]:
Context: [what the orchestrator already knows about the task]
Question type: [tradeoff / failure mode / constraint / analogy]
Specific question: [the one thing this subagent is better at than the orchestrator]
Output constraint: [how many words / what format the answer should take]

Examples: 

@deep-reasoner:
Context: We're adding a webhook delivery system to our platform.
Question type: failure mode
Specific question: What are the three most likely failure modes in
  webhook delivery, and which is hardest to detect automatically?
Output constraint: 3 bullet points, 20 words each

@fast-worker:
Context: The API schema for the user profile is at api/schemas/user.ts
Question type: constraint
Specific question: Which fields are optional today, and what would
  breaking each one cost us in terms of migrations?
Output constraint: A table with field, optional, migration cost

Pattern selector — which to use

When in doubt, run this:

  1. Is the output a decision the orchestrator should make?

    Yes → Partial delegation. No → continue.

  2. Would two independent perspectives make this better?

    Yes → Parallel synthesis. No → continue.

  3. Did the cheap model fail? Should the expensive model try?

    Yes → Sequential escalation. No → continue.

  4. Is context getting tight?

    Yes → Context compression. No → Full delegation is fine.
