The Orchestrator Pattern

L1 Mental model · ~8 min · Lesson 1 of the Multi-Model Orchestration series

Most people use one model for everything. That's like hiring a senior architect to also paint walls, sweep floors, and file expense reports. The senior gets paid the same either way — but the floors still get swept, and the work that needed the architect gets rushed.

This lesson teaches the orchestrator pattern: route different types of work to different models. Quality stays high on the hard problems. Cost stays low on the bulk. The orchestrator never paints the wall.

Four roles, four models

Each model has a different cost/capability profile. The trick is matching the kind of thinking a task requires to the model that does that thinking best:

Key insight
The orchestrator (Fable 5) should do minimal actual work. Its job is to plan, decompose, delegate, and synthesize. Think of it as a tech lead who never writes code — only decides who writes what.

The routing decision

When a task comes in, the orchestrator asks one question: What kind of thinking does this require?

  1. Reasoning-heavy → Opus (deep-reasoner)

    Architecture decisions, debugging complex race conditions, designing algorithms, analyzing tradeoffs. These need deep, careful thinking.

  2. Mechanical work → Sonnet (fast-worker)

    Writing tests, generating boilerplate, formatting code, making simple edits. High-volume, low-complexity tasks.

  3. High-stakes decision → Opus + Codex in parallel

    When a decision really matters, get two independent perspectives. Task both on the same problem; synthesize. Neither sees the other's answer.

  4. Need a fresh take → Codex

    Codex (OpenAI) has a different training distribution than Claude. When you suspect tunnel vision, send it to Codex for an orthogonal perspective.

The cost math

Here's why this matters for the wallet. Say you're building a feature that involves:

  • 1 architecture decision (hard reasoning) — Opus
  • 10 implementation files (mechanical) — Sonnet
  • 5 test files (mechanical) — Sonnet
  • Orchestration overhead — Fable 5 (lean context)

Without orchestration, you'd run everything through your most expensive model. With orchestration, only the hard 10% gets expensive tokens. The remaining 90% runs on Sonnet at a fraction of the cost.

Anti-pattern
Don't let the orchestrator (Fable 5) do the work itself. If Fable starts writing implementation code, you're paying orchestrator prices for worker work. Keep Fable's context lean — plan, delegate, synthesize.

Knowledge check

A teammate asks you to review a complex database migration that's timing out. Which model handles this?
