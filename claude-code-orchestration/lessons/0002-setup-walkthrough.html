Setup Walkthrough

L2 Hands-on · ~15 min · Lesson 2 of the Multi-Model Orchestration series

Lesson 1 explained the orchestrator pattern as a mental model. This lesson runs the setup on your machine. Four steps, each with a verification check so you know it actually worked.

By the end, Fable 5 will be your orchestrator, two subagents will be pinned, the Codex plugin will be installed, and your CLAUDE.md will tell Fable 5 how to delegate.

Before you start
You need Claude Code v2.1.170 or later (Fable 5 requires it). Run claude --version to check, and claude update to upgrade. You also need Node.js 18+ if you don't have the Codex CLI installed yet, and an OpenAI account with either a ChatGPT subscription or an API key for !codex login.

Step 1 — Pin Fable 5 as the orchestrator

Open Claude Code. Run /model with no argument to open the picker, then select Fable 5. Press Enter to save Fable 5 as your default for new sessions.

Or skip the picker and run /model fable directly. Both are equivalent.

Then crank the reasoning effort. Fable 5 ships with a default effort of high; for orchestrator duty you want max: 

/effort max
Verify it worked
Look at the spinner area. The current effort level is displayed next to the logo, e.g. with max effort. If you don't see that line, run /effort with no argument and the slider should show max.
If Fable 5 isn't in the picker
Your Claude Code is too old. Run claude update and restart. If it's still missing after the upgrade, your account may not have Fable 5 access yet — check the [model configuration docs](https://docs.anthropic.com/en/docs/claude-code/model-config) under "Work with Fable 5" for the access requirements.

Step 2 — Create the two subagents

Subagents are Markdown files with YAML frontmatter. The /agents command opens a tabbed interface that creates them for you, but it's faster and more reproducible to write the files directly. Both paths produce the same result.

Path A — write the files directly (recommended)

Pick a scope. User-level (~/.claude/agents/) makes the subagents available in every project on your machine. Project-level (.claude/agents/) is fine if you want them tied to one repo. The recipe below assumes user-level. 

---
name: deep-reasoner
description: Reasoning-heavy subagent. Use for architecture, debugging complex issues, algorithm design, and tradeoff analysis. Thinks thoroughly and returns a concise conclusion the orchestrator can act on.
model: opus
effort: high
---

You are the deep reasoner in a multi-model orchestration. You are pinned to Opus and
configured for hard thinking. Take your time. State assumptions. Return a conclusion
the orchestrator can act on — not a 500-line exploration.

---
name: fast-worker
description: Mechanical work subagent. Use for boilerplate, tests, formatting, simple edits, and other high-volume, low-complexity tasks. Execute efficiently and return a diff.
model: sonnet
---

You are the fast worker in a multi-model orchestration. You are pinned to Sonnet and
configured for throughput. Skip the preamble. Show the diff or the file. Don't restate
the task.

Save both files. The next time you start a Claude Code session, the subagents will be loaded automatically. You can verify by running /agents and looking in the Library tab — you should see deep-reasoner and fast-worker listed.

Path B — the /agents interactive flow

  1. Run /agents in Claude Code.
  2. Switch to the Library tab, select Create new agent, then choose Personal.
  3. Either Generate with Claude (paste a description like "reasoning-heavy subagent for architecture and debugging") or skip the generator and fill in the fields manually.
  4. On the model step, pick Opus for the deep-reasoner and Sonnet for the fast-worker.
  5. Save. Repeat for the second subagent.
Verify it worked
Run /agents. In the Library tab you should see both subagents with the right models pinned. If a subagent shows the wrong model, open it for editing and change the model field.

A note on the effort field in deep-reasoner: this overrides the session effort level when the subagent is active. We set it to high rather than max on purpose — max on Opus is fine for a single architecture call, but if the orchestrator fans out to the deep-reasoner many times in a session, the cost adds up. Leave max on the orchestrator (Fable 5) where it pays for itself by keeping context lean.

Step 3 — Install the Codex plugin

Run these four commands in order, inside Claude Code. The first three add and install the plugin; the fourth verifies everything is connected. 

/plugin marketplace add openai/codex-plugin-cc
/plugin install codex@openai-codex
/codex:setup

The /codex:setup command is the one that actually does the work of checking the connection. It will tell you one of three things:

  1. Codex is installed and authenticated. You're done — skip to Step 4.
  2. Codex is not installed but npm is available. It will offer to install Codex for you. Accept.
  3. Codex is installed but not authenticated. Run !codex login and sign in with a ChatGPT account or an API key.

If you'd rather install the Codex CLI yourself before running /codex:setup: 

npm install -g @openai/codex
codex login
Verify it worked
Run /codex:status in Claude Code. It should report that Codex is ready and the session is connected. Then run /codex:rescue --background "echo hello" — a trivial task — to confirm Codex actually responds.
Common pitfall
If /codex:setup says "Codex is installed but not authenticated," the login step is on the Codex side, not the Claude Code side. Run !codex login (with the exclamation mark) so it runs in the shell, not in the Claude session. Then re-run /codex:setup.

Step 4 — Drop the orchestration snippet into CLAUDE.md

The subagents and the plugin know what they do. They don't yet know how Fable 5 should delegate to them. That's the job of CLAUDE.md — a Markdown file in the project root that Claude Code reads at the start of every session.

Create or open CLAUDE.md in the project you'll be orchestrating from. The snippet below is the minimum the orchestrator needs. Adapt the rest of your CLAUDE.md as usual — this block just adds the delegation rules. 

## Orchestration workflow
You (Fable) are the orchestrator. Plan, decompose, synthesize.
Reasoning-heavy phases → deep-reasoner
Mechanical work → fast-worker
Codex (/codex:rescue --background) is a cracked engineer on
par with deep-reasoner, from a different perspective.
Treat as a peer, not a reviewer.
High-stakes decisions: task Opus + Codex on the same problem
in parallel, synthesize the best of both, without showing
either the other's answer. Keep your own context lean.
Verify it worked
Start a new Claude Code session in that project. Fable 5 should respond by acknowledging the orchestration rules before you ask your first question — or, more commonly, by behaving like an orchestrator on the first task (decomposing before answering, mentioning when it routes to a subagent).

End-to-end smoke test

All four setup steps are done. Run this one prompt to confirm the whole system is wired up. It exercises the orchestrator + the deep-reasoner + the fast-worker in a single pass. 

Goal: Add input validation to the createUser function in src/users.ts.
Context: Express + zod. The function currently takes (req, res) and
         inserts into Postgres with no validation.
You're the lead. Plan first, then execute. Delegate reasoning to
deep-reasoner, grunt work to fast-worker.

You should see Fable 5 produce a plan, route the schema-design question to deep-reasoner (Opus), and route the actual implementation to fast-worker (Sonnet). If Fable 5 just writes the code itself, your CLAUDE.md snippet isn't being read — check the file is at the project root and named exactly CLAUDE.md.

Micro-experiment — one number, one data point

The mission says you want to measure cost vs. quality. The smallest possible data point is a single-task A/B: run the same task twice, once with orchestration and once without, and log the tokens.

  1. Run the task un-orchestrated

    Switch to /model opus (or whatever your most expensive default is). Run the task. Note the input/output/total tokens from the session footer.

  2. Run the task orchestrated

    Switch back to /model fable with the orchestration rules in CLAUDE.md. Run the same task. Note the tokens again.

  3. Log both runs

    Open learning-records/0003-cost-experiment-first-run.md and record: task description, two token counts, two wall-clock times, two-line quality assessment. Just one run, both arms.

One data point is not a benchmark. But it converts "the orchestrator pattern saves money" from a claim into a measurement, and the act of recording it is what separates reading the lesson from doing the work.

Troubleshooting quick reference

Symptom Likely cause Fix
Fable 5 missing from /model picker Claude Code < v2.1.170, or no Fable access on the account Run claude update; check account access
with max effort line missing Effort not set, or model is not Fable 5 / Opus 4.7+ / Opus 4.8 Run /effort max after confirming the model
Subagent shows wrong model in /agents Frontmatter model field set to wrong alias Edit the file, use opus / sonnet / haiku / fable (no quotes)
/codex:setup says not authenticated !codex login never ran, or token expired Run !codex login in the shell, re-run setup
Fable 5 writes implementation instead of delegating CLAUDE.md snippet missing or in wrong location Confirm the file is at the project root, named exactly CLAUDE.md, with the snippet from Step 4
Subagent never invoked even when it should be Subagent description too vague Make the description specific: include the trigger phrases (e.g. "Use for X, Y, Z")
