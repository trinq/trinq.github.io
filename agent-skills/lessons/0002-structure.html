Structure

Steps and reference, nothing else · the second axis of the skill checklist
~5 min Lesson 0002 glossary · checklist

Once you've decided who fires the skill, the next axis is structure: what does the inside of SKILL.md look like? Matt's discipline is to recognise exactly two kinds of content, and to keep the file minimal by pushing everything else out behind context pointers.

The two units

Steps are imperative. The agent does these.

Reference is descriptive. The agent reads these to inform its actions.

# Inside a skill:
1. Read the failing test.                 ← step
2. Read ./reference/error-glossary.md.     ← step (also a context pointer)
3. Match the error to a category.         ← step
4. Apply the fix template.                ← step

# Inside reference/error-glossary.md:
- TIMEOUT: ...                            ← reference
- NULL_POINTER: ...                       ← reference

That's it. If a paragraph is neither an action the agent takes nor material the agent reads to take an action, it's in the wrong place.

Why two units? Steps are the skeleton. Reference is the muscle. Mixing them is how skills bloat: a paragraph that "explains why" creeps into the steps section, then the reference section grows a duplicate of it, then neither can be deleted without leaving the other stranded.

The branches problem

The real test of structure is what happens when the skill has branches — different paths the agent might take based on context.

Suppose a debug skill can route errors into TIMEOUT, NULL_POINTER, or PERMISSION_DENIED. The fix templates for each are wildly different. You have three options:

Inline everything

  • All three templates in the main file
  • The agent reads all three even if only one applies
  • Bloats every run, bloats every model-invoked description

Push behind context pointers

  • Main file has the routing logic (the only universal part)
  • Each template in its own file, linked
  • Agent loads only the file that matches its branch

The second option is what Matt means by "make skill.md minimal". A skill with branches should look like a router, not a phone book.

Three rules for a minimal skill.md

  1. Steps only at the top. Imperative sentences. Each one is an action.
  2. Reference only behind context pointers. Link to it, don't inline it. The agent will load it if needed.
  3. If a paragraph applies to only one branch, push it out. The main file should be readable in one screen.

What "context pointer" actually means

A context pointer is anything that lets the agent load more context on demand:

  • A markdown link to another file (see ./reference/glossary.md)
  • An instruction to read a specific file (Read the PRD template at ./templates/PRD.md)
  • A pointer to an external resource (see https://...)

What it is not: a paragraph that explains the same content inline. If you can move it behind a pointer without changing the agent's behaviour, move it.

Try it now

Pick one of your skills. The largest one. Open it. Now:
  1. Highlight every paragraph. Classify each: step, reference, or other.
  2. For each "other" paragraph, ask: which branch of the skill is this for? If only one, push it behind a context pointer.
  3. For each step, ask: is the action imperative, or is it actually advice dressed up as an instruction? Move advice to reference.
  4. Goal: main file readable in one screen.

If your skill has 5+ branches and the main file is still under a screen, congratulations — you've already done the work. If it's two screens of prose, you have a structure problem, not a pruning problem.

Common trap: "but the agent needs to know X"

The pushback you'll give yourself: "but if I don't include this paragraph, the agent won't know X". Two responses:

  1. Move it to reference. The agent will read it when it needs to.
  2. If the agent always needs it, it's not branch-specific — but it might still be reference, not a step. The agent reads it once at the top, then acts.

Default to pushing out. You'll be surprised how rarely the agent actually needed that paragraph inline.

Takeaway

Two units. Steps and reference. Anything else is structure waiting to be cleaned up.
