The Skill Checklist, retold

The whole framework, in one lesson · my retelling of the talk, with citations to the original
~10 min Lesson 0005 glossary · checklist · deep dive

Lessons 0001–0004 took the framework apart axis by axis. This lesson reassembles it as one essay — closer in shape to a talk, with the four axes knitted together the way Matt Pocock knit them together in "Building Great Agent Skills". If you only have ten minutes, read this lesson. If you have an hour, read it, then watch the talk.

Read these first: Trigger, Structure, Steering, Pruning. This lesson is a synthesis, not a replacement.

The argument

The talk opens on a diagnosis. Developers, Matt argues, keep walking into the same trap in a new disguise. A few years ago it was tutorial hell: you read everything, you can't piece it together. Before that it was framework hell: every ten minutes a new JavaScript framework, you can't keep up. The version we're walking into now he calls skill hell.

We have all of these skills available freely that you can download, contribute to, you can figure out on your own — but you don't really know how the pieces all work together. You can't tell a good skill from a bad skill.— Matt Pocock, opening of the talk

Skill hell isn't a property of any individual skill. It's the state of an ecosystem: a community where the supply of skills has outrun the ability of users to evaluate them. The fix isn't more skills. "Just one more skill, bro," Matt says, is what we've been telling ourselves, and it's not working.

He also names what he thinks the missing piece is. Not better authoring tools. Not more contributors. A rubric — a shared way to look at a skill and say what makes it good or bad.

The thing we're missing is we don't know what makes a skill great. We can't yet look at a skill and go, okay, this skill is doing these good things and these bad things. There's no shared rubric.— Matt Pocock

The checklist that follows is his attempt to fill that gap. It's a procedure you can run on any skill: four audits, in a fixed order, that give you a verdict and a direction for fixing it. The order matters because each audit depends on the previous being roughly right.

Axis 1 — Trigger

First question: who fires this skill? You, the pilot, or the agent?

The mechanism is small. If a skill has a description in its frontmatter, that description sits in the agent's context on every request — a pointer the agent can choose to follow. That's a model-invoked skill. If the description is hidden (disable-model-invocation: true), only you can fire it. That's a user-invoked skill.

Both have a cost. Model-invoked adds a description to every context window — context load. User-invoked requires you to remember to fire it — cognitive load. Choosing is partly economics: with a hundred model-invoked skills you've spent a hundred descriptions' worth of attention; with a hundred user-invoked skills you've reserved a hundred slots in your head.

But Matt's real reason for preferring user-invoked is sharper than the economics.

Every time you have a context pointer pointing from one resource to another, the model may just choose not to follow it. Even if it's absolutely perfect for the task, it may just choose not to invoke the skill.— Matt Pocock

A model-invoked skill is a probabilistic commitment that the agent will pick it up. A user-invoked skill is a deterministic commitment that you will. The unpredictability isn't a tax — it's a class of bugs. To use a model-invoked skill responsibly, you have to run evals to verify the agent fires it at the right times. Matt doesn't run evals on his skills. So his skills are user-invoked. He is removing a class of bugs from existing.

I much prefer removing that level of unpredictability, imposing a bit more cognitive load on the user. You're removing a class of problem from even being a problem.— Matt Pocock

His decision rule, distilled: if the agent would reliably know when to fire it without seeing the description, user-invoke it. Otherwise, model-invoke it and accept the eval burden.

Axis 2 — Structure

Once you've decided who fires the skill, the next axis is what's inside the file. Matt recognises two units: steps (imperatives the agent does) and reference (descriptive material the agent reads). Some skills are all steps. Some are all reference. Most are mixes.

You can have skills that have no steps and are only reference. And you can have skills that are no reference and only a set of simple steps. But if you start thinking of skills as composed of these two units, it really helps just break them down a lot more.— Matt Pocock

The push to keep skill.md minimal comes from two forces. Small files are easier to maintain, audit, and read. And every word you shave is a token shaved — per request, when the skill is model-invoked.

Every time you shave off a word, that is a token shaved. That multiple token shave from your skill's cost.— Matt Pocock

The lever he reaches for is branches. If a skill has multiple routes it can take (different paths based on context), the reference material that applies to only one route is a candidate for moving out of the main file. His domain-modeling skill, for example, sometimes updates a context glossary and sometimes writes an ADR. Each template lives in its own file, linked from the main skill only when the relevant branch fires.

The test is simple. Walk through the branches in your head. For each piece of reference material, ask: does this apply to every branch? If yes, leave it in. If no, push it behind a context pointer. The result is a main file small enough to read in one screen — and an agent that loads only the material it needs for the branch it's actually on.

Axis 3 — Steering

Now the most opinionated axis. Even with the right trigger and a clean structure, the agent often does what you literally said instead of what you meant. Matt's response is a technique he calls leading words.

The agent doesn't do what I want. I specify something in the skill. I think that I've been clear and then it just doesn't do the thing.— Matt Pocock

A leading word is a precise term — usually two or three words — that packs a procedure into a phrase the agent already half-knows. Matt's canonical example is vertical slice. The naive instruction is "don't code layer by layer, make a small slice first". The leading-word version repeats vertical slice everywhere. The agent, having seen the term in prior training, picks it up — and you can verify it did, because vertical slice shows up in its reasoning trace.

You say vertical slice in your skill and then you'll notice in the reasoning traces that the agent is saying, "Okay, we're going to do this as a thin vertical slice."— Matt Pocock

Matt borrows the term from literary theory — Leitwort, a guiding word that recurs and signals intent. The same move works across prose and agents: a precise, repeated term activates more useful priors than a paragraph of explanation.

There's a second lever on this axis. Sometimes the agent doesn't do enough leg work on a step. The example he keeps returning to is plan mode: the agent has two steps (ask clarifying questions, then create a plan), and it sees the future goal. So it rushes through clarifying questions, asks you two things, and eagerly produces a half-formed plan. The fix is to split the skill so the agent doing step 1 cannot see step 2. Hiding the future goal forces depth on the present goal.

We have step one and step two, but the agent only sees one step at a time… It's not always necessary to split skills into individual steps, but in particular cases where you really want an extra chunk of leg work — there is no technique like it.— Matt Pocock

Two steering tools, both compact. Leading words change what the agent hears you say. Splitting skills changes what the agent sees ahead of it.

Axis 4 — Pruning

Last axis, cheapest in spirit but costliest in attention. Pruning catches the sediment that accumulates when many people edit a file for a long time. Three failure modes, in increasing levels of insidiousness.

Single source of truth. Don't repeat material across branches or across skills. If a PRD template lives in two places, extract it. If "what is a test seam" is explained twice in the same file, decide where it canonically belongs.

Sediment. Legacy material that doesn't apply anymore. People add their bit, no one feels brave enough to delete anyone else's, and the file accretes. Look for dated references — old APIs, old branches, old workflows.

No-ops. The worst offender. Paragraphs that look like they're doing something, but if you deleted them the agent's behaviour wouldn't change. Matt's example: a paragraph in an implement skill that tells the agent to "write a long detailed commit message". If you delete it, the agent would probably still write a long detailed commit message. The paragraph is theatre.

Things inside the skill that appear to do something, but don't actually influence the agent's behavior inside the context of the skill.— Matt Pocock

The deletion test is the one technique that consistently catches no-ops. Save the skill's current output. Delete a paragraph. Re-run the skill. If the output is materially the same, the paragraph was a no-op. Leave it deleted.

Matt closes the talk on the same question with which he opened it. People ask him how his skills get so small. He lists four contributors — in this order:

Using deletion tests, using making sure that I compact things into leading words, I don't have anything irrelevant in there, and I don't have any no ops.— Matt Pocock

Notice: not the deletion tests alone. The deletion tests catch what's already there. To keep skills small in the first place, you lead with compaction — choosing leading words while you write, not just pruning after.

What he didn't cover

Three honest gaps. Matt doesn't quantify the cost of model-invoked skills at scale. "A hundred descriptions in context" is an example, not a measurement. He compares his repo to superpowers structurally — user-invoked versus model-invoked — but never runs the checklist on a third-party skill. And he doesn't address what to do when the deletion test gives a false positive — a paragraph that looks like a no-op but is silently load-bearing.

These gaps matter because they remind us the framework is one expert's opinion. It's a well-reasoned opinion, but it's not yet a measured one. Use the checklist. Watch for counter-examples. When you find one, that's signal, not noise.

The order as a cost sequence

The order — Trigger, Structure, Steering, Pruning — is also a cost sequence. Trigger changes rewrite frontmatter, which is cheap. Structure changes reshape the file. Steering changes are vocabulary edits. Pruning changes require running the deletion test, which is medium effort. Audit in this order so cheap fixes don't get buried under expensive ones.

The escape

The closing image, used twice in the talk, is escape. Skill hell, Matt says, is the latest version of a recurring pattern. Tutorial hell was the previous one. Framework hell before that. Each had a moment where the community had the knowledge but no rubric — no shared way to look at the tool and say what made it good.

I hope that what I've given you is enough to help you escape from skill hell, or at least try to make the bitter journey out of there.— Matt Pocock, closing

The checklist doesn't make that escape automatic. It gives you a procedure for escaping. Whether you run it on your skills, your team's skills, or the skills of strangers — that's the choice the talk hands you at the end.

Takeaway

Four audits, one order, one goal. Audit the trigger — who fires this, and what's the bug class? Audit the structure — are steps and reference separated, is branching material behind pointers? Audit the steering — are you using leading words, is the agent seeing too much future? Audit the pruning — are there no-ops, sediment, or duplicates?

Run it on the next skill you touch. Then run it on a community skill you've been hesitant to trust. The rubric exists because no one had one. Now you do.
