Trigger

The first axis of the skill checklist · who fires the skill, and what's that costing?
~5 min Lesson 0001 glossary · checklist

Every skill has a trigger: the moment when its instructions enter the agent's context. Two fundamentally different mechanisms, two fundamentally different costs. The first axis of Matt's checklist is choosing deliberately between them — and being honest about which cost you'd rather pay.

The two mechanisms

User-invoked. You, the pilot, type /skill-name (or otherwise tell the agent which skill to run). The skill's description is not in the agent's context. The harness loads the full SKILL.md only when you fire it.

Model-invoked. A short description in the skill's frontmatter sits in the agent's context on every request. The agent reads that description, decides the skill is relevant, and pulls the full SKILL.md into context itself.

The description is a context pointer. It lives in the agent's permanent context and points at the file the agent can load — but doesn't have to. Whether the agent actually loads it is its choice.

What each costs

Matt's framing is sharp: each style of trigger imposes a different kind of load.

Trigger Cost Who pays
Model-invoked Context load — every description is on every request The agent
User-invoked Cognitive load — you must remember to fire it You

If you have 100 model-invoked skills, you have 100 descriptions in your context window on every request. Tokens, but also attention: the agent now has 100 things to think about. The agent's judgment about which to fire becomes a failure mode in itself — sometimes it'll load the wrong skill, sometimes it'll load none and improvise badly.

If you have 100 user-invoked skills, you have to remember all 100. That's a lot of headspace reserved for tooling.

Frontmatter tells you which one you have

---
name: writing-great-skills
description: Audit an existing agent skill against the four-axis checklist...
# (no disable-model-invocation) → model-invoked
---

---
name: grill-me
description: Stress-test a plan by interviewing the user relentlessly...
disable-model-invocation: true   # user-invoked
---

Open any skill in your environment and check the frontmatter. If the description is there and there is no disable-model-invocation: true, you have a model-invoked skill sitting in your context right now.

The decision rule

Matt's heuristic, distilled:

  1. Will the agent reliably know when to fire this skill without seeing the description?
  2. If yes → user-invoked (cheaper context).
  3. If no → model-invoked (cheaper cognitive load).

Most "I just want this to happen when X is true" skills are user-invoked. Most "I want this fire pattern to be discoverable" skills are model-invoked. The mistake is leaving the default in place because you didn't think about it.

Why the order in the checklist matters. Trigger is axis one because every later axis can be wasted work if the skill never fires (user-invoked and you forget it) or fires wrong (model-invoked and the description is sloppy). Audit this first.

Try it now

Audit: what triggers are in your environment right now?

Run this in your shell:

find ~/.claude/skills ~/.cursor/skills-cursor ~/.agents/skills \
  ~/.cursor/projects/*/mcps 2>/dev/null \
  -name "SKILL.md" -o -name "skill.md" \
  | head -50

For the first 10 skills:

  1. Open the file. Read the frontmatter.
  2. Does it have a description:? Is disable-model-invocation: true set?
  3. Classify: user-invoked or model-invoked.
  4. For each model-invoked one, ask: would the agent reliably fire this on its own, or is this a tax I'm paying on every request?

Expected finding for most environments: 60–80% of skills are model-invoked by default, and half of those would be cheaper as user-invoked. That's the context-load bloat Matt is calling out.

Common trap

"I'll just make everything model-invoked — more flexible!" The trap: every model-invoked skill adds a description to every request. The agent's ability to choose well degrades as the menu grows. Matt explicitly prefers user-invoked skills; the tradeoff is that he has to remember them. That's a price he pays deliberately for cheaper agent context.

If you've never made this tradeoff consciously, you haven't audited your trigger yet.

Takeaway

Open the next skill you write and decide: who fires this? The answer should appear in the frontmatter, not in the default.
