Steering

Leading words and leg work · the third axis of the skill checklist
~6 min Lesson 0003 glossary · checklist

Structure tells you what's in the file. Steering tells you whether the agent does what you mean. Matt calls steering "the main thing" — the technique that crosses every other axis and turns into a habit.

The problem steering fixes

The classic failure: you write a skill. You explain something clearly, in plain English, in a paragraph. The agent reads it, then does something close to but not what you wanted. You add more words. The agent gets more verbose about doing the wrong thing.

Why? Because long explanations of behaviour get diluted in the agent's reasoning. The agent hears "lots of words about X" and pattern-matches to "this is the kind of thing where I do whatever is typical". A vague paragraph about doing X well is easier to ignore than a precise term.

The technique: leading words

Pick a leading word — a precise, dense term the agent already half-knows from prior training. Pack your meaning into that term. Use it consistently throughout the skill.

The agent will echo the term in its reasoning trace: "I'll do this as a thin vertical slice..." And that echo changes its behaviour — because the term activates a richer prior than your prose did.

Matt's canonical example. The leading word is vertical slice. The problem is agents coding layer by layer — all the DB, then all the API, then all the UI. The naive instruction is "don't code layer by layer; create a small slice first". The leading-word version just says vertical slice, everywhere. The agent's prior on that phrase does the rest.

What makes a good leading word

  • Already exists in the agent's prior. The agent has seen this phrase. It carries weight. Vertical slice, red-green-refactor, event sourcing, type-driven design, evidence before assertions.
  • Carries a procedure, not just a fact. A leading word should imply how to act. Test seam implies "find a place where you can isolate and verify behaviour".
  • Short enough to repeat. You will write it many times. One to three words is right.
  • Different from your neighbour's. If you use vertical slice and so does every other skill in your repo, the term gets diluted. Pick terms that are yours.

How to use them

Put them in the skill's prose. The agent will echo them. Verify by reading the agent's reasoning trace.

# A skill with weak steering:
When implementing a feature, don't build the whole data layer first.
Make a small, working piece first that goes from the user interface
all the way down to the database. Get it working end-to-end. Then
extend it. Don't build layer by layer.

# Same skill with leading-word steering:
Use a vertical slice. The first slice must be thin, end-to-end,
and runnable. Each subsequent slice extends the previous one.
Repeat until the feature is complete.

Both paragraphs say the same thing. The second one has a leading word that the agent already half-knows. The first one is just… words.

Building the habit

This is the one technique in the framework that crosses the boundary into every other skill — it becomes a habit, not a one-off fix. Three habits to install:

  1. Pick 2–4 personal leading words. Don't recycle the obvious ones — pick the ones that match how you think. Examples: minimal pivot, evidence before assertions, narrow waist, smallest diff that still teaches me something.
  2. Repeat them across skills. When the agent sees vertical slice in three skills, it generalises. When it sees narrow waist in five, that's your lens.
  3. Audit the trace. After running a skill, read the agent's reasoning. Are your leading words present? If not, the steering is too weak — make the words more prominent, or pick better ones.

The leg-work lever: hiding the future goal

Sometimes the problem isn't what the agent does, it's how little effort it puts in. Matt's example: every plan-mode implementation he's tried has ask clarifying questions doing too little leg work, because the agent can see the future step create plan and rushes toward it.

The fix: split the skill. Two skills instead of one. The first skill does step 1 (and only step 1). The second skill does step 2. The agent running skill 1 cannot see step 2, so it cannot shortcut toward it.

This is not always the right move. Use it when:

  • The agent visibly under-invests in a step.
  • The shortcut is specifically caused by seeing the future goal.
  • You're willing to pay the cost of two skills (more to maintain, more to remember).

Try it now

Audit the leading words in your three highest-leverage skills.
  1. Read each skill. Underline every term that sounds like it carries a procedure.
  2. For each underlined term, ask: does this appear in the agent's reasoning trace after I run the skill? If you've never checked, run the skill once and check.
  3. Find one place where a long paragraph could be replaced by one or two leading words. Rewrite it.
  4. Re-run. Did the agent's behaviour change in the direction you wanted? If yes, that leading word is doing real work.

Common trap: leading words that don't lead

It's tempting to invent your own jargon. "We always do a contextual rehydration before refactoring". The agent has never seen this phrase. It carries no prior. It's not a leading word — it's an in-group idiom, and the agent will treat it as decorative.

If you want to coin a term, you have to also teach it inside the skill — define it, give an example, repeat. That's expensive. If you can't justify the cost, pick a term the agent already half-knows.

Takeaway

Leading words are the one technique that becomes a habit. Pick two, repeat them everywhere, watch the trace. Split a skill only when the agent is rushing toward a future goal it shouldn't see.
