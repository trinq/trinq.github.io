Pruning

Sediment, crud, and no-ops · the fourth axis of the skill checklist
~5 min Lesson 0004 glossary · checklist

Pruning is the last axis because it only matters once Trigger, Structure, and Steering are roughly right. Pruning a skill with a broken trigger is just rearranging deck chairs.

Pruning is also the axis where Matt's question — "how do you keep your skills so small?" — gets its one clean answer: deletion tests. Run them, delete what survives them.

Three failure modes

Sediment

Irrelevant legacy paragraphs. Most common in skills that multiple people have edited: each person adds their bit, nobody deletes anyone else's, and the file accretes. A paragraph about a deprecated branch. A workflow that worked three versions ago. A note from when the harness loaded skills differently.

Sediment is the easiest failure to spot because it usually dates itself — references to old APIs, old branches, old file paths.

Crud

Generic material inside the skill that doesn't apply to every branch of its use cases. A paragraph that helps in 30% of cases but bloats the skill for the other 70%. The fix is almost always to push it behind a context pointer (Structure-axis work).

Crud is the failure mode where Structure and Pruning meet — when you find crud, the answer is structure.

No-ops

Instructions that look like they do something but don't actually change the agent's behaviour. Matt's example: a paragraph in an implement skill that says "write a long, detailed commit message". If you delete that paragraph, the agent probably still writes a decent commit message — because the agent already does this without being told. The paragraph is a no-op. Delete it.

No-ops are the hardest to spot by eye. You need the deletion test.

The deletion test

One procedure, applied paragraph by paragraph:

  1. Pick a paragraph in a skill.
  2. Save a copy of the skill's current output (the agent's response on a representative task).
  3. Delete the paragraph.
  4. Run the skill again on the same task.
  5. Compare the output to the saved copy.
  6. If the output is materially the same → the paragraph is a no-op → leave it deleted.
  7. If the output is materially different → the paragraph was doing real work → put it back.
The deletion test is the one technique that consistently catches no-ops. Build it into your review workflow for any skill — yours or community-authored. A skill you've never deletion-tested is a skill you don't actually know.

Single source of truth

The non-obvious pruning rule: every piece of reference material should live in exactly one place. If a glossary term appears in two skills, one of them should reference the other. If a PRD template appears in two places, extract it.

Why: duplicated reference goes stale. One copy gets updated, the other doesn't. The agent then reads two slightly-different versions and doesn't know which to trust. Single source of truth means there's only one version to update, and the others are guaranteed consistent.

The quickfire pruning checklist

  • Massive skill — symptom of one of the other three failures. Find the cause first.
  • Repeated material — single-source-of-truth violation. Extract.
  • Sediment — legacy paragraphs. Look for dated references (old API names, old branches). Kill.
  • Crud — material that helps one branch but bloats all. Push behind a context pointer.
  • No-ops — paragraphs the agent ignores. Deletion test, then delete.

Try it now

Pick the same skill you structured in lesson 0002.
  1. Save the current output of the skill on a representative task.
  2. Delete one paragraph at a time. Re-run. Compare.
  3. For each paragraph that survives the deletion test: leave it deleted. For each that doesn't: put it back, but consider whether you can shorten it without breaking it.
  4. Look for paragraphs that exist in two skills. Extract to a single reference file.
  5. Final goal: the main file is one screen.

You will almost certainly find 2–3 no-ops per skill. That's normal. Matt's skills are small partly because he runs this test every time he edits.

Common trap: "but I want the agent to do X"

The hardest no-ops to delete are the ones that reflect your taste, not the agent's behaviour. "I always want the agent to write tests first." If the agent writes tests first without being told, that instruction is a no-op — keep the behaviour, delete the paragraph.

Test of whether it's a no-op: would removing the paragraph change what the agent actually does? If not, it's a no-op, even if removing it makes you feel less safe.

Takeaway

You now have the full checklist.

Trigger → Structure → Steering → Pruning. In that order. Five minutes per skill.

The discipline is to keep the loop short: every skill you touch, audit it. Every new skill you write, audit it before you commit it. Every skill you read from a community, deletion-test it before you trust it.

Matt's whole point: skill hell is what happens when nobody has a rubric. You have one now. Use it.
