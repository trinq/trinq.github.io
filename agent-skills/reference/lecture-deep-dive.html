The Lecture Itself — A Deep Dive

The second-order insights from Matt Pocock's "Building Great Agent Skills" · read this once and you'll catch lines the lessons glossed over

This document is what you get if you read the talk at 0.5× speed and underline twice. It is the ground the four lessons stand on. Use it when you're auditing a skill and one of the lessons feels too neat — this doc has the messy parts.

The frame Matt sets up

The talk opens with three hells. Tutorial hell (you can't piece things together). Framework hell (the next hot thing every ten minutes). Skill hell: you have many skills, but you can't tell good from bad, so the skills don't deliver on their promise.

The thing we're missing is we don't know what makes a skill great. There's no shared rubric. — Matt Pocock

The frame is important because it licenses the rest of the talk. Without it, "use a four-axis checklist" sounds like personal taste. With it, the checklist is an escape from a recognised pathology. Matt is presenting a rubric to fix an industry-wide failure mode — not to optimise his own skills.

This is also why the talk feels opinionated. He's filling a gap that didn't exist a year ago. There's no peer-reviewed work on skill writing yet. The checklist is one expert's well-reasoned answer.

The Trigger axis — what the lessons underplayed

The lesson stopped at the frontmatter distinction. The transcript has three lines worth pausing on:

Every time you have a context pointer pointing from one resource to another, the model may just choose not to follow it. Even if it's absolutely perfect for the task, it may just choose not to invoke the skill.— Matt Pocock

This is the *real* reason he prefers user-invoked. Not the context-cost math (he mentions it, then moves on). The reason is unpredictability. A model-invoked skill is a probabilistic bet that the model will pick it up. A user-invoked skill is a deterministic bet that you will pick it. He prefers deterministic bets when the cost of getting it wrong is high.

I much prefer removing that level of unpredictability, imposing a bit more cognitive load on the user. You are removing a class of problem from even being a problem.— Matt Pocock

That last phrase is the giveaway: the goal isn't "small context". The goal is "fewer classes of bug". A skill that can't misfire is a class of bug eliminated, not just a token cost saved.

The follow-on implication: if you write a model-invoked skill, you implicitly take on the eval burden. You have to verify the model picks it up at the right times. Matt says this "leaves people to need to eval their skills to make sure they're being called at the right time which is really nasty and it's a problem that I prefer to avoid." Most skill authors don't run evals. So most model-invoked skills are silently misfiring.

The Trigger question reframed: not "which is cheaper", but "which class of bug am I willing to own?". User-invoked owns the recall problem (you forget to fire it). Model-invoked owns the recognition problem (the agent forgets to fire it). Pick the bug class you can debug.

The Structure axis — what the lessons underplayed

The lesson framed steps and reference as the two units. The transcript makes a sharper claim:

You can have skills that have no steps and are only reference. And you can have skills that are no reference and only a set of simple steps. But if you start thinking of skills as composed of these two units, it really helps just break them down a lot more.— Matt Pocock

Read that carefully. Skills can be pure reference. The lesson's structure mental model was "steps with reference attached". The talk's mental model is broader: any skill is a mix, but the ratio is up to you. A pure-reference skill is a glossary the agent loads on demand. A pure-step skill is a checklist. Most skills are mixes.

Smaller skills are just easier to maintain, easier to audit, fewer words to think about. And every time you shave off a word, that is a token shaved. That multiple token shave from your skill's cost.— Matt Pocock

The "multiple token shave" line is doing work. Every model-invoked description is on every request. Every word shaved from a description saves tokens on every request — across all your sessions. Across all your sessions and across all the agents using that skill. Matt is hinting that small skills have network effects on cost, not just line-item savings.

The branches reasoning has a second layer the lesson skipped:

If you feel like your skill is going to be used in lots of different ways, then take the reference material that's relevant for those branches and hide them behind context pointers.— Matt Pocock

Two distinct branches tests. The first: does every branch need this material? If yes, inline. If only one branch, push out. The second test (which Matt doesn't state, but is implied): does the agent reliably know which branch it's on before loading anything? If yes, the routing happens in skill.md and the branches live behind pointers. If no, the agent has to load everything to figure out what to do — at which point the inline material is back in the main file regardless.

The Steering axis — what the lessons underplayed

The transcript has a subtle aside:

The idea of leading words or Leitwort if you like literary theory.— Matt Pocock

Leitwort is a real concept from literary criticism — a "lead word" or "guiding word" that recurs in a text and signals thematic continuity. Matt is borrowing deliberately. This isn't a fresh coinage; it's a known technique from a different field, ported in. The transfer is legitimate because both cases (novels, agent context) are trying to inject a specific meaning into a reader's reasoning by repetition.

Agents are actually pretty good at helping you think of them.— Matt Pocock

This is a permission slip. If you're struggling to invent leading words, ask the agent. The agent knows its own prior. "What 3-word phrases from software engineering pack a procedure into them?" is a great prompt to run.

Sometimes the agent just doesn't do enough leg work.— Matt Pocock

The leg-work problem is broader than plan mode. Anywhere a skill has steps that depend on each other, the agent sees the future step and shortcuts toward it. Matt says he's found this "almost everywhere" plan mode exists, but the principle is general: sequence your steps by splitting them when an earlier one isn't pulling its weight.

There's a counter-tactic Matt doesn't mention: instead of splitting the skill, you can split the steps by giving the earlier steps a different leading word. If "ask clarifying questions" is tagged as grilling — and the user invokes a separate skill for "create plan" tagged as focused — the agent has two different frames. This is the same trick as leg-work splitting, but achieved through vocabulary, not skill boundaries.

The Pruning axis — what the lessons underplayed

People ask me a lot how I get my skills so small and it's just using these techniques, using deletion tests, using making sure that I compact things into leading words, I don't have anything irrelevant in there, and I don't have any no ops.— Matt Pocock

The lessons made the deletion test sound like the one trick. But Matt lists four contributors to small skills: deletion tests, leading words (compaction), no-irrelevant-content, no-no-ops. Deletion tests catch the no-ops. But to get small skills in the first place you have to lead with compaction while writing, not just prune at the end.

The next failure mode is really common when an agent writes your skills...— Matt Pocock

This is the part that should scare you if you author with agents. Agents love to add paragraphs that sound important. The deletion test is the guard. But the meta-lesson is: if you used an agent to write a skill, plan to audit it ruthlessly. Don't ship AI-authored skills without running the deletion test on every paragraph.

The hierarchy of fixes. Matt's sequence (Trigger → Structure → Steering → Pruning) is also a cost sequence. Trigger changes rewrite frontmatter (cheap). Structure changes reshape the file (cheap). Steering changes are vocabulary edits (cheap). Pruning changes are deletion tests (medium). The earlier the axis, the cheaper the fix. Audit in this order so cheap fixes don't get buried under expensive ones.

What the talk does not address

Three things Matt doesn't cover — gaps a careful reader should notice.

  1. No measurement. He never quantifies the context cost of N model-invoked skills. No benchmarks, no rule of thumb beyond "if you have a hundred".
  2. No skill examples outside his own repo. The contrast with superpowers is structural (user- vs model-invoked), not deep. There's no audit of a third-party skill using his checklist.
  3. No failure recovery. What do you do if you delete a paragraph and the agent stops doing the right thing? Matt doesn't address false positives of the deletion test — paragraphs that look like no-ops but are silently load-bearing.

These aren't criticisms — the talk is 20 minutes. They're gaps future sessions can fill: a session measuring context cost at scale, a session auditing a third-party skill with the checklist, a session probing the limits of the deletion test.

Lines worth saving

  • "Skill hell is what happens when nobody has a shared rubric." — your mission statement.
  • "Removing a class of problem from even being a problem." — the Trigger justification, in one phrase.
  • "Every time you shave off a word, that is a token shaved. That multiple token shave from your skill's cost." — the network effect of small skills.
  • "English is a pretty wide API." — a permission slip to experiment with leading words.
  • "Use deletion tests, compact into leading words, no irrelevant content, no no-ops." — Matt's four-part formula for small skills, in order.
