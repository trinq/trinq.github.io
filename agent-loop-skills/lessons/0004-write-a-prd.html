Lesson 0004
Write a PRD: Capture the Destination Before the Journey

Turn shared understanding into a durable document your future self and your AI agents can rely on

Prerequisite

This lesson builds on Lesson 0003 (Grill Me). If you haven't used grill-me to reach shared understanding with your AI, start there first.

The Gap Between Understanding and Implementation

After a good grilling session, you and the AI both understand the problem. But that understanding lives in a volatile context window — it will evaporate the moment the conversation moves on.

The write-a-prd skill fixes this. It takes shared understanding and crystallizes it into a document that:

  • Survives beyond the current conversation
  • Gives future agents a clear target to work toward
  • Serves as a contract between you and the AI about what "done" means

PRD ≠ plan. A plan describes the journey. A PRD describes the destination. Don't prescribe implementation details here — let the AI discover the best path during implementation.

The Skill

write-a-prd — invoked when the user wants to create a PRD.

You may skip steps if you don't consider them necessary.

  1. Ask the user for a long, detailed description of the problem and desired outcome.
  2. Explore the repo to verify their assertions and ground the PRD in reality.
  3. Interview the user relentlessly (grill-me) if anything is unclear.
  4. Sketch the major modules you will need to build or modify.
  5. Once you have a complete understanding of the problem and solution, use the template below to write the PRD and submit it as a GitHub issue.

The PRD Template

A PRD has four sections. Each serves a specific purpose:

1. Problem Statement

Why does this need to change? What's broken, missing, or painful today? Be concrete — avoid vague framings like "improve performance."

2. Solution Overview

What does the end state look like? Describe the behavior, not the implementation. Use "the system will..." statements.

3. User Stories

How does each stakeholder interact with the new feature? Written in plain language, describing desired behavior — not technical requirements.

4. Implementation Decisions

Constraints, non-negotiables, and known unknowns. What architectural decisions have already been made? What's still uncertain?

Example: Split Pane Document Editor

Here's a real-world example from a course platform build. After the grilling session surfaced 16 questions, the resulting PRD opened like this:

Problem: The article writing page currently regenerates the entire document on every AI interaction. Users lose their work-in-progress content.

Solution: Add a split pane document editing experience. Chat stays on the left. A new document panel appears on the right, showing the rendered output, live-updating as the AI writes.

Notice the user stories — they describe behavior, not code:

  • "As a course author, I can see the rendered article while the AI writes, so I can catch errors early."
  • "As a course author, I can edit the document panel directly and those changes are preserved."

Don't over-prescribe implementation decisions. The more you specify how in the PRD, the more likely the implementation will drift from reality as you learn. The PRD describes the destination — the journey (issues, TDD cycles) comes next.

Where the PRD Lives

The skill outputs the PRD as a GitHub issue. This is deliberate:

  • PRDs as issues can reference other issues (blocking relationships, parent-child links)
  • Agents can fetch it with gh issue view at any time
  • It lives alongside the code in the same repository

From there, the PRD feeds into the prd-to-issues skill — which breaks the destination into a journey of vertical slices.

How This Fits the Workflow

grill-me
reach shared understanding → write-a-prd
capture the destination → prd-to-issues
plan the journey → tdd
implement with confidence

Each skill does exactly one thing. Grill-me surfaces the full scope. Write-a-prd locks it down. The rest follows.

Practice Exercise

Think of a small feature you've been meaning to add to a project. Before the next lesson:

  1. Invoke grill-me Use the skill from Lesson 0003. Ask the AI to interview you about this feature. Don't stop until it asks 10+ questions.
  2. Say "write a PRD" When the interview feels complete, ask the AI to write a PRD as a GitHub issue.
  3. Review the result Does the PRD accurately reflect what you want? Are the user stories in plain language? Is anything missing?
