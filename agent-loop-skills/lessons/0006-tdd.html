Lesson 0006
TDD: Red Green Refactor with Agents

The single most effective way to raise the code quality of anything an agent produces

Prerequisite

This lesson builds on Lesson 0005 (PRD to Issues). You should have a set of vertical slices (GitHub issues) ready to implement. Each issue is the input to the TDD loop.

Why TDD with Agents Is Different

Humans do TDD for many reasons: to drive design, to document intent, to enable safe refactoring. With agents, there's one dominant reason:

TDD is the most consistent way to improve agent output quality. It forces the agent to write one failing test, then the minimum code to pass it. The loop creates tight feedback discipline that agents otherwise lack.

Agents have a peculiar weakness: they're precious about code in their own context window. Clear the context, and they forget. Keep it, and they resist changing it. TDD fights this by forcing small, testable steps. Each iteration is too small to be precious about.

The Red Green Refactor Loop

The TDD skill prompts the agent to follow this exact loop for every issue:

Red
write a failing test → Green
write minimum code to pass → Refactor
clean up without changing behavior
repeat until acceptance criteria met
→ Done

The TDD Skill

tdd — invoked when implementing a GitHub issue. Follows red green refactor until all acceptance criteria pass.

  1. Confirm interface changes needed with the user. This is critical — agents need a clear surface to target.
  2. Confirm which behaviors to test.
  3. Design the interfaces for testability (thin interfaces on top of deep modules).
  4. Write one failing test. Run it. Watch it fail.
  5. Write the minimum code to make the test pass. Run the test. Watch it pass.
  6. Refactor — look for candidates: duplicate logic, long functions, unclear names. Clean without changing behavior.
  7. Repeat from step 4 until all acceptance criteria are covered by tests.

Interface Changes Are the Most Important Step

Before writing a single test, the skill asks: what interface changes are needed? This question forces clarity about:

  • What is exported from each module?
  • What do callers actually use?
  • Where are the test boundaries?

Agents working on a well-structured codebase — with deep modules and thin interfaces — produce much better output than agents working on a codebase with many small, undifferentiated files.

TDD is hard in badly structured codebases. If the module boundaries are unclear, the test boundaries are unclear too. The improve-codebase-architecture skill (Lesson 0007) exists specifically to address this.

Refactoring: Where Agents Struggle

The refactor step is where agents underperform humans. Because the code the agent just wrote lives in its own context window, it becomes "precious" — the agent is reluctant to change it.

Techniques to work around this:

  • Clear the context window before the refactor step — the agent forgets what it just wrote and evaluates the code freshly
  • Name things specifically: "Extract duplicate logic in parseAndFormat" is better than "refactor for cleanliness"
  • Ask explicitly: "List 3 refactor candidates in this module and pick one"

The output is real: In one real project, a single TDD cycle on a document editing engine produced 28 passing tests covering all acceptance criteria — in one agent loop run.

Mocking and Deep Modules

The TDD skill includes guidance on:

  • Mocking at boundaries: Mock external services, not internal collaborators. The test should describe behavior, not implementation.
  • Deep modules: Prefer modules with rich interfaces over modules with many shallow functions. Deep modules are easier to test at their single surface.

How This Fits the Workflow

grill-me
surface the full scope → write-a-prd
lock the destination → prd-to-issues
plan the vertical slices → tdd
implement each slice with tests

The workflow chain is complete. TDD is the final skill — the one that actually builds working code.

Practice Exercise

  1. Pick an unblocked issue From your PRD breakdown (Lesson 0005), find an issue with no blocking dependencies.
  2. Confirm the interface Before writing tests, answer: what functions will callers use? What should the module export?
  3. Write a failing test Start with the simplest acceptance criterion. Write the test, watch it fail.
  4. Make it pass Write the minimum code to make the test pass. Don't optimize yet.
  5. Refactor After the test passes, ask: what would make this code cleaner? Rename, extract, consolidate.
