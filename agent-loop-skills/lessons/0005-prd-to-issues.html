Lesson 0005
PRD to Issues: Turn a Destination into a Journey

Break a PRD down into vertical slices so your agents can work independently, in parallel, and without getting stuck on unknown unknowns

Prerequisite

This lesson builds on Lesson 0004 (Write a PRD). You should have a PRD — ideally a real one from your own project — to work through the breakdown process.

The Problem with Horizontal Slices

When developers break a PRD into tasks, the naive approach is horizontal slices:

  • "Create the database schema"
  • "Build the API endpoints"
  • "Write the frontend components"
  • "Add tests"

This looks organized. It's not. Each slice touches every layer, which means:

  • You can't test anything until all slices are done
  • If layer B is blocked, layer A sits idle
  • You don't discover integration problems until the very end

Vertical slices cut through all layers. Each issue does one thing end-to-end: database → API → frontend → test. This gives you working, testable code after every single issue.

The Tracer Bullet Principle

The goal of prd-to-issues is to identify the unknowns as early as possible. The fastest path to learning what you don't know:

  1. Pick the task that touches the most uncertain part of your system
  2. Build it as a thin, end-to-end vertical slice
  3. If it works, you have a proven path. If it fails, you know immediately and can pivot

Known as the tracer bullet approach — you fire through the whole system at once and see where the smoke comes out.

The Skill

prd-to-issues — takes a PRD and turns it into a canon board of independent issues.

  1. Locate the PRD. Fetch it if not in the current context.
  2. Draft vertical slices. Each slice should cut through all integration layers — not be a single-layer task.
  3. Prioritize unknown unknowns first. If you're integrating something new, do that as early as possible.
  4. Establish blocking relationships. Which issues can run in parallel? Which must complete before others start?
  5. Create GitHub issues referencing the parent PRD, with user stories embedded in each issue.

What a Good Breakdown Looks Like

Here's a real breakdown from a split-pane document editor PRD. Four slices, each a full end-to-end working feature:

Slice 1: Document Editing Engine

Core engine with 28 tests. Pure function. No UI. Establishes whether the approach is feasible before any UI work begins.

⚠️ Do this first — it answers "can we?"

Slice 2: UI Split Pane Layout

Basic layout — chat left, document panel right. No AI integration yet. Establishes the visual shell.

→ Blocked by: nothing (parallel to 1)

Slice 3: AI → Engine Integration

Wire the AI output into the editing engine. Once this works, the core loop is live.

→ Blocked by: Slice 1 (needs working engine)

Slice 4: Monaco Editor Toggle

Add Monaco editor mode for power users. Depends on the split pane existing.

→ Blocked by: Slice 2 (needs split pane)

Notice: slices 1 and 2 can run in parallel. Slices 3 and 4 each have exactly one blocker. This means two agents can work simultaneously on the first two slices, then converge once their dependencies are resolved.

Blocking Relationships Are a Feature

Blocking relationships make the work visible to autonomous agents. When a ralph-loop picks up an issue, it:

  • Fetches the parent PRD to understand context
  • Comments on the issue when it starts
  • Closes the issue when done — which unblocks the next one
  • Moves to the next unblocked issue automatically

You get a self-propelling development machine where the board state itself drives what gets done next.

QA issues are often forgotten. When you finish all the feature slices, remember to add a QA or testing issue at the end — and block it on all the feature slices. QA often falls through the cracks because it only appears after everything else is done.

How This Fits the Workflow

grill-me
surface the full scope → write-a-prd
lock the destination → prd-to-issues
plan the vertical slices → tdd
implement each slice

The chain from grill-me to tdd is a complete development methodology: surface → capture → slice → build. Each skill hands off to the next.

Practice Exercise

  1. Find or create a PRD Use Lesson 0004 to create a PRD for something you want to build. A small feature is fine.
  2. Draft 3–4 vertical slices For each slice, ask: does this cut through all layers? Does it have a single blocker? Could an agent pick this up independently?
  3. Name the blockers Which slices can run in parallel? Which must wait? Write it down explicitly.
  4. Optional: create real issues If you have a real project, create GitHub issues using gh issue create with the PRD in the body and blocking relationships set via --blocking.
