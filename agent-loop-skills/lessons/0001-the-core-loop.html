Lesson 0001
The Core Loop: Why Iteration Beats Cleverness

Master the 5 core components and the "propose → score → keep" mindset behind every loop

Core Insight

This is the first thing to understand, before looking at any code:

Agents aren't powerful because of a clever prompt. Agents are powerful because they iterate — propose a change, run it, measure the result, keep it if it's better, loop back.

Karpathy ran ~700 experiments in 2 days with a simple markdown prompt. Not because the prompt was brilliant — but because it looped. Every iteration, real-world feedback fed into the next step.

Agent Loop Skills packages that mindset into a reusable unit: the loop.

The 5 Core Components

Every loop has exactly 5 components. Missing one means the loop is missing information.

1. Program

Orchestration guide. In Agent Loop Skills: the SKILL.md file.

2. Artifact Slot

What gets improved. Not hardcoded — passed in at invocation time.

3. Feedback Signal

Signal that decides the next step. Prioritize objective + locally runnable.

4. Run Ledger

Append-only log: which change, what result, what decision. Never overwrites.

5. Termination

When to stop: threshold reached, plateau (no improvement for K rounds), out of budget, or loop-until-dry.

The Basic Flow

Here's the flowchart for every loop — regardless of which loop in the repo:

Propose change → Run it → Score it
better ↙             ↘ worse
✓ Keep + log ↔ ✗ Revert
stop?
→ Done

Key discipline: signal decides. Not intuition, not "looks fine". A specific number comparing against the baseline. Keep if better, revert if not.

Minimal Example

Say you want to optimize a SQL query. The loop goes through these steps:

# Each iteration:
1. Propose: "Try JOIN instead of subquery"
2. Run: execute both queries, measure latency
3. Score: compare new latency vs baseline
4. Keep if <baseline; revert if ≥baseline
5. Log: ledger records [iteration, change, latency_old, latency_new, decision]
6. Stop: when latency drops < 5% for 3 consecutive rounds

This is exactly optimize-loop in the repo. You bind the SQL query, specify how to measure latency, and the loop runs itself. No need to rewrite propose/run/score logic every time.

Bindings: How a Loop Receives a Task

A loop doesn't hardcode the artifact. Instead, when invoking the loop, you bind specific values:

BindingMeaning
artifactYour specific file/task
signalHow to measure "better"
run_cmdCommand to run the artifact
budgetMaximum number of iterations

Bindings are resolved at invocation time — no code changes needed. Bind correctly → loop runs the right task.

Loops vs Regular Skills

Regular skills (e.g. create-blog-post) do one thing and stop. Loops iterate with feedback. Use when:

Use a loop when: you know what "better" looks like and want the machine to find it across multiple attempts.

Use a regular skill when: one run is enough, no measurement needed between attempts.
