The Core Loop: Why Iteration Beats Cleverness

Nắm được 5 thành phần cốt lõi và tư duy "propose → score → keep" đằng sau mọi loop

Insight nền tảng

Đây là điều đầu tiên cần nắm, trước khi nhìn bất kỳ code nào:

Agents không mạnh vì một prompt khéo léo. Agents mạnh vì lặp lại — propose một thay đổi, chạy nó, đo kết quả, giữ lại nếu tốt hơn, quay lại.

Karpathy chạy ~700 experiments trong 2 ngày với một markdown prompt đơn giản. Không phải vì prompt đó quá thông minh — mà vì nó lặp. Mỗi lần lặp, kết quả thực tế feedback vào bước tiếp theo.

Agent Loop Skills là framework đóng gói tư duy đó thành một đơn vị có thể reuse: loop.

5 thành phần cốt lõi

Mọi loop đều có đúng 5 thành phần. Thiếu một là loop thiếu thông tin.

1. Program
Hướng dẫn điều phối. Trong Agent Loop Skills: file SKILL.md
SKILL.md
2. Artifact slot
Thứ được cải thiện. Không hardcode — được truyền vào lúc gọi loop
<artifact>
3. Feedback signal
Tín hiệu quyết định bước tiếp theo. Ưu tiên objective + locally runnable
<signal>
4. Run ledger
Log append-only: thay đổi nào, kết quả ra sao, quyết định gì. Không ghi đè.
ledger.tsv
5. Termination
Khi nào dừng: threshold đạt, plateau (không cải thiện K vòng), hết budget, hoặc loop-until-dry
<budget> | threshold | plateau | loop-until-dry

Key discipline: signal decides. Không phải intuition, không phải "trông ổn". Một con số cụ thể so sánh với baseline. Giữ lại nếu tốt hơn, revert nếu không.

Ví dụ tối giản

Giả sử bạn muốn tối ưu một câu SQL query. Loop đi qua các bước:

# Mỗi vòng lặp:
1. Propose: "Thử JOIN thay vì subquery"
2. Run: chạy cả hai query, đo latency
3. Score: so sánh latency mới vs baseline
4. Keep nếu

Đây chính là optimize-loop trong repo. Bạn bind câu SQL vào, chỉ định cách đo latency, và loop tự chạy. Không cần viết lại logic propose/run/score mỗi lần.

Bindings: cách loop nhận task

Loop không hardcode artifact. Thay vào đó, khi gọi loop, bạn bind các giá trị cụ thể:

📦
artifact
File/task cụ thể của bạn
⚡
signal
Cách đo "tốt hơn"
🔧
run_cmd
Lệnh chạy artifact
⏱️
budget
Số vòng tối đa

Bindings được resolve khi gọi loop — không cần sửa code. Bind đúng → loop chạy đúng task.

Loops vs Skills thường

Skills thường (ví dụ: create-blog-post) làm một việc và dừng. Loops lặp đi lặp lại với feedback. Dùng khi:

Dùng loop khi: bạn biết "tốt hơn" trông như thế nào và muốn machine tìm nó qua nhiều attempts.

Dùng skill thường khi: một lần chạy là đủ, không cần đo lường giữa các attempts.

Bài tập thực hành

Trước bài tiếp theo, thử suy nghĩ:

🎯 Task của bạn: Bạn có task gì trong công việc hoặc dự án cá nhân mà bạn lặp đi lặp lại thủ công? (VD: tối ưu prompt, clean data, viết draft, debug)

Chọn 1-2 task đó. Chúng ta sẽ map chúng vào loop phù hợp hoặc viết loop mới ở bài tiếp.
