Lesson 0007
Improve Codebase Architecture: Build the Foundation Agents Need

The skill that makes every other skill — including TDD — work better. A well-structured codebase is the single best investment you can make in agent output quality.

Prerequisite

This lesson builds on Lesson 0006 (TDD). Understanding why TDD is hard in badly structured codebases sets up why this skill exists.

The Feedback Loop Nobody Talks About

Here's a pattern that plays out constantly:

  1. You hire a new AI agent on a messy codebase
  2. It produces messy code — inconsistent, duplicative, poorly tested
  3. You blame the AI

But step back. A senior human engineer handed a messy codebase will also produce messy code — or spend weeks just understanding the structure before writing anything useful.

Garbage codebase → garbage agent output. This is not a criticism of AI. It's the same rule that applies to humans. If you want better outputs, invest in the structure the AI works within.

The Core Insight: Shallow vs. Deep Modules

John Ousterhout's A Philosophy of Software Design introduces a concept that transforms how you think about architecture: shallow modules vs. deep modules.

Shallow Module

A small file with many tiny exported functions. The interface is wide — lots of things to learn. But each piece does almost nothing alone. Tests are scattered and hard to write.

Deep Module

A larger module with a thin, focused interface. Rich behavior lives behind it. You learn a few functions and get a lot of capability. Easy to test at the boundary.

When a codebase has many shallow modules, the AI sees a fog of undifferentiated files. It doesn't know where to start, what depends on what, or where bugs are likely to hide.

When a codebase has deep modules, the AI sees clear boundaries. It knows exactly what to test and where to look for bugs.

The Skill

improve-codebase-architecture — surfaces opportunities to deepen shallow modules.

  1. Explore the codebase as an agent would. Look for confusions — places where understanding one concept requires bouncing around many small files.
  2. Present candidates: numbered list of deepening opportunities. Each is a candidate for a refactor.
  3. User picks a candidate. Then: spawn three sub-agents in parallel, each producing a radically different interface design for that module.
  4. Compare the three designs. Give your recommendation — which is strongest and why. If elements combine well, propose a hybrid.
  5. Create a GitHub issue: an RFC describing the proposed refactor as a destination, then use prd-to-issues to break it into a journey.

Finding Candidates: The Confusion Checklist

When exploring, the skill asks questions a senior engineer would ask about any codebase:

  • Where does understanding one concept require bouncing around many small files? — signals a shallow module that should be deepened
  • Where have pure functions been extracted just for testability, but the real bugs hide in how they're called? — signals misaligned test boundaries
  • Where do tightly coupled modules create integration risk in the seams between them? — signals missing abstractions
  • Where does the AI repeatedly ask "which file should I put this in?" — signals structural confusion

The skill surfaces what the AI finds confusing so you can resolve it — together.

Three Designs, One Decision

The most powerful step: spawn three sub-agents in parallel, each producing a radically different interface design for the same module. This forces genuine exploration rather than the first reasonable idea.

You don't need to be an interface design expert to benefit from this. Comparing three concrete options reveals trade-offs you wouldn't see otherwise. Then you apply your judgment:

  • Which design matches how callers actually use the module?
  • Which is easiest to test at its boundary?
  • Which is most resilient to future changes?

The skill asks for a recommendation — but with rationale. It forces you to think, not just accept.

Only One at a Time

Architecture refactors are cognitively expensive. The skill recommends doing one at a time:

  • Pick the highest-confusion candidate
  • Design the new interface
  • Write the GitHub issue
  • Break it into slices with prd-to-issues
  • Implement with tdd
  • Then move to the next candidate

Running this once a week — or whenever a surge of new features creates a new wing of the codebase — keeps the structure clean enough for agents to navigate.

How This Fits the Workflow

grill-me
surface the full scope → write-a-prd
lock the destination → prd-to-issues
plan the vertical slices → tdd
implement each slice

But wait — this skill runs outside the main workflow. It runs in the background, continuously improving the codebase so that when you invoke the main workflow, the AI is working in a cleaner environment. Think of it as maintenance, not feature development.

Invest early. The best time to deepen modules is before they become critical paths in your workflow. Once a shallow module is depended on everywhere, refactoring it becomes risky. Catch it early.

The Full Skill Chain

Here's the complete picture of the Claude Code for Real Engineers workflow:

Planning Phase

grill-me → write-a-prd → prd-to-issues

Surface, capture, and slice the work before writing any code.

Implementation Phase

tdd with ralph-loop

Implement each slice with red green refactor, one issue at a time.

Maintenance Phase

improve-codebase-architecture

Continuously deepen shallow modules so future work happens in a cleaner environment.

Practice Exercise

  1. Explore your own codebase Pick one module you work with often. Can you describe its interface in one sentence? Or do you have to navigate many files to understand it?
  2. Identify one confusion point Where does understanding one concept require bouncing between 3+ files? That's a deepening candidate.
  3. Sketch two alternative interfaces Without spawning sub-agents yet, think: how could this module's surface look differently? Which is cleaner?
