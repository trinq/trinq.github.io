Verification Loops

Reference 0004 · Print-friendly cheat sheet · From Lesson 0011

The 2×2 matrix

Computational (deterministic)Inferential (AI-driven)
Guides (before action) Linter rules, file templates, schema validation System prompt, coding conventions, architectural constraints
Sensors (after action) Unit tests, type checkers, CI pipelines, build checks LLM-as-judge, review agent, screenshot comparison

The three verification types

TypeSpeedCostCatchesMisses
Rules-based ~seconds Free Syntax, type, logic (if tested), style Semantic correctness, design quality
Visual ~seconds Low Layout, missing elements, style regressions Logic, data correctness
LLM-as-judge ~10s Medium Spec compliance, pattern consistency, semantic drift Subtle logic bugs, performance

Layering order

Agent makes changes ↓ Layer 1: Linter → Type checker → Tests → Build (fast, free) ↓ all pass Layer 2: Screenshot → Compare to spec (if UI task) ↓ visual match Layer 3: LLM-as-judge with rubric (slow, semantic) ↓ pass DONE → commit

LLM-as-judge best practices

PracticeWhy
Use different model familyAvoids self-preference bias
Force chain-of-thoughtImproves reliability, makes failures debuggable
Give a rubric, not "is this good?"Specific criteria → consistent evaluations
Judge cannot edit codePrevents judge from introducing its own bugs
Fresh context windowNo context rot from working session
Cap retries at 2Prevents unbounded token burn

Anti-patterns

Anti-patternFix
Soft verification ("try to fix, proceed if you can't")Hard gate: agent cannot proceed until checks pass
Vague judge prompt ("review this code")Structured rubric with specific criteria from PRD
Unbounded retriesCap at 2 retries, then escalate to human
Only guides, no sensorsAdd at least one computational sensor (test/lint gate)
