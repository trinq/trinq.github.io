Agentic Engineering Vocabulary

Reference 0001 · The 10 words Matt uses that you should too

Reference Source: Matt Pocock's Agentic Engineering Workflow (18 Jun 2026)

This is the spine of the course. Every lesson links back to one of these. Print it. Tape it to your monitor. Use these words in standups, PR descriptions, and CLAUDE.md comments — they are the shared vocabulary that makes the rest of the work legible.

Core distinction

Harness/ˈhɑː.nɪs/

Everything outside the model that shapes its behaviour: prompts, skills, the codebase, the tooling, the rules of engagement. The opposite of the model is the harness, and Matt's whole claim is that the harness is 50% of the work, fully under your control, and most people ignore it.

Everyone's obsessed with the model and I think they should be more interested in the harness, what you can do to get the most out of the harness, giving it the right prompts, giving it the right skills to work with and improving the environment in which the model runs.

Model

The thing you can't easily change this month — Opus 4.8, Fable, GPT-6, whatever. Useful but not the lever. "The engine of the Formula 1 car."

The two kinds of work

Tactical programming

Day-to-day coding. Syntax, debugging, commits, scaffolding a feature. This is what AI has eaten. Don't compete on this; delegate it.

AI has basically eaten tactical programming. It's gone, right? It's all gone.

Strategic programming

Winning the war, not the battle. How does the codebase need to look? What slows us down? What module boundaries make the agent's job easier? This is your job now.

You need to be great at strategic programming in order to get the most out of this infinite fleet of tactical programmers that you now have access to.

Vibe coder

Someone who ships with AI but has weak software fundamentals. They can read code, run a terminal, and prompt. They cannot design a module boundary, write a test that survives a refactor, or scope a task. The whole "build a scheduling app" demo in the talk is Matt pretending to be one to show the teach skill filling in the gap.

The two kinds of skills

Procedure skill

A skill you invoke by typing /name. The model only does the work when you trigger it. Matt strongly prefers this. The model's context window does not get polluted with the skill's description unless you call it.

I prefer my skills as procedures. I like to be the one in control.

Ability skill

A skill the model can invoke itself when it thinks the situation matches. The skill's description is always loaded into the model's context window so it can decide. Useful for "React coding standards" that the agent should silently follow. Expensive at scale — 100 abilities = 100 leaked descriptions.

Every single ability… you're going to be leaking 100 descriptions into the context window.

disable-model-invocation

The YAML frontmatter flag that turns an ability into a procedure. disable-model-invocation: true in a SKILL.md means only the human can run it. Use this for anything that decides how you work (grill-me, to-prd, triage) — not for "the agent should remember our test style."

How you work with the agent

Human-in-the-loop (HITL)

You are sitting with the agent. You answer its questions, grant permissions, steer it. Good for planning, unscoped work, and "I don't know what I want yet."

AFK (away from keyboard)

You ping an agent with a task and walk away. The agent runs in a sandbox (Docker, Vercel, GitHub Actions). You come back to a pull request. The moment Matt discovered this, his output multiplied.

The moment that I discovered AFK was the moment I really got into AI coding and the moment I was really able to massively increase my output because then instead of me having to sit in the loop, handle all the permissions requests… I've paralyzed myself. Suddenly there are two of me, you know, or three of me, four of me, five of me.

Sandbox

An isolated environment the agent runs in. Without one, the agent can wipe your home directory or exfiltrate your env vars. Matt's tool is called Sandcastle — it lets you spin up Docker/Podman/Vercel sandboxes for Claude Code. A GitHub Actions workflow is the lightweight version of the same idea.

How you model the work

Queue

A backlog of well-scoped tasks. Items come in (from bug reports, user requests, your own ideas), get triaged, get picked up by an agent or a human, and come off when their PR merges. This is how Matt actually thinks about development. Not an infinite loop — a queue.

That's all development is really. You just have a queue of tasks that you need to get done. Project managers add more stuff to the queue. You complete the tasks in the queue.

Loop

The viral "Ralph loop" pattern: a while that re-invokes the agent with the same prompt until done. Useful for the narrow case of "do this one thing, then check, then repeat." Matt's claim: most of what people call a loop is really just an AFK agent on a single task. The loop is an implementation detail, not a paradigm.

This idea that loops are the only way to do it is crazy, you know, like we're essentially talking about the history of this goes back to Jeffrey Huntley… the Ralph loop. And what I realized is I don't really need to run this as a loop, right. The only thing I need out of this is the AFK agent to take on a specific task and do that task.

Tracer bullet

A thin vertical slice through the whole stack — UI, API, DB, tests — that proves the path works end-to-end before you build out the breadth. Each TDD iteration should be a tracer bullet, not a horizontal layer.

How you design the codebase

DX (developer experience)

How easy the codebase is for a human to work in. Clear module boundaries, good tests, deep modules with thin interfaces, intentional naming. The traditional software-design goal.

AX (agent experience)

How easy the codebase is for the agent to work in. Massive overlap with DX — good module boundaries help both. The differences: agents need clearer seams (to know where to put a change), better docs at the top of files (so they don't have to re-derive intent), and tighter feedback loops (so they can self-correct).

I think of there as being a difference between DX developer experience and AX, right? Agent experience. And so agent experience is the experience that the agent has working in the codebase.

Deep module

A module with a small public interface and a lot of complex behaviour behind it. The opposite of a "shallow" module where every internal concept is exposed. John Ousterhout's term, and the thing /improve-codebase-architecture is hunting for.

Failure mode to avoid

The bitter lesson trap

Sutton's bitter lesson says compute always wins — so don't bother optimising. Matt's warning: don't fall for it on the harness side. Yes, wait for the next model. Also build the harness so the next model is 10x more effective when it arrives.

AI slop

Code that compiles, passes tests, and ships — but adds nothing, ignores the project's conventions, and slowly corrupts the codebase. The opposite of "codebases agents love." Matt's /improve-codebase-architecture and /tdd are the antidotes.

Context bloat

Loading your CLAUDE.md, your skill descriptions, your MCP servers, and your custom instructions into every conversation. The thing Matt says is the #1 mistake. The fix: delete everything, observe the agent, add only what you personally decided.
