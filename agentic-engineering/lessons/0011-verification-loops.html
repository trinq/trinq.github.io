Verification Loops: The 2–3× Quality Lever

Lesson 0011 · ~16 min · Give the agent a way to check its own work

Deepening Source: Boris Cherny (Anthropic), Martin Fowler & Birgitta Böckeler (Thoughtworks), LLM-as-judge research, Akshay Pachaar synthesis

Lesson 0009 named verification loops as component 10 of the harness. Lesson 0010 showed how context management prevents the agent from getting dumb. This lesson is about the opposite problem: the agent does something confidently wrong, and nothing catches it.

"Giving the model a way to verify its work improves quality by 2–3×." — Boris Cherny, Head of Claude Code, Anthropic

That's the single highest-leverage claim in the entire harness literature. Not 10%. Not 20%. Two to three times. The reason: without verification, the agent produces output and moves on. With verification, the agent produces output, checks it, and either confirms or self-corrects. The check doesn't need to be expensive. It just needs to exist.

1. The Fowler framework: guides vs. sensors

Martin Fowler and Birgitta Böckeler at Thoughtworks formalized the cleanest framing for verification in early 2026. They split all harness controls into two types:

Guides (feedforward)Sensors (feedback)
When Before the agent acts After the agent acts
Purpose Steer toward correct behavior Detect incorrect behavior
Examples System prompts, coding conventions, file templates, linter rules, CLAUDE.md constraints Tests, type checkers, CI pipelines, review agents, screenshots
Failure mode Agent ignores the guide Sensor doesn't catch the error

Both guides and sensors can be computational (deterministic, fast, CPU-based: linters, tests, type checkers) or inferential (AI-driven, semantic: LLM-as-judge, design evaluation).

COMPUTATIONAL INFERENTIAL (deterministic) (AI-driven) GUIDES Linter rules System prompt (feedforward) File templates Coding conventions Schema validation Architectural constraints SENSORS Unit tests LLM-as-judge (feedback) Type checkers Review agent CI pipelines Screenshot comparison

This 2×2 matrix is the mental model. Every verification mechanism in your harness fits into one of these four quadrants. Most developers over-invest in guides (writing detailed prompts) and under-invest in sensors (letting the agent check its own output). The 2–3× quality lift comes from adding sensors.

2. Three types of verification loops

From Anthropic's documentation and the practitioner community, there are three distinct types of verification. They're ordered by cost and coverage:

Type 1: Rules-based (computational sensors)

Deterministic checks that give ground truth. The agent runs tests, linters, or type checkers after making changes, and the harness feeds the results back into the loop.

Agent edits code ↓ Harness runs: npm test ↓ Tests fail (3 of 12) ↓ Error output fed back to agent as a ToolMessage ↓ Agent reads failure, adjusts code ↓ Harness runs: npm test ↓ Tests pass (12 of 12) ↓ Agent proceeds to next sub-task

Why it works: Tests don't hallucinate. A failing test is an unambiguous signal. The agent doesn't need to interpret — it needs to fix. This is the fastest feedback loop: run, fail, fix, run again.

What it catches: Syntax errors, type errors, logic errors covered by tests, style violations, import errors, build failures.

What it misses: Semantic correctness ("the code compiles but does the wrong thing"), design quality, adherence to non-goals, UI fidelity.

The linter-as-hard-gate pattern. In Boris Cherny's loop engineering, if the linter log is not empty, the loop does not complete. The error is fed back into the agent for automated correction. The agent cannot "finish" until all deterministic checks pass. This is a computational sensor used as a gate, not just a signal.

Type 2: Visual (computational + inferential sensors)

For UI tasks, the agent takes screenshots of its work using tools like Playwright, then evaluates whether the result matches the spec. This bridges the gap between "code compiles" and "UI looks right."

Agent edits React component ↓ Harness runs: playwright screenshot → captures PNG ↓ Agent sees the screenshot in its context ↓ Agent compares against the spec: "The button should be centered with 16px padding" ↓ Agent notices: button is left-aligned ↓ Agent adjusts CSS, retakes screenshot ↓ Visual check passes → proceed

Why it works: Seeing is believing. The model is surprisingly good at evaluating visual output when given a screenshot and a clear spec. This catches layout issues, missing elements, and style inconsistencies that no test suite would cover.

When to use it: Any task that produces visible output — UI components, reports, charts, generated documents. If a human would check it by looking, the agent can too.

Type 3: LLM-as-judge (inferential sensors)

A separate agent evaluates the output of the working agent. The judge doesn't have the working agent's biases, assumptions, or accumulated context rot. It gets a fresh window and a focused evaluation prompt.

Working agent produces: auth middleware (src/middleware/auth.ts) ↓ Harness spawns Judge agent with fresh context: - The PRD (what was requested) - The diff (what changed) - The test results (what passed) - Evaluation rubric: "Does this diff satisfy the PRD? Does it violate any non-goals? Is it consistent with the existing codebase patterns?" ↓ Judge returns: "PASS — satisfies requirements. NOTE: Error messages are hardcoded in English. PRD says nothing about i18n, but existing codebase uses i18n keys in all other middlewares." ↓ Working agent receives judge's note, adds i18n keys ↓ Proceed

Why it works: The judge operates in a fresh context window (no context rot from the working session), evaluates against the spec (not the agent's assumptions), and catches semantic issues that tests miss. The judge in the example above caught an inconsistency with codebase patterns that no test would flag.

The fresh-context advantage. The working agent has been deep in the code for 20+ turns. Its context is full of implementation details. The judge starts clean, reads only the spec and the diff, and asks: "Does this actually do what was requested?" This is the same advantage a code reviewer has over the code author — distance from the implementation.

Production best practices for LLM-as-judge:

  • Use a different model family if possible. Same-model judges exhibit self-preference bias.
  • Force chain-of-thought. Make the judge explain its reasoning before giving a verdict. This improves reliability and makes failures debuggable.
  • Give the judge a rubric, not a vague "is this good?" Specific criteria produce consistent evaluations. Vague prompts produce inconsistent ones.
  • The judge cannot edit code. It can only evaluate and comment. This prevents the judge from "fixing" things and introducing its own bugs.

3. Layering: the production pattern

Production harnesses don't pick one type. They layer all three in a specific order:

Agent makes changes │ ▼ Layer 1: COMPUTATIONAL SENSORS (fast, free, deterministic) ├── Linter (eslint, ruff, etc.) ├── Type checker (tsc, mypy, etc.) ├── Unit tests └── Build check │ │ If any fail → feed errors back, agent self-corrects, re-run Layer 1 │ If all pass ↓ ▼ Layer 2: VISUAL SENSORS (if applicable) ├── Playwright screenshot └── Compare against spec │ │ If visual mismatch → agent adjusts, re-run from Layer 1 │ If pass ↓ ▼ Layer 3: INFERENTIAL SENSORS (slow, expensive, semantic) ├── LLM-as-judge (fresh context) └── Evaluates: spec compliance, codebase consistency, non-goal violations │ │ If judge flags issues → agent addresses them, re-run from Layer 1 │ If pass ↓ ▼ DONE — commit, push, open PR

The order matters: cheap before expensive. Every layer that catches an error saves the cost of the layers below it. A linter catching a syntax error in 100ms is infinitely cheaper than an LLM judge catching it in 10 seconds.

This is Gather-Act-Verify. Anthropic describes Claude Code's cycle as: gather context (search files, read code), take action (edit files, run commands), verify results (run tests, check output), repeat. The verification step is not optional — it's a first-class phase of the cycle.

5. The Gather-Act-Verify cycle in practice

Here's what a single iteration of a well-verified AFK agent looks like, combining lessons 0008, 0010, and this lesson:

One sub-task, fully verified

  1. Gather: Agent reads the current sub-task from the issue plan. Uses JIT retrieval (lesson 0010) to load only the relevant files. Reads the PRD's done-means for this specific sub-task.
  2. Act: Agent edits the code. Makes the change.
  3. Verify (Layer 1): Agent runs the linter. If errors, fixes them. Runs the test suite. If failures, reads the error output, adjusts the code, re-runs. Loops until all computational checks pass.
  4. Verify (Layer 2, if UI): Agent takes a screenshot. Compares against the spec. If misaligned, adjusts and re-screenshots.
  5. Verify (Layer 3, on commit): Agent commits. The review agent in CI evaluates the diff against the PRD rubric. If flagged, the working agent receives the comments and addresses them.
  6. Proceed: Move to the next sub-task. The context management strategies from lesson 0010 keep the window clean for the next iteration.

Each verification layer is a filter. Layer 1 catches 80% of errors (the ones tests and linters were written for). Layer 2 catches visual regressions. Layer 3 catches semantic drift — the code does something subtly different from what was specified. Without all three, errors leak through to the PR.

6. The anti-patterns

Three common mistakes that undermine verification loops:

Anti-pattern 1: No hard gates

Soft verification

"Run the tests. If some fail, try to fix them, but proceed if you can't." The agent learns it can skip failures. Quality degrades to whatever the agent can get away with.

Hard gate

"Run the tests. Do not proceed until all tests pass. If you cannot fix a failure after 2 retries, stop and report the error." The agent cannot advance past a failing check.

Anti-pattern 2: Vague judge prompts

Vague

"Review this code and check if it looks good." The judge has no criteria. Its evaluation will be inconsistent and biased toward approval.

Rubric-based

"Evaluate this diff against the following criteria: (1) Does it satisfy the done-means from the PRD? (2) Does it violate any non-goals? (3) Does it follow existing codebase patterns for error handling and i18n? Explain your reasoning, then give PASS or FAIL."

Anti-pattern 3: Unbounded retries

Cap your retries. A verification loop without a retry limit will burn tokens endlessly on a problem the agent can't solve. Stripe's production harness caps at 2 retries. After that, the error is escalated to a human. Remember lesson 0009: 99% per-step success gives only 90.4% over 10 steps. Unlimited retries make the math worse, not better.

7. The meta-principle

The 2–3× quality claim from Boris Cherny isn't about any single verification technique. It's about the existence of verification as a first-class phase. Without it, the agent's output is a first draft — plausible but unchecked. With it, the agent's output is a reviewed draft — the agent has caught its own mistakes before you see them.

This maps directly to Matt Pocock's lesson 0001 axiom. The verification loop is harness infrastructure. It uses the same model, the same weights. The quality lift comes entirely from the harness giving the model a way to check itself. You're not making the model smarter. You're making the system around it more rigorous.

What this lesson taught

  • Guides vs. sensors: Guides steer before action (prompts, templates). Sensors check after action (tests, judges). The 2–3× lift comes from sensors.
  • Three verification types: Rules-based (tests, linters — fast, deterministic), visual (screenshots — for UI), LLM-as-judge (fresh context, semantic evaluation).
  • Layer cheap before expensive: Linter → tests → screenshots → LLM judge. Each layer filters errors before hitting the next.
  • Hard gates, not soft suggestions: The agent cannot proceed until verification passes. Soft verification degrades to no verification.
  • Cap retries at 2. Unbounded retries burn tokens without progress. Escalate to human after 2 failures.
  • Give judges rubrics, not vibes. Specific criteria produce consistent evaluations.

Quiz

  1. What's the difference between a guide and a sensor? Give one example of each from your Claude Code setup.
  2. Why should the LLM-as-judge operate in a fresh context window rather than the working agent's context?
  3. Your AFK agent produces code that passes all tests but subtly violates the PRD's non-goals. Which verification layer failed, and how would you fix it?
Answers (try first)
  1. A guide steers the agent before it acts (feedforward). Example: your CLAUDE.md rules, or a PRD's non-goals section. A sensor checks after the agent acts (feedback). Example: running npm test after editing code. Guides prevent errors; sensors detect them. Both are necessary — guides alone can be ignored, sensors alone catch errors too late.
  2. The working agent's context is polluted by 20+ turns of implementation detail and its own assumptions. It's likely experiencing context rot (lesson 0010). The judge starts with a fresh window: it reads only the spec, the diff, and the test results. This gives it the same advantage a code reviewer has over the code author — distance from the implementation, no inherited biases, no mid-window attention degradation.
  3. Layer 3 (inferential/LLM-as-judge) failed. Tests (Layer 1) passed because the code is functionally correct. But the semantic violation — touching files the non-goals forbid, or implementing behavior outside scope — is something only a judge evaluating against the PRD can catch. Fix: give the review agent a structured rubric that explicitly includes: "Does the diff modify any files or behaviors listed in the PRD's non-goals section? List them." Or make non-goals machine-checkable (a file-touch allow-list) to catch them in Layer 1.
