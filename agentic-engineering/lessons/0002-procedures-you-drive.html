Procedures You Drive, Abilities You Lend

Lesson 0002 · ~12 min · The two kinds of skills, and how to keep yourself in the loop

Procedural Source: Matt Pocock's Agentic Engineering Workflow at 11:40, 12:25, 42:50

Every skill you install is a small choice about who is in control. A procedure is one you trigger by typing /name — the agent does the work because you asked, and only because you asked. An ability is one the agent can trigger on its own — when it thinks the situation matches, it pulls the skill into the work without you.

The distinction sounds academic. It isn't. Every ability you install leaks its description into the model's context window on every turn. Matt's standing rule: procedures you drive, abilities you lend. This lesson teaches you how to tell them apart, when to convert one to the other, and how to clean up your list from lesson 0001 using that lens.

1. The leak you don't see

Picture your ~/.claude/skills/ folder. Whatever's in there has a SKILL.md with a description at the top — usually one or two sentences telling the model what the skill is for. If the skill is an ability, that description is loaded into every conversation, every turn, so the model can decide when to use the skill.

Every single ability — let's say we have a list. Let's say we have a 100 different skills. You're going to be leaking 100 descriptions into the context window. — Matt Pocock, Agentic Engineering Workflow, 12:25

100 skills × ~50 words per description = ~5,000 tokens of standing noise. That doesn't sound like much until you remember it's paid for on every turn and it sits there competing for attention with your actual repo, your actual task, and your actual CLAUDE.md. The signal drowns.

Procedures have the same descriptions, but they're hidden behind a flag: disable-model-invocation: true in the YAML frontmatter. That flag tells the model "don't try to call this — only the human can." The description is invisible until you type /name, at which point the skill loads into the context for that one turn only.

2. The taxonomy, in one table

Dimension Procedure Ability
Who invokes You (the human) The model, when it decides
Frontmatter flag disable-model-invocation: true (flag absent or false)
Description in context Hidden until /name Always loaded
Cost at scale Zero until used Linear in # of skills
You control the flow Yes — you're driving No — model decides when
Best for Decision-shaped work: interview me, write the PRD, scope the issue Convention-shaped work: "our tests look like X", "our naming is Y"

Matt's preference, stated directly in the talk: "I tend to prefer my skills as procedures. I like to be the one in control." The reason is not dogmatic — it's the cost. If a skill is decision-shaped (it changes how the agent thinks about a problem, asks you questions, writes a doc), you want to invoke it deliberately. If it's convention-shaped (the agent should just always do X), an ability is fine.

3. The decision rule

When you look at a skill on your list, ask two questions:

  1. Does the skill make a decision, or does it encode a convention?
  2. Would you notice if the agent invoked it without you?

If the skill is making a decision — "interview me about this feature," "scope this into issues," "triage this bug report" — you almost always want it to be a procedure. The whole point is that you chose to run it. If the agent decided on its own, the work would be different.

If the skill encodes a convention — "our React components use functional syntax with hooks, never classes" — the agent should just do that without asking. An ability is fine. The model notices the situation, applies the rule, moves on. You don't need to be in the loop on a per-task basis.

The interesting case: skills that look like both. Matt's /grill-me encodes a convention (interview me aggressively) but is decision-shaped (the result changes what gets built). He makes it a procedure so he can choose when to enter that mode.

4. The flag in practice

Open any skill in ~/.claude/skills/<name>/SKILL.md. You'll see YAML frontmatter at the top, before the markdown body:

---
name: grill-me
description: Turn the agent into an adversarial interviewer...
disable-model-invocation: true
---

# Grill Me

...

Three fields matter here:

  • name — what you type to invoke it (/grill-me).
  • description — the bit that leaks into context if it's an ability. Make it short. Make it specific.
  • disable-model-invocation — true means procedure, absent/false means ability.

If you want to convert a skill from ability to procedure, add the flag. One line. No code changes. The skill now loads only when you type its name.

5. The cleanup, applied to your list

You have a list from the lesson 0001 audit. For each skill on it, run this:

Re-audit by kind

  1. Pick the first skill on your audit list. Open its SKILL.md.
  2. Read its description field. Is it a convention ("React coding standards", "use Prettier") or a decision ("interview me", "scope this", "review my PR")?
  3. Check whether it has disable-model-invocation: true. If it's a decision-shaped skill and the flag is missing, add the flag. That's your edit. No other changes.
  4. If the skill is convention-shaped and you use it daily, leave it as an ability. The context cost is the price of consistency.
  5. If the skill is convention-shaped and you've used it once in the last 60 days, delete it. The model will infer the convention from your codebase anyway.
  6. Repeat for the next 5 skills. Stop after 5. That's enough pattern to feel the rule.

The point is not to do a heroic refactor in one sitting. The point is to feel the difference between a skill you want to drive and a skill you want to lend. After 5, you'll know which side of the line each remaining skill sits on. You can come back for the rest later.

6. The description is the cost

When you write or modify a skill, the description field is the most expensive part. Every word in it is a word the model sees on every turn (for abilities) or on every invocation (for procedures, which still pay once when you call them). Matt's rule of thumb, paraphrased from the talk: descriptions should be specific enough that the model can match the situation, short enough that the cost is negligible.

Compare:

# Bad — vague, leaks a lot of context for little signal
description: "Coding best practices"

# Bad — too long, the cost exceeds the value
description: "Apply React best practices including hooks, memo, useCallback, useMemo when needed, avoid stale closures, prefer composition over inheritance, use TypeScript..."

# Good — short, specific, matches one situation
description: "Apply our React component conventions (functional only, hooks, no classes)"

The third one is roughly 12 words. The second is roughly 30. If you have 50 React-component-related skills in your harness, the difference is 18 × 50 = 900 tokens on every turn. That's not nothing.

The compounding insight. Procedures have zero context cost when not in use. Abilities have a flat cost that scales with the number of skills you install. The first time you delete an unused ability is the most expensive context-bloat fix you'll ever do, because every subsequent turn also gets the benefit. Audit one skill per week and your context stays clean without you noticing.

Quiz

Don't over-think these. First instinct.

  1. What frontmatter flag turns an ability into a procedure?
  2. You install a skill called /pr-review that runs a thorough PR review checklist. Should it be a procedure or an ability?
  3. You install a skill called /react-style that reminds the agent to use functional components with hooks. Procedure or ability?
  4. True or false: the bigger your skills folder, the worse your agent performs, regardless of which kind of skills you install.
Answers (try first)
  1. disable-model-invocation: true
  2. Procedure. You want to choose when a full PR review runs, and the review itself produces a decision-shaped artifact. Letting the model invoke it at will would change the workflow.
  3. Ability. It's a convention. The agent should just apply it on every React task without you invoking it. The context cost is worth the consistency.
  4. False. A large folder of procedures has zero context cost until you invoke one. A large folder of abilities has linear cost. The kind matters, not just the count.
