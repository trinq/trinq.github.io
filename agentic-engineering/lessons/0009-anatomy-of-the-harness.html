Anatomy of the Agent Harness

Lesson 0009 · ~18 min · The 12 components inside the box you've been calling "the harness"

Deepening Source: Akshay Pachaar's synthesis thread (Apr 2026) — cross-referencing Anthropic, OpenAI, LangChain, Beren Millidge, Stanford NLP

Lesson 0001 gave you the axiom: the harness is 50% of the work. Lesson 0008 showed you the harness running an AFK agent end-to-end. Both treated the harness as a black box — you knew it mattered, but not what was inside.

This lesson opens the box. We trace the 12 components of a production agent harness, as documented across Anthropic (Claude Code), OpenAI (Agents SDK / Codex), LangChain (LangGraph / Deep Agents), and the practitioner community. By the end, you'll be able to name every part, see how they connect, and diagnose which part is failing when your agent misbehaves.

The metaphor that makes it stick

In 2023, Beren Millidge wrote an essay called "Scaffolded LLMs as Natural Language Computers." His argument: we have reinvented the Von Neumann architecture.

ComputerAgent
CPUThe LLM (processes tokens)
RAMContext window (fast, limited)
DiskExternal databases, files (large, slow)
Device driversTool integrations
Operating systemThe harness

A raw LLM is a CPU with no RAM, no disk, and no I/O. The harness is the operating system that makes it useful. This maps directly to what Matt Pocock said in lesson 0001: the model is the engine, the harness is the chassis. Same idea, different metaphor, different year.

"If you're not the model, you're the harness." — Vivek Trivedy, LangChain

Three levels of engineering

Before the 12 components, the framing. There are three concentric levels of engineering around the model:

  1. Prompt engineering — crafts the instructions the model receives. The innermost ring.
  2. Context engineering — manages what the model sees and when. Wider. Includes retrieval, context window strategy, memory placement.
  3. Harness engineering — encompasses both, plus the entire application infrastructure: tool orchestration, state persistence, error recovery, verification loops, safety enforcement, lifecycle management.

Prompt engineering is a subset of context engineering. Context engineering is a subset of harness engineering. When Matt said "the harness is the chassis," he was talking about level 3. When he said "have a codebase that's easier to make changes in," that's level 2 leaking into level 3. This lesson covers level 3 — the whole system.

The 12 components

Synthesized from Anthropic, OpenAI, LangChain, and the practitioner community. Every production harness has these, whether or not the authors name them explicitly.

1. The Orchestration Loop

The heartbeat. Implements the Thought-Action-Observation (TAO) cycle, also called the ReAct loop:

assemble prompt → call LLM → parse output → execute tool calls → feed results back → repeat until done

Mechanically, it's often just a while loop. Anthropic describes their runtime as a "dumb loop" where all intelligence lives in the model. The harness just manages turns. The complexity lives in everything the loop manages, not the loop itself.

You met this in lesson 0008: the inner loop that wraps well-scoped work (sub-task → code → test → commit). That's this component.

2. Tools

The agent's hands. Tools are defined as schemas — name, description, parameter types — injected into the LLM's context so the model knows what's available. The tool layer handles:

  • Registration and schema validation
  • Argument extraction from model output
  • Sandboxed execution
  • Result capture and formatting back into LLM-readable observations

Claude Code provides tools across six categories: file operations, search, execution, web access, code intelligence, and subagent spawning. OpenAI's Agents SDK supports function tools (@function_tool), hosted tools (WebSearch, CodeInterpreter, FileSearch), and MCP server tools.

The tool scoping trap. More tools often means worse performance. Vercel removed 80% of tools from v0 and got better results. Claude Code achieves 95% context reduction via lazy loading. Principle: expose the minimum tool set needed for the current step.

3. Memory

Memory operates at multiple timescales:

TimescaleMechanismExample
Short-termConversation history within a sessionThe messages array
Long-termPersists across sessionsCLAUDE.md, MEMORY.md, LangGraph JSON Stores, OpenAI Sessions (SQLite/Redis)

Claude Code implements a three-tier hierarchy: a lightweight index (~150 characters per entry, always loaded), detailed topic files pulled in on demand, and raw transcripts accessed via search only.

Critical design principle: the agent treats its own memory as a "hint" and verifies against actual state before acting. This is why lesson 0004's top-of-file context blocks matter — they're the ground truth that memory checks against.

4. Context Management

This is where many agents fail silently. The core problem is context rot: model performance degrades 30%+ when key content falls in mid-window positions.

Stanford's "Lost in the Middle" finding: LLMs perform best when relevant information is at the beginning or end of the input. Middle content is effectively invisible. — Liu et al., "Lost in the Middle: How Language Models Use Long Contexts," TACL 2024

Even million-token windows suffer from instruction-following degradation as context grows. Five production strategies:

  1. Compaction — summarizing conversation history when approaching limits. Claude Code preserves architectural decisions and unresolved bugs while discarding redundant tool outputs.
  2. Observation masking — JetBrains' Junie hides old tool outputs while keeping tool calls visible.
  3. Just-in-time retrieval — maintaining lightweight identifiers and loading data dynamically. Claude Code uses grep, glob, head, tail rather than loading full files.
  4. Sub-agent delegation — each subagent explores extensively but returns only 1,000–2,000 token condensed summaries.
  5. Structured note-taking — the agent writes structured notes to files (progress files, scratchpads), then references them instead of re-reading conversation history.

Anthropic's context engineering guide states the goal: find the smallest possible set of high-signal tokens that maximize likelihood of the desired outcome.

5. Prompt Construction

Assembles what the model actually sees at each step. It's hierarchical:

System prompt (highest priority) ↓ Tool definitions ↓ Developer instructions (AGENTS.md, 32 KiB limit) ↓ Memory files ↓ Conversation history ↓ Current user message

Important context is positioned at the beginning and end of the prompt (the "Lost in the Middle" finding). This is the internal assembly logic — you don't write it, but you need to know it exists because it explains why the order of your CLAUDE.md sections matters.

6. Output Parsing

Modern harnesses rely on native tool calling — the model returns structured tool_calls objects rather than free-text that must be parsed. The harness checks: are there tool calls? Execute them and loop. No tool calls? That's the final answer.

For structured outputs, both OpenAI and LangChain support schema-constrained responses via Pydantic models. Legacy approaches like RetryWithErrorOutputParser (which feeds the original prompt, the failed completion, and the parsing error back to the model) remain available for edge cases.

7. State Management

How the agent tracks durable state. Three approaches in the wild:

  • LangGraph — typed dictionaries flowing through graph nodes, with reducers merging updates. Checkpointing at super-step boundaries for resume and time-travel debugging.
  • OpenAI — four mutually exclusive strategies: application memory, SDK sessions, server-side Conversations API, or lightweight previous_response_id chaining.
  • Claude Code — git commits as checkpoints and progress files as structured scratchpads. You saw this in lesson 0008: the agent commits after each sub-task. That's state management.

8. Error Handling

The compounding problem. A 10-step process with 99% per-step success still has only ~90.4% end-to-end success. Errors compound fast. This is why Stripe's production harness caps retry attempts at two.

LangGraph distinguishes four error types:

TypeResponse
TransientRetry with backoff
LLM-recoverableReturn error as ToolMessage so the model can adjust
User-fixableInterrupt for human input
UnexpectedBubble up for debugging

Anthropic catches failures within tool handlers and returns them as error results to keep the loop running. The model sees the error and self-corrects. This is the same pattern your AFK agent uses when tests fail: it doesn't crash, it tries a different approach.

9. Guardrails and Safety

Anthropic separates permission enforcement from model reasoning architecturally. The model decides what to attempt; the tool system decides what's allowed. Claude Code gates ~40 discrete tool capabilities independently, with three stages:

  1. Trust establishment at project load
  2. Permission check before each tool call
  3. Explicit user confirmation for high-risk operations

OpenAI's SDK implements three levels: input guardrails (run on first agent), output guardrails (run on final output), and tool guardrails (run on every tool invocation). A "tripwire" mechanism halts the agent immediately when triggered.

10. Verification Loops

"Giving the model a way to verify its work improves quality by 2–3×." — Boris Cherny, creator of Claude Code

This is what separates toy demos from production agents. Three approaches:

  • Rules-based feedback — tests, linters, type checkers. Deterministic ground truth.
  • Visual feedback — screenshots via Playwright for UI tasks.
  • LLM-as-judge — a separate subagent evaluates output. Catches semantic issues but adds latency.

You know this from lesson 0008: the review agent. That's an LLM-as-judge verification loop. The test suite in the inner loop is a rules-based verification loop. Production harnesses use both.

11. Subagent Orchestration

Claude Code supports three execution models:

  • Fork — byte-identical copy of parent context
  • Teammate — separate terminal pane with file-based mailbox communication
  • Worktree — own git worktree, isolated branch per agent

OpenAI's SDK supports agents-as-tools (specialist handles bounded subtask) and handoffs (specialist takes full control). LangGraph implements subagents as nested state graphs.

12. Prompt Engineering (yes, it's a component too)

The actual instructions the model receives. This is component #12, not component #1, because in a production harness it's the least likely to need manual tuning. Most problems that look like "bad prompting" are actually failures in components 3–11.

If your agent forgets what it did three steps ago — that's component 4 (context management), not component 12. If tool calls fail silently — that's component 8 (error handling). If the agent does too much — that's component 9 (guardrails). Fix the right component.

The loop in motion: one cycle, step by step

Now that you know the components, trace how they work together in a single TAO cycle:

Step 1 │ PROMPT ASSEMBLY (components 3, 4, 5) │ System prompt + tool schemas + memory + history + user message │ Important context at beginning and end ("Lost in the Middle") ▼ Step 2 │ LLM INFERENCE (the model, not the harness) │ Input tokens → output tokens (text, tool calls, or both) ▼ Step 3 │ OUTPUT CLASSIFICATION (component 6) │ Tool calls? → Step 4. No tool calls? → done. Handoff? → restart. ▼ Step 4 │ TOOL EXECUTION (components 2, 9) │ Validate args → check permissions → execute in sandbox → capture results │ Read-only ops run concurrently; mutating ops run serially ▼ Step 5 │ RESULT PACKAGING (component 8) │ Format results as LLM-readable messages │ Catch errors, return as error results for self-correction ▼ Step 6 │ CONTEXT UPDATE (component 4, 7) │ Append results to history. If near limit → trigger compaction ▼ Step 7 │ LOOP (component 1) │ Return to Step 1. Repeat until termination.

Termination conditions are layered: the model produces a response with no tool calls, maximum turn limit exceeded, token budget exhausted, a guardrail tripwire fires, the user interrupts, or a safety refusal is returned.

A simple question: 1–2 turns. A complex refactoring task: dozens of tool calls across many turns. The lesson 0008 AFK agent runs this cycle for each sub-task in the issue's plan.

The evidence that the harness is the product

Lesson 0001 gave you Matt's claim. Here's the empirical evidence from the broader community:

  • LangChain's TerminalBench jump — changed only the infrastructure wrapping their LLM (same model, same weights) and jumped from outside the top 30 to rank 5 on TerminalBench 2.0. Twenty ranking positions from harness changes alone.
  • Meta-harness research — a separate project hit 76.4% pass rate by having an LLM optimize the infrastructure itself, surpassing hand-designed systems.
  • ACON research — showed 26–54% token reduction while preserving 95%+ accuracy by prioritizing reasoning traces over raw tool outputs in context management.

Two products using identical models can have wildly different performance based solely on harness design. This is what Matt meant by "have a codebase that's easier to make changes in" — the codebase is part of the harness, and improving it improves every model that touches it.

The co-evolution principle

Models are now post-trained with specific harnesses in the loop. Claude Code's model learned to use the specific harness it was trained with. Changing tool implementations can degrade performance because of this tight coupling.

The "future-proofing test" for harness design: if performance scales up with more powerful models without adding harness complexity, the design is sound. If you need to add more scaffolding every time the model improves, something is wrong.

The field is moving toward thinner harnesses as models improve. Manus was rebuilt five times in six months, each rewrite removing complexity. Complex tool definitions became general shell execution. "Management agents" became simple structured handoffs. This is the scaffolding metaphor made real: scaffolding is removed when the building is complete.

But the harness itself isn't going away. Even the most capable model needs something to manage its context window, execute its tool calls, persist its state, and verify its work.

Seven decisions that define every harness

These are the architectural choices you face when building or tuning your own harness. Each one maps back to the 12 components:

#DecisionGuidance
1 Single-agent vs. multi-agent Both Anthropic and OpenAI say: maximize a single agent first. Split only when tool overload exceeds ~10 overlapping tools or clearly separate task domains exist.
2 ReAct vs. plan-and-execute ReAct interleaves reasoning and action (flexible, higher per-step cost). Plan-and-execute separates them (LLMCompiler reports 3.6× speedup over sequential ReAct).
3 Context window strategy Time-based clearing, summarization, observation masking, structured note-taking, or sub-agent delegation. Choose based on task length.
4 Verification loop design Computational (tests, linters) for deterministic ground truth. Inferential (LLM-as-judge) for semantic issues. Best harnesses use both.
5 Permission architecture Permissive (fast, risky) vs. restrictive (safe, slow). Depends on deployment context. AFK agents lean permissive within the sandbox.
6 Tool scoping Expose the minimum tool set needed for the current step. Fewer tools = better performance.
7 Harness thickness How much logic lives in the harness vs. the model. Anthropic bets thin; graph-based frameworks bet thick. Anthropic regularly deletes planning steps as new models internalize that capability.

For your Claude Code setup (lesson 0001–0008), decisions 1, 5, and 7 are already made: single agent, permissive-in-sandbox, thin harness. Decisions 3, 4, and 6 are where your CLAUDE.md and skill design do the most work. Decision 2 is implicit in the /grill-me → /to-prd → /to-issues → AFK pipeline: the planning is separated from execution. You're already using plan-and-execute at the workflow level, even if the agent itself uses ReAct at the step level.

The diagnostic rule

When your agent fails, don't blame the model. Look at the harness.

Agent forgets what it did three steps ago? → Component 4 (Context Management)
Tool calls fail silently? → Component 8 (Error Handling)
Context window fills up with garbage? → Component 4 (Context Management)
Agent does the wrong thing despite correct instructions? → Component 5 (Prompt Construction — check where your instructions land in the window)
Agent does too much? → Component 9 (Guardrails) or Component 2 (Tool Scoping)
Agent produces correct code that's semantically wrong? → Component 10 (Verification Loops — add an LLM-as-judge)

Quiz

  1. Name the 12 components without looking. (Hint: group them by function — brain, hands, memory, safety, team.)
  2. Your AFK agent keeps making changes to files the PRD's non-goals section explicitly forbids. Which component is failing, and what would you change?
  3. What does "context rot" mean, and which research finding explains it?
Answers (try first)
  1. Brain: orchestration loop, prompt construction, output parsing. Hands: tools. Memory: memory, context management, state management. Safety: error handling, guardrails. Team: verification loops, subagent orchestration. Plus: prompt engineering. (Twelve total.)
  2. Component 9 (Guardrails). The harness should enforce non-goals as a machine-checkable constraint — a file-touch allow-list, a pre-commit check, or a review-agent prompt that greps the diff against the PRD's non-goals section. If the model can attempt the action, the guardrail layer should block it. Lesson 0008 covered this exact failure mode.
  3. Context rot is the degradation of model performance as context fills with low-signal content. Stanford's "Lost in the Middle" finding (Liu et al., 2024) showed that performance degrades 30%+ when key content falls in mid-window positions, even in million-token windows. The fix is aggressive compaction, observation masking, and just-in-time retrieval.

What to do next

  1. Audit your harness. Open your CLAUDE.md and map each line to one of the 12 components. Lines that don't map to any component are probably noise — delete them. Lines that are doing double duty across components might need to be split.
  2. Diagnose a recent failure. Pick an AFK session that produced bad output. Walk through the diagnostic rule above. Which component failed? What would you change?
  3. Check the Harness Components Reference for a one-page cheat sheet you can print and stick next to your monitor.
  4. Ask your teacher anything that wasn't clear. The 12-component model is a lens, not a law — if something in your setup doesn't fit neatly, that's worth talking through.
