Write a PRD the Agent Can Read

Lesson 0006 · ~10 min · The document that tells the AFK agent what the code can't

Procedural Source: derived from Matt Pocock's Agentic Engineering Workflow at 11:40; /write-a-prd from his skills repo

You have a queue item from /grill-me. It's specific, bounded, and verifiable. You could hand it to an AFK agent right now. The agent will read the queue item, look at the codebase, and start working. Two hours later you'll come back to a PR that solves the letter of the request but misses the point entirely.

The point isn't in the code. The code is what you asked for. The point is why you asked for it, what you decided to not do, and what you assumed would be true. None of that lives in the codebase. If you don't write it down, the agent makes it up.

This lesson teaches you to write the document the AFK agent reads before it touches the code: a PRD that captures the things the queue item can't.

1. What a PRD is, here

A PRD (Product Requirements Document) in this workflow is not the 30-page corporate document. It's a short, decision-shaped file — typically 200–800 lines — that lives next to the code it describes. Matt calls it a PRD because that's the word the industry uses; you could call it the spec, the design doc, or the context file. The label matters less than the structure.

The PRD's job is to be the file the AFK agent reads first. It's how you put the "why" into a form the agent can ingest, so that when it makes the inevitable mid-task decision, it makes the same decision you would have made.

2. The skill, in skeleton

# ~/.claude/skills/write-a-prd/SKILL.md

---
name: write-a-prd
description: Turn a queue item (from /grill-me) into a short PRD that captures the context an AFK agent can't infer from the code.
disable-model-invocation: true
---

# Write a PRD

Given a queue item (specific, bounded, verifiable), produce a
PRD with these sections:

1. Context — why this work exists. Who asked, what
   triggered it, what changes if it ships.
2. Non-goals — what is explicitly out of scope.
   This is the most important section.
3. Decisions — what was decided, when, and why.
   Including rejected alternatives.
4. Open questions — things still uncertain.
   If any, the work isn't ready for AFK.
5. Done means / not done means — copied
   verbatim from the queue item.
6. Plan — short numbered list of file-level changes.

Same frontmatter pattern as /grill-me — disable-model-invocation: true. Procedure. You invoke it when you have a queue item you want to commit to. You don't let the model invoke it on its own.

3. The sections that matter most

Sections 1, 5, and 6 are mechanical: context is what you know, done-means is the line from the grill-me session, plan is a list of files you'll touch. The interesting ones are 2, 3, and 4.

3.1 Non-goals (the most important section)

The agent's job is to maximize "did the work get done" given a fixed context window. If you don't tell it what not to do, it will infer the scope from the codebase and its training data. Both sources will be slightly wrong. The result: the agent does more than you asked, touches files it shouldn't, and produces a PR that solves adjacent problems you didn't want solved.

Non-goals fix this. They are explicit, written, scoped-out statements. For the SSO example from lesson 0005:

## Non-goals

- Mobile app SSO. Web only.
- Migrating existing users off email/password.
- SSO for admin users (they keep using email/password + MFA).
- Refactoring the existing auth module beyond the minimum
  surface needed for the new Okta provider.
- Any change to billing, permissions, or session state.
- Adding new analytics events.

Notice the specificity. "We won't do mobile SSO" is too vague — does that mean iOS only, or Android too, or the mobile web? "Mobile SSO. Web only." is the version that stops the agent from guessing.

3.2 Decisions, including rejected alternatives

Mid-task, the agent will face a fork: "should I use the existing logger or add a new one?" "Should this go in auth.ts or in a new file?" Without context, the agent picks something reasonable and you review it later. With context, the agent picks the thing you would have picked, which is the point of the whole exercise.

Decisions section captures the forks you've already resolved. Example:

## Decisions

- **Okta over Auth0**: existing vendor relationship, SAML
  already configured in Okta dashboard.
- **New file `oktaAuthProvider.ts` over extending `auth.ts`**:
  keeps the existing email/password path untouched, which is
  the lowest-risk change for non-SSO users.
- **No state library**: session cookie + server check is
  sufficient for this scope. Rejected: Zustand (overkill),
  React Context (causes re-renders).
- **Login button on `/login`, not in nav**: marketing wants
  SSO and password users to land on the same page.

The "rejected" parts are the expensive ones to discover from scratch. They save the agent from spending 10 minutes considering Auth0.

3.3 Open questions

If there are open questions, the work isn't ready for AFK. That's the rule. An agent handed a PRD with open questions will answer them — plausibly, confidently, and often wrongly. The grill-me session (lesson 0005) should have caught most of them. If any survive into the PRD, the work isn't ready and you should finish the scoping before proceeding.

Open questions can also be a deliberate handoff. "We don't know the rate limit; pick a sensible default and document your choice in the PR description." That's a legitimate open question with a fallback decision. The agent can proceed.

4. Why a doc, not a prompt

You could put all of this in a long prompt. The agent would read it once. If it needed the same context again ten minutes later — because the prompt scrolled off or because a sub-task forgot the parent context — it would re-derive it, badly.

A doc on disk has different properties:

  • The agent can re-read it whenever it needs to.
  • Multiple agents (parallel AFK workers) can share the same context.
  • The doc survives across sessions — it's the artifact that remains when the conversation is gone.
  • You can edit it without re-running the skill. If a decision changes, you change one paragraph.
  • It doubles as documentation for humans — your future self, six months from now, when the AFK agent is long gone.

Lesson 0001 said strategic programming compounds. The PRD is the slowest-compounding artifact in the workflow — the one that pays back over months, not hours. But it pays back hardest.

5. PRD vs the codebase context doc (lesson 0004)

Lesson 0004 said every file should have a 5–10 line top-of-file context block. The PRD is the same idea at a different altitude. The hierarchy:

Layer Lives in Scope Audience
Top-of-file context Each module file One module Any reader of that file
PRD docs/prd/<feature>.md One feature or change Any agent working on this feature
Architecture decision record docs/adr/ System-wide decisions Any reader of the system

ADRs are the slowest layer. PRDs sit in the middle. Top-of-file context is the fastest layer. All three are needed; none replaces the others.

6. The "PRD or no PRD" decision

Not every queue item deserves a PRD. The rule of thumb:

Skip the PRD
  • One-line bug fix.
  • Typo / docs-only change.
  • Refactor with no behaviour change.
  • Task fits in a single file edit.
Write the PRD
  • New feature touching 3+ modules.
  • Anything with multiple stakeholders.
  • Anything where the "why" isn't obvious from the code.
  • Anything you'd otherwise have to re-explain in 6 months.

Most queue items are the "skip" case. PRDs are for the work that, without context, the agent will get wrong.

7. Exercise: write one PRD, by hand

Take the queue item from lesson 0005 and produce a PRD

  1. Open the queue item you wrote during the lesson 0005 exercise. If you didn't do it, pick any non-trivial feature request from your backlog.
  2. Write the six sections in order. The two hardest are Non-goals and Decisions — those are where you should spend the most time.
  3. Read the PRD back to yourself and ask: if I were an agent with no prior context, would this tell me what to do, what not to do, and why? If not, the missing section is your next 15 minutes.
  4. Save it to docs/prd/<feature-name>.md in your repo and commit. Don't open a PR yet — the PRD is its own commit so reviewers can see the design before the code.
  5. Optional: hand the PRD to Claude Code with the prompt "Read this PRD and tell me what you'd do, what questions you have, and what the riskiest decision is." Notice where the agent's questions match your Open Questions section.
The compounding effect. Six months from now, when you need to change the SSO integration, you open the PRD. The PRD tells you what was decided and why. You don't re-derive the architecture from the code. You update the PRD, then update the code to match. The codebase and the doc drift together — but slowly, and intentionally.

Quiz

  1. What frontmatter flag should /write-a-prd have, and why?
  2. Why is the "Non-goals" section the most important?
Answers (try first)
  1. disable-model-invocation: true. It's a procedure — you invoke it deliberately when a queue item is ready to commit to a doc. Letting the model invoke it on its own would cause it to write PRDs unsolicited, polluting the repo with speculative docs.
  2. Because the agent's job is to maximize "done" given a fixed context. Without explicit non-goals, the agent will infer scope from the codebase and its training data — both slightly wrong — and produce a PR that solves adjacent problems you didn't ask for.
