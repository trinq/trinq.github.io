The to-spec Skill

From conversation to published spec — no interview, just synthesis.

Lesson 0006 taught you write-a-prd. That skill still exists in spirit, but Matt Pocock renamed and tightened it in mid-2026: to-spec. Same job — turn a conversation into a written artifact — but the new framing draws a sharper line between interviewing and synthesizing, and the chain it sits in is now /grill-with-docs → /to-spec → /to-tickets → /implement.

This lesson is the upgrade. You already know the destination (a written spec). This lesson teaches you the new vocabulary, the new chain, and three concrete behaviors that make to-spec produce better specs than write-a-prd did.

1. The rename and the reframe

Three things changed in the same commit that introduced to-spec:

  1. write-a-prd → to-spec. The artifact is still a spec — the skill opens with "you may know this document as a PRD" for discoverability — but spec is now the through-line term. PRDs felt like a corporate artifact; specs feel like a code artifact. The rename is small, the framing shift is large.
  2. create-issues → to-tickets. The decomposition skill absorbed the old to-plan skill and is now medium-agnostic: it emits a set of tickets that read as a sequential plan (a local tickets.md, edges as text) or as a parallelisable DAG (a real tracker with native blocking links). The medium only decides whether anything acts on the dependencies in parallel.
  3. The chain is now explicit. ask-matt routes idea → to-spec → to-tickets → implement. The intermediate PRD step and the issues step are both gone in name. The work is the same; the seams are cleaner.
Why the rename matters for you. If you keep using write-a-prd in prompts, agents that have the new skill loaded will misroute. Pin to-spec in your vocabulary. Update your team docs. The word matters because the skill name is the trigger.

2. The skill, in skeleton

The full SKILL.md is 38 lines. Here is the structure that matters:

---
name: to-spec
description: Turn the current conversation into a spec
  and publish it to the project issue tracker — no
  interview, just synthesis of what you've already discussed.
disable-model-invocation: true
---

This skill takes the current conversation context and
codebase understanding and produces a spec. Do NOT
interview the user — just synthesize what you already know.

1. Explore the repo to understand the current state of the
   codebase, if you haven't already. Use the project's
   domain glossary vocabulary throughout the spec, and
   respect any ADRs in the area you're touching.

2. Sketch the seams at which you're going to test the
   feature. Existing seams should be preferred to new
   ones. Use the highest seam possible. If new seams are
   needed, propose them at the highest point you can.
   The fewer seams across the codebase, the better — the
   ideal number is one.

3. Write the spec using the template below, then publish
   it to the project issue tracker. Apply the
   `ready-for-agent` triage label — no need for
   additional triage.

Three instructions. That's the whole skill. The rest of the SKILL.md is a template for the spec body.

3. The "no interview" rule is the whole point

This is the single most important behavior, and the one most likely to drift if you don't pin it.

Lesson 0006 taught you write-a-prd, which used to interview you and write the document. That coupling is gone. to-spec assumes the interview already happened — in grill-me or grill-with-docs — and that the domain language is settled. It will not ask you another round of questions. It will not pause to confirm scope. It will read the conversation, read the repo, sketch the seams, and write the spec.

If you find to-spec asking you questions, your grill step was incomplete. Stop, go back to grill-me or grill-with-docs, finish the interview, then re-invoke to-spec in the same conversation. The interview context is what the skill is synthesising — clear the context and you've erased the input.

The reason this rule exists is the same reason write-a-prd was split in the first place: interviewing and synthesizing are different modes, and mixing them in one skill produces mediocre output of both. grill-me interviews; to-spec writes. Trying to merge them is the regression the Matt Pocock team explicitly rejected.

4. The seam sketch is the new behavior

This is the part write-a-prd did not do. Before writing the spec, to-spec sketches the seams at which the feature will be tested. It prefers:

  1. Existing seams to new ones. If the repo already has a test seam (a port, a fake, a public test surface), use it. Don't add a second one.
  2. The highest seam possible. Higher = closer to the user-facing surface. A test at the API boundary catches more than a test at the utility function.
  3. Fewer seams across the codebase, the better — the ideal number is one. Every new seam is a permanent cost. If the change can be tested through one seam, it should be.

After sketching, to-spec checks with you: do these seams match your expectations? That is the only question it asks. Once you confirm, it writes the spec with those seams as the testing decisions.

Why this matters. A spec that names its seams in advance produces tickets that the implementer can actually verify. A spec that says "add a dark mode toggle" produces tickets where the implementer has to invent the test surface, which is where scope creep hides. The seam sketch is what makes the spec implementable rather than just describable.

5. The spec template

The body of every to-spec artifact has the same six sections. Memorize the order; the order is the workflow.

Problem Statement

The problem the user is facing, from the user's perspective. Not the technical problem. Not the system problem. What is the user trying to do, and what's stopping them?

Solution

The solution to the problem, from the user's perspective. One or two paragraphs. The user should be able to read this section alone and understand what the feature will do for them.

User Stories

A LONG, numbered list. Each story in the format As a [role], I want [capability], so that [outcome]. The list is intentionally exhaustive — it should cover every aspect of the feature, not just the happy path.

Implementation Decisions

The technical decisions that were made during the grill. Modules touched, interfaces changed, schema migrations, API contracts, architectural decisions. The rule: no specific file paths or code snippets, because they go stale fast. Exception: if a prototype produced a snippet that encodes a decision more precisely than prose (a state machine, a reducer, a schema), inline it and note that it came from a prototype.

Testing Decisions

What makes a good test (only external behavior, not implementation details), which modules will be tested, and prior art — similar tests already in the codebase. The seams from step 2 live here.

Out of Scope

What the spec explicitly is not doing. This is the most-skipped section in any spec, and the one that prevents the most scope creep. The agent's job is to maximize "done" given a fixed context; without explicit non-goals, the agent will infer scope from its training data — which is slightly wrong — and produce a PR that solves adjacent problems you didn't ask for.

7. Exercise: rewrite your lesson-0006 PRD as a spec

Take the PRD from lesson 0006 and run it through the new skill

  1. Open the PRD you wrote during the lesson 0006 exercise. (If you didn't, pick any non-trivial feature from your backlog.)
  2. Read it with the new lens: does the Problem Statement name the user, not the system? Does the Solution say what the user gets, not what the code does? Are the User Stories exhaustive, not aspirational?
  3. Re-label the section headers to the new names: Problem Statement, Solution, User Stories, Implementation Decisions, Testing Decisions, Out of Scope. If you have to invent content for a section that didn't exist before (most likely User Stories or Out of Scope), write it now — that's the gap.
  4. Add the seam sketch. For each Testing Decision, name the seam: an existing one, or a new one proposed at the highest point you can. Count them. If the count is more than one, ask whether any of them can collapse.
  5. Commit the rewritten file to docs/spec/<feature-name>.md (move it from docs/prd/ if you used the old path). Open a PR with the title spec: <feature-name>. Notice how the reviewer conversation changes when the spec is shaped this way.
  6. Optional but powerful: in the same PR, add a second commit that only contains the user stories, formatted as a checklist. That checklist is the seed for the to-tickets skill in the next lesson.
The compounding effect, redux. The PRD from lesson 0006 was a destination. The spec from this lesson is a destination plus a contract — the seam sketch is what makes it executable, and the exhaustive user stories are what make the tickets testable. Same document, sharper shape. Future-you opens the spec to change the SSO integration and immediately sees which seams to preserve.

Quiz

  1. What frontmatter flag does /to-spec have, and why does it matter?
  2. If to-spec starts asking you clarifying questions, what went wrong upstream?
  3. Why does the seam sketch happen before writing the spec, not during implementation?
Answers (try first)
  1. disable-model-invocation: true. to-spec is a procedure — you invoke it deliberately when a grill session is complete. Letting the model invoke it on its own would cause it to write specs unsolicited, polluting the issue tracker with speculative artifacts and bypassing the interview step.
  2. The grill was incomplete. The decision tree still has unresolved branches. Stop, return to grill-me or grill-with-docs, finish the interview in the same conversation (so the context is preserved), then re-invoke to-spec. The skill's input is the conversation; an empty conversation produces a generic spec.
  3. Because the seams are the testing decisions. If the implementer invents the test surface during coding, scope hides inside the tests. A spec that names the seams up front makes the testing decisions a contract the implementer has to honor, not a discovery the implementer has to make.

Primary source

The skill itself is Matt Pocock's to-spec on GitHub. The rename and the chain unification are documented in commit 386d4ff ("refactor: unify planning skills into /to-spec + /to-tickets"), and the community discussion that prompted the split is in discussion #333.

For grounding in why "no interview" is the right default, the relevant background is the grill-me skill itself: 3 sentences, devastatingly effective, and explicitly designed to be the interview that hands off to to-spec. The two skills are a pair.

What to do next

  1. Run the exercise above. Rewrite your lesson-0006 PRD as a spec, with a seam sketch and exhaustive user stories. Commit it as docs/spec/<feature-name>.md.
  2. The natural next lesson is to-tickets — the procedure that takes the spec (this lesson) and turns it into one or more tickets that name their blocking edges. That's where the spec becomes a queue that the implementer (or an AFK agent) can consume in order or in parallel.
  3. Ask your teacher anything that wasn't clear. The seam sketch is the hardest part to internalize — it requires thinking about the test surface before the implementation exists, which feels backwards. It's not. It's why to-spec produces specs the agent can actually build, and write-a-prd often didn't.
