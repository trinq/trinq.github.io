Create Issues, Atomic Work Units

Lesson 0007 · ~10 min · The procedure that turns a PRD into one-merge-per-PR work

Procedural Source: derived from Matt Pocock's Agentic Engineering Workflow at 11:40, 24:10; /create-issues from his skills repo

You have a PRD (lesson 0006) that captures the why, the non-goals, the decisions, the plan. The PRD is 400 lines. The plan section lists six files that need to change. Hand that single PRD to an AFK agent and you get a single 800-line PR that touches all six files at once.

That's the worst case for review. Six files of changes, no clean point to revert, no way to bisect if a regression lands three weeks later, no way to merge the safe half while the risky half gets more review. Lesson 0003 already said: well-scoped = one PR, mergeable on its own. The PRD is not yet one PR. It's a feature.

This lesson teaches you to decompose the PRD into atomic issues — each one sized to a single PR that can be reviewed, merged, and reverted independently. The skill that does this is /create-issues, and it's the last procedure in the queue-prep chain before the AFK agent takes over.

1. The decomposition principle

The rule, stated once:

Each issue should be mergeable on its own. If the agent's PR for issue N is rejected, every other issue (N+1, N+2, ...) should still make sense as its own PR.

This is the same rule from lesson 0003 about queue items — now applied at a finer grain, inside a single PRD. The PRD is a feature. The issues are the steps to build the feature. Each step is independently mergeable.

The boundary you should feel while decomposing is the order of dependencies. If issue B requires the file changes from issue A, you can still merge them separately — issue B's branch just sits behind A's branch until A merges. That is fine and normal. What you cannot do is have issue C that requires both A and B to be merged at the same time. That is one issue disguised as two.

2. The skill, in skeleton

# ~/.claude/skills/create-issues/SKILL.md

---
name: create-issues
description: Take a PRD and decompose it into atomic GitHub issues, each sized to one mergeable PR.
disable-model-invocation: true
---

# Create Issues

Given a PRD path, produce N issues where:

1. N >= 1. Single-issue PRDs are allowed but rare.
2. Each issue references the PRD by path.
3. Each issue ends with: Done means / Not done means.
4. Dependency edges between issues are explicit
   (issue N+1 may depend on N).
5. No two issues depend on each other circularly.
6. Issues are filed with: gh issue create using
   the org's template.

Same frontmatter as the other two procedures — disable-model-invocation: true. You invoke it when a PRD is committed and ready to be broken into work units. You do not let the model invoke it on its own.

3. The order matters, but only weakly

Decomposition order is one of the few decisions where you have real choice. The defaults:

Decomposition axis Best for Risk
By layer (bottom-up) New features with a clear data flow Early PRs have no UI to click
By user-visible slice (vertical) PRDs with multiple user flows Later slices rework earlier ones
By risk (risky first) PRDs with an unknown-decision inside Risky issue may block simpler ones
By parallel (independent first) PRDs that need to be split across multiple agents Risk of duplication if boundaries slip

Matt's default, in the talk, is vertical: each issue ships a thin slice of the user flow. The reasoning: every merged PR gives you a checkpoint where the system is in a working state, even if the feature is partial. That's better than shipping a complete data layer and no UI, then a complete UI and no data, then trying to merge them and discovering the contract was wrong.

For the SSO example, the vertical split might be:

#1: Add Okta SAML config + env vars (config only, no UI)
     → verifiable by: env loads, config validates
#2: Add /api/auth/okta/callback route + handler
     → verifiable by: hitting the route returns 302 to Okta
#3: Add "Sign in with Okta" button to /login
     → verifiable by: clicking it starts the flow
#4: Add session-cookie wiring + redirect to /dashboard
     → verifiable by: completing flow lands user on dashboard
#5: Add tests for the happy path + a sad-path case
     → verifiable by: new tests pass, no regressions

Each is one PR. Each is mergeable on its own. The dependency graph is a chain (#5 needs #1–#4), but no two issues depend on each other circularly. The system is in a working state at every merge — except #2 and #3 might temporarily expose a button that doesn't fully work, which is acceptable if you wrap them in a feature flag.

4. The "feature flag" escape hatch

When a vertical slice is incomplete, you have two choices: merge anyway and accept that the feature is half-built in main, or hide it behind a feature flag. The flag is the safer default for any user-visible work.

// auth.ts
if (featureFlags.oktaSso && user.clickedOktaButton) {
  return oktaAuthenticate();
}
return passwordAuthenticate();

This is the only acceptable way to merge PRs that depend on each other in the user-visible sense but are independent in the codebase sense. The flag is the temporary contract. It comes out in the last issue, after the full flow works end-to-end.

Lesson 0004's deep-module rule still applies: the flag should live in one place (often featureFlags.ts), not be sprinkled across files. The agent will probably reach for the sprinkled version if you don't say otherwise — this is exactly the kind of decision the PRD's "Decisions" section should have captured.

5. When to use a single issue instead

Not every PRD needs to be decomposed. Use one issue when:

  • The PRD is already small (under ~150 lines).
  • The plan touches <3 files.
  • There is no clean vertical cut — the change is genuinely one thing.
  • Splitting would force a feature flag for a 50-line change that doesn't justify the complexity.

In that case, /create-issues still runs — it just produces one issue, and the issue body is the PRD's plan section verbatim. The skill is the boundary between "PRD-level thinking" and "issue-level execution"; running it confirms you considered the decomposition and decided against it.

6. The issue body format

Matt's repo includes an issue template. The skeleton that works:

## Context
[1–2 paragraphs from the PRD's Context section]

## Plan
[numbered list from the PRD's Plan section]

## Done means
[verbatim from the PRD]

## Not done means
[verbatim from the PRD]

## Depends on
[#issue-number, if any]

## Reference
- PRD: docs/prd/okta-sso.md
- Related decisions: see PRD §Decisions
- Touch-this-file-when: see PRD §Non-goals

The Done/Not done lines are the load-bearing ones. They get copy-pasted into the PR description by the agent later (lesson 0008). If you change the wording between the PRD and the issue, you have introduced a place for the agent to get confused. Keep them identical.

7. Filing: gh CLI, not the web UI

Use gh issue create from the terminal, not the GitHub web form. Reasons:

  • The body is local-first — you can edit it in your editor with full tooling.
  • You can dry-run the body, eyeball it, then file.
  • The PR description later (lesson 0008) is generated by the agent reading the issue body — so the body is the source of truth, not the GitHub-rendered version.
  • Scriptable. If you have 5 issues to file from one PRD, you script them all.

The skill's gh issue create line in §2 is the directive. Don't web-UI it.

8. The queue, finally

Once issues are filed, they go on the queue from lesson 0003. The PR references the issue number (with "Closes #N"), the issue body becomes the PR description, and the queue item comes off when the PR merges.

PRD (1 file, decision-shaped)
  ↓
create-issues
  ↓
Issues (N files, atomic, mergeable)
  ↓
Issue tracker (the queue from lesson 0003)
  ↓
AFK agent picks one, opens PR
  ↓
PR description = issue body
  ↓
Review, merge, item comes off the queue

You're at the bottom of the queue-prep chain. Lessons 0005–0007 are the three procedures that turn "a vague request" into "the next item the AFK agent should work on." Tomorrow's lesson (0008) is the AFK agent itself — what it reads, what it does, what it produces.

9. Exercise: decompose your PRD from lesson 0006

Take the PRD from lesson 0006 and decompose it

  1. Open the PRD you committed in lesson 0006's exercise. Re-read the Plan section.
  2. Decide on an axis (vertical, by-layer, by-risk, by-parallel). State which one and why in one sentence at the top of your decomposition.
  3. For each issue, write: title, 1-paragraph context, plan, done-means, not-done-means, depends-on. Aim for 3–5 issues. Fewer is fine. More than 7 means your PRD was actually two features.
  4. Run the "mergeable on its own" test on each issue: if this PR is rejected, do the remaining issues still make sense? If not, you've drawn a dependency wrong.
  5. File them with gh issue create from your terminal. Use the body format from §6. Reference the PRD path in each one.
  6. Add a label like prd:okta-sso to each so they're findable as a group.
The label trick. Once you've filed 4–5 issues from one PRD, add a shared label (e.g., prd:okta-sso). Now the AFK agent can be pointed at "all issues with this label" and work them in order. The label is how you turn a PRD into a queue of queue items — a queue of queues.
The signal that you decomposed wrong. If you find yourself writing "and then update the tests" in more than one issue's plan, the test work is its own issue. If you find yourself writing "and add a feature flag for X" in more than one issue, the flag is its own issue. The decomposition test is: can the agent implement this issue without reading the other issues' plans? If no, the boundaries are wrong.

Answers (try first)
  1. Each issue should be mergeable on its own. If the agent's PR for issue N is rejected, every other issue (N+1, N+2, ...) should still make sense as its own PR.
  2. disable-model-invocation: true. It's a procedure — you invoke it deliberately when a PRD is ready to be decomposed. Letting the model invoke it on its own would cause it to decompose PRDs unsolicited, fragmenting work the user hasn't approved.
