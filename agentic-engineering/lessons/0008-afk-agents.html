AFK Agents, the Loop Closes

Lesson 0008 · ~12 min · The agent that consumes everything the previous lessons produced

Operational Source: Matt Pocock's Agentic Engineering Workflow at 24:10 (Ralph-loop pushback), 43:35 (AFK quote); Sandcastle (his tool) from transcript §83

You have a queue (lesson 0003) of well-scoped issues (lesson 0007), each backed by a PRD (lesson 0006), each driven by a grill-me conversation (lesson 0005). Your harness is set up (0001). You know which skills are procedures (0002). Your codebase has AX (0004). The only thing left is: who actually does the work?

An AFK agent does. "AFK" means Away From Keyboard — you start the agent, walk away, come back to a PR. The agent reads the issue, reads the PRD, explores the codebase, writes the code, writes the tests, opens the PR. When you return, you review, not author.

"AFK is just incredible way to work. It's just takes a little bit of setup, but once it's set up, it's just goes crazy." — Matt Pocock, 43:35

This lesson is the loop closing. Lessons 0001–0004 set up the world; lessons 0005–0007 produced the queue; this lesson runs the consumer.

1. What the AFK agent reads, in order

The agent's first action is to ingest its context. The order matters — the agent should read the most-focused doc first, then the wider context, then the code:

  1. The issue body — the plan, the done-means, the not-done-means. The most specific.
  2. The PRD (linked from the issue) — the why, the non-goals, the decisions, the rejected alternatives.
  3. The codebase — top-of-file context blocks (lesson 0004), then the relevant modules, then anything the agent needs to extend.
  4. The issue's prior PRs in the same label — if issues share a prd: label (lesson 0007's trick), the agent should check what the previous issue's PR did, to maintain consistency.

This order is the one you set in the harness. If the harness's startup prompt says "read .claude/CLAUDE.md first," that file should encode this read-order. If it doesn't, the agent will invent its own order, and you will spend the first 20 minutes of every AFK session re-deriving what the agent should already know.

2. The agent's environment: a sandbox

The AFK agent runs in a sandbox — an environment isolated from your machine, your main branch, and your secrets. Two reasons:

  • Safety. The agent is going to run shell commands. Most of them are fine. Some aren't. The sandbox is the boundary that lets you walk away without worrying.
  • Reproducibility. The agent needs to start from a clean state every time. Your local working tree has uncommitted changes, stale branches, half-finished work. The sandbox doesn't.

Matt's tool for this is Sandcastle — runs Claude Code inside Docker/Podman sandboxes locally, or in Vercel sandboxes for remote AFK agents. The pattern, not the tool, is the lesson:

# pseudocode for an AFK agent invocation
agent.spawn({
  environment: "sandbox",           # isolated, ephemeral
  startingPoint: "main",            # clean checkout
  context: [
    "issue: #1234",                 # the queue item
    "prd: docs/prd/okta-sso.md",   # the design
    "constraints: .claude/CLAUDE.md" # your harness rules
  ],
  tools: ["bash", "edit", "github", "test-runner"],
  budget: { tokens: 200_000, wallTime: "2h" }
})

When the agent finishes — either successfully or by hitting the budget — the sandbox is destroyed. The commits it produced are pulled back into your local workspace. You review the PR there.

3. The loop, as the agent runs it

Inside the sandbox, the agent runs a loop. Each iteration:

  1. Pick the next sub-task from the issue's plan.
  2. Read the relevant code (file-level context, then function-level).
  3. Make the change.
  4. Run the tests.
  5. If tests pass, commit. If not, debug (up to N retries).
  6. Move to the next sub-task.

This is the "Ralph loop" pattern — the while loop that keeps invoking the agent until the task is done. Lesson 0003 said you don't need a Ralph loop at the queue level. But at the agent level, you absolutely do. The contradiction is fake: the outer loop (queue → PR → merge → next item) is replaced by your issue tracker; the inner loop (sub-task → code → test → commit) is the agent's own. Both loops exist. They exist at different scales.

"Half of it I would say is like the research labs selling more tokens. You know basically you should be running loops to pay us more endless tokens. Stop prompting your agents. Figure out what loops it can run forever permanently. Half of it could be useful." — Matt Pocock, 24:08

Read that quote again. "Half of it could be useful." The inner loop (this lesson) is the half that's useful. The outer loop (Ralph loops running the whole workflow) is the half that's not. The distinction is what the loop is wrapping: well-scoped atomic work, or vague user intent. Wrap atomic work — useful. Wrap vague intent — expensive.

4. What the agent produces

When the agent finishes, it should have produced:

  • A branch with commits matching the issue's plan.
  • Tests covering the new code — the done-means line should be testable as written.
  • A PR description that is the issue body verbatim, plus a "What changed" section listing the commits.
  • A "Closes #N" reference, so merging the PR closes the issue and the item comes off the queue (lesson 0003).
  • Updated top-of-file context blocks (lesson 0004) for any file the agent created or significantly changed. This is the AX payoff — the next agent that reads the file inherits the new context.

If the agent produces a PR without tests, or with a PR description that's a rewrite of the issue body (not a copy), or without updating context blocks — you know the harness isn't enforcing the contracts. Fix the harness, not the agent's prompt. Lesson 0001's rule applies: spend the time on the system, not the call.

5. The review agent

Once the AFK agent opens the PR, a second agent reviews it. This is where the second loop lives: the review agent runs in CI (Matt uses GitHub Actions for this — see transcript §84), reads the diff, checks against the issue's done-means / not-done-means, leaves comments, and either approves or requests changes.

The review agent is cheaper than the AFK agent (less context, less reasoning), runs unattended, and gives you a first-pass filter before you read the diff yourself. You still review — the review agent doesn't replace you, it just ensures you don't waste 30 minutes on a PR that's clearly broken.

Without review agent

You come back to 5 PRs. 2 are obviously broken (failing tests, missing files). You spend 10–15 minutes on each before realizing they need a complete rewrite. The other 3 are reasonable. Total time: 90 minutes.

With review agent

You come back to 5 PRs. 2 have "REQUEST CHANGES" comments and a list of specific fixes. You skip them or send them back. 3 are approved. You review the 3 reasonable ones. Total time: 40 minutes.

6. The "what if it fails" cases

The AFK agent will fail in three ways, and the harness should be designed to handle each:

Failure mode Signal Recovery
Wrong code PR merges, tests pass, but the behaviour is wrong Revert, rewrite the issue (the scoping was bad), re-run
Stuck loop Agent burns budget on the same sub-task, makes no progress Kill the sandbox, review what it tried, fix the harness or the issue, re-run
Out of scope PR touches files the non-goals section forbids Reject the PR, fix the harness to enforce non-goals as a pre-commit check, re-run

The third case is the most common. The agent doesn't know what it doesn't know, and without explicit enforcement, it will quietly expand scope. The fix is to make non-goals machine-checkable — a lint rule, a file-touch allow-list, or a review-agent prompt that greps the diff against the PRD's non-goals section.

8. Exercise: run one AFK session, end to end

The whole chain, in one sitting

  1. Pick a fresh vague request (a feature you've been avoiding). Run /grill-me (lesson 0005). Get the queue item.
  2. Run /write-a-prd (lesson 0006). Commit the PRD.
  3. Run /create-issues (lesson 0007). File 3–5 issues with gh. Add a shared label.
  4. Set up an AFK harness. Sandcastle (Matt's tool), or Claude Code in a Docker container, or a GitHub Actions workflow that runs on issue labeled. The harness should: clone the repo, checkout main, read the issue, run the agent, push the branch, open the PR.
  5. Point the harness at issue #1. Start it. Walk away for 1–2 hours.
  6. Come back. Review the PR. If it satisfies the done-means line, merge. If not, leave comments and re-run.
  7. Watch the issue close automatically when the PR merges. The queue item comes off. The next issue is the next AFK session.
The throughput jump. Once the harness is set up, your week looks like: 1 hour of scoping (lessons 0005–0007), 5–10 hours of AFK sessions (lesson 0008), 1–2 hours of reviewing the PRs. Net: a feature that would have taken 2–3 weeks of focused work gets shipped in a week, with most of your time spent on the parts only you can do — the scoping, the reviewing, the design calls. The middle (the typing) is what the agent is for.
The failure mode to watch. AFK agents make it cheap to produce code. Cheap code production is not the same as cheap shipping. If your review loop is slow, your merge queue is full, your test infrastructure is flaky — AFK agents just push more broken PRs into the bottleneck faster. The workflow compounds. So does the garbage. Fix the slow parts of the loop before scaling up the agent count.

Quiz

  1. In what order does the AFK agent read its context? Why does order matter?
  2. What's the difference between the "Ralph loop" at the agent level and the "Ralph loop" at the workflow level? Which one is useful?
Answers (try first)
  1. Issue body → PRD → codebase → sibling PRs (in the same label). Order matters because the agent should ingest the most-specific scope first, then the wider context. If it reads the codebase first, it will infer scope from the code and miss the PRD's non-goals. If it reads sibling PRs first, it will pattern-match from previous decisions that may not apply.
  2. The agent-level loop wraps atomic, well-scoped work (one sub-task at a time, within a single issue). It's useful: it lets the agent iterate without re-prompting. The workflow-level loop wraps the whole queue (issue after issue, forever). It's not useful: it bypasses your scoping step, runs on vague intent, and produces unbounded work. Lesson 0003's queue replaces the workflow-level loop; this lesson keeps the agent-level loop.
