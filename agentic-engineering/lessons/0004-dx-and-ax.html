DX and AX, or: Design for the Reader Who Reads Everything

Lesson 0004 · ~12 min · Codebases that humans love usually already love agents too — with a few adjustments

Architectural Source: Matt Pocock's Agentic Engineering Workflow at 38:30, 38:50

A queue of well-scoped tasks (lesson 0003) gets you nothing if the codebase the agent inherits is hostile. A well-scoped task on a hostile codebase produces a hostile PR. You end up reviewing twice as much because every change touches five modules and breaks two conventions.

Matt's answer is to design codebases with two readers in mind: the human (DX, developer experience) and the agent (AX, agent experience). The good news: good DX usually already is good AX. The bad news: there are adjustments only a senior dev who has actually worked with agents will know to make.

This lesson teaches you to read your codebase through both lenses at once.

1. The two readers, stated plainly

I think of there as being a difference between DX — developer experience — and AX, right? Agent experience. And so agent experience is the experience that the agent has working in the codebase. — Matt Pocock, Agentic Engineering Workflow, 38:30

DX is the traditional software-design goal: clear module boundaries, good tests, intentional naming, small interfaces, deep modules. You already know most of this. AX is the same goal with one extra reader: an agent that has no prior context, no implicit knowledge of "the way we do things here," and no instinct for what broke last time someone touched this file.

Matt's other quote from the same stretch of the talk:

Often people forget about improving the codebase actually for better AX. They forget about all the edges you can get with like good software fundamentals and so that's where the senior will be useful, because the senior knows how to build good DX. — Matt Pocock, 38:50

That last sentence is the thesis. A senior dev who knows good DX has most of what AX needs. The work is mostly adjustment, not rebuild.

2. The overlap is huge

List things that help DX. Then ask: does this help AX too?

DX practice Helps AX too? Why
Clear module boundaries Yes Agent knows where to put a change without asking
Good test coverage Yes Agent can verify its own work without you in the loop
Intentional naming Yes Agent doesn't have to guess what helper2 means
Types / contracts Yes Agent catches its own mistakes at compile time
DRY / single source of truth Yes Agent edits one place, not five
Top-of-file context docs Even more so Agent reloads context on every task; humans can hold context in their head
Tight feedback loops (fast tests, fast lint) Even more so Agent iterates many times in one task; humans iterate once or twice

Most senior devs already do the left column. The bottom two rows are where AX adds new pressure. Both are easy to fix; neither is intuitive without seeing an agent fail at them.

3. The "top-of-file context" pattern

An agent reading a 600-line file for the first time doesn't know what it's looking at. You, the human author, have the file's purpose in your head — you've been working on it all week. The agent starts cold.

The fix is small: write 5–10 lines at the top of every file that say what it is, what it owns, what it depends on, and what it deliberately does not do. Not a JSDoc per function — a single block at the top.

Without context
// users.ts

import { db } from './db';
import { mailer } from './mailer';

export async function createUser(...) { ... }
export async function deleteUser(...) { ... }
export async function updateUser(...) { ... }

An agent landing here has to read the whole file to figure out scope. It might reach for mailer or db when it shouldn't.

With context
// users.ts
//
// Owns: user CRUD (create, read, update, delete) and auth helpers.
// Depends on: db (users table only), mailer (welcome emails).
// Does NOT: handle permissions (see permissions.ts),
//   billing (see billing.ts), or session state (see sessions.ts).
// Touch this file when: adding user fields, changing
//   validation, or adjusting the welcome flow.

import { db } from './db';
import { mailer } from './mailer';

export async function createUser(...) { ... }

An agent landing here knows the boundaries immediately. It doesn't reach for permissions through this file. It knows when it's allowed to touch this file.

This is not documentation for documentation's sake. It's how you give the agent the context you have in your head, in a form it can ingest in one read.

4. The tight feedback loop

When you, the human, edit a file, you run the tests, see a red bar, fix it, run again. That's a slow loop for you — three round-trips in a row is annoying.

An agent does this loop twenty times in one task. The agent writes code, runs the test, reads the failure, fixes the code, runs again. If each loop takes 60 seconds, the task takes 20 minutes. If each loop takes 5 seconds, the task takes 2 minutes. The math compounds.

What makes the loop slow:

  • Test suite that takes 10 minutes to run end-to-end
  • Linter that hangs for 30 seconds before failing
  • Build step that has to compile the whole monorepo
  • Tests that need a live database to start
  • Tests that don't isolate state and so one failure cascades

What makes the loop fast:

  • Unit tests that run in <1s
  • Tests that mock external services, not real ones
  • Lint and typecheck on save (instant feedback)
  • A test runner that watches files and re-runs only changed tests
  • CI that fails in 2 minutes, not 20

You, the human, would prefer a 2-minute CI too. The agent needs it. This is the AX adjustment that pays for itself the first time you ask an agent to do something it would have abandoned at the 15-minute mark.

5. Module boundaries — the rule that's the same for both

Ousterhout's deep modules idea (which Matt cites): a module with a small public interface and a lot of complex behaviour behind it. The opposite is a "shallow" module where every internal concept leaks out.

Both DX and AX prefer deep modules. For DX: humans don't have to read 200 functions to understand the surface. For AX: the agent doesn't have to decide which of 200 functions is the right entry point for this task.

Concretely: if you're tempted to export an internal helper because "the tests need it," wrap it. The tests should import from the module's public interface, not from its internals. If they can't, that's a signal the module is doing too much and should be split.

6. Exercise: audit one module for AX

Pick your worst file, read it like an agent

  1. Open a module in your repo that an agent would plausibly be asked to modify. Something with non-trivial logic. (Not index.ts, not a config file.)
  2. Read it cold, pretending you've never seen the codebase. For each function, ask: what is this for, and when would I call it? If you can't answer in 5 seconds, the function is missing context.
  3. Check the top of the file. Is there a 5–10 line block explaining what the module owns, what it depends on, and what it deliberately does not do? If not, write one. Commit it as its own PR.
  4. Run the tests for this module alone. Time them. If they take more than 10 seconds, find the slowest test and ask why.
  5. Hand the file to Claude Code with the prompt: "Add a function archiveThing(id: string) that marks a thing as archived (don't actually delete it). Don't ask me any questions." Watch what the agent does. The places it gets stuck are exactly the AX problems in your codebase.
The signal. If the agent finishes the task in one shot, your AX is good. If it asks three clarifying questions, those are the three places where the file's context was missing. If it produces code that doesn't match the rest of the module's conventions, that's a deeper-module problem. None of these are the agent's fault — they're the codebase's fault, and yours to fix.

7. The strategic-programming layer

This is the strategic-programming move from lesson 0001 in concrete form. Tactical programming is "make this change work." Strategic programming is "make this change easy for the next person — or the next agent — to make too." AX is just strategic programming with a specific reader in mind.

When you add the top-of-file context block, you're not adding documentation. You're investing in the harness. When you tighten the feedback loop, you're not optimizing CI. You're investing in the harness. When you extract a deep module from a shallow one, you're not refactoring for fun. You're investing in the harness.

These investments compound. The next agent task on the same module moves faster. The task after that, faster still. That's the bet Matt is making: that the senior dev who quietly raises the AX of their codebase will out-ship the senior dev who chases the next model.

Quiz

  1. What's the difference between DX and AX?
  2. True or false: a codebase with great DX automatically has great AX with no additional work.
  3. What's the cheapest single change you can make to a file to raise its AX?
  4. An agent's task is taking 20 minutes when you expected 2. The agent is doing 60 round-trips of "write code, run test, read failure." What's the most likely cause?
Answers (try first)
  1. DX is the experience of a human developer working in the codebase. AX is the same idea applied to an AI agent — a reader who has no prior context, no implicit knowledge, and no instinct for what "the way we do things here" means.
  2. False. Most DX practices transfer, but two are extra-important for AX: top-of-file context docs and tight feedback loops. These need specific attention.
  3. Add a 5–10 line block at the top of the file explaining what the module owns, what it depends on, and what it deliberately does not do.
  4. The feedback loop is too slow. Each round-trip costs seconds; over 60 round-trips, that's the difference between 2 minutes and 20. Fix the test suite, mock the external services, get the loop under 10s.
