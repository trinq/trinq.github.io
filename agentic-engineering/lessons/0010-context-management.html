Context Management: The Silent Killer

Lesson 0010 · ~15 min · The #1 failure point in production agents, and five strategies to fix it

Deepening Source: Anthropic context engineering guide, JetBrains Research (Junie), ACON framework (arXiv 2025), Stanford "Lost in the Middle" (TACL 2024)

Lesson 0009 named 12 harness components. When I asked which component causes the most agent failures, the answer was clear: component 4, context management. This is the one that fails silently — the agent doesn't crash, it just gets progressively dumber as the context window fills with low-signal tokens.

This lesson goes deep on context management alone. By the end, you'll understand why your agent degrades in long sessions, name the five production strategies to fix it, and know which one to reach for in each situation.

1. The problem: context rot

Context rot is the degradation of model performance as the context window fills with low-signal content. It's not a dramatic failure — it's a slow fade. The agent starts strong, makes good decisions, calls the right tools. Twenty turns later, it's repeating itself, ignoring instructions, and making choices that contradict its own earlier reasoning.

Two research findings explain why:

The "Lost in the Middle" effect

LLMs perform best when relevant information is at the beginning or end of the input. Content in the middle of the context window is effectively invisible — performance degrades 30%+ on retrieval tasks when key facts fall in mid-window positions. — Liu et al., "Lost in the Middle: How Language Models Use Long Contexts," Stanford NLP, TACL 2024

This is a U-shaped attention curve. The model pays most attention to the first tokens (primacy) and the last tokens (recency). Everything in between gets proportionally less attention. Even models with million-token windows exhibit this pattern.

This has a direct practical implication: the order of your CLAUDE.md sections matters. Critical rules should be at the top (primacy position) or reinforced at the bottom. Rules buried in the middle of a long file are in the danger zone.

The instruction-following cliff

Even when the model can retrieve facts from its context, instruction-following ability degrades as context grows. Chroma research found that instruction compliance dropped consistently as context windows filled — the model becomes harder to steer, not just harder to inform.

This explains a pattern you've probably seen: the agent follows your constraints perfectly in the first 5 turns, then starts violating them by turn 20. It hasn't "forgotten" the constraints. They're still in the window. But the model's ability to follow them has degraded because the constraint instructions are now in the mid-window dead zone, buried under 15 turns of conversation.

2. The 80/20 rule of context windows

Experienced Claude Code users treat 80% context utilization as the danger zone. Beyond this point, performance degradation becomes noticeable: the model starts repeating itself, missing constraints, and making contradictory decisions.

0–50%
Full reasoning capacity. The sweet spot.
50–80%
Gradual degradation begins. Consider compaction.
80–100%
Danger zone. Save state and restart, or compact aggressively.

Monitor this with /context in Claude Code. When you see the bar creeping toward 80%, it's decision time: compact, delegate, or restart.

3. The five strategies

Five production-proven strategies for managing context. They're not mutually exclusive — the best harnesses use several in combination.

Strategy 1: Compaction

What it does: Summarizes conversation history when approaching the context limit, replacing verbose message blocks with a concise summary.

How Claude Code does it: When compaction triggers, Claude Code preserves architectural decisions, unresolved bugs, and the current plan while discarding redundant tool outputs and repetitive exchanges. The summary is a lossy compression — you keep the signal, drop the noise.

Before compaction (~12,000 tokens)

Turn 1: User asks for auth middleware
Turn 2: Agent reads 3 files (full contents)
Turn 3: Agent proposes approach
Turn 4: User says "looks good"
Turn 5: Agent writes middleware (full diff)
Turn 6: Agent runs tests (full output, 200 lines)
Turn 7: Tests pass, agent confirms

After compaction (~800 tokens)

Auth middleware implemented in src/middleware/auth.ts. Uses JWT validation via jose library. Tests pass (7/7). User approved the approach. Remaining: add rate limiting (next task).

When to use it: Long, multi-turn tasks where you need to stay in the same session. The agent needs continuity but not the raw history.

The risk: Lossy compression means detail loss. If a subtle constraint was discussed in an early turn, the summary might drop it. Mitigation: put durable constraints in CLAUDE.md, not in conversation.

Strategy 2: Observation masking

What it does: Hides old tool outputs while keeping the record of tool calls visible. The agent remembers what it did but not the verbose output it received.

How JetBrains' Junie does it: When the agent reads a file, the file contents are available for the current reasoning step. In the next step, the harness replaces the file contents with [masked]. The tool call record stays: "read_file(src/auth.ts) → [masked]". If the agent needs the file again, it re-reads it fresh.

Turn 5: tool_call: read_file("src/auth.ts") tool_result: [287 lines of TypeScript] ← visible now Turn 8: tool_call: read_file("src/auth.ts") tool_result: [masked] ← replaced tool_call: edit_file("src/auth.ts", ...) tool_result: "File edited successfully" ← visible now

Why it works: JetBrains Research found that observation masking matches or exceeds the solve rates of more expensive LLM-based summarization on benchmarks like SWE-bench. Zero extra compute. The agent can always re-fetch if needed.

When to use it: Any task with heavy tool use (file reads, search results, command output). The tool outputs are the biggest context hogs — they're often hundreds of lines, and most of that is consumed once and never referenced again.

The connection to your setup: This is why Claude Code uses grep, glob, head, tail rather than loading full files. It's observation masking by design — the agent gets just enough to make a decision, not the entire file permanently stuck in context.

Strategy 3: Just-in-time (JIT) retrieval

What it does: Maintains lightweight identifiers (file paths, function names, line numbers) in context rather than full content. The agent fetches data dynamically when it needs to act on it.

Eager loading

Agent reads all 15 files in src/ upfront, loading 4,000 lines into context. Spends 40% of the window on file contents it may never reference again.

JIT retrieval

Agent runs grep -rn "authenticate" src/. Gets 12 matches with file:line references (~200 tokens). Reads only the 2 files it needs to modify. Context stays lean.

How Claude Code does it: The tool set is designed for this. grep and glob return lightweight pointers. head -n 50 gives a file preview without loading everything. read_file with line ranges loads exactly what's needed. The agent builds a mental model from summaries, then drills into specifics just before acting.

When to use it: Codebase exploration, debugging, any task where the agent needs to survey a large surface before focusing. JIT retrieval is the difference between an agent that reads 50 files and one that reads 5.

Anthropic's goal, stated directly: "Find the smallest possible set of high-signal tokens that maximize likelihood of the desired outcome." JIT retrieval is how you implement this. Don't load it until you need it. Don't keep it after you've used it.

Strategy 4: Sub-agent delegation

What it does: Spawns a child agent with its own context window to handle a bounded subtask. The child explores extensively (reads files, runs commands, reasons through options), then returns only a 1,000–2,000 token condensed summary to the parent.

Parent agent (main context window) │ ├── "Research the auth middleware options" │ └── Sub-agent spawns with its own 200K window │ Reads 12 files, 3 docs, runs 2 commands │ Returns: "Use jose for JWT, passport for OAuth, │ middleware goes in src/middleware/auth.ts, │ see test pattern in src/__tests__/auth.test.ts" │ (~400 tokens returned to parent) │ ├── "Implement the auth middleware" │ └── Parent acts on the summary, reads only 2 files │ └── Context window stays at 35% utilization

Why it works: The sub-agent does the expensive exploration in an isolated context window. The parent gets the compressed output. The parent's window never sees the 12 files the sub-agent read — it only sees the conclusion.

When to use it: Any task that requires significant exploration or research before a focused action. "Understand the codebase" is a sub-agent task. "Write the code" stays in the parent.

The connection to your setup: Lesson 0008's Fork and Teammate execution models are sub-agent delegation. The AFK agent itself is a sub-agent of your workflow. The pattern nests: your workflow delegates to the AFK agent, the AFK agent delegates to sub-agents for research, each returning only the compressed result.

Strategy 5: Structured note-taking

What it does: The agent writes structured notes to files (progress files, scratchpads, decision logs), then references the files instead of keeping everything in conversation history.

How it works in practice: Anthropic's "Ralph Loop" pattern uses this for long-running tasks that span multiple context windows. The agent writes to a progress file at the end of each session:

# Progress — Auth Feature
## Completed
- [x] JWT middleware (src/middleware/auth.ts)
- [x] Tests (7/7 passing)

## In Progress
- [ ] Rate limiting (next)

## Decisions
- Using jose over jsonwebtoken (smaller bundle, ESM native)
- Middleware runs before route handlers, not at app level

## Known Issues
- Token refresh endpoint not yet implemented
- Error messages need i18n

When the next session starts (or after compaction), the agent reads this file to orient itself. The file is the durable state; the conversation is the ephemeral workspace.

When to use it: Tasks that span multiple sessions or context windows. Also useful within a single long session as a way to "checkpoint" reasoning before compaction discards the details.

The connection to your setup: Your top-of-file context blocks (lesson 0004) are structured notes for the agent. Your CLAUDE.md is a structured note. The PRD (lesson 0006) is a structured note the agent reads before acting. You're already doing this — the lesson here is to do it intentionally as a context management strategy, not just as documentation.

Production harnesses combine strategies. Claude Code uses compaction + JIT retrieval + sub-agent delegation. Your AFK setup (lesson 0008) uses all five: the agent reads the issue and PRD (structured notes), greps the codebase (JIT retrieval), reads specific files (observation masking happens under the hood), delegates research to sub-agents, and runs compaction when the window fills.

5. The ACON insight: what to keep, what to drop

The ACON framework (Agent Context Optimization, arXiv 2025) ran experiments on what to preserve during compaction. Their finding:

Prioritize reasoning traces over raw tool outputs. ACON achieved 26–54% token reduction while preserving 95%+ task accuracy by keeping the agent's reasoning (why it made decisions) and dropping the raw observations (file contents, command outputs) that informed those decisions.

This is counterintuitive. You'd expect the raw data to be more important than the reasoning about it. But the reasoning is already compressed — the agent has already extracted the signal. The raw output is just evidence for a conclusion that's already been drawn.

Think of it like your own note-taking. After reading a 30-page paper, do you keep the paper in your working memory, or your notes about the paper? Your notes. They're smaller and contain the decisions, not the raw evidence.

When designing compaction in your own harness or reviewing Claude Code's built-in compaction: keep the "why" and "what I decided," drop the "what I read."

6. Practical: context hygiene for Claude Code

Based on the five strategies, here's a context hygiene checklist for your daily Claude Code use:

Context hygiene checklist

  1. Monitor with /context. Check usage before complex operations. If you're above 60%, consider whether you need to compact or restart.
  2. Use /clear between unrelated tasks. Don't carry a debugging session's context into a feature implementation. They pollute each other.
  3. Keep CLAUDE.md under 200 lines. Every line competes for attention in the primacy zone. Audit regularly: does this line still earn its place?
  4. Put critical rules at the top of CLAUDE.md. Not in the middle. Not at the bottom. The top is the highest-attention zone.
  5. Use sub-agents for research. "Explore the codebase and tell me about the auth system" should be a sub-agent task, not a main-session task. The main session gets the summary.
  6. Don't paste large files into chat. Let the agent use grep and read_file with line ranges. Tool-based retrieval is maskable; pasted text is not.
  7. Write progress files for long tasks. Before you hit 80%, have the agent write a PROGRESS.md capturing decisions made and work remaining. If you need to restart, the new session reads this file instead of the old conversation.
  8. Restart aggressively. A fresh session with a good PROGRESS.md outperforms a degraded session with full history. Restarting is not failure — it's hygiene.

7. The meta-lesson: context is a managed resource

Matt Pocock's axiom from lesson 0001 was: the harness is 50% of the work. Context management is the single biggest chunk of that 50%. It's also the component that improves the most from conscious effort.

The shift is from treating the context window as "just a chat history that accumulates" to treating it as a managed resource with a budget. Every token in the window either contributes to the next decision or dilutes it. There is no neutral content. Anthropic's goal — "find the smallest possible set of high-signal tokens" — is the north star.

What this lesson taught

  • Context rot is real, silent, and the #1 agent failure mode. It's caused by the "Lost in the Middle" effect and instruction-following degradation as context grows.
  • Five strategies to fight it: compaction, observation masking, JIT retrieval, sub-agent delegation, and structured note-taking. Production harnesses use several in combination.
  • The ACON insight: keep reasoning traces, drop raw tool outputs. Reasoning is already compressed signal.
  • The 80/20 rule: treat 80% context utilization as the danger zone. Monitor, compact, or restart before you get there.
  • Context is a managed resource, not a passive accumulation. Every token either helps or hurts.

  2. Your AFK agent is on turn 35 of a 50-turn task and starts ignoring the PRD's non-goals. The non-goals are in the PRD that was loaded at turn 1. What's happening, and what would you do?
  3. When compacting context, should you prioritize keeping raw tool outputs or the agent's reasoning about those outputs? Why?
Answers (try first)
  1. The "Lost in the Middle" effect (Liu et al., Stanford, 2024) is the finding that LLMs attend most strongly to content at the beginning and end of the context window, while content in the middle gets 30%+ less attention. Place your most important instructions at the top of CLAUDE.md (primacy position) and reinforce them in the current user message if needed (recency position). Never rely solely on mid-window placement.
  2. The PRD's non-goals were loaded 34 turns ago and are now in the mid-window dead zone. The agent can still technically "see" them, but instruction-following has degraded. Three fixes: (a) have the harness re-inject non-goals into the current turn's context when they're relevant (JIT retrieval of the constraint), (b) make non-goals machine-checkable via a pre-commit hook so they're enforced by rules not memory (lesson 0009, component 9), (c) trigger compaction but explicitly preserve the non-goals in the summary.
  3. Keep the reasoning, drop the raw outputs. ACON research showed 26–54% token reduction with 95%+ accuracy preserved using this approach. The reasoning is already compressed signal — the agent has extracted what matters from the raw data. The raw data (file contents, command outputs) can always be re-fetched via JIT retrieval if needed.
