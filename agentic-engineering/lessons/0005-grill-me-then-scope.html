Grill Me, Then Scope It

Lesson 0005 · ~11 min · The procedure that turns a wish into a queue item

Procedural Source: derived from Matt Pocock's Agentic Engineering Workflow at 11:40; pattern names from Matt's repo

You have a clean harness (lesson 0001). You know which skills are procedures and which are abilities (0002). You think in queues, not loops (0003). Your codebase has decent AX (0004). The only thing left is the moment you actually start work: a vague request lands in your inbox. "We need a notification system." "Make the dashboard faster." "Add SSO."

Lesson 0003 said: the agent needs a well-scoped task. Lesson 0003 did not say: how do you produce one, in 5 minutes, without going to 12 meetings?

Matt's answer, in the talk, is to lean on a procedure skill called /grill-me. The agent becomes an adversarial interviewer. You answer. The conversation turns a half-formed wish into a queue item that's specific, bounded, and verifiable. This lesson teaches you what the skill does, why it's a procedure (not an ability), and how to write your own version.

1. The problem in one sentence

The gap between "I want to add SSO" and "a one-PR change to auth.ts that integrates with Okta SAML and adds a login button to LoginPage.tsx" is huge, and the gap is where most failed agent tasks live. An agent handed the first version will spend 30 minutes asking what you meant and produce nothing.

You, the human, can fill the gap. But you can also avoid the gap by never letting yourself hand the agent the first version. The work is turning the first version into the second version before you hand anything over.

That's what /grill-me does. It gives you a structured way to interrogate your own request until you've made it well-scoped.

2. The skill, in skeleton

# ~/.claude/skills/grill-me/SKILL.md

---
name: grill-me
description: Interview the user adversarially about a plan or design until the design is well-scoped enough to hand to an agent.
disable-model-invocation: true
---

# Grill Me

You are an adversarial interviewer. The user has a request — a feature,
a refactor, a bug. Your job: ask hard questions until the request is
well-scoped (specific, bounded, verifiable).

Rules:
1. One question at a time. Never batch.
2. Push back on vagueness. If the answer is "TBD", ask what would
   change the answer.
3. After 5 questions, write the queue item yourself and ask the
   user to confirm or correct.
4. The queue item must end with: <done means> + <not done means>.

Read it again. Notice: disable-model-invocation: true. This is a procedure (lesson 0002). The model only runs it when you type /grill-me. The skill does not exist to be triggered by the model on its own — it exists to be triggered by you when you have a request you don't yet know how to scope.

3. The four kinds of questions

The skill works because the questions are constrained. Without a constraint, "adversarial" becomes "rude" and the user disengages. The four kinds of questions are:

Question kind What it forces Example
Concrete Replace abstraction with specifics "You said 'fast' — in milliseconds, for which screen, in which percentile?"
Boundary Find what's in scope and what isn't "Does this touch the mobile app, or just web?"
Inverse Define done by defining not-done "If the user sees X, is the feature shipped or not?"
Risk Find the failure mode that would force a redo "What breaks if we ship this and it's wrong?"

One question at a time (rule 1) is the constraint that keeps this from feeling like an interrogation. The user answers one thing, the agent pushes on the answer, repeat. Five questions in, most requests have crystallized.

4. The "done means / not done means" ending

Rule 4 is the one that converts a conversation into a queue item. The agent is required to end with two sentences:

  • Done means: ...
  • Not done means: ...

This is the test for "verifiable" from lesson 0003. If you can write the two sentences clearly, the agent can test its way to done. If you can't, the task isn't ready to hand off yet.

Example, for the SSO request:

Done means: A user with a valid Okta account can click "Sign in
with Okta" on /login, complete SAML auth, and reach /dashboard
with a session cookie set. Authenticated via the new
oktaAuthProvider.ts module. Old email/password login still works.

Not done means: User can complete the flow but no session cookie
is set. Old login is broken. Tests don't pass. PR touches more
than 3 files outside of auth/ and the login page.

That last line ("PR touches more than 3 files") is a boundary clause — a hard limit on the scope. If the agent's PR would have to violate that boundary, the task is too big and should be split. The "grill me" conversation should have caught that earlier, but the done/not-done sentence catches it late.

5. Why it's a procedure, not an ability

Lesson 0002's rule: decision-shaped skills that change the workflow should be procedures. /grill-me is the purest case.

  • The act of running the skill is a decision you make consciously.
  • The output is a queue item that the rest of the workflow depends on.
  • The cost of letting the model invoke it at will is high — it would interrupt every conversation to ask you clarifying questions.

If you make /grill-me an ability, you'll see the model invoke it mid-task when it's confused, and the result is the agent turning your one-line prompt into a 12-question survey. That's the wrong direction. The right direction is: you see a vague request, you invoke /grill-me explicitly, you answer 5 questions, you get a queue item, you put it on the queue (lesson 0003), you hand it to an AFK agent.

6. The contrast: features vs bugs

The skill is most useful for features, where the user typically has a vague wish. For bugs, the situation is often reversed: the user has a specific symptom ("login is broken") and the grill-me process is shorter. For refactors, the questions are mostly boundary questions: "what's in scope, what's out, what's the test that proves the refactor preserves behaviour?"

Matt's /create-issues skill (mentioned in the same repo, also a procedure) is the step after: take the queue item the grill-me produced and turn it into a GitHub issue. The two skills chain. Grill me first, then create-issues, then the AFK agent picks it up.

7. Exercise: grill yourself, once

Pick one vague request, run the four question kinds on it

  1. Pick one item from your current backlog that you've been avoiding because it's vague. Write the one-line version here, before you read further. (If nothing is vague, pick any "improve X" task and treat it as the prompt.)
  2. Run the four question kinds in order. Concrete → Boundary → Inverse → Risk. One question at a time. Don't batch.
  3. After the four questions, write the done-means / not-done-means pair. If you can't, that's a signal the request is still vague — ask one more risk question and try again.
  4. Open Claude Code, type /grill-me, paste your one-line request, and let the agent drive the same process. Compare its questions to yours. Notice what it caught that you missed.
  5. Take the resulting queue item and add it to your real issue tracker. (If you don't have a tracker, the next lesson covers the lighter-weight version of this workflow.)
The surprising thing. Most people who try this discover that the first 2–3 questions they ask themselves are surface questions ("what does the user see?"). The useful questions are risk questions ("what breaks if we ship this wrong?"). Risk questions are where the real scope hides. If you only do one kind of question, do risk.

8. The whole sequence, in one diagram

3. Queue item    →   specific, bounded, verifiable

4. /create-issues    →   GitHub issue filed

5. Issue goes to the queue (lesson 0003)

6. AFK agent picks it up, opens PR

Steps 1–3 are this lesson. Steps 4–5 are the next two lessons. Steps 6–7 are AFK agents, which is lesson 0008. You're roughly halfway through the workflow. The earlier lessons were about the harness and the codebase; the next three are about the agent loop itself.

Quiz

  1. What frontmatter flag should /grill-me have, and why?
  2. Name the four kinds of questions the skill should ask.
Answers (try first)
  1. disable-model-invocation: true. It's a procedure — you invoke it when you have a vague request. Letting the model invoke it on its own would cause it to interrupt every task with clarifying questions. The cost of an ability that "interviews you" is unbounded context and broken flow.
  2. Concrete (force specifics), Boundary (find scope), Inverse (define done by defining not-done), Risk (find the failure mode).

What to do next

  1. Run the exercise above on one vague request. Put the resulting queue item in your tracker.
  2. The next lesson is writing a PRD — the procedure skill that turns a grill-me queue item into a design document the agent can read to understand context the code doesn't show. PRDs are how the AFK agent gets the "why" the codebase can't tell it.
  3. Ask your teacher anything that wasn't clear. /grill-me is the most-used procedure in Matt's setup, and the one with the widest quality range between a good and bad version — small wording changes matter a lot.
