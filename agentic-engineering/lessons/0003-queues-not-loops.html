Queues, Not Loops

Lesson 0003 · ~10 min · How to model your work so agents can pick it up

Structural Source: Matt Pocock's Agentic Engineering Workflow at 25:40, 26:00, 27:15, 43:35

There is a version of AI-assisted development that looks like this: you write a prompt, the agent writes some code, you check it, the agent fixes it, you check it again, and you keep doing this until the work is done. A while-loop with a human inside it.

Matt calls this the loop pattern, after the Ralph Wiggum coding pattern from 2025 that made it viral. It's seductive because it feels productive — the agent is always working. But it's also the pattern that makes you feel like you're babysitting the agent instead of working with it.

The alternative Matt describes is the queue pattern: your work is a backlog of well-scoped tasks. You add items. Agents pick them up when they're ready. Items come off the queue when their pull request merges. That's it. No loop. No babysitting.

This lesson teaches you to stop thinking in loops and start thinking in queues — and gives you the structural setup to make that switch without losing control.

1. Why the loop pattern feels good and performs badly

Here's what a loop session looks like in practice. You start a Claude Code session and ask it to build a feature. It writes code. You review it. It asks a question. You answer. It writes more code. You review again. Three hours later you have a feature and you're exhausted, because you were in the loop the whole time.

The loop pattern isn't wrong — it's just expensive in a specific way. Every time the agent needs you to answer a question, make a decision, or grant a permission, the loop has to pause. If you're the kind of senior dev who has strong opinions about module boundaries, naming, and test strategy, you're going to be in the loop a lot. The agent is fast; you're the bottleneck.

The moment that I discovered AFK was the moment I really got into AI coding and the moment I was really able to massively increase my output because then instead of me having to sit in the loop, handle all the permissions requests… I've paralyzed myself. Suddenly there were two of me, you know, or three of me, four of me, five of me. — Matt Pocock, Agentic Engineering Workflow, 43:35

"Paralyzed myself." That's the loop tax. Every round-trip is a context switch, and if you're the one doing the round-tripping, the agent isn't really working for you — you're working with it, which is a different thing.

2. The queue model, illustrated

Your queue (backlog):
  [ ] Refactor auth module into a service class
  [ ] Add rate limiting to the /api/search endpoint
  [ ] Write integration tests for payment flow
  [x] Update README with new environment variables  (PR merged)
  [ ] Write a design doc for the notification system

Agent picks up: one task at a time
  ↓
Agent works AFK: sandbox, git push, PR opens
  ↓
You review PR — you’re back in only when it matters
  ↓
PR merges — task comes off the queue

The key property: the agent works on one task at a time, in a sandbox, without you in the loop. You come back for the PR review. The review is a focused, bounded decision — does this PR solve the task? Yes or no. Much less cognitively expensive than sitting in the loop for three hours.

That's all development is really. You just have a queue of tasks that you need to get done. Project managers add more stuff to the queue. You complete the tasks in the queue. — Matt Pocock, 26:00

3. The scoping requirement

The queue pattern only works if the tasks are well-scoped. A well-scoped task is one the agent can pick up and complete without asking you a question it doesn't already know the answer to.

The contrast is an ill-scoped task: "improve the auth system." The agent will spend the first 20 minutes asking you what "improve" means, what the current state is, and what "good" looks like. That's you in the loop again, just slower.

Well-scoped tasks look like this:

  • Specific: "Refactor auth.js to extract a TokenService class with generate() and validate() methods" — not "improve auth."
  • Bounded: one module, one feature, one fix — not "rewrite the frontend."
  • Verifiable: the agent can write a test that proves the work is done — not "make it better."

Note what's in that list: not "small." A task can be large as long as it's specific, bounded, and verifiable. The Ralph loop works best for small tasks precisely because the scope is tight. The queue works for tasks of any size as long as the scope is clear.

4. The medieval kingdom framing

If you deployed a minister into some region, far region and you never heard from him… ultimately as the king in that medieval kingdom, you want to do the Q approach. You want to have people come to you and say we have a problem — there's an upcoming invasion, or there's a famine in this region — and you have this queue of problems, and you are still in charge. — Matt Pocock, 27:15

This framing is worth sitting with. In the loop pattern, you're inside the system. You see every decision as it happens. You feel productive but you're actually executing, not directing. In the queue pattern, you're the king. The ministers — your agents — go out, do the work, and come back with a result. You make the decisions that matter: which task to prioritize, what to do about the PR, when to change direction.

The skill of being a good king is different from the skill of being a good worker. Matt is explicitly teaching the king skill here, not the worker skill.

5. What the queue actually is in practice

For most solo devs, the queue lives in their issue tracker. GitHub Issues, Linear, Trello — doesn't matter. The structure that matters is:

Queue element What it contains Who adds it
Task Well-scoped description (specific, bounded, verifiable) You or the agent
State needs-scoping → ready-for-agent → in-review → done You
Context Links to relevant files, prior decisions, the original PRD You (when scoping)
Exit criteria What "done" looks like — tests pass, PR open, etc. You (implicit in well-scoped tasks)

The state machine above is Matt's /triage skill in practice. Tasks in needs-scoping are not ready for the agent — they're waiting for you to make them well-scoped. Tasks in ready-for-agent are fair game for AFK agents. Tasks in in-review are the moment you come back into the loop.

6. Exercise: Audit your current work model

Find out if you're looping or queuing

  1. Open your terminal and run a mental experiment: what is the longest you've ever been in a Claude Code session, actively answering questions and making decisions? Write that time down.
  2. For each session in the last week, ask: was I scoping work, reviewing work, or executing work? If you were executing — answering the agent's questions, approving its changes, guiding its decisions — you were in the loop. That's the tax.
  3. Pick one task you did this week. Write it as a queue item: specific, bounded, verifiable. Does it fit in one PR? If yes, it's a well-scoped queue item. If no, what's missing from the description?
  4. This week, before starting any new work, write the queue item first. Not a prompt — a task description. Then hand it to the agent and step away. See what comes back.
The goal. A healthy solo dev session looks like: (1) you scope a task in 5 minutes, (2) you hand it to an AFK agent and go do something else, (3) you come back to a PR in 20–60 minutes, (4) you review and merge. If your sessions look like this, the queue pattern is working. If you're still in the loop for most of it, something about the scoping broke down.

Quiz

  1. What's the main cognitive cost of the loop pattern for a solo senior dev?
  2. Name the three properties of a well-scoped queue item.
  3. True or false: a queue item can be large (many files, many changes) as long as it's specific, bounded, and verifiable.
  4. What's the state a queue item should be in before an AFK agent picks it up?
Answers (try first)
  1. The solo dev is the bottleneck. Every agent question, decision, and permission request requires a context switch from the human. The agent is fast; the human is the tax.
  2. Specific (clearly describes the change), Bounded (one module/feature/fix), Verifiable (the agent can test its way to done).
  3. True. Size is not the scoping variable. The Ralph loop works for small tasks because they're tightly bounded by accident — a large task is fine on the queue if the scope is explicit.
  4. ready-for-agent (or equivalent). Tasks in needs-scoping are not ready — the agent will ask questions that bring you back into the loop.

What to do next

  1. Audit one past session. Write it as a queue item. Notice what's missing when you try to scope it.
  2. Next time you start a session, write the queue item first, then hand it to the agent.
  3. The next lesson is DX and AX — how to design codebases that agents love working in, which is the other half of making the queue work. A perfectly scoped task still fails if the codebase gives the agent a bad experience.
  4. Ask your teacher anything that wasn't clear. The queue pattern takes practice — the first few times you try to hand off a task and step away, you'll feel the urge to check in. That's normal.
