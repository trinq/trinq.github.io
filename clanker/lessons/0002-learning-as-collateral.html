Learning as the Collateral

The Tobi review that taught Shopify what was actually worth keeping.

The Tobi story is the most concrete moment in the whole talk. It is also the one you should be ready to retell when a teammate asks "why are we throwing away code that works?" Farhan tells it at 03:13.

The story

Pre-LLMs. Shopify has a team of about 50 engineers. They spend 18 months building a feature for merchants. They are about to launch. Tobi — the founder and CEO — walks through the architecture with them. Approves it.

As the team is walking out the door, Tobi stops them:

"If you could start over, how would you build it?"

Someone walks to the whiteboard. They describe a completely different architecture. Simpler. Tobi says: "Why don't you build that? Just delete what you did and start over."

What he was trying to show: the learning was the collateral, not the code.

What "collateral" means here

Collateral is what you get to keep when you give everything else back. In a loan, collateral is the house. In an 18-month engineering project, Farhan argues, the collateral is not the codebase — it is the team's understanding.

Shopify did not actually delete the code (that is too hard). They invented a process: the team was allowed to look at the old code, but not cut and paste from it. They rebuilt the system in 3 months. The result was a simpler, more elegant solution — because the second build was informed by everything the first build had taught them.

Why this lands now

Farhan returns to the same idea at the end of the talk, at 25:22. He reframes it as a weekly cadence:

"You know how some people talk about shipping every week? We talk about learning every week. What is it that you learned this week? What is unintuitive? What is a secret that you believe that others don't because you looked at the data, you've tried and prototyped something, you put something in front of a customer?" — Farhan Thawar

The pattern: ship-or-learn collapses into just-learn, because shipping got cheap.

The before / after frame

Before — code is the artifact
  • Code in main is the deliverable
  • Rewrites are failure, sunk cost
  • Engineers protect their PRs
  • Throwaway code is "waste"
  • Success = lines shipped
After — learning is the artifact
  • The team's mental model is the deliverable
  • Rewrites are expected, sign of learning
  • Engineers protect their insights
  • Prototype code is fine — it's disposable
  • Success = what the team now understands

Farhan makes the AI connection explicit at 25:22: when AI is writing the code, "you can use these tools to build whatever you want as quickly as you want." Which means code is no longer the scarce resource. Understanding is.

The walk-out-the-door question

The Tobi story contains a small detail that matters: the question came after approval, as the team was leaving. That is on purpose. The question is not "is this build correct?" — that's been answered. The question is: "now that you know what you know, would you do it differently?"

That second question is what produces learning. It only works if the team is honest. It only matters if the team is willing to throw away the approved work.

The trap: most teams never ask this question because they cannot afford to throw away 18 months of work. The pre-LLM reality made the question theoretical. AI makes it cheap enough to be a real practice — and that is the shift Farhan is pointing at.

How this maps to your team

You will not run the exact Tobi experiment with your team. But the principle has a direct application. Three concrete moves you can make this month:

  1. Build prototypes and throw them away on purpose. Frame prototypes as learning vehicles, not commit candidates. If a prototype accidentally becomes the real implementation, you've skipped the most useful step.
  2. Reward rewrites when they are informed. The 3-month rebuild at Shopify worked because the team had the 18-month learning to draw on. Rewrites that happen after deep work are not waste — they are evidence that the team learned.
  3. Run the walk-out-the-door question in reviews. After a design doc is approved, before implementation starts, ask: "Knowing what we know now, would we write this doc the same way?" Treat the answer as a learning artifact, not a critique of the original.

The "LGTM is the four scariest letters" connection

Farhan makes a related claim at 20:46: humans were not actually reviewing code well before AI entered the picture. The default PR approval is "LGTM" and the code may still be broken.

Why does this matter for the learning-as-collateral frame? Because if humans were not extracting learning from code review, then code review was not generating the collateral either. It was just a ritual.

The Council of LLMs at Shopify — multiple AI judges looking for different things — is partly a recognition that the human review process was not the source of learning people thought it was. The learning has to come from somewhere else: planning, prototyping in front of customers, the walk-out-the-door question.

Exercise: Run the walk-out-the-door question

  1. Pick one design doc your team has approved in the last 30 days. (If you don't have design docs, pick a sprint goal or a PRD.)
  2. Schedule a 30-minute slot with the people who worked on it. Frame it explicitly: "this is not a critique, this is a learning exercise."
  3. Open with: "Knowing what we now know, would we write this the same way? What would change?"
  4. Write down what surfaces. Do not commit to any action. The output is the artifact.
  5. At your next team retro, share one observation from the exercise. Ask the team: what does this tell us about our learning?

What you should walk away with: a one-line answer to "what should we keep when we throw away code?" — the team's mental model. And a small practice (the walk-out-the-door question) you can introduce this month without asking for permission.

Ask follow-up questions

  • "How do I run the walk-out-the-door question without it feeling like a gotcha?"
  • "What if the answer is 'no, we'd build it the same way' — is that failure?"
  • "How does this connect to my team's existing retro format?"
  • "Show me how this would land differently in a team that has just shipped something vs a team that has been blocked for months."
