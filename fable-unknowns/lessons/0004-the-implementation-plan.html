The Implementation Plan

L4 Pre-implementation closeout · ~10 min · Lesson 4 of the Finding Your Unknowns series

Lessons 1–3 got you from a fuzzy problem to a chosen direction. Lesson 4 turns that direction into a doc Fable can execute against. This is where the pre-implementation phase closes. Once the plan is written, you start a fresh session, paste it in, and let Fable run.

The plan is the most underestimated artifact in agentic coding. Skipped, the implementation drifts: surprises aren't surfaced, decisions get re-made, the finished work looks like a first draft. Written well, the plan lets Fable move fast without re-asking you — the doc answers the questions Fable would otherwise have to guess at.

The shape of a useful plan

Wood's one-liner on the format: "write an implementation plan in HTML, but lead with the decisions I'm most likely to tweak with — data model changes, new type interfaces, and anything user-facing. Bury the mechanical refactoring at the bottom; I trust you on that part." Three principles:

  1. Lead with what you'll want to change

    Above the fold of the plan doc — the first thing you read — goes the load-bearing decisions. Things you'd want to revise after reading the implementation: data model, type interfaces, public API, UX flows, anything user-facing. These are the sections you'll re-read three times during implementation.

  2. Bury the mechanical

    Renames, file moves, test scaffolding, internal helpers — all the work that's tedious to read but doesn't need your judgment. Put these at the bottom. Wood's "I trust you on that part" is doing real work: it tells Fable not to ask about these decisions at all.

  3. Write it in HTML

    Not markdown. The plan lives alongside your other lesson artifacts; matching the format means it gets read like the rest of the docs. Wood: "I've previously written about using HTML with Claude, in almost all of these cases, a HTML artifact is the best way to visualize and represent it."
Why HTML over markdown
A markdown plan embeds into the chat scroll and disappears. An HTML plan is addressable, can be linked, has structure you can skim, and rounds out the artifact set the same way each lesson does in this workspace.

What goes at the top of the plan

Your tweak-points are not all the same. Rank them. The order in the plan is the order you'll re-read during implementation:

Section What it covers Why it goes first
Data model Schema, migrations, indices, relationships Wrong here means a costly rollback. The plan should describe the model as if you were explaining it to a new teammate — names, types, the invariants.
Type interfaces / API shape Function signatures, request/response shapes, error contracts Public surface. Changing a public type after the fact is a breaking change.
UX flows Page flows, edge cases, error states, empty states, loading states Anything user-facing lives here. The Unknown Knowns from Lesson 3 are usually answered in this section.
Behavior of uncertain edges Race conditions, retries, idempotency, concurrency These don't always fit "data model" but are load-bearing — they go in the same band as the model.

The pattern: tweak-points are above the line; mechanical is below. If you read the plan top-to-bottom and want to skip ahead, the parts you'd skip are the parts you should bury.

The replayable plan

Two readers will look at your plan:

  • Fable, during execution, treating the doc as a spec.
  • You, two weeks later, trying to remember why you decided what you decided.

A good plan serves both. Three properties make that possible:

  1. Decisions are explicit, not inferred

    "Store users in a Postgres users table, indexed by email; soft-delete via deleted_at" beats "users live in Postgres." The first makes the decisions searchable; the second makes them guesses.

  2. Cross-references are URLs, not prose

    "See Lesson 2's section 4 for the prototype" is fine in chat; in a doc it's better as an anchor link. Fable can navigate it; you can re-find it.

  3. The plan names the verification step

    "Tests pass" or "the dev server responds with 200 on /api/users" or "the prototype renders identically in Safari and Chrome." The verification is the spec's exit condition. Without it, the plan isn't done — it's a draft.
The plan is a deliverable
Wood calls the plan out as separate from the work: "I make a new session and pass any artifacts to the prompt." The plan is one of the artifacts. It travels with the prototype, the spec, and the implementation notes. You're handing Fable a small bundle — not a chat history.

Decisions you might want to revise

Lesson scope — one lesson, not two

L5 covers Wood's "during" (implementation notes) and "post" (pitch, explainer, quiz) in a single lesson. This was the L3 rehearsal's commitment (Direction B in the brainstorm). It is revisable: if you decide the post-material wants its own lesson (because pitch and quiz are richer than I'm modeling here), L5 splits and L6 is born. The cost: ~10 more minutes of reading and an extra quiz.

Unifying claim — the lesson's thesis sentence

"Every surprise during execution deserves a log line; every shipped change deserves an artifact the next person can pick up." This sentence appears near the top of the lesson and is what carries the connection between the two halves. If it doesn't land, the lesson feels like two separate topics.

The implementation-notes artifact format

Two options under consideration: a literal implementation-notes.md file (Wood's example uses .md), or a section of the lesson itself that shows a representative notes file. I'm leaning toward the literal .md because it makes the technique reusable across projects; the lesson then shows what a representative one looks like. **This is the most likely thing you'll want to revise.** Decide which.

The pitch artifact — in-lesson or separate?

Wood's post-implementation artifact is a separate HTML page (the "packaged doc for Slack"). But the lesson in this workspace inherits the convention that artifacts live inside lessons. Three options to consider:

  1. Embed the pitch as a section of L5 (matches workspace convention; loses the "reviewers can paste this in Slack" property).
  2. Produce a separate reference/ship-and-close-pitch.html (matches Wood; inconsistent with sibling workspaces).
  3. Show a pitch template in L5 plus the actual lesson's working pitch as an artifact — the lesson teaches the template, references the artifact.

My recommendation: option 3. Templates > artifacts for lessons, artifacts > templates for shipped work. Decide which.

The quiz on the shipped change

Wood: "I only merge after I pass the quiz perfectly." L5 needs a quiz that tests whether the user understands what happened, not just what they shipped. Two design decisions:

  • How many quiz questions? Three (the minimum to validate understanding across multiple cells of the 2×2) or five (L4 had five; L1–3 had four each).
  • Quiz on the lesson's example pitch (synthetic, easy to grade) or on the user's own work (real, but harder to validate in a self-paced lesson)?

My recommendation: synthetic quiz on the lesson's own example pitch (the "color grading launch video" Wood uses), grounded in the post-quiz Wood describes for the Fable launch video. Decide.

Reading this back

Notice four things about the plan above:

  1. The first paragraph of each section is a decision, not an instruction

    "Lesson scope — one lesson, not two" is the decision. The body of the section explains why and what's revisable. If you start with "Write a lesson that..." you've written a task description, not a plan.

  2. The decisions I labeled "most likely to revise"

    Three sections: notes artifact format, pitch format, quiz design. These are where I'd cut if I had to shorten the lesson. They're also where I'd patch first if Fable's output didn't match my expectations.

  3. The mechanical section is a checklist, not a paragraph

    No prose around file renames and footer updates. The work is enumerated. Fable can take the list and do it; I don't have to re-decide any of it.

  4. The verification is the spec's exit condition

    "Open the HTML, the styles render, the quizzes respond" are specific, observable, testable. The plan is done when the verification passes — not when Fable says it's done.
If the plan needs prose to explain the mechanical
The mechanical should be a list. If you're writing prose around file renames, the prose is hiding a decision you should have surfaced.

How the four pre-implementation lessons compose

Lessons 1–4 form a sequence. Each closes a phase:

  1. L1: Map vs. Territory

    Mental model. The 2×2 of unknowns. The iterate-don't-plan loop.

  2. L2: Blindspot pass + Interview

    Populates the Unknown Unknowns cell. The two passes: broad, then narrow.

  3. L3: Brainstorm + Prototype + Reference

    Populates the Unknown Knowns cell. The "see something before you commit" pattern.

  4. L4 (this): The Implementation Plan

    Synthesizes everything. Turns the chosen direction into a doc Fable executes.

After L4, pre-implementation is done. You start a new session, paste the plan (and prototype, if any), and let Fable run. That's where Wood's "during implementation" lessons take over — Lesson 5.
