Brainstorm, Prototype, Reference

L3 Pre-implementation · ~14 min · Lesson 3 of the Finding Your Unknowns series

Lesson 2 taught you how to surface Unknown Unknowns — things you haven't even considered. This lesson teaches the techniques for the cell next to it: Unknown Knowns — things you know so deeply you'd never write them down, but you'd recognize the wrong version of instantly.

The unifying idea is simple and uncomfortable: you have to see something to know what you want. Most Unknown Knowns can't be articulated in a prompt. They can only be recognized when shown a candidate. So the technique class for this cell is "produce a candidate, react to it, refine."

Wood offers three moves in this cell: brainstorm to widen the option space, prototype to make one direction concrete enough to react to, and reference to point at an already-good version someone else made. Each is cheap. Each runs before any wiring happens.

1. Brainstorm — widen before you write

A brainstorm is a request for Fable to produce multiple directions for the same problem, in a format you can react to. The reaction is the point — you are gathering options you wouldn't have generated yourself, so you can prune them with judgment you already have but can't articulate.

The pattern that matters

I want a dashboard for this data but I have no visual taste and don't
know what's possible. Make me an HTML page with 4 wildly different design
directions so I can react to them.

Three things to notice:

  1. "Wildly different"

    This is the load-bearing adjective. Without it, Fable produces four minor variations on a theme — the same layout, the same color palette, the same density. With it, Fable reaches for directions that don't share a skeleton: a spartan data-dense table; a tile-based magazine; a story-driven narrative; a controls-and-charts engineering view.

  2. Disclose what you don't know

    "I have no visual taste and don't know what's possible" tells Fable to explain why each direction exists, not just to render four pages. The explanations are part of the artifact.

  3. "So I can react to them"

    The framing matters. You're not asking for a recommendation; you're asking for a set of stimuli. That framing changes how Fable calibrates confidence — it's showing you options, not picking one.

Why brainstorm before implementation

Wood's central claim
Small changes in a feature or spec can cause drastically different implementations in code, and it can be more difficult for your agent to revert previous changes. A brainstorm is the cheapest place to make those changes. A prototype is the next cheapest. Implementation is the most expensive.

The cost order is the discipline. You don't think from inside one direction and hope it's right. You widen first, react, narrow, and only then commit to code.

Brainstorm as scope-setter

Wood also uses brainstorming at session start, even when there's no UI question: 

Here's my rough problem: users churn after onboarding. Search the codebase
and brainstorm 10 places we could intervene, from cheapest to most ambitious.
I'll tell you which ones resonate.

This is a different use of the technique. The brainstorm isn't about visual reactions; it's about scope. The "from cheapest to most ambitious" axis gives you a way to prune. The "which ones resonate" handoff keeps you in the loop without forcing a premature decision.

2. Prototype — make one direction concrete

A brainstorm gives you options. A prototype takes one option and makes it tangible — concrete enough that you can react to specifics rather than to abstractions. Wood's worked examples are mostly visual: 

Before wiring anything up, make a single HTML file mocking the new editor
toolbar with fake data. I want to react to the layout before you touch
the real app.

The shape: one file, fake data, no backend, no state. You are isolating the unknown-known surface — the part you know when you see it but can't articulate — from everything that's already known. The toolbar's layout is the Unknown Known. The persistence layer, the state machine, the auth flow — all knowns, all distractions.

Why "single HTML file" is a useful constraint
Fable can produce prototypes in many formats: a real component file, a Figma mock, a Storybook story. Single-HTML-file is the cheapest constraint that still gives you something you can open in a browser and react to. It strips away build systems, routing, and state plumbing. The Unknown Known sits alone.

The prototype isn't a deliverable. It's a stimulus. You'll throw it away. What you keep is the *reaction*: "I like the spacing but the typography is too dense," or "this works for the home view but won't scale to the table view." Those reactions are the start of the implementation plan — which is Lesson 4.

3. Reference — point at something already good

Sometimes the Unknown Known can't be discovered by brainstorming; you only know it when you see it, and what you have in mind is an existing thing. The reference technique is: point Fable at that thing and tell it what to look for. 

This Rust crate in vendor/rate-limiter implements the exact backoff
behavior I want. Read it and reimplement the same semantics in our
TypeScript API client.

Three reasons this works better than describing the behavior in prose:

  1. Code as ground truth

    A prose description of backoff semantics is ambiguous. The Rust crate has no ambiguity — the code is the spec. The fewer interpretive steps between you and Fable, the fewer guesses Fable has to make.

  2. Cross-language reference is fine

    Wood is explicit: "you can point it at a folder and tell it what to look for, even if it's in a different language." Fable reads the code, not the syntax. The reference transfers semantics, not syntax.

  3. "Tell it what to look for"

    Unfocused reading produces shallow mimicry. Naming the specific behavior ("the exponential backoff with jitter on lines 80–120") tells Fable which part of the reference is load-bearing.
Claude Design works the same way
Wood notes that Claude Design reads the underlying markup of pages you point it at, not just the screenshot. The same principle: code is richer than pixels. When you have a reference, point at the code.

4. A live brainstorm + prototype, run on the rest of this series

The remaining lessons of this series — 4 and 5 — are themselves an Unknown Knowns problem. I know roughly what they should cover, but the *shape* of each lesson (how much content, what stays separate, what combines) is something I'll only know when I see options. So I ran a brainstorm.

B Compress during+post into one lesson Honest about content density; the reader sees fewer lessons One lesson carries implementation notes + pitch + quiz. Density.

Prototype step — what Direction B actually looks like

The brainstorm made me curious whether Direction B would survive a concrete mock. So I prototyped it: one paragraph describing what Lesson 5 would be if I combined during and post.

Lesson 5 — Shipping and Closing the Loop. The last lesson of the series. Two halves. First half: implementation notes — a small file your agent keeps during execution so surprises are logged instead of lost. Second half: the post-ship artifacts — a pitch/explainer for reviewers, a quiz for yourself. The unifying claim: every surprise during execution deserves a log line; every shipped change deserves an artifact the next person can pick up.

Reaction: that one paragraph has more content than L4 (plan) has on its own. Direction B isn't just "compress to save a lesson" — it's "the right lesson size for what these techniques actually contain." Direction A would have produced two thin lessons instead of one good one.

Decision

Direction B. The series is now 5 lessons, not 6. This is the decision the brainstorm-and-prototype was designed to produce: a commitment to a direction I could defend, made by reacting to a concrete option rather than thinking from inside the problem.

What the prototype wasn't
The prototype was a single paragraph. It did not include the actual lesson HTML. That's the discipline: prototype enough to react, not enough to ship. If I'd written the full Lesson 5 HTML, I'd have over-committed and lost the ability to change direction cheaply.

5. How the three techniques compose

After Lesson 2's blindspot pass + interview (which populated Unknown Unknowns), you have a narrower problem but a still-unclear criterion. Lesson 3's three moves populate Unknown Knowns by giving you something to react to:

  1. Brainstorm

    Multiple directions, wildly different, in a format you can react to. Best when you don't know what shape the answer should take.

  2. Prototype

    One direction, one file, fake data, no wiring. Best when you know roughly what you want but can't articulate the details until you see them.

  3. Reference

    Point at code that already exists. Best when you have a model in mind and want its semantics without its syntax.

Pick one. They don't all need to run on every task. The discipline is see something before you commit — which one of the three you use depends on what kind of "something" is available.

When you've reacted enough to commit to a direction, you're ready for Lesson 4: the implementation plan. That's where the prototype (or the chosen brainstorm direction) becomes an executable doc.

Knowledge check

Each quiz targets the Unknown Knowns cell or a trade-off inside one of the three techniques.

You ask Fable for "four directions for the editor toolbar" but you don't say "wildly different." What goes wrong?
A teammate asks Fable to prototype a dashboard by writing the real React component with mocked API responses. What does this cost them?
You point Fable at a TypeScript library and tell it to "reimplement the rate-limiter logic." Without naming what to look for, what happens?
You ran a brainstorm and got four directions, then prototyped one. The prototype changed your mind about which direction was best. What did the prototype buy you?
