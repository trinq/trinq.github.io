Glossary

Reference · ~5 min skim · One page · All vocabulary across L1–L5

The five lessons ship ~30 terms worth knowing. Each entry below gives a short definition, names the lesson(s) where the term is taught in depth, and (where the term is a technique) the literal prompt phrase Wood uses for it.

Use this page two ways: as a check after finishing the series (did I miss any vocabulary?) and as a lookup while working with Fable (what was that phrase again?). The cross-references go in both directions — click through to the lessons for technique, click back here for the one-line definition.

What's here and what isn't
In: technique names, trigger phrases, concept names (the 2×2, the time windows), and the unifying claims. Out: generic English words, terms used once and never defined in the lessons, and meta-vocabulary about the curriculum itself.

Mental model

Known Knowns
Things you know and can articulate. They go in the prompt as direct instructions. The easy quadrant. Source: L1.
Known Unknowns
Things you know you don't know. They go in the prompt as explicit questions, constraints, or "find out X before implementing" directives. Better than Unknown Unknowns because they're at least visible. Source: L1.
Unknown Knowns
Things you'd recognize the wrong version of but can't articulate up front. Visual taste, "good API ergonomics," the rhythm of a conversation. Populated by reacting to candidates — brainstorm, prototype, or reference. Source: L1, L3.
Unknown Unknowns
Things you haven't even considered. The biggest risk to shipped work because they become guesses the model makes on your behalf. Populated by a broad pass first (blindspot) and a narrow pass second (interview). Source: L1, L2.
Map vs. Territory
The core mental model. The prompt + context you hand the model is the map. The codebase + real-world constraints it has to navigate are the territory. The series teaches you to tighten the map when they diverge. Source: L1.
The 2×2
The four-quadrant frame (Known/Known × Known/Unknown). The single most useful diagnostic: where in the grid does this task sit? Different cells warrant different techniques. Source: L1.
Iterate, don't plan
The core insight from L1: most prompts are under-iterated, not under-planned. The cost of one more loop (blindspot, brainstorm, prototype) is much lower than the cost of getting the prompt wrong on the first try. Source: L1.

Time windows

Pre-implementation
Before any code is written. The window for L2 (blindspot + interview) and L3 (brainstorm + prototype + reference). Also where L4 (the plan) closes the window with a deliverable. Source: L1–L4.
During implementation
While the implementation runs. The window for implementation notes — log surprises, deviations, judgment calls. Source: L5.
Post-implementation
After the code ships. The window for the pitch (artifact for the next person), the explainer, and the post-quiz. The user passes the quiz before merging. Source: L5.

Techniques — pre-implementation

Blindspot pass
A broad, low-stakes pass that surfaces Unknown Unknowns before any code is written. Cheap to run; high payoff if it surfaces anything. Source: L2.
Interview
A narrow, targeted pass that asks one question at a time. Used after the blindspot pass to resolve residual ambiguity. The one question at a time framing matters — multi-question prompts produce thinner answers. Source: L2.
Brainstorm
A request for multiple directions on the same problem, in a format you can react to. The "wildly different" adjective is load-bearing — without it, the result is four minor variations on one theme. The reaction is the point. Source: L3.
Prototype
A single file (typically HTML) with fake data and no wiring, made to isolate the unknown-known surface from the rest of the system. The artifact is a stimulus, not a deliverable. Source: L3.
Reference
Pointing Fable at existing code (or a Claude Design page) that demonstrates the desired behavior or style. Code is richer than pixels; cross-language references transfer semantics. Source: L3.
Implementation plan
The synthesis artifact that turns a chosen direction into a doc Fable can execute. Wood's ordering: lead with what you'll tweak (data model, type interfaces, UX), bury the mechanical. Source: L4.

Techniques — during implementation

Implementation notes
A literal implementation-notes.md file (or folder) kept open during execution. Logs surprises, deviations, judgment calls. Functions as a drift detector against the plan. Source: L5.
Surprises and decisions
Section of the notes file. Per-execution entries that capture what happened and why. Judgment-call entries are written by the user, not the agent. Source: L5.
Deviations
Section of the notes file. What changed vs. the plan, and why. The drift detector against L4. Source: L5.

Techniques — post-implementation

Pitch
An HTML artifact (in the project's reference/ folder) a teammate can paste into Slack to understand what shipped. Four blocks: lede, problem, differentiator, push-back. Source: L5.
Explainer
A longer-form artifact (often a separate doc or video) for deeper context on the shipped change. Distinct from the pitch: the pitch is for quick context, the explainer is for full context. Source: L5.
Post-quiz
A short quiz the user takes before merging the work. Tests judgment on what the user just shipped, not vocabulary about the project. Categories: what surprised you, what would you change next, what would you push back on. Source: L5.

Trigger phrases

These are literal prompt phrases from Wood. Paraphrasing produces thinner output — the literal phrasing is load-bearing.

Trigger phrase Technique Lesson
"Run a blindspot pass on this." Blindspot pass L2
"Interview me one question at a time." Interview L2
"Make me an HTML page with 4 wildly different design directions so I can react to them." Brainstorm L3
"Before wiring anything up, make a single HTML file mocking X with fake data." Prototype L3
"Look at <reference> and reimplement X. Tell me what to look for." Reference L3
"Write an implementation plan in HTML, but lead with the decisions I'm most likely to tweak with — data model changes, new type interfaces, and anything user-facing. Bury the mechanical refactoring at the bottom." Implementation plan L4
"Log this to implementation-notes.md." Implementation notes L5
"I only merge after I pass the quiz perfectly." Post-quiz L5
Why "literally"
The L2 lesson surfaces this as the first lesson in the series: paraphrasing the blindspot-pass phrase produces a model that skims rather than scans. The same applies to the other phrases above. Treat them as recipes, not summaries.

Unifying claims

Two sentences that anchor the series. If you remember these, the rest of the vocabulary hangs together.

L4's unifying claim (about the plan): Tweak-points first, mechanical last, HTML over markdown, named verification. The plan is a deliverable that travels — not the chat, one of the bundle of artifacts the next session reads.

L5's unifying claim (about shipping): Every surprise during execution deserves a log line; every shipped change deserves an artifact the next person can pick up.

How this glossary relates to the lessons

The glossary is a reference, not a substitute. Each lesson teaches its technique with worked examples, live rehearsals, and quizzes; the glossary compresses the vocabulary into one page. If a term here feels unfamiliar, the linked lesson is where the unfamiliarity gets resolved.

The glossary is also the seed for the cheatsheet that implementation-notes.md defers. When this glossary stops being enough — when the user wants something printable, one page, that fits on a wall — the cheatsheet is the next iteration. For now, the glossary plus the index is enough.
